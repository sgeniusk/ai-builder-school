## 도입 — 학습자에게 말 거는 한 장면

작은 챗 UI를 만들었어요. 입력창에 질문을 넣으면 AI가 답하고, 화면에는 그럴듯한 말풍선이 뜹니다. 그런데 코드를 다시 보니 브라우저 쪽 파일 안에 API 키가 들어가 있어요. 처음에는 “내 컴퓨터에서만 쓰니까 괜찮겠지” 싶지만, 개발자 도구를 열어 보면 요청과 응답이 그대로 보입니다. 배포라도 하면 키가 밖으로 새고, 누군가 그 키로 비용을 써버릴 수 있어요.

또 다른 문제도 있습니다. API가 잠깐 느려지면 버튼은 계속 로딩 중이고, 사용자는 아무 설명 없이 기다립니다. 모델이 에러를 반환하면 화면에는 긴 기술 메시지가 그대로 뜹니다. “코드는 연결됐는데 앱이라고 부르기엔 불안한” 상태예요.

오늘은 이 틈을 닫습니다. 프론트엔드가 AI 모델을 직접 부르지 않고, 우리 서버의 작은 문을 통과해 부르게 만들 거예요. 이 문이 키를 숨기고, 입력을 확인하고, 실패했을 때 사용자에게 보일 말과 개발자가 볼 로그를 나눠 줍니다.

## Phase 4 다음에 왜 이 레슨인가

[Lesson 4.1](/lessons/coding-agent-setup) 에서 코딩 에이전트가 움직일 환경을 만들었고, Phase 4 후반에는 테스트와 버그 재현 루프까지 다뤘어요. 이제 그 루프를 실제 앱 안에 넣기 시작합니다. 여기서 중요한 변화는 “AI에게 코드 조각을 써 달라”가 아니라 “사용자가 누르는 버튼 뒤에 AI 호출 흐름을 설계한다”는 점이에요.

AI 앱의 첫 번째 기준은 화려한 UI가 아닙니다. 키가 안전한지, 실패가 복구 가능한지, 비용이 폭주하지 않는지가 먼저예요. [Lesson 1.4](/lessons/checks-before-trusting-ai-output) 의 트러스트 체크를 앱 호출에도 붙이고, [Lesson 2.2](/lessons/enforcing-output-format) 의 출력 형식 습관을 서버 응답에도 적용합니다. 프롬프트에서만 쓰던 안전 루틴이 이제 코드 경계로 들어오는 거예요.

## 핵심 개념 — Server Route 와 Graceful Failure

환경변수부터 짚고 갈게요. 환경변수는 코드 파일에 직접 쓰지 않는 설정값입니다. 예를 들어 `OPENAI_API_KEY`나 `ANTHROPIC_API_KEY` 같은 비밀 키를 `.env.local`에 두고, 서버에서만 읽게 만들어요. 중요한 건 “변수 이름을 정했다”가 아니라 “브라우저 번들에 들어가지 않는다”입니다. 클라이언트 컴포넌트에서 `process.env`를 읽는 방식은 조심해야 해요. 공개해도 되는 값이 아니라면 서버 라우트 안에서만 다룹니다.

Server Route는 프론트엔드와 AI API 사이의 중계 창구입니다. 사용자는 `/api/chat` 같은 우리 앱의 주소로 요청을 보내고, 이 라우트가 모델 호출을 대신합니다. 여기서 입력 길이, 필수 필드, 모델 이름, 최대 토큰, 타임아웃을 확인할 수 있어요. 사용자가 보낸 내용을 그대로 모델에 던지는 대신, 앱이 감당할 수 있는 모양으로 정리한 뒤 넘깁니다.

Graceful Failure는 실패를 부드럽게 다루는 기술입니다. 실패를 숨기자는 뜻이 아니에요. 사용자에게는 “잠시 후 다시 시도해 주세요”처럼 행동 가능한 메시지를 주고, 개발자 로그에는 실제 에러 코드와 요청 ID를 남기는 방식입니다. 네트워크 오류와 입력 오류와 과금 한도 오류는 서로 다르게 다뤄야 해요. 재시도도 무조건 하면 안 됩니다. 같은 요청을 세 번씩 다시 보내면 비용이 조용히 늘어납니다.

## 그래서 오늘 할 일

오늘 만들 것은 “AI 연결 기본 라우트”입니다. Next.js 기준으로는 `app/api/chat/route.ts`가 되고, Python 서버를 쓰는 사람은 같은 역할의 `/chat` 엔드포인트로 바꿔 생각하면 돼요.

작업은 작게 나눕니다. 먼저 환경변수 이름과 필요한 모델을 정합니다. 다음으로 서버 라우트에서 `message` 같은 입력을 검증합니다. 그다음 모델 호출을 넣고, 타임아웃을 명시합니다. 실패 시에는 사용자용 메시지와 로그용 메시지를 분리합니다. 마지막으로 아주 단순한 UI에서 `fetch("/api/chat")`로 이 라우트를 호출해 봅니다.

산출물 [`api-connect-checklist.md`](/lessons/connect-ai-api) 에는 키 보관 위치, 서버 라우트 설계, 실패 처리, 비용 제한, 첫 수동 테스트를 한 장으로 적습니다. 비전공자라면 코드 전체를 이해하려고 하기보다 “비밀은 서버에만 있다”, “실패해도 UI가 멈추지 않는다”, “재시도는 제한된다” 세 문장을 먼저 확인하세요.

## 마무리

AI 앱 만들기는 모델을 한 번 부르는 순간 시작되지 않습니다. 안전한 호출 경계를 만든 뒤에 시작돼요. 오늘 만든 라우트는 앞으로 Phase 5의 나머지 레슨이 올라갈 바닥입니다. [Lesson 5.2](/lessons/streaming-response-ui) 에서는 같은 호출을 기다리는 UI가 아니라 흐르는 UI로 바꾸고, [Lesson 5.3](/lessons/structured-output-handling) 에서는 답을 앱이 읽을 수 있는 구조로 고정합니다. Phase 6에서는 이 호출에 문서 검색과 임베딩이 붙으면서, “답하는 앱”이 “내 자료를 읽고 답하는 앱”으로 자라게 됩니다.
