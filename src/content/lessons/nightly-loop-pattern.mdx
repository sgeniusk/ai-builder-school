## 도입 — 학습자에게 말 거는 한 장면

저녁 11시입니다. 오늘 하려던 성능 개선을 다 끝내지 못했어요. `train.py`는 아직 느리고, 어떤 변경이 효과가 있는지 하나씩 실험해야 합니다. 평소라면 내일 다시 열어야 합니다. 그런데 이번에는 목표, 검증 명령, 시간 한도를 한 장으로 정리해 두고 자리를 비웁니다.

아침에 돌아와 보니 에이전트가 여러 작은 시도를 해봤습니다. 어떤 변경은 검증을 통과했고, 어떤 변경은 실패해서 버려졌습니다. 중요한 건 “AI가 밤새 알아서 다 했다”가 아닙니다. 내가 정해 둔 목표와 검증 기준 안에서 반복이 돈 거예요. Phase 4에서 손으로 돌리던 작은 루프를 세션 밖으로 잠깐 꺼낸 것입니다.

이 레슨은 Phase 4에서 가장 어려운 편입니다. 그래도 겁먹을 필요는 없어요. 핵심은 자동화 기술보다 기준입니다. 목표가 좁고, 검증이 자동이고, 예산이 정해져 있으면 자동 루프는 무리한 도박이 아니라 잘 묶인 실험이 됩니다.

## Lesson 4.6 다음에 왜 이 레슨인가

[Lesson 4.5](/lessons/write-tests-with-coding-agent) 에서는 실패 테스트를 먼저 만들고 통과시키는 루프를 배웠습니다. [Lesson 4.6](/lessons/bug-reproduction-loop) 에서는 버그를 재현하고 회귀 테스트로 잠그는 루프를 만들었어요. 둘 다 내가 세션 안에서 지켜보는 루프였습니다.

오늘은 한 발 더 나갑니다. 사람이 옆에서 계속 보고 있지 않아도 돌아갈 수 있는 루프를 설계합니다. 외부에서는 이런 흐름을 test-time compute scaling이라고 부르기도 해요. 같은 모델이라도 더 많은 시도와 검증을 허용하면 결과가 좋아질 수 있다는 뜻입니다. 우리 식으로 말하면, 한 번의 멋진 요청보다 **작은 기준을 두고 여러 번 시도하는 구조**가 더 강하다는 말이에요.

단, 자동 루프는 아무 작업에나 쓰면 안 됩니다. 배포, 결제, 고객 데이터, 외부 전송처럼 영향이 큰 일은 사람 승인 없이 돌리지 않습니다. 이 레슨에서 다루는 대상은 작고 되돌릴 수 있는 코딩 실험입니다. 예를 들어 느린 테스트 개선, 작은 리팩터링 후보 탐색, 문서 링크 점검, 버그 재현 후보 찾기 같은 작업이에요.

## 핵심 개념 — 세션 밖으로 나가는 세 가지 루프

첫 번째는 Ralph Loop입니다. 가장 단순한 모양은 아래처럼 같은 프롬프트를 반복해서 에이전트에 넣는 방식입니다.

```bash
while :; do cat PROMPT.md | claude-code; done
```

이 패턴의 핵심은 명령 자체가 아니라 멈춤과 학습입니다. Stop Hook으로 위험한 지점에서 멈추게 하고, AGENTS.md 같은 규칙 문서에 매 회차 배운 내용을 남깁니다. 그러면 다음 회차는 같은 실수를 조금 덜 합니다. 입문자에게는 “같은 요청을 무한히 던진다”가 아니라 “매 회차 실패와 배움을 다음 입력으로 넘긴다”로 이해하면 충분해요.

두 번째는 fixed-budget research loop입니다. 시간을 먼저 정합니다. 예를 들어 10분 동안 한 가지 변경을 시도하고, 검증이 좋아지면 보존하고, 나빠지면 되돌립니다. 원래 연구 코드에서 많이 보이는 방식이지만, 우리에게는 작은 실험 관리법으로 쓰면 됩니다. 핵심은 시간 예산과 되돌리기 기준이에요.

세 번째는 Codex `/goal`입니다. plan, act, test, review, iterate 흐름이 도구 안에 묶여 있는 내장 루프라고 보면 됩니다. 잘 쓰려면 목표 문장이 좋아야 해요. 그래서 [Lesson 2.5](/lessons/meta-prompting) 가 여기서 다시 중요해집니다. AI에게 먼저 좋은 `/goal` 문장을 만들게 하고, 사람이 검증 조건과 제한을 읽은 뒤 실행합니다.

세 루프 모두 공통 준비물이 있습니다. 목표, 검증 조건, 토큰 예산, 시간 한도, 멈춤 조건입니다. 이 다섯 가지가 없으면 자동 루프가 아니라 긴 대화가 됩니다.

## 그래서 오늘 할 일

오늘 만들 것은 `nightly-loop-design.md`입니다. 실제로 밤새 돌리기 전에 설계서를 먼저 만듭니다. 처음부터 긴 실행을 하지 말고 10분짜리 드라이런으로 시작하세요.

먼저 작업 하나를 고릅니다. “앱 성능 개선”은 너무 큽니다. “느린 테스트 파일 하나의 실행 시간을 줄인다”처럼 좁혀야 합니다. 그다음 어떤 루프가 맞는지 고릅니다. 반복 프롬프트와 규칙 누적이 중요하면 Ralph식, 정해진 시간 안에서 작은 실험을 많이 해볼 거면 fixed-budget 방식, 도구가 제공하는 계획과 반복 흐름을 쓰고 싶으면 `/goal`이 맞습니다.

검증 조건을 적습니다. 예를 들어 `npm test`가 통과하고 특정 테스트 시간이 20% 줄어드는지, 타입체크가 깨지지 않는지, 변경 파일이 지정 범위를 벗어나지 않는지 같은 조건입니다. 사람이 읽는 느낌보다 명령이 판단할 수 있는 조건이 좋아요.

마지막으로 예산을 정합니다. 토큰 한도, 시간 한도, 최대 회차, 실패가 계속될 때 멈추는 기준을 적습니다. [Lesson 4.4](/lessons/plan-with-ai) 의 계획 습관을 그대로 가져오면 됩니다. 자동 루프도 결국 계획, 실행, 테스트, 리뷰, 회고의 반복이에요.

## 마무리

이 레슨으로 Phase 4는 닫힙니다. 환경을 세팅하고, 이슈를 작업지시서로 바꾸고, 코드베이스를 읽히고, 계획을 세우고, 테스트와 버그 재현 루프를 만들었습니다. 이제 그 루프를 사람이 지켜보는 세션 안에서만 쓰지 않고, 제한된 범위에서 세션 밖으로 확장하는 법까지 봤어요.

다음 [Lesson 5.1](/lessons/connect-ai-api) 부터는 AI 앱을 직접 만들기 시작합니다. 그때 자동 루프는 더 조심스럽게 다뤄야 합니다. 프로덕션 시스템 안으로 들어갈수록 목표, 검증, 비용, 멈춤 조건이 더 중요해져요. 오늘의 산출물은 “밤새 맡기는 법”이 아니라 “내가 자리를 비워도 기준이 남아 있게 하는 법”입니다.
