## "테스트요? 나중에 짤게요"

기능 하나를 코딩 에이전트한테 맡겼어요. 30분 만에 코드가 나옵니다. 잘 돌아가는 것 같아요. "테스트도 짜줘"라고 하니까 에이전트가 방금 짠 코드를 보고 **통과만 하는 테스트**를 만들어줍니다. 모든 케이스가 초록불. 안심하고 머지했는데, 일주일 뒤 엣지 케이스에서 버그가 터져요. 테스트는 통과했는데 왜?

답은 간단해요. **구현을 보고 짠 테스트는 구현의 실수를 같이 복사합니다.** 구현이 "빈 문자열을 무시한다"면, 테스트도 빈 문자열 케이스를 안 넣어요. 둘 다 같은 맹점을 가지게 됩니다.

{/* Source · BOOK · Kent Beck "Test-Driven Development: By Example" (2002) · Red-Green-Refactor 사이클의 원전 */}

해결책은 순서를 뒤집는 것이에요. **테스트를 먼저 짜고, 그다음에 구현을 시킨다.** 이게 TDD(Test-Driven Development)의 핵심이고, AI 코딩 에이전트와 함께 쓰면 특히 잘 맞는 패턴이에요.

## 왜 AI가 테스트 짜기에 강한가

AI 코딩 에이전트는 구현보다 테스트를 더 잘 짜요. 이유가 있어요.

**테스트는 "의도를 서술"하는 글쓰기**예요. "이 함수에 빈 배열을 넣으면 0을 반환해야 한다" — 이건 자연어에 가깝습니다. AI가 자연어를 코드로 바꾸는 것에 강하다는 건 [LLM이 잘하는 일과 못하는 일](/lessons/what-llms-are-good-and-bad-at)에서 봤죠. 테스트 코드는 그 강점이 가장 잘 먹히는 영역이에요.

반면 구현은 **시스템의 맥락**이 필요합니다. 다른 모듈과 어떻게 연결되는지, 성능 제약이 뭔지, 기존 코드 스타일이 뭔지. 에이전트가 놓치기 쉬운 부분이죠. 테스트에는 이 맥락이 덜 필요합니다.

| 영역 | AI의 강점이 먹히는 이유 |
| --- | --- |
| 테스트 | 의도를 서술 → 자연어 ↔ 코드 변환 | 
| 구현 | 시스템 맥락 필요 → 에이전트의 약점 |

그래서 **테스트를 AI에게 먼저 시키고, 구현도 AI에게 시키되 테스트가 가드레일이 되는** 흐름이 가장 안전해요.

## Red → Green → Refactor — AI와 도는 TDD 사이클

{/* Source · BOOK · Kent Beck "Test-Driven Development: By Example" (2002) · 3단 사이클 */}

TDD의 사이클은 세 단계예요.

1. **Red** — 실패하는 테스트를 먼저 짠다
2. **Green** — 그 테스트만 통과시키는 최소 구현을 짠다
3. **Refactor** — 통과를 유지하면서 코드를 정리한다

AI 코딩 에이전트와 함께 돌면 각 단계가 이렇게 바뀝니다.

| 단계 | 사람이 하는 것 | 에이전트가 하는 것 |
| --- | --- | --- |
| Red | 의도와 엣지 케이스를 자연어로 정의 | 테스트 코드 작성 |
| Green | 테스트 결과 확인, 방향 판단 | 최소 구현 작성 |
| Refactor | 코드 리뷰, 수용/거부 결정 | 리팩터링 제안 |

핵심은 사람의 역할이 **"코드를 직접 짜는 것"에서 "의도를 정의하고 결과를 판단하는 것"**으로 바뀐다는 점이에요. 이건 [AI와 함께 구현 계획 세우기](/lessons/plan-with-ai)의 Plan-then-Execute 패턴과 같은 원리입니다 — 방향을 먼저 잡고, 실행은 에이전트에게.

## 의도 먼저, 엣지 케이스 두 번째 — 2단계 테스트 요청

에이전트에게 테스트를 시킬 때 한 번에 다 시키면 안 돼요. **두 번에 나눠서** 시킵니다.

### 1단계: Golden Path (행복 경로)

```
이 함수의 정상 동작 테스트를 먼저 짜줘.
- calculateTotal(items) → 각 item의 price * quantity 합산
- 정상 입력: [{price: 1000, quantity: 2}, {price: 500, quantity: 3}] → 3500
- 빈 배열 → 0

테스트만 짜고 구현은 하지 마.
```

이 단계에서 나오는 테스트를 먼저 확인합니다. "이 테스트가 통과하면 이 함수가 기본적으로 맞다"고 동의할 수 있는지.

### 2단계: Edge Cases (경계 케이스)

```
좋아, 이제 엣지 케이스를 추가해 줘.
- 음수 가격이 들어오면?
- quantity가 0이면?
- price가 소수점이면 (부동소수점 문제)?
- items가 null이면?
- item에 price 필드가 없으면?

각 테스트에 "// 의도: ..." 주석을 달아줘.
```

왜 두 번에 나누냐면, **1단계를 확인하지 않고 2단계로 가면 엣지 케이스의 "정답"이 틀릴 수 있어요.** 기본 동작의 합의가 먼저 있어야 경계 동작의 기대값을 정할 수 있습니다.

## "// 의도:" 주석 — 테스트의 생명줄

{/* Note: "의도 주석" 패턴은 일반적인 소프트웨어 엔지니어링 관행을 한국어로 정리한 것입니다. */}

에이전트가 짠 테스트에서 가장 중요한 건 코드가 아니에요. **그 테스트가 왜 존재하는지**입니다.

```typescript
// 의도: 빈 배열은 합계 0을 반환해야 한다 (에러가 아님)
test("빈 배열 → 0", () => {
  expect(calculateTotal([])).toBe(0);
});

// 의도: 음수 가격은 거부해야 한다 (할인은 별도 필드로)
test("음수 가격 → 에러", () => {
  expect(() => calculateTotal([{ price: -100, quantity: 1 }])).toThrow();
});
```

의도 주석이 없으면 3개월 뒤에 이 테스트가 깨졌을 때 "이걸 고쳐야 하나, 지워야 하나"를 판단할 수 없어요. 의도가 남아있으면 "아, 이건 음수 가격을 막으려는 테스트구나. 비즈니스 규칙이 바뀌지 않았으니 구현을 고쳐야 한다"고 바로 판단됩니다.

에이전트에게 항상 이렇게 시키세요: **"각 테스트 케이스 위에 `// 의도:` 주석을 달아줘."**

## 커버리지 숫자의 함정

테스트 커버리지가 90%라고 안심하면 안 됩니다. 커버리지는 **"이 줄이 실행됐느냐"**만 봐요. "이 줄이 맞는 결과를 냈느냐"는 안 봅니다.

```typescript
// 커버리지 100%이지만 아무것도 검증하지 않는 테스트
test("runs without error", () => {
  calculateTotal([{ price: 1000, quantity: 2 }]);
  // assert가 없다!
});
```

이런 테스트는 커버리지를 높이지만 버그를 잡지 못해요. AI 에이전트가 "구현을 보고" 테스트를 짜면 이런 게 자주 나옵니다. 그래서 **구현 전에 테스트를 시키는 것**이 중요해요 — 에이전트가 참고할 구현이 없으니 의도 중심으로 짤 수밖에 없습니다.

커버리지와 신뢰도를 분리해서 보세요.

| 지표 | 보는 것 | 한계 |
| --- | --- | --- |
| 커버리지 | 실행된 코드 라인 비율 | assert 없는 테스트도 포함 |
| 신뢰도 | 이 테스트가 진짜 버그를 잡을 수 있나 | 수치화하기 어려움 |

신뢰도를 높이는 가장 쉬운 방법은 **"이 테스트를 고의로 깨뜨릴 수 있는가"**를 확인하는 것이에요. 구현의 핵심 로직 한 줄을 주석 처리했을 때 테스트가 빨간불이 되면 그 테스트는 신뢰할 수 있어요.

## 그래서 오늘 만들 한 폴더

오늘 만들 건 **한 함수에 대한 AI-짠 테스트 + 실행 결과 기록**이에요.

| 항목 | 무엇 |
| --- | --- |
| 대상 함수 | 본인 프로젝트의 함수 1개 (또는 연습용 `calculateTotal`) |
| 1단계 | Golden Path 테스트 요청 → 에이전트 출력 → 확인 |
| 2단계 | Edge Case 테스트 요청 → 에이전트 출력 → 확인 |
| 실행 | 테스트 실행 → Red 확인 → 구현 요청 → Green 확인 |
| 의도 주석 | 모든 테스트에 `// 의도:` 주석 존재 |
| 깨뜨리기 확인 | 핵심 로직 한 줄 주석 처리 → 테스트 빨간불 확인 |

이 한 폴더가 생기면 다음부터 "테스트요? 나중에 짤게요"가 사라집니다. **테스트가 먼저고, 구현이 나중이에요.**
