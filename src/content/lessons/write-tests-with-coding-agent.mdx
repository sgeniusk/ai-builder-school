## 도입 — 학습자에게 말 거는 한 장면

에이전트가 기능을 고쳤고, 마지막에 테스트도 추가했다고 말합니다. 겉으로는 좋아 보여요. 그런데 테스트를 열어보니 이미 구현된 결과에 맞춰 “통과하는 확인”만 있습니다. 버그가 다시 생겨도 잡히지 않을 테스트예요. 테스트가 있는 것과 신뢰할 수 있는 것은 다릅니다.

AI 에게 구현을 먼저 맡기면 테스트는 종종 뒤처집니다. 모델은 자기가 방금 만든 코드가 통과하도록 테스트를 맞추기 쉬워요. 그래서 오늘은 순서를 바꿉니다. 구현 전에 실패 테스트를 먼저 쓰게 하고, 사람이 그 테스트가 진짜 요구사항을 잡는지 확인한 뒤, 최소 구현으로 통과시키는 흐름을 만듭니다.

## Lesson 4.4 다음에 왜 이 레슨인가

[Lesson 4.4](/lessons/plan-with-ai) 에서 목표, 변경 파일, 검증 방법을 합의했어요. 이제 그 계획을 코드로 바로 옮기기 전에 “성공을 어떻게 증명할지”를 테스트로 고정합니다. 계획이 약속이라면 테스트는 약속을 자동으로 확인하는 장치예요.

여기서 [Lesson 2.2](/lessons/enforcing-output-format) 의 스키마 감각을 다시 씁니다. 테스트도 출력 형식입니다. 입력, 기대값, 실패 조건이 명확해야 해요. 오늘 산출물 [`tdd-with-agent.md`](/lessons/write-tests-with-coding-agent) 는 에이전트에게 Red → Green 순서를 지키게 하는 체크리스트입니다.

## 핵심 개념 — Red → Green 과 Golden Path vs Edge

### Red → Green — 먼저 실패해야 믿을 수 있다

Red 는 테스트가 실패하는 상태예요. 아직 구현이 없거나 버그가 남아 있어서 실패해야 정상입니다. Green 은 최소 구현으로 그 테스트를 통과한 상태예요. 이 순서를 지키면 “테스트가 진짜 문제를 잡는가?”를 확인할 수 있습니다.

에이전트에게 이렇게 말해보세요.

```md
지금은 구현하지 말고 실패 테스트만 작성해 주세요.
테스트가 실패하는 이유를 한 줄씩 주석으로 남기고 멈춰 주세요.
내가 테스트를 리뷰한 뒤 구현을 요청하겠습니다.
```

이 멈춤이 중요합니다. 테스트와 구현을 한 번에 맡기면 에이전트가 둘을 서로 맞춰버릴 수 있어요.

### Golden Path — 가장 정상적인 길

Golden Path 는 사용자가 의도대로 움직였을 때의 대표 시나리오입니다. 로그인 폼이라면 올바른 이메일과 비밀번호로 성공하는 경우예요. 이 테스트가 없으면 기능의 중심이 흔들립니다.

### Edge Case — 작은 틈에서 깨지는 길

Edge Case 는 비어 있는 입력, 너무 긴 값, 권한 없는 사용자, 네트워크 실패처럼 경계에서 생기는 상황입니다. AI 는 행복 경로는 잘 떠올리지만, 도메인 특유의 엣지 케이스는 놓치기 쉬워요. [Lesson 1.4](/lessons/checks-before-trusting-ai-output) 의 Bias 와 Scope 체크처럼 “무엇이 빠졌나?”를 사람이 봐야 합니다.

### 커버리지보다 신뢰도

커버리지 숫자가 높아도 중요한 버그를 못 잡으면 의미가 적어요. 반대로 테스트 3개라도 핵심 위험을 잡으면 가치가 큽니다. 오늘은 숫자를 올리는 것이 아니라, 계획에서 정한 수용 조건을 테스트로 잠그는 것이 목표입니다. 이 흐름은 다음 [Lesson 4.6](/lessons/bug-reproduction-loop) 의 회귀 테스트와 이어집니다.

## 그래서 오늘 할 일

오늘 만들 것은 “코딩 에이전트와 TDD 를 하는 작업 카드”입니다.

1. [Lesson 4.4](/lessons/plan-with-ai) 의 계획서에서 수용 조건 1개를 고릅니다.
2. Golden Path 테스트 1개를 먼저 정의합니다.
3. Edge Case 테스트 2개 이상을 추가합니다.
4. 에이전트에게 실패 테스트만 작성하고 멈추게 합니다.
5. 사람이 테스트 이름, 기대값, 실패 이유를 검토합니다.
6. 승인 후 최소 구현을 요청합니다.
7. 산출물 [`tdd-with-agent.md`](/lessons/write-tests-with-coding-agent) 에 Red → Green 기록을 남깁니다.

비전공자라면 테스트 코드를 모두 이해하지 못해도 괜찮아요. 대신 테스트 이름과 시나리오를 읽고 “내가 원하는 행동을 확인하나?”를 볼 수 있어야 합니다. 에이전트가 코드 문법을 돕고, 사람은 요구사항을 지킵니다.

## 마무리

테스트를 먼저 쓰게 하는 순간 에이전트의 역할이 달라집니다. 그냥 코드를 빨리 쓰는 도구가 아니라, 요구사항을 증명하는 파트너가 돼요. 다음 [Lesson 4.6](/lessons/bug-reproduction-loop) 에서는 이미 생긴 버그를 재현하고, 최소 예제와 회귀 테스트로 다시 나타나지 않게 잠그는 루프로 확장합니다.
