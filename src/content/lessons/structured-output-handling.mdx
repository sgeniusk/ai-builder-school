## 도입 — 학습자에게 말 거는 한 장면

AI가 추천 결과를 JSON으로 줍니다. 화면에는 카드 3개가 예쁘게 떠야 해요. 첫 번째 실행은 잘 됩니다. 두 번째 실행도 괜찮습니다. 그런데 세 번째 실행에서 `title`은 있는데 `reason`이 빠졌고, 네 번째 실행에서는 배열이어야 할 값이 문장 하나로 왔어요. UI는 빈 칸을 보여 주거나, 더 나쁘게는 에러 화면으로 넘어갑니다.

프롬프트에 “반드시 JSON으로 답해”라고 적었는데도 이런 일이 생깁니다. AI는 형식을 잘 맞추는 편이지만, 앱은 “대체로 잘 맞음”으로는 움직일 수 없어요. 버튼, 카드, 저장 로직, 결제 후 생성 결과처럼 downstream 단계가 붙으면 작은 필드 하나가 빠져도 전체 흐름이 흔들립니다.

오늘은 AI 답변을 믿고 바로 쓰지 않고, 스키마로 검사한 뒤 앱에 넣는 루프를 만듭니다. [Lesson 5.2](/lessons/streaming-response-ui) 에서 흐르는 답을 다뤘다면, 이번에는 앱이 읽을 수 있는 단단한 모양을 다룹니다.

## Lesson 5.2 다음에 왜 이 레슨인가

스트리밍은 사용자가 기다리는 경험을 좋게 만듭니다. 하지만 제품을 만들 때는 “잘 보인다” 다음에 “잘 저장되고, 잘 필터링되고, 잘 재사용된다”가 필요해요. 그때 텍스트 덩어리만 있으면 한계가 옵니다. 제목, 요약, 태그, 위험도, 다음 행동 같은 필드가 안정적으로 나와야 합니다.

[Lesson 2.2](/lessons/enforcing-output-format) 에서는 프롬프트 레벨에서 출력 형식을 강제했어요. 오늘은 그 습관을 서버 코드로 가져옵니다. 프롬프트가 “이렇게 답해 줘”라고 요청한다면, 스키마 검증은 “정말 그렇게 왔는지 확인한다”는 문입니다. [Lesson 1.4](/lessons/checks-before-trusting-ai-output) 의 트러스트 체크가 사람의 검토 루틴이었다면, 여기서는 앱이 자동으로 돌리는 검토 루틴이 됩니다.

## 핵심 개념 — Schema-Driven Output 과 Validation Loop

스키마는 앱과 AI 사이의 계약서입니다. 예를 들어 추천 카드라면 `title`, `summary`, `confidence`, `nextActions` 같은 필드가 있고, 각 필드가 문자열인지 숫자인지 배열인지 정합니다. TypeScript에서는 Zod 같은 도구로 이 계약을 코드로 적을 수 있어요. 비전공자라면 “AI 답변을 받을 때 필요한 칸 이름과 칸의 종류를 미리 정한다”고 이해하면 충분합니다.

중요한 차이가 하나 있습니다. 파싱은 문자열을 데이터로 바꾸는 일이고, 검증은 그 데이터가 약속한 모양인지 확인하는 일입니다. `JSON.parse`가 성공했다고 해서 앱이 안전한 것은 아니에요. JSON 문법은 맞지만 필드가 빠졌거나 값이 엉뚱할 수 있습니다. 그래서 파싱 다음에 스키마 검증이 필요합니다.

Validation Loop는 검증 실패를 다루는 반복입니다. 실패하면 곧바로 사용자에게 에러를 보여 줄 수도 있지만, 한 번 정도는 모델에게 “이 스키마에 맞게 다시 출력해”라고 재요청할 수 있어요. 다만 무한 재시도는 금지입니다. 재시도 횟수, 실패 로그, 사용자 대체 메시지가 있어야 합니다. 구조화 출력은 엄격할수록 안정적이지만, 너무 엄격하면 사소한 이탈에도 기능이 멈춥니다. 그래서 필수 필드와 선택 필드를 나누는 감각도 중요합니다.

## 그래서 오늘 할 일

오늘 만들 것은 “스키마 기반 파이프라인”입니다. 작은 기능 하나를 고릅니다. 예를 들어 “사용자 질문을 받아 학습 카드 3개를 추천한다”처럼 카드 UI로 보일 수 있는 기능이 좋아요.

먼저 출력 스키마를 정의합니다. 필수 필드와 선택 필드를 나누고, 숫자 범위나 배열 길이도 가능하면 적습니다. 다음으로 [Lesson 5.1](/lessons/connect-ai-api) 의 서버 라우트에서 모델 응답을 받은 뒤 파싱과 검증을 실행합니다. 실패하면 재요청을 최대 1회만 하고, 그래도 실패하면 사용자에게 안전한 대체 메시지를 보냅니다. 클라이언트에서는 검증된 데이터만 카드 UI에 넣습니다.

산출물 [`structured-output-handler.md`](/lessons/structured-output-handling) 에는 스키마, 성공 응답 예시, 실패 응답 예시, 재시도 규칙, UI fallback을 적습니다. 특히 “필드가 빠졌을 때 앱이 어떻게 보일지”를 직접 테스트하세요. 구조화 출력은 성공 케이스보다 실패 케이스에서 실력이 드러납니다.

## 마무리

AI 앱은 자연어를 다루지만, 앱 자체는 구조를 먹고 삽니다. 텍스트가 카드가 되고, 카드가 저장되고, 저장된 값이 다음 추천에 쓰이려면 모양이 안정적이어야 해요. 오늘 만든 검증 루프는 앞으로 Phase 6의 RAG 결과, Phase 7의 에이전트 액션, Phase 11의 제품 지표까지 이어지는 기본 안전장치입니다. 다음 [Lesson 5.4](/lessons/conversation-storage-design) 에서는 이 검증된 대화와 결과를 어떻게 저장할지 설계합니다.
