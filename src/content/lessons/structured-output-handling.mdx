## "JSON은 왔는데 카드 UI가 깨졌어요"

[출력 형식(Output Format)을 강제하는 법](/lessons/enforcing-output-format)에서 3단 방어선을 만들었죠 — 프롬프트 가드 · API 스키마 강제 · 재시도 그물. 터미널에서 테스트할 때는 완벽했어요. 그런데 이걸 **React 앱**에 붙이는 순간 새로운 문제가 터집니다.

모델이 보낸 JSON의 `tags` 필드가 때로 문자열이고 때로 배열이에요. `null`이 올 때도 있어요. 카드 컴포넌트가 `tags.map`을 부르다 `TypeError: tags.map is not a function`으로 흰 화면. 사용자는 "이 앱 뭐야?" 하고 떠납니다.

3단 방어선은 **서버에서 모델의 출력을 잡는 것**이었어요. 오늘은 그 다음 단계 — **잡은 출력을 앱 상태로 안전하게 가져오는 법**을 만듭니다.

## L06의 3단 방어선 → 앱 안의 4번째 벽

{/* Source · DOC · Anthropic "Structured outputs" — https://docs.claude.com/en/docs/build-with-claude/structured-outputs */}
{/* Source · DOC · OpenAI "Structured model outputs" — https://developers.openai.com/api/docs/guides/structured-outputs */}

[출력 형식(Output Format)을 강제하는 법](/lessons/enforcing-output-format)의 3단 방어선을 다시 볼게요.

| 방어선 | 위치 | 역할 |
| --- | --- | --- |
| 1선 — 프롬프트 | 서버 (호출 전) | 형식·예시·금지 명시 |
| 2선 — API 스키마 | 서버 (호출 시) | 디코더 레벨 강제 |
| 3선 — 재시도 | 서버 (호출 후) | 파싱 실패 시 수정 재호출 |
| **4번째 벽 — 앱 상태** | **클라이언트** | **Zod 검증 → 타입 안전 → UI 바인딩** |

{/* Note: "4번째 벽" 명칭은 이 학교 운영자의 정리이며, 외부 표준명은 아닙니다. */}

1~3선이 "모델이 올바른 JSON을 뱉게 만드는 것"이라면, 4번째 벽은 "그 JSON을 React 상태로 안전하게 변환하는 것"이에요. 서버에서 검증을 했더라도, 네트워크 전송 중 깨지거나 버전 불일치가 있을 수 있으니 클라이언트에서도 한 번 더 거르는 게 핵심입니다.

## Zod — 스키마가 곧 타입이 되는 마법

{/* Source · DOC · Zod "Introduction" — https://zod.dev */}

[출력 형식(Output Format)을 강제하는 법](/lessons/enforcing-output-format)에서 Zod를 처음 만났죠. 오늘은 Zod를 **React 앱의 상태 경계**에 배치합니다.

```typescript
import { z } from "zod";

// 1. 스키마 정의 = 런타임 검증 + TypeScript 타입 동시 생성
const RecipeCardSchema = z.object({
  title: z.string().min(1),
  description: z.string(),
  difficulty: z.enum(["easy", "medium", "hard"]),
  ingredients: z.array(z.string()).min(1),
  cookTimeMinutes: z.number().int().positive(),
  tags: z.array(z.string()).default([]),
});

// 2. 타입 추출 — 컴포넌트에서 이 타입을 씁니다
type RecipeCard = z.infer<typeof RecipeCardSchema>;
```

이 한 정의에서 **런타임 검증**과 **TypeScript 타입**이 동시에 나옵니다. 스키마와 타입이 따로 놀면 "서버 응답은 되는데 타입이 안 맞아" 같은 사고가 나요. Zod를 쓰면 한 곳에서 관리됩니다.

## 안전한 파싱 패턴 — safeParse

`schema.parse(data)`는 실패하면 예외를 던집니다. React 컴포넌트 안에서 예외가 날아가면 흰 화면이에요. 대신 **`safeParse`** 를 씁니다.

```typescript
function parseAIResponse(raw: unknown): RecipeCard | null {
  const result = RecipeCardSchema.safeParse(raw);

  if (result.success) {
    return result.data; // 타입 안전한 RecipeCard
  }

  // 실패 시 — UI에 안전한 피드백
  console.error("AI 응답 검증 실패:", result.error.issues);
  return null;
}
```

```tsx
// React 컴포넌트에서
const [card, setCard] = useState<RecipeCard | null>(null);
const [error, setError] = useState<string | null>(null);

async function onGenerate() {
  const raw = await fetchAI(prompt);
  const parsed = parseAIResponse(raw);

  if (parsed) {
    setCard(parsed);
    setError(null);
  } else {
    setError("AI가 올바른 형식으로 답하지 못했어요. 다시 시도해 주세요.");
  }
}
```

`safeParse`의 핵심: **예외 대신 분기**. 성공하면 타입 안전한 데이터, 실패하면 에러 정보 — 컴포넌트가 어느 쪽이든 안전하게 처리할 수 있어요.

## 서버-클라이언트 이중 검증 패턴

실전에서는 **서버에서 한 번, 클라이언트에서 한 번** — 총 두 번 검증합니다.

```
[AI 모델] → 서버: Zod parse (3선 재시도 포함)
               ↓ 검증된 JSON
          → 클라이언트: Zod safeParse (방어적)
               ↓ 타입 안전한 상태
          → React 컴포넌트: RecipeCard 타입으로 렌더링
```

"서버에서 이미 검증했는데 왜 또?"라고 생각할 수 있어요. 세 가지 이유가 있습니다.

1. **네트워크 전송 중 변형** — 프록시·CDN이 JSON을 건드릴 수 있음
2. **API 버전 불일치** — 서버가 새 필드를 추가했는데 클라이언트가 모름
3. **방어적 프로그래밍** — 어디서 깨져도 흰 화면이 아니라 안내 메시지가 뜨게

## 대체 경로(Fallback) 설계

검증이 실패했을 때 "다시 시도해 주세요"만 보여주는 건 게으른 UX예요. 더 나은 대체 경로를 만들어 봅시다.

| 실패 유형 | 대체 경로 |
| --- | --- |
| 필드 1~2개 누락 | `.default()`로 기본값 채우기 |
| 타입 불일치 (string↔number) | `.coerce`로 자동 변환 시도 |
| 완전히 깨진 JSON | 에러 메시지 + "다시 생성" 버튼 |
| 3회 연속 실패 | 마지막 성공한 결과 캐시에서 보여주기 |

```typescript
// Zod의 .default()와 .coerce 활용
const RobustRecipeSchema = z.object({
  title: z.string().min(1),
  description: z.string().default("설명이 생성되지 않았습니다."),
  difficulty: z.enum(["easy", "medium", "hard"]).default("medium"),
  ingredients: z.array(z.string()).default([]),
  cookTimeMinutes: z.coerce.number().int().positive().default(30),
  tags: z.array(z.string()).default([]),
});
```

`.default()`와 `.coerce`를 전략적으로 쓰면, "약간 틀린" 응답도 UI가 살아남습니다. 물론 핵심 필드(`title` 같은)는 default 없이 엄격하게 — 이 균형이 중요해요.

## 스키마 진화 — 필드가 바뀔 때

앱이 자라면 스키마도 바뀝니다. `tags`를 추가하거나 `difficulty`를 제거하거나. 이때 **이전 버전의 응답**이 캐시에 남아 있으면 깨져요.

| 전략 | 설명 |
| --- | --- |
| 선택적 필드 | 새 필드는 `.optional()`로 시작 → 안정되면 `.required()`로 |
| 버전 필드 | `{ _version: 2, ... }` — 버전별 분기 파서 |
| 마이그레이션 함수 | 이전 버전 데이터를 새 스키마로 변환하는 함수 |

이 전략은 [대화 기록 저장 구조 설계](/lessons/conversation-storage-design)에서 대화 저장소 마이그레이션과 다시 만나요. 데이터가 영속되는 순간부터 스키마 진화는 피할 수 없습니다.
