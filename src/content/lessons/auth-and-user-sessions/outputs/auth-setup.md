# 사용자 인증·세션 설계 시트

> 이 파일은 지음 · Stage 6 "사용자 인증과 세션" 에서 제공되는 출발 템플릿입니다.
> 당신 앱에 맞춰 자유롭게 수정하세요.

## 1. 인증 방식 선택

| 항목 | 내용 |
| --- | --- |
| 앱 스택 | 예: Next.js 15 App Router |
| 선택한 라이브러리 | 예: Auth.js / Supabase Auth / Clerk |
| 선택 이유 | 예: Next.js와 통합이 쉽고 무료 시작 가능 |
| 로그인 방식 | 예: 이메일+비밀번호 / 구글 OAuth / 매직 링크 |

- [ ] 직접 구현하지 않고 검증된 라이브러리를 쓴다
- [ ] 라이브러리 공식 문서의 시작 가이드를 따라 설치했다

## 2. 로그인·세션·보호 라우트 체크

- [ ] 회원가입 화면이 동작한다
- [ ] 로그인 화면이 동작한다
- [ ] 로그아웃이 세션을 실제로 끊는다
- [ ] 로그인 후 새로고침해도 로그인 상태가 유지된다 (세션 지속)
- [ ] 로그인하지 않은 사용자가 보호 라우트에 접근하면 로그인 화면으로 보낸다
- 보호해야 할 라우트 목록
	- 예: /dashboard
	- 예: /chat
	-

## 3. 사용자별 데이터 격리 점검

모든 저장 데이터에 `user_id`가 묶여 있는지 확인한다.

| 데이터 종류 | user_id 열 있음 | 읽을 때 user_id 조건 있음 |
| --- | --- | --- |
| 예: conversations | [ ] | [ ] |
| 예: messages | [ ] | [ ] |
| | [ ] | [ ] |

- [ ] 새 데이터를 저장할 때 현재 로그인 사용자의 user_id를 기록한다
- [ ] 데이터를 읽는 모든 쿼리에 "현재 사용자의 것만" 조건이 들어간다
- [ ] 다른 계정으로 로그인해 옆 사람 데이터가 보이지 않는지 직접 테스트했다

## 4. 보안 점검

- [ ] 비밀번호는 평문이 아니라 해시로 저장된다 (라이브러리 기본값 확인)
- [ ] 세션 토큰에 만료 시간이 설정되어 있다
- [ ] 세션 쿠키에 HttpOnly·Secure 속성이 켜져 있다
- [ ] API 키·세션 시크릿이 코드가 아니라 환경 변수에 있다
- [ ] 인가 점검 — 관리자 전용 기능이 일반 사용자에게 막혀 있는가
