# 스트리밍 UI 패턴 카드

> AI Builder School · Lesson 5.2 출발 템플릿입니다.
> 긴 AI 응답을 기다리는 화면에서 “살아 있는 신호”, 중단, 재시도를 설계합니다.

## 0. 내 기능 맥락

| 항목 | 내용 |
| --- | --- |
| 기능 이름 | |
| 평균 응답 길이 | 짧음 / 중간 / 김 |
| 사용자가 기다릴 수 있는 시간 | *(예) 10초 안에 첫 글자가 보여야 함* |
| 중단이 꼭 필요한 이유 | |

---

## 1. UI 상태표

| 상태 | 화면 표시 | 가능한 사용자 행동 |
| --- | --- | --- |
| `idle` | 입력창과 보내기 버튼 | 질문 입력 |
| `streaming` | 누적되는 답변 + 중단 버튼 | 중단 |
| `done` | 완성된 답변 + 재시도 버튼 | 다시 묻기, 복사 |
| `error` | 복구 가능한 오류 메시지 | 재시도 |
| `cancelled` | 중단됨 표시 | 다시 보내기 |

---

## 2. 이벤트 흐름

```txt
사용자 입력.
POST /api/chat/stream 호출.
첫 chunk 수신.
assistant 메시지 생성.
chunk마다 content 뒤에 이어 붙임.
완료 이벤트 수신.
상태를 done으로 변경.
```

중단 흐름.

```txt
사용자가 중단 버튼 클릭.
AbortController로 요청 취소.
상태를 cancelled로 변경.
부분 답변은 회색 상태로 보존.
재시도 버튼을 제공.
```

---

## 3. 스트리밍 중 하지 않을 일

- 완성되지 않은 JSON을 매 chunk마다 억지로 파싱하지 않기.
- 사용자가 스크롤을 올려 읽는 중인데 계속 맨 아래로 끌어내리지 않기.
- 중단 버튼을 UI 상태만 바꾸는 장식으로 두지 않기.
- 같은 요청을 자동으로 여러 번 재시도하지 않기.

---

## 4. 수동 검증

| 테스트 | 기대 결과 | 실제 결과 |
| --- | --- | --- |
| 긴 질문 전송 | 첫 chunk가 빠르게 보인다 | |
| 중간에 중단 | 실제 요청이 취소된다 | |
| 중단 후 재시도 | 새 응답이 깨끗하게 시작된다 | |
| 모바일 화면 | 입력창과 중단 버튼이 겹치지 않는다 | |
| 네트워크 오류 | error 상태와 재시도 버튼이 보인다 | |
