## "일단 짜 봐" 하고 시켰더니 3시간이 사라졌어요

간단한 기능 하나를 코딩 에이전트한테 맡겼어요. "로그인 페이지에 비밀번호 찾기 버튼 추가해 줘." 5분이면 될 줄 알았는데, 에이전트가 "겸사겸사" 인증 모듈 전체를 리팩터링하기 시작합니다. 파일 12개가 바뀌고, 테스트 3개가 깨지고, 원래 하려던 버튼은 아직 없습니다. 3시간 뒤에 `git reset --hard`를 치며 처음으로 돌아옵니다.

{/* Source · DOC · Anthropic "Claude Code overview" — https://docs.claude.com/en/docs/claude-code/ */}

문제는 에이전트의 실력이 아니에요. **계획 없이 실행부터 시킨 것**이 진짜 원인입니다. 사람도 "일단 짜 봐"라고 들으면 눈앞에 보이는 것부터 건드려요. 에이전트도 마찬가지예요 — 범위를 안 정해주면 자기가 "좋아 보이는" 범위를 만들어버립니다.

오늘 만들 건 **코드 한 줄 짜기 전에 AI와 5~15분간 계획을 합의하는 패턴**이에요. 이 의식 하나가 되돌리는 시간을 줄이는 데 큰 도움이 됩니다.

## 왜 계획이 실행 시간을 줄이는가

{/* Source · DOC · Anthropic "Extended thinking" — https://docs.claude.com/en/docs/build-with-claude/extended-thinking */}

직관에 반하는 얘기죠. "5분 더 쓰는데 어떻게 시간이 줄어?" 실제 실행 시간의 상당 부분은 **되돌리기**에 쓰이는 경우가 많아요. 잘못된 방향으로 10분 달린 뒤 5분 되돌리고 다시 10분 달리면 — 25분 중 유효 작업은 10분이에요. 계획이 방향을 미리 잡아주면 되돌리기가 줄어듭니다.

[Lesson 14](/lessons/github-issue-to-ai-brief)에서 이슈를 에이전트 브리프로 바꾸는 법을 배웠죠. 그 브리프가 "무엇을 할지"였다면, 오늘의 계획은 **"어떻게 할지"**예요. 어느 파일을 건드리고, 어느 순서로 하고, 어디서 멈출지.

| 단계 | 무엇 | 시간 |
| --- | --- | --- |
| 브리프 (L14) | 무엇을 · 왜 · 제약 | 5분 |
| **계획 (오늘)** | 어떤 파일 · 어떤 순서 · 어디서 멈춤 | 5~15분 |
| 실행 | 코드 작성 | 나머지 |

이 두 단계가 앞에 깔리면 실행이 한 방향으로 흐릅니다.

## Plan Mode — 에이전트의 "생각 먼저" 모드

{/* Source · DOC · Anthropic "Claude Code overview" (Plan mode) — https://docs.claude.com/en/docs/claude-code/ */}

Claude Code에는 **Plan Mode**가 있어요. 에이전트에게 "코드를 짜지 말고 계획만 세워"라고 지시하는 모드입니다. 이 모드에서 에이전트는 파일을 읽고, 구조를 파악하고, 변경 계획을 텍스트로 출력합니다 — 코드 한 줄 건드리지 않고.

Plan Mode를 쓰는 가장 쉬운 방법은 프롬프트 앞에 명시하는 것이에요.

```
[Plan Mode] 아래 작업의 구현 계획을 세워줘. 코드를 수정하지 마.

작업: 로그인 페이지에 비밀번호 찾기 버튼 추가
제약: 기존 인증 모듈 건드리지 않음
```

에이전트가 돌려주는 계획에는 보통 이런 것들이 담겨요.

- 변경할 파일 목록
- 각 파일에서 바꿀 부분의 요약
- 새로 만들 파일이 있으면 그 목적
- 예상되는 부작용이나 위험

이 계획을 읽고 **동의하거나 수정한 뒤에** 실행을 시키면, 에이전트의 범위가 명확하게 잠깁니다.

## Extended Thinking — 복잡한 작업의 내부 추론

{/* Source · DOC · Anthropic "Extended thinking" — https://docs.claude.com/en/docs/build-with-claude/extended-thinking */}

Plan Mode가 "계획을 텍스트로 보여달라"는 것이라면, **Extended Thinking**은 에이전트가 답을 내놓기 전에 내부적으로 더 깊이 추론하게 만드는 기능이에요. 복잡한 아키텍처 결정이나 여러 파일에 걸친 변경에서 특히 효과가 큽니다.

둘의 차이를 정리하면:

| 기능 | 하는 일 | 언제 쓰나 |
| --- | --- | --- |
| Plan Mode | 계획을 텍스트로 출력, 코드 수정 안 함 | 범위 합의가 필요할 때 |
| Extended Thinking | 답 전에 내부 추론 단계를 더 깊게 | 복잡한 결정이 포함될 때 |

둘을 같이 쓸 수도 있어요. "Plan Mode로 계획을 세우되, extended thinking으로 깊이 생각해"라고 하면 가장 신중한 계획이 나옵니다.

## implementation-plan.md — 계획의 구체적 형태

{/* Note: implementation-plan.md 패턴은 이 학교 운영자가 정리한 템플릿이며, 외부 표준명은 아닙니다. */}

계획이 대화창에만 있으면 사라져요. 파일로 남겨야 합니다. `implementation-plan.md` 한 장이면 충분해요.

```markdown
# Implementation Plan: 비밀번호 찾기 버튼

## 목표
로그인 페이지에 "비밀번호 찾기" 링크를 추가한다.

## 변경 파일
| 파일 | 변경 내용 | 위험도 |
| --- | --- | --- |
| src/app/login/page.tsx | 버튼 컴포넌트 추가 | 낮음 |
| src/components/ForgotPassword.tsx | 새 파일 생성 | 없음 |
| src/app/login/page.test.tsx | 테스트 케이스 추가 | 낮음 |

## 건드리지 않는 것
- src/lib/auth.ts (인증 로직)
- src/middleware.ts (라우팅)

## 테스트 전략
1. 버튼 렌더링 확인 (unit)
2. 클릭 시 이동 확인 (integration)

## 롤백 포인트
- 커밋 abc1234 (현재 상태)

## Stop Points
- [ ] 파일 목록 합의 → 여기서 한 번 멈춤
- [ ] UI 완성 후, API 연결 전 → 여기서 한 번 멈춤
```

핵심은 **"건드리지 않는 것"** 섹션이에요. 에이전트의 범위 폭주를 막는 가장 강력한 장치입니다. [Lesson 05](/lessons/structure-of-good-prompts)의 Constraints 축과 같은 원리 — 금지를 명시하면 허용 범위가 좁아져요.

## Stop Points — 에이전트를 중간에 세우는 장치

에이전트에게 전체를 한 번에 시키면 중간에 방향이 틀어져도 알 수가 없어요. **Stop Point**는 "여기까지 하고 멈춰서 나한테 보여줘"라는 체크포인트입니다.

좋은 Stop Point의 조건:

- **검증 가능**: 멈춘 시점에 내가 확인할 수 있는 산출물이 있어야 해요
- **되돌리기 쉬움**: 여기서 잘못됐으면 이 지점으로 돌아올 수 있어야 해요
- **의사결정 포함**: "다음 단계를 A로 갈지 B로 갈지"가 있는 지점이 좋아요

```
Stop Points:
1. 계획 합의 — 파일 목록과 순서를 확인한 뒤 진행
2. 스캐폴딩 완료 — 빈 파일/함수 시그니처만 만든 상태에서 확인
3. 핵심 로직 완료 — 테스트 전에 한 번 확인
4. 테스트 통과 — 최종 리뷰
```

이 네 지점이 깔리면 에이전트가 3시간 동안 혼자 달리는 일이 없어집니다.

## Plan-then-Execute 사이클 실전

실제로 해보면 이 흐름이에요.

| 단계 | 시간 | 하는 일 |
| --- | --- | --- |
| 1. 브리프 전달 | 2분 | L14 브리프를 에이전트에게 줌 |
| 2. Plan Mode | 5~10분 | 에이전트가 계획 출력, 사람이 검토 |
| 3. 합의 | 2~3분 | "건드리지 않는 것" 확인, Stop Points 확정 |
| 4. 실행 | 나머지 | Stop Point마다 중간 확인 |
| 5. 기록 | 3분 | 계획 vs 실제 차이를 plan에 메모 |

마지막 단계가 중요해요. **계획과 실제의 차이를 기록**하면 다음 계획이 더 정확해집니다. "파일 3개 바꿀 줄 알았는데 5개 바뀜 — 이유: 공유 컴포넌트를 몰랐음" 같은 메모 한 줄이면 됩니다.

## 그래서 오늘 만들 한 장

오늘 만들 건 **한 작업의 plan-then-execute 사이클 기록**이에요.

| 항목 | 무엇 |
| --- | --- |
| 작업 선정 | 지금 하고 있는 실제 코딩 작업 1개 (또는 연습용 작업) |
| Plan Mode 결과 | 에이전트가 출력한 계획 원문 |
| 합의 메모 | 내가 수정하거나 확인한 부분 |
| Stop Points | 실제로 멈춘 지점과 그때 확인한 것 |
| 계획 vs 실제 | 달랐던 점 + 이유 한 줄 |

이 한 장이 생기면 다음부터 코딩 에이전트에게 "일단 짜 봐"라고 시키는 일이 사라집니다. **짧은 계획 한 장이 큰 되돌림을 줄여줘요.**
