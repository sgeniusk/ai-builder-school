## 도입 — 학습자에게 말 거는 한 장면

에이전트에게 “이 이슈 고쳐줘”라고 했더니 바로 파일을 수정하기 시작합니다. 처음에는 시원해 보여요. 그런데 조금 지나면 원래 요청하지 않은 리팩터링이 들어가고, 테스트는 나중에 붙겠다고 하고, 변경 파일이 예상보다 많아집니다. 중간에 멈추려 해도 이미 어디까지 바뀌었는지 파악하기 어렵습니다.

AI 와 일할 때 가장 위험한 말은 “일단 해봐”예요. 사람 개발자에게도 계획 없는 구현은 위험하지만, 에이전트는 속도가 빨라서 더 빨리 멀리 갑니다. 오늘은 구현 전에 멈춰서, 변경 파일, 테스트, 롤백, Stop Point 를 한 장으로 합의하는 루틴을 만듭니다.

## Lesson 4.3 다음에 왜 이 레슨인가

[Lesson 4.2](/lessons/github-issue-to-ai-brief) 에서 작업지시서를 만들고, [Lesson 4.3](/lessons/letting-ai-read-codebase) 에서 코드베이스를 읽히는 지도를 만들었어요. 이제 에이전트는 무엇을 해야 하는지와 어디를 봐야 하는지를 압니다. 하지만 아직 “어떤 순서로, 어디까지, 무엇으로 검증할지”는 합의하지 않았어요.

이 레슨은 구현 전 의식을 만듭니다. [Lesson 1.4](/lessons/checks-before-trusting-ai-output) 의 검토 습관을 실행 전에 당겨오는 거예요. 산출물 [`plan-with-ai.md`](/lessons/plan-with-ai) 는 에이전트가 코드를 고치기 전에 반드시 제출해야 할 계획서 템플릿입니다.

## 핵심 개념 — Implementation Plan 과 Stop Points

### Implementation Plan — 변경 전 합의 문서

좋은 구현 계획은 길 필요가 없어요. 대신 구체적이어야 합니다. “컴포넌트 수정”이 아니라 “`CheckoutButton.tsx` 의 모바일 레이아웃 클래스 수정”처럼 파일과 의도가 보이면 충분해요.

계획에는 최소 다섯 가지가 들어갑니다.

| 항목 | 질문 |
| --- | --- |
| 목표 | 이번 변경으로 무엇이 달라지나요? |
| 변경 파일 | 어떤 파일을 읽고, 어떤 파일을 수정하나요? |
| 접근 순서 | 무엇부터 확인하고 무엇을 나중에 하나요? |
| 검증 | 어떤 명령과 수동 확인을 하나요? |
| 롤백 | 문제가 생기면 무엇을 되돌리나요? |

이 구조는 [Lesson 2.2](/lessons/enforcing-output-format) 의 출력 강제 패턴과 비슷합니다. 에이전트가 자유롭게 장황한 계획을 쓰지 않고, 필요한 칸을 빠뜨리지 않게 하는 거예요.

### Stop Points — 반드시 사람에게 묻는 지점

Stop Point 는 에이전트를 불신해서 만드는 장치가 아니에요. 자동화가 멈춰야 안전한 지점을 미리 정하는 겁니다. 예를 들어 의존성을 바꾸려 할 때, 설정 파일을 바꾸려 할 때, 원래 범위 밖 파일을 고쳐야 할 때는 멈춰야 합니다.

이 기준이 없으면 에이전트가 “필요해 보여서” 일을 넓힙니다. 기준이 있으면 “여기서 사람 확인이 필요합니다”라고 보고합니다. [Lesson 4.1](/lessons/coding-agent-setup) 의 권한 정책이 실제 작업 계획 안으로 들어오는 순간이에요.

### 계획과 실행은 다른 단계다

계획을 작성한 에이전트가 바로 구현하지 않게 하세요. 먼저 사람이 계획을 읽고 “이 범위로 진행해요”라고 승인합니다. 이 짧은 멈춤이 품질을 올립니다. 다음 [Lesson 4.5](/lessons/write-tests-with-coding-agent) 에서도 같은 원리로 테스트를 먼저 쓰고 구현을 나중에 합니다.

## 그래서 오늘 할 일

오늘 만들 것은 “AI 와 합의하는 구현 계획 템플릿”입니다.

1. [Lesson 4.2](/lessons/github-issue-to-ai-brief) 에서 만든 브리프 하나를 고릅니다.
2. [Lesson 4.3](/lessons/letting-ai-read-codebase) 의 Scout Run 결과를 붙입니다.
3. 변경 후보 파일을 읽기 전용과 수정 후보로 나눕니다.
4. 테스트 전략을 자동 검증과 수동 확인으로 나눕니다.
5. Stop Point 를 3개 이상 적습니다.
6. 롤백 기준과 되돌릴 단위를 정합니다.
7. 산출물 [`plan-with-ai.md`](/lessons/plan-with-ai) 로 실제 계획 1건을 작성합니다.

계획서를 만들 때 중요한 질문은 하나예요. “이 계획만 읽고도 다른 사람이 같은 구현을 시작할 수 있나?” 그렇다면 에이전트도 훨씬 안정적으로 움직입니다.

## 마무리

계획은 속도를 늦추는 문서가 아니라, 잘못된 속도를 줄이는 장치입니다. 오늘 합의한 계획이 있으면 다음 [Lesson 4.5](/lessons/write-tests-with-coding-agent) 에서 테스트를 먼저 쓰게 만들 수 있어요. 구현 전에 무엇이 성공인지 정했으니, 이제 그 성공을 증명할 테스트를 에이전트와 함께 만들 차례입니다.
