## 만든 것을 보여줘야 존재한다

4주가 지났다. 배포했다.

```
GitHub 저장소에 코드가 있다.
README에는 "프로젝트 설명을 여기 적으세요."가 그대로다.
링크를 공유하기 창피하다.
```

배포가 끝이 아니다. 공개가 끝이다.

좋은 포트폴리오는 "나 이런 거 만들었어요"가 아니라
"당신이 이런 문제 있을 때 이게 도움이 됩니다" 로 시작한다.

## 포트폴리오 README 4구조

### 1. 무엇 (한 줄 요약)

```
[제품명] — 회의 녹음을 올리면 5초 만에 요약과 결정 사항을 뽑아줍니다.
```

### 2. 왜 (어떤 문제)

```
클라이언트 미팅 후 회의 정리에 매번 30분이 사라집니다.
한 달 뒤에 뭘 결정했는지 기억이 안 나는 경험, 있으시죠?
```

### 3. 어떻게 (기술 스택·핵심 결정)

```
기술 스택: Next.js + FastAPI + pgvector + Claude API
핵심 결정:
- Whisper로 STT → Claude로 요약 + 결정 사항 추출
- pgvector에 임베딩 저장 → 나중에 검색 가능
```

### 4. 데모 (링크 + 스크린샷)

```
🔗 데모: https://...
📸 스크린샷: [이미지]
```

## 회고 문서 작성

`docs/retrospective.md` 를 4축으로 채운다.

### 지표

```
- 배포 후 실제 사용자 수: N명
- 피드백 받은 수: N건
- 월 API 비용: $X
- 가장 많이 쓰인 기능: [기능명]
```

초기라 숫자가 작아도 괜찮다. 기록이 있는 게 없는 것보다 낫다.

### AI vs 사람 비율

```
코드 비율:
- AI 작성: 약 70%
- 내가 수정/작성: 약 30%

결정 비율:
- AI 제안으로 결정: 약 40%
- 내가 직접 결정: 약 60%

수정이 가장 많았던 부분:
- [무엇] — AI 코드가 맥락을 몰라서 직접 수정
```

### 한계

```

- 기술적으로 해결 못 한 것: [있으면]
- 예상과 달랐던 것: [있으면]
```

### 다음 버전

```
v2에서 한다면:
```
