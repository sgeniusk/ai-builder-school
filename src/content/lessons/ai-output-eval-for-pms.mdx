## "품질이 안 좋아요"는 기준이 아니다

개발팀에게 "AI 출력 품질이 안 좋아요"라고 말하면 돌아오는 말이 있다.

"어떤 기준으로요?"

PM이 느끼는 '안 좋음'을 코드로 측정 가능한 기준으로 번역하는 것 — 이게 Eval 루브릭의 역할이다.

좋은 소식이 있다. **Eval 기준을 정의하는 건 코드가 아니라 비즈니스 판단이다.** 코드를 몰라도 된다.

---

## 비즈니스 목표에서 역으로

AI 기능이 존재하는 이유에서 품질 기준이 나온다.

**예시: 고객 상담 요약 AI**

비즈니스 목표: 상담원이 이전 상담 내역을 파악하는 시간을 줄인다.

이 목표에서 역으로:
- 요약을 읽는 데 30초 이내여야 한다 → **길이 기준: 100~200자**
- 핵심 불만이 빠지면 안 된다 → **핵심 내용 포함 여부**
- 틀린 정보가 있으면 안 된다 → **정보 정확도**

AI에게 이 과정을 도움받을 수 있다.

```
이 AI 기능의 비즈니스 목표는 '상담원 상담 이력 파악 시간 단축'이야.
이 목표에서 역으로, 측정 가능한 AI 출력 품질 기준 5개를 제안해줘.
각 기준은 0/1 또는 1~5점으로 판단할 수 있어야 해.
```

---

## Eval 루브릭 만들기

AI가 제안한 기준을 편집해 루브릭으로 만든다.

**나쁜 기준 (측정 불가능):**
- 요약이 자연스러워야 한다
- 도움이 되어야 한다
- 이해하기 쉬워야 한다

**좋은 기준 (측정 가능):**

| 항목 | 기준 | 점수 |
|------|------|------|
| 길이 | 100~200자 사이 | 0 / 1 |
| 핵심 불만 포함 | 리뷰에 언급된 주요 불만이 1개 이상 요약에 있음 | 0 / 1 |
| 정보 정확도 | 리뷰에 없는 내용이 요약에 없음 | 0 / 1 |
| 고유명사 정확도 | 상품명, 고객명이 원본과 일치 | 0 / 1 |
| 액션 포인트 | 상담원이 취해야 할 행동이 명시됨 | 0 / 1 |

두 사람이 같은 샘플을 독립적으로 채점했을 때 같은 결과가 나오면 좋은 기준이다.

## 스프레드시트 Eval

코드 없이 품질을 측정하는 가장 빠른 방법이다.

1. AI 출력 샘플 10~20개를 스프레드시트 A열에 입력
2. 원본 입력을 B열에
3. C~G열에 루브릭 항목별 점수(0/1)
4. H열에 합계
5. 평균 점수로 현재 품질 수준 파악

```
| 출력 | 원본 | 길이 | 핵심불만 | 정확도 | 고유명사 | 액션 | 합계 |
|------|------|------|---------|--------|---------|------|------|
| ...  | ...  |  1   |    1    |   0    |    1    |  1   |  4/5 |
| ...  | ...  |  0   |    1    |   1    |    1    |  0   |  3/5 |
```

10개 샘플의 평균이 3.2/5 → 목표를 4.0/5로 잡고 개발팀에게 요청한다.

## 개발팀에게 전달할 때

PM의 루브릭을 개발팀이 코드로 구현하려면 더 구체적인 형태가 필요하다.

**PM 루브릭:**
> "요약이 100~200자 사이여야 한다"

**개발팀 전달 형태:**
```python
def check_length(output: str) -> bool:
    return 100 <= len(output) <= 200
```

AI에게 이 변환도 맡길 수 있다.

```
아래 Eval 루브릭을 Python 함수로 변환해줘.
각 항목이 True/False를 반환하게 해줘.

루브릭:
1. 길이 100~200자
2. 원본에서 추출 가능한 내용만 포함 (외부 정보 추가 없음)
3. ...
```

## 주간 Eval 루틴

Eval을 한 번만 하면 의미가 없다. 주간 루틴으로 만든다.

1. **매주 월요일 10분:** 지난 주 AI 출력 샘플 10개 선택
2. 스프레드시트에 채점 (루브릭 기준)
3. 점수 트렌드 기록 (이번 주 vs 지난 주)
4. 점수가 목표 이하로 떨어지면 개발팀에 알림

월 1회 루브릭 자체를 검토한다. 기준이 여전히 비즈니스 목표와 일치하는지 확인한다.

## 핵심 원칙

> "안 좋아요"는 피드백이 아니다. 기준이 필요하다.
> Eval 루브릭은 코드가 아니라 비즈니스 판단이다.
> 두 사람이 같은 결과를 낼 수 있어야 좋은 기준이다.
