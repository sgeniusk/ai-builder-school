## "같은 문서인데 검색 결과가 완전히 달라요"

회사 기술 가이드(30페이지)를 RAG에 넣었어요. "인증 방식이 뭐야?"라고 물으면 정확한 답이 나옵니다. 그런데 동료가 같은 문서를 다른 방식으로 잘라 넣었더니 엉뚱한 답이 나와요. 문서가 같고, 질문이 같고, 모델이 같은데 **자르는 방식만 달라서** 답이 달라졌습니다.

[Lesson 23](/lessons/understanding-embeddings)에서 임베딩이 "의미의 좌표"를 만든다고 했죠. 그 좌표의 품질은 **무엇을 임베딩하느냐** — 즉 청크의 모양에 달려 있어요. 너무 짧게 자르면 맥락이 사라지고, 너무 길게 자르면 [Lesson 07](/lessons/feeding-long-documents)의 "Lost in the Middle"이 다시 켜집니다.

오늘 만들 건 **같은 문서를 3가지 청킹 전략으로 잘라 검색 품질을 비교한 노트**예요.

## 청킹이 검색 품질을 좌우하는 이유

{/* Source · PAPER · Lewis et al. (2020) "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" — https://arxiv.org/abs/2005.11401 */}
{/* Source · DOC · Anthropic "Contextual Retrieval" (2024) — https://www.anthropic.com/news/contextual-retrieval */}

RAG 파이프라인은 이런 흐름이에요:

```
문서 → [청킹] → [임베딩] → 벡터 DB → [검색] → LLM에 주입 → 답변
```

이 파이프라인에서 **청킹은 첫 번째 병목**이에요. 아무리 좋은 임베딩 모델을 써도, 청크 안에 관련 정보가 없으면 검색이 안 잡아요. 반대로 청크가 너무 크면 관련 없는 정보까지 같이 들어와서 LLM이 핵심을 놓칩니다.

Anthropic의 Contextual Retrieval 연구에서도 청킹 전략이 검색 정확도에 미치는 영향이 크다는 점을 강조합니다. 전략 하나만 바꿔도 검색 실패율이 크게 줄어들 수 있어요.

## 3가지 청킹 전략

### 1. 고정 크기 (Fixed-size)

{/* Source · DOC · LangChain "RecursiveCharacterTextSplitter" — https://python.langchain.com/docs/how_to/recursive_text_splitter/ */}

가장 단순합니다. 문서를 N글자(또는 N토큰) 단위로 자릅니다. LangChain의 `RecursiveCharacterTextSplitter`가 대표적이에요.

```
원문 3000토큰
├─ 청크 1: 0~500토큰
├─ 청크 2: 400~900토큰 (100토큰 오버랩)
├─ 청크 3: 800~1300토큰
├─ ...
└─ 청크 6: 2500~3000토큰
```

- **장점**: 구현이 쉽고 예측 가능. 어떤 문서든 돌아감.
- **단점**: 문단 중간이 잘려 맥락이 깨짐. 표나 코드 블록이 반으로 나뉨.

시작점: **500~1000 토큰 청크 + 10~20% 오버랩**. 자기 도메인에서 조정하세요.

### 2. 구조 기반 (Structure-aware)

문서의 **구조를 존중**해서 자릅니다. Markdown이면 `##` 헤더 단위, HTML이면 `<section>` 단위, PDF면 페이지/섹션 단위.

```
원문 (Markdown)
├─ 청크 1: ## 개요 (전체 섹션)
├─ 청크 2: ## 인증 방식 (전체 섹션)
├─ 청크 3: ## 에러 처리 (전체 섹션)
└─ 청크 4: ## FAQ (전체 섹션)
```

- **장점**: 맥락이 자연스럽게 보존됨. 사람이 읽어도 의미 있는 단위.
- **단점**: 섹션 크기가 들쭉날쭉. 한 섹션이 5000토큰이면 그 자체가 너무 큼. 구조가 없는 문서(순수 텍스트)엔 못 씀.

Markdown·HTML·코드 파일처럼 **구조가 명확한 문서**에 강합니다.

### 3. 의미 기반 (Semantic)

문단 사이의 **임베딩 유사도**를 계산해서, 의미가 크게 바뀌는 지점에서 자릅니다.

```
원문
├─ 문단 1~3: 유사도 높음 → 하나의 청크
├─ 문단 4: 유사도 급락 (주제 전환) → 새 청크 시작
├─ 문단 4~6: 유사도 높음 → 하나의 청크
└─ ...
```

- **장점**: 주제 전환 지점을 자동으로 감지. 가장 "의미 있는" 단위.
- **단점**: 구현 복잡. 임베딩 호출 비용 추가. 청크 크기 예측 불가.

구조가 없는 긴 텍스트(회의록, 인터뷰 전사본)에 효과적이에요.

## 오버랩 — 경계에서 정보가 잘리는 걸 막는 장치

{/* Note: 오버랩 비율 10~20%는 일반적 관행이며 정식 표준은 아닙니다. */}

세 전략 모두 **경계 문제**가 있어요. "인증은 OAuth 2.0을 사용하며"가 청크 1의 끝에, "만료 시간은 3600초"가 청크 2의 시작에 걸리면 — 둘 다 검색에서 약해집니다. 하나의 사실이 두 조각에 걸친 거예요.

해결책이 **오버랩(overlap)**이에요. 청크 사이에 겹치는 구간을 두면 경계 정보가 양쪽 청크에 다 들어갑니다.

| 오버랩 | 효과 | 비용 |
| --- | --- | --- |
| 0% | 경계 정보 손실 ↑ | 저장 공간 최소 |
| 10~20% | 대부분의 경계 문제 해결 | 약간의 저장 증가 |
| 50% 이상 | 경계 문제 거의 없음 | 저장·임베딩 비용 2배 가까이 |

보통 **10~20%**로 시작해서, 검색 실패가 잦으면 올립니다.

## Contextual Retrieval — 청크에 맥락을 더하기

{/* Source · DOC · Anthropic "Contextual Retrieval" (2024) — https://www.anthropic.com/news/contextual-retrieval */}

Anthropic이 2024년에 제안한 방법이에요. 청크를 자른 뒤, 각 청크 앞에 **"이 청크가 전체 문서에서 어떤 위치인지"를 설명하는 맥락 문장**을 LLM으로 생성해 붙입니다.

```
[원래 청크]
OAuth 2.0 인증 플로우는 다음과 같다...

[맥락 추가 후]
이 청크는 "API 보안 가이드" 문서의 "3장 인증 방식" 섹션에 속합니다.
OAuth 2.0 인증 플로우는 다음과 같다...
```

이렇게 하면 청크 단독으로도 "이게 어디서 온 건지"를 알 수 있어서 검색 정확도가 올라가요. 비용(LLM 호출)이 추가되지만, 검색 실패율 감소 폭이 큽니다.

## 셋의 트레이드오프 — 한 표로

{/* Note: 아래 평점은 대략적 감을 위한 것이며 모델·도메인에 따라 다릅니다. */}

| 전략 | 구현 난이도 | 맥락 보존 | 검색 정밀도 | 적합 문서 |
| --- | --- | --- | --- | --- |
| 고정 크기 | 낮음 | 낮음 | 중간 | 모든 문서 (범용 시작점) |
| 구조 기반 | 중간 | 높음 | 높음 | Markdown·HTML·코드 |
| 의미 기반 | 높음 | 높음 | 높음 | 구조 없는 긴 텍스트 |

"항상 의미 기반이 답"이 아니에요. 문서에 `##` 헤더가 잘 잡혀 있으면 구조 기반이 더 낫고, 빠르게 프로토타입 하려면 고정 크기로 시작하는 게 맞습니다.

## 실전 청킹 사이즈 가이드

| 문서 종류 | 권장 청크 크기 | 오버랩 | 전략 |
| --- | --- | --- | --- |
| 기술 문서 (Markdown) | 헤더 단위 (500~1500토큰) | 10% | 구조 기반 |
| 회의록 / 인터뷰 | 800~1200토큰 | 15% | 의미 기반 또는 고정 |
| 법률 / 계약서 | 조항 단위 | 20% | 구조 기반 |
| FAQ | 질문-답변 쌍 단위 | 0% | 구조 기반 |
| 코드 파일 | 함수/클래스 단위 | 0% | 구조 기반 |

(이 표는 시작점이에요. 자기 도메인에서 실험해서 조정하세요.)
