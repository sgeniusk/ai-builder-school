# 장기 기억 설계도

> 이 파일은 AI Builder School · Stage 6 "장기 기억과 상태" 에서 제공되는 출발 템플릿입니다.
> 당신이 만드는 AI 서비스에 맞춰 자유롭게 수정하세요.

## 1. 저장 vs 기억 — 무엇을 어디에 둘 것인가

대화 원문은 "저장"하고, 재사용할 사실만 "기억"으로 승격한다.

| 항목 | 저장(원문 보관) | 기억(추출·구조화) |
| --- | --- | --- |
| 무엇을 | 모든 메시지 그대로 | 계속 참일 사실만 |
| 형태 | 메시지 행(시간순 로그) | 구조화된 메모리 레코드 |
| 목적 | 감사·복기·디버깅 | 다음 대화에 회수 |
| 예시 | "사용자: 저 채식해요" | `{선호: 채식, 범위: 음식 추천}` |

- 내 서비스에서 "저장"에 해당하는 것:
- 내 서비스에서 "기억"으로 승격할 것:
- 기억 레코드 한 건의 필드 (예: `id / 사실 / 범주 / 출처 대화 / 생성일 / 최종확인일`):

## 2. 사실 추출 기준 — 무엇을 기억으로 올릴까

대화가 끝날 때마다 "이 안에 계속 참일 사실이 있나?"를 묻는다.

- [ ] 시간이 지나도 참인가 (선호·제약·맥락 → 기억 / 그 순간뿐 → 폐기)
- [ ] 다음 대화에서 다시 쓸 일이 있는가
- [ ] 한 줄로 정제 가능한가 (잡담·맥락 없이 사실만)

기억할 사실 범주 (예시 — 내 서비스에 맞게 수정)

| 범주 | 기억할 것 | 예시 |
| --- | --- | --- |
| 선호 | 사용자가 원하거나 싫어하는 것 | 매운 음식 X |
| 제약 | 지켜야 할 한계·조건 | 월 예산 5만 원 |
| 맥락 | 사용자의 고정된 배경 | 부산 거주 |
| | | |

- 추출하지 않을 것 (기억 오염 방지):

## 3. 관련 기억 회수 — 다음 대화에 무엇을 넣을까

기억 전부가 아니라, 지금 사용자가 하는 말과 관련된 것만 골라 넣는다(RAG 검색 원리).

- 회수 트리거 (언제 기억을 찾는가, 예: 매 사용자 발화 직전):
- "관련 있다"의 판단 기준 (예: 같은 범주 / 키워드 일치 / 의미 유사도 → 다음 레슨 임베딩):
- 한 번에 프롬프트에 넣을 기억 최대 개수 (컨텍스트·비용 한계 고려):
- 회수된 기억을 프롬프트 어디에 넣을지 (예: 시스템 프롬프트 상단):

> 주의 — 대화 원문을 통째로 재주입하지 않는다. 비용이 폭발하고 컨텍스트 한도에 부딪힌다.

## 4. 기억 수명 정책 — 갱신과 망각

사실은 변한다. 모순되는 새 정보와 노후한 기억을 어떻게 처리할지 미리 정한다.

- 모순 처리 — 새 사실이 옛 사실과 부딪힐 때 어느 쪽을 살릴지 (예: 최신 발화 우선):
- 갱신 방식 — 기존 레코드를 덮어쓸지, 새 레코드를 추가하고 옛것을 비활성화할지:
- 망각 기준 — 며칠/몇 달간 재확인되지 않은 기억을 폐기할지 (예: 180일 미확인 시 보류):
- 사용자 권한 — 사용자가 자기 기억을 직접 보고 수정·삭제할 수 있는가:
