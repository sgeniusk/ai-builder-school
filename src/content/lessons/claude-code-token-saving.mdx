## "1번째 메시지 500토큰이 30번째에 232,000토큰"

Claude Code로 작업하다 한 시간도 안 됐는데 "토큰을 모두 소모하였습니다" 메시지가 떠요. 돈 내고 쓰는데, 클로드 탓도 플랜 탓도 아닙니다 — **쓰는 방식이 문제**예요.

핵심 사실 한 줄.

> Claude는 매번 전체 대화를 재로딩합니다. 30번째 메시지는 첫 번째보다 31배 비싸요.

오늘 만들 건 **Claude Code 토큰 절약 12가지 핵심**이에요. 외부에서 모은 52가지 팁 중에서 가장 효과 큰 것만 추렸습니다.

## 왜 토큰이 그렇게 빨리 녹나

{/* Source · DOC · Anthropic "Context windows" — https://platform.claude.com/docs/en/build-with-claude/context-windows */}
{/* Note: 본문의 토큰 시뮬레이션 수치(500/27,500/232,000)는 외부 사례로부터의 표본이며, 실제는 모델·메시지 길이·도구 호출 빈도에 따라 다릅니다. */}

카카오톡 단톡방은 새 메시지만 읽어요. 그런데 Claude는 메시지를 보낼 때마다 **1번 메시지부터 전부 다시 읽습니다**. 메시지당 평균 500 토큰 가정으로.

| 메시지 # | 누적 토큰 |
| --- | --- |
| 1번째 | 500 |
| 10번째 | 27,500 (5,000이 아님) |
| 30번째 | **232,000** |

10번째에서 이미 첫 메시지의 55배. 30번째에서는 31배 더 비싸진 비용이 한 메시지에 들어갑니다. **재로딩이 핵심**이에요.

이걸 알고 나면 12가지 팁이 왜 효과 있는지가 자연스럽게 풀려요.

## 초급 4가지 — 즉시 적용

### 1. `/clear`가 가장 빠른 절약

작업 끝났으면 다음 작업 시작 전에 무조건 `/clear`. 토큰을 절약하는 가장 쉽고 빠른 방법이에요.

| 메시지 | 토큰 |
| --- | --- |
| 1번째 | 500 |
| 30번째 (clear 안 함) | 232,000 |
| 30번째 (clear 후 첫 메시지) | 500 |

기능 하나 끝났으면 `/clear`. 이 한 습관만 들여도 비용이 절반으로 떨어져요.

### 2. `.claudeignore` 파일 만들기

Claude는 탐색 시 `node_modules`, `.next`, `dist` 같은 파일까지 전부 읽으려 해요. 큰 프로젝트면 수십 기가 — 토큰이 순식간에 녹습니다.

```
node_modules/
.next/
dist/
build/
*.log
*.lock
__pycache__/
coverage/
```

프로젝트 루트에 한 번만 만들면 계속 유효해요. ".gitignore처럼 클로드가 안 읽을 파일을 지정"이라고 생각하면 됩니다.

### 3. CLAUDE.md를 200줄 이하로 유지

CLAUDE.md는 세션 시작 때만 읽히는 게 아니에요. **매 메시지마다** 읽혀요. 5,000줄짜리 장황한 CLAUDE.md vs 700토큰짜리 간결한 CLAUDE.md는 메시지당 4,300토큰 차이입니다.

다이어트 4가지.

- "이 프로젝트는..." 같은 서술형 문장 제거
- 헤딩·리스트·테이블로 구조화
- 규칙은 최소 단위로 축약, 코드 예시 과도 포함 금지
- 200줄 넘기 시작하면 다이어트 타이밍

### 4. 상세 지식은 스킬로 분리

CLAUDE.md를 200줄 이하로 유지하면서 나머지는 어디에? **스킬 파일**로 분리해요.

```
# 항상 로드
CLAUDE.md

# 필요할 때만 로드
docs/api-guide.md
docs/db-guide.md
docs/deploy-guide.md
```

CLAUDE.md는 항상 로드 → 핵심 규칙만, 스킬 파일은 필요할 때만 메모리 로드. 세션당 적지 않은 토큰을 절약할 수 있어요.

## 중급 4가지 — 한 단계 더

### 5. Compaction 타이밍 알기

세션이 길어지면 Claude가 자동으로 대화를 요약해 압축해요. 이걸 **Compaction**이라고 부릅니다. 이상적 타이밍을 알면 토큰을 더 절약할 수 있어요.

- 한 작업 묶음이 끝나면 `/compact`로 직접 트리거
- 도구 호출 결과가 길게 누적됐다면 `/compact`
- 단, **`/compact` 직전 핵심 결정은 메모로 따로 저장** — 압축 후 사라질 수 있음

### 6. 5가지 절약 규칙을 CLAUDE.md에 명시

직접 넣어두면 클로드가 스스로 낭비를 줄여요.

```markdown
## Token saving rules
1. 이미 읽은 파일은 다시 확인하지 않는다
2. 불필요한 도구 호출은 하지 않는다
3. 가능한 도구 호출은 동시에 실행한다
4. 20줄 이상의 불필요한 출력은 서브에이전트에 위임한다
5. 사용자가 이미 설명한 내용을 다시 반복하지 않는다
```

특히 1번과 5번이 효과 큽니다.

### 7. 도구 호출 병렬화

Claude는 기본적으로 도구를 순차로 호출해요. 4개 파일을 읽어야 하면 4번에 걸쳐 읽습니다. 병렬화하면 토큰은 같지만 응답 시간이 4배 빨라요.

CLAUDE.md에 한 줄.

```markdown
## 도구 호출
- 독립적인 도구 호출은 병렬로 실행 (parallel function calling)
```

### 8. 서브에이전트로 컨텍스트 분리

긴 작업은 메인 세션에 다 넣지 마세요. 별도 서브에이전트로 던지면 그 작업의 토큰이 메인 세션에 누적되지 않아요. 코드 검토·테스트 작성 같은 격리 가능한 작업이 적합합니다.

## 고급 4가지 — 큰 프로젝트 필수

### 9. QMD — 코드베이스 사전 인덱싱

```
"인증 관련 코드 어디 있어?" 물어보면 무슨 일이 생기나요?
1. glob으로 전체 파일 목록 읽기 (탐색에 낭비되는 토큰)
2. grep으로 키워드 검색 (또 낭비)
3. 파일 통째로 읽기 (의미 있는 단계는 이거뿐)
```

QMD는 1·2단계 토큰 낭비를 통째로 건너뛰는 도구예요. 프로젝트의 모든 파일을 미리 색인해놓고 클로드가 검색 엔진처럼 바로 원하는 코드를 찾을 수 있게 해줍니다. 외부 사례에서 탐색 토큰의 90%대 절감이 보고됐어요(인증 로직 탐색 91%, DB 스키마 파악 93% 등). 본인 프로젝트에서 직접 측정 후 도입을 판단하세요.

설치 후 CLAUDE.md에 한 줄 추가.

```markdown
"파일을 읽기 전에 항상 qmd로 먼저 검색하라."
```

### 10. read-once 훅 — 파일 재읽기 방지

Claude는 같은 파일을 반복해서 읽어요. `package.json`을 읽고, 변경하고, 또 읽어서 확인하고, 연관 파일 작업 때 또 읽고. 매번 읽을 때마다 그 파일의 전체 토큰 비용이 발생합니다.

read-once 훅은 이미 읽은 파일을 추적하고 중복 재읽기를 차단해요. diff 모드도 있어서, 파일 읽기를 완전히 막는 대신 마지막으로 읽은 이후 변경된 줄만 반환하게 할 수도 있습니다.

### 11. 3 에이전트 팀 구조

큰 프로젝트는 한 에이전트로 다 짜지 마세요. 3 역할 분리.

| 역할 | 무엇 |
| --- | --- |
| **Architect** | 큰 그림 결정, 인터페이스 정의 |
| **Implementer** | 코드 구현 |
| **Reviewer** | 검증·테스트·리뷰 |

각 에이전트가 자기 컨텍스트만 들고 다녀서 메인 세션이 가벼워져요.

### 12. 훅 자동화로 반복 절약

Claude Code의 훅(hook) 시스템으로 반복 작업을 자동화하면 매번 사람이 같은 지시를 안 해도 돼요. PreToolUse / PostToolUse 같은 훅에 토큰 절약 규칙을 박아두면 자동으로 작동합니다.

| 항목 | 현재 | 다음 액션 |
