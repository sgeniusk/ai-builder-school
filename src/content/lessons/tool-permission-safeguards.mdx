## 도입 — 학습자에게 말 거는 한 장면

미니 에이전트가 꽤 잘 움직입니다. 검색하고, 요약하고, 초안을 저장해요. 그래서 도구를 하나 더 붙이고 싶어집니다. 이메일 보내기, 파일 삭제하기, 고객 레코드 수정하기, 외부 API 호출하기 같은 도구입니다. 여기서부터 분위기가 달라져요. 이제 에이전트는 답을 만드는 존재가 아니라 실제 시스템을 바꾸는 존재가 됩니다.

도구는 직원에게 쥐어주는 회사 카드와 비슷합니다. 카드가 있으면 일을 빠르게 처리할 수 있지만, 한도와 용처가 없으면 사고도 빠르게 납니다. 점심 결제용 카드로 서버 비용을 결제하면 안 되고, 테스트용 카드로 고객 환불을 처리하면 안 돼요. 에이전트 도구도 같습니다. 도구마다 허용 범위와 확인 절차가 있어야 합니다.

오늘은 에이전트에게 쥐여 줄 도구 권한 프로파일을 만듭니다. “무엇을 할 수 있는가”보다 “어디서 멈춰야 하는가”를 먼저 정하는 레슨이에요.

## Lesson 7.2 다음에 왜 이 레슨인가

[Lesson 7.2](/lessons/mini-agent-loop) 에서는 검색, 요약, 저장 도구를 가진 작은 루프를 만들었습니다. 루프가 생기면 같은 도구를 여러 번 부를 수 있고, 실패를 복구하려고 다른 도구를 시도할 수도 있어요. 이 힘은 편리하지만, 권한이 흐리면 위험합니다.

이미 우리는 비슷한 일을 해봤습니다. [Lesson 4.1](/lessons/coding-agent-setup) 에서 코딩 에이전트에게 파일 읽기, 파일 수정, 터미널 실행, 비밀 파일 접근의 울타리를 쳤어요. 그때 만든 원칙을 이제 우리가 만든 에이전트에도 적용합니다. 코딩 에이전트한테 친 울타리를, 제품 안의 에이전트한테도 치는 거예요.

[Lesson 4.5](/lessons/write-tests-with-coding-agent) 의 테스트 습관도 여기서 중요합니다. 권한 정책은 문서로만 있으면 쉽게 어긋납니다. 위험 동작이 실제로 차단되는지, 사람 확인 없이 실행되지 않는지 테스트해야 해요.

## 핵심 개념 — Allow List, Deny List, Human Confirm Step

Allow List는 에이전트가 해도 되는 일을 적은 목록입니다. 읽기 전용 문서 검색, 공개 데이터 조회, 임시 초안 저장처럼 되돌리기 쉽고 위험이 낮은 일을 넣습니다. 허용 목록은 좁게 시작하는 편이 좋아요. 나중에 필요한 도구를 하나씩 열면 되지만, 처음부터 넓게 열어 둔 권한을 줄이는 일은 어렵습니다.

Deny List는 절대 자동 실행하면 안 되는 일을 적은 목록입니다. 비밀 파일 읽기, 고객 데이터 삭제, 결제 변경, 운영 DB 직접 수정, 외부로 대량 전송 같은 동작입니다. 여기에 있는 일은 모델이 아무리 그럴듯한 이유를 말해도 실행하지 않습니다. “정말 필요해 보인다”는 판단은 모델이 아니라 사람이 해야 합니다.

Human Confirm Step은 위험 동작 직전에 사람 확인을 끼우는 단계입니다. 모든 것을 막자는 뜻이 아니에요. 예를 들어 이메일 초안 작성은 자동으로 해도 되지만, 실제 발송은 확인을 받습니다. 파일 삭제 후보 목록을 만드는 것은 허용하지만, 삭제 실행은 확인을 받습니다. 이 작은 멈춤 지점이 제품 사고를 많이 줄여요.

권한은 도구 이름만으로 정하면 부족합니다. 같은 `writeFile` 도구라도 임시 폴더에 쓰는 것과 운영 설정 파일을 바꾸는 것은 다릅니다. 그래서 경로, 데이터 종류, 외부 전송 여부, 되돌릴 수 있는지까지 함께 봐야 합니다.

## 그래서 오늘 할 일

오늘 만들 것은 “도구 권한 프로파일”입니다. 먼저 에이전트가 가진 도구를 모두 적습니다. 검색, 요약, 저장, 이메일, 파일 수정, 외부 API 호출처럼 실제 이름으로 적어요. 그다음 각 도구를 세 칸으로 나눕니다. 항상 허용, 확인 후 허용, 항상 금지입니다.

다음으로 위험 동작 예시를 만듭니다. “운영 고객 100명에게 이메일 발송”, “`.env` 파일 읽기”, “지난달 주문 데이터 삭제”, “외부 URL의 지시문 실행” 같은 문장을 써 봅니다. 에이전트가 이런 요청을 받았을 때 어떤 메시지로 멈출지 정합니다.

마지막으로 로그 형식을 붙입니다. 도구 이름, 요청자, 인자 요약, 권한 판정, 확인 여부, 실행 결과를 남깁니다. 권한 시스템은 사고를 막는 장치이면서, 사고가 났을 때 되짚는 기록 장치이기도 합니다.

산출물 [`tool-permission-policy.md`](/lessons/tool-permission-safeguards) 에는 도구 분류표, 확인 플로우, 금지 동작, 테스트 케이스를 적습니다. 처음 정책은 보수적으로 만들어도 괜찮아요. 에이전트가 못 해서 불편한 일은 나중에 열 수 있지만, 이미 실행된 위험 동작은 되돌리기 어렵습니다.

## 마무리

도구 권한은 에이전트 제품의 안전벨트입니다. 사용자는 빠른 자동화를 원하지만, 제품은 멈춰야 할 곳에서 멈출 줄 알아야 해요. 오늘 만든 정책은 다음 [Lesson 7.4](/lessons/prompt-injection-defense) 의 바닥이 됩니다. 외부 문서나 사용자 입력이 에이전트를 속이려 해도, 권한 프로파일이 있으면 민감한 도구 앞에서 한 번 더 멈출 수 있습니다. Phase 11에서 실제 사용자가 있는 제품을 출시할 때도 이 정책은 기능 목록만큼 중요합니다.
