## "이번엔 마크다운 표였는데 다음엔 또 글이네"

자동화 파이프라인을 짭니다. AI가 회의록을 받아 Action Item JSON을 뱉으면 그걸 Notion API에 그대로 던지는 흐름이에요. 첫 5번은 잘 됩니다. 그러다 6번째 호출에서 모델이 친절하게 "다음은 정리한 결과입니다:" 한 줄을 앞에 붙이는 순간 — `JSON.parse` 가 깨지면서 파이프라인 전체가 멈춰요.

[좋은 프롬프트의 구조](/lessons/structure-of-good-prompts)의 4축 중 **Output 축**이 자동화에서 가장 까다로운 이유가 여기 있습니다. 사람이 읽을 답은 "어느 정도 흔들려도" 괜찮지만, **기계가 읽을 답은 99.9% 같은 모양**이 나와야 해요. 한 번의 이탈이 전체를 멈춥니다.

오늘은 그 99.9%를 만드는 기법을 한 장에 정리합니다.

:::callout

**📌 비개발자라면 — 20분 루트**

코드 없이도 *출력 형식 강제*의 80%는 달성됩니다. 아래 3단계만 보고 닫아도 됩니다. JSON Schema·Zod·Pydantic은 *기계가 다음 단계를 자동 실행*해야 할 때만 필요합니다.

1. **프롬프트 끝에 마크다운 표 형식 명시** — "다음 형식으로 답하세요. `| 항목 | 내용 |`" 식으로. AI가 표를 흐트러뜨릴 일이 거의 없어집니다.
2. **ChatGPT/Claude의 Custom GPT 또는 Project에 출력 양식 지시문을 한 번만 박아두기** — 매번 다시 쓰지 않아도 됩니다.
3. **결과를 그대로 Notion DB / Google Sheets 열에 붙여넣기** — 사람이 검수 후 다음 셀로 넘기는 *반자동* 루프가 됩니다.

이 3단계로 *회의록 → Action Item 표*, *리서치 → 근거 표*, *고객 문의 → 분류 라벨* 같은 *비-개발 자동화*는 이미 안정적으로 운영 가능합니다. 아래 1~3선 방어는 *API 호출*과 *코드 파이프라인*이 등장한 뒤부터 필요해집니다.

:::

## "예시 한 줄" 만으로는 부족할 때

{/* Source · DOC · OpenAI "Structured model outputs" — https://developers.openai.com/api/docs/guides/structured-outputs · 2024년 GA, JSON Schema 강제 */}
{/* Source · DOC · Anthropic "Structured outputs" — https://platform.claude.com/docs/en/build-with-claude/structured-outputs */}

L05에서 "예시 한 줄을 보여주면 답이 그 모양을 따라온다"고 했죠. 사람을 상대로 한 작업이라면 그걸로 충분해요. 하지만 자동화 파이프라인에서는 두 가지가 더 필요합니다.

**1. 형식 강제 (Format enforcement)** — 모델이 "벗어날 수 없게" 만드는 가드.
**2. 실패 회복 (Failure recovery)** — 그래도 벗어났을 때 깨지지 않게 받치는 그물.

이 둘을 합쳐 **3단 방어선**을 짭니다. 안에서 바깥으로:

| 방어선 | 무엇 | 실패율 감소 |
| --- | --- | --- |
| 1선 — 프롬프트 | 형식·예시·금지 명시 | 들쭉날쭉 → 대체로 일정 |
| 2선 — 스키마 강제 | API의 structured output 기능 | 대체로 일정 → 거의 항상 일치 |
| 3선 — 재시도 | 파싱 실패 시 자동 수정 호출 | 거의 항상 → 사실상 100% |

각 방어선이 잡아주는 비율은 모델·도메인마다 다르지만, 셋을 다 깔면 자동화 파이프라인이 "야간에도 안 깨지는" 수준이 됩니다.

## 1선 — 프롬프트로 막는 가장 가벼운 방어

{/* Source · PAPER · Brown et al. (2020) "Language Models are Few-Shot Learners" — https://arxiv.org/abs/2005.14165 · 예시 기반 in-context learning */}

API 기능을 쓰기 전에, 프롬프트만으로도 절반은 잡힙니다. 세 줄을 명시하세요.

```
Output:
- 형식: 순수 JSON. 앞뒤 설명·코드펜스·이모지 금지.
- 스키마: { "title": string, "owner": string, "due_date": "YYYY-MM-DD" }
- 예시:
  {"title": "Q3 보고서 초안", "owner": "김태우", "due_date": "2025-09-15"}
```

이 패턴의 핵심은 **세 가지 금지를 명시한 첫 줄**이에요.

- "앞뒤 설명 금지" — "다음은 정리한 결과입니다:" 같은 친절한 문구를 막습니다.
- "코드펜스 금지" — \`\`\`json ... \`\`\` 같은 마크다운 펜스 안 붙이게 합니다.
- "이모지 금지" — 모델이 종종 키 옆에 ✨를 붙입니다.

이 세 줄이 빠지면 모델은 "친절한 답"으로 회귀합니다. 사람한테는 좋지만 파서한테는 사고예요.

## 2선 — API의 스키마 강제 기능

{/* Source · DOC · OpenAI "Structured model outputs" (response_format=json_schema) — https://developers.openai.com/api/docs/guides/structured-outputs */}
{/* Source · DOC · Anthropic "Tool use" (input_schema) — https://platform.claude.com/docs/en/build-with-claude/tool-use */}

요즘 주요 LLM API는 **스키마를 직접 받는 모드**가 있어요. 프롬프트로 부탁하는 게 아니라 API 호출 자체에 JSON Schema를 첨부합니다.

**OpenAI** — `response_format: { type: "json_schema", json_schema: {...} }`
**Anthropic** — Tool use의 `input_schema` 를 활용

```typescript
// OpenAI 예시 (개념만 — 실제 호출은 SDK 문서 참고)
const response = await openai.chat.completions.create({
  model: "gpt-4o",
  messages: [...],
  response_format: {
    type: "json_schema",
    json_schema: {
      name: "action_item",
      schema: {
        type: "object",
        properties: {
          title: { type: "string" },
          owner: { type: "string" },
          due_date: { type: "string", format: "date" },
        },
        required: ["title", "owner", "due_date"],
        additionalProperties: false,
      },
      strict: true,
    },
  },
});
```

이 모드의 차이는 **"부탁이 아니라 제약"** 이에요. 프롬프트는 모델이 무시할 수 있지만, API 레벨 스키마는 디코더에서 형식을 맞춰 토큰을 강제합니다. 1선에서 새던 케이스의 대부분이 여기서 잡혀요.

(OpenAI의 `strict: true` 모드는 키 누락·여분 키·타입 불일치까지 막습니다. Anthropic의 tool use는 같은 효과를 도구 호출 형태로 얻어요.)

## 3선 — 재시도 그물

{/* Note: 재시도 패턴은 LangChain·Instructor 등 여러 라이브러리의 공통 관행. */}

스키마 강제를 깔아도 이탈이 0이 되진 않습니다. 모델이 너무 긴 출력을 생성하다 잘리거나, 스키마 외부에서 답이 비어 있는 경우가 생겨요. 마지막 그물이 필요합니다.

```typescript
async function callWithRetry(prompt: string, schema: ZodSchema, maxRetries = 2) {
  let lastError: unknown;
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const raw = await callLLM(prompt);
      const parsed = JSON.parse(raw);
      return schema.parse(parsed); // Zod 검증
    } catch (e) {
      lastError = e;
      // 두 번째 시도부터는 "이전 출력이 왜 깨졌는지" 를 같이 줍니다
      prompt = buildRepairPrompt(prompt, e);
    }
  }
  throw lastError;
}
```

핵심은 **재시도 호출에 "왜 깨졌는지"를 같이 넣는 것**이에요. 모델은 자기가 한 실수를 자기가 고치는 데 강합니다. 단순히 한 번 더 부르는 것보다 "이전 출력이 X 키가 빠졌어. 그것만 고쳐서 JSON만 줘"가 훨씬 잘 됩니다.

`maxRetries`는 보통 2면 충분해요. 그 이상에서 깨지면 모델 문제가 아니라 스키마/프롬프트 문제일 가능성이 높습니다.

## Zod / Pydantic — 런타임 검증을 같이 깔자

{/* Source · DOC · Zod (TypeScript-first schema validation) — https://zod.dev */}
{/* Source · DOC · Pydantic (Python data validation) — https://docs.pydantic.dev/ */}

API 스키마 강제만으로는 비즈니스 규칙까지는 안 잡혀요. 예를 들어 "due_date는 오늘 이후여야 한다", "owner는 우리 팀 명단에 있어야 한다" 같은 제약. 이런 건 받은 뒤 **런타임 검증 라이브러리**로 한 번 더 거릅니다.

- TypeScript/JS — **Zod**
- Python — **Pydantic**

```typescript
import { z } from "zod";

const ActionItemSchema = z.object({
  title: z.string().min(1).max(200),
  owner: z.enum(["김태우", "박민지", "이수민"]), // 팀 명단 강제
  due_date: z.string().refine((d) => new Date(d) > new Date(), {
    message: "due_date는 오늘 이후여야 합니다",
  }),
});

type ActionItem = z.infer<typeof ActionItemSchema>;
```

이 한 장이 깔리면 자동화의 마지막 신뢰가 닫힙니다. 모델이 "거의" 맞춰서 줘도, 런타임 검증이 비즈니스 규칙까지 들여다보고 막아 줘요.

## 그래서 오늘 만들 미니 예제

오늘 만들 건 **3단 방어선이 다 들어간 30줄 예제**예요. `examples/structured-output/` 폴더 안에 놓고, 다음 자동화에서 그냥 복사해 쓸 수 있는 형태로.

| 파일 | 역할 |
| --- | --- |
| `schema.ts` | Zod 스키마 정의 (단일 소스) |
| `prompt.md` | 1선 — 형식·금지·예시 명시 프롬프트 |
| `call.ts` | 2선 — API의 structured output 호출 |
| `retry.ts` | 3선 — 파싱 실패 시 자동 수정 재호출 |
| `test.ts` | 위 3선이 의도한 대로 동작하는지 확인하는 미니 테스트 |

이 다섯 파일이 한 자리에 있으면, 다음 자동화 작업 (Slack 봇, 이메일 분류기, 일정 추출기 등)에서 처음부터 짜지 않습니다. 폴더를 복사하고 스키마만 갈아끼우면 됩니다.
