- [ ] 새 파라미터는 항상 옵셔널로 추가한다

- [ ] 깨뜨릴 변경(breaking change)이 필요하면 도구에 버전 태그를 붙인다

## 3. 에러 처리

조용한 실패를 막는 게 목표. 각 항목을 점검한다.

- [ ] 실패 시 빈 결과 대신 명확한 에러 메시지를 반환한다
- [ ] 에러 메시지에 "무엇이 / 왜 / 다음에 무엇을" 이 들어있다
	- 나쁜 예: "입력 오류"
	- 좋은 예: "date는 YYYY-MM-DD 형식이어야 합니다"
- [ ] 외부 의존(API·DB) 실패와 입력 오류를 구분해 알린다
- 자주 나는 에러 Top 3와 대응 문구:
	1.
	2.
	3.

- [ ] 지난주 실패율이 평소보다 높은 도구가 있는가
