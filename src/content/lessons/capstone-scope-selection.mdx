## 결정이 제일 어렵다

Phase 1-11을 마쳤다. 이제 캡스톤을 만들 차례다.

```
후보 A: 회사 보고서 자동화 도구
후보 B: 개인 생산성 AI 앱
후보 C: 커뮤니티 Q&A 봇
```

셋 다 만들 수 있을 것 같다. 어떤 걸 골라야 하나?

잘못된 기준으로 고르면 중간에 멈추게 된다. "이것보다 저게 더 좋겠는데" 라는 생각에 바꾸기를 반복한다.

---

## 프로젝트를 중간에 멈추는 3가지 이유

1. **범위가 너무 넓다** — 4주 안에 끝낼 수 없는 걸 시작했다
2. **내가 실제로 안 쓴다** — 만들고 나서 본인도 안 쓰는 제품
3. **배울 게 없다** — 이미 다 아는 기술로만 만드는 것

이 세 가지를 사전에 확인하는 게 3축 매트릭스다.

---

## 3축 평가 매트릭스

각 후보를 1-5점으로 평가한다.

| 축 | 질문 | 1점 | 5점 |
|----|------|-----|-----|
| 실현 가능성 | 4주 안에 혼자 공개 가능한가? | 불가능 | 확실히 가능 |
| 임팩트 | 내가 또는 타인이 실제로 쓸 것인가? | 아무도 안 쓸 것 같다 | 당장 쓰고 싶다 |
| 학습 가치 | 아직 익히지 못한 것을 배우는가? | 아무것도 새로울 게 없다 | 핵심을 처음 배운다 |

**동점이면 임팩트 우선.** 실제로 쓰이는 게 가장 중요하다.

---

## 범위 압축: "이 기능이 없으면 MVP가 아닌가?"

선택된 프로젝트의 기능 목록을 쓴다.

```
초기 기능 목록:
- 문서 업로드 + 요약
- 대화형 Q&A
- 팀 공유
- 이메일 알림
- 모바일 앱
- 사용량 대시보드
```

각 기능에 대해 "이게 없으면 MVP가 아닌가?"를 묻는다.

```
압축 후:
✅ 문서 업로드 + 요약  ← 핵심
✅ 대화형 Q&A         ← 핵심
❌ 팀 공유            ← 2주차 이후
❌ 이메일 알림        ← 나중에
❌ 모바일 앱          ← v2에서
❌ 사용량 대시보드    ← 나중에
```

기능을 줄이는 게 아깝게 느껴지면 이렇게 생각한다:
"4주 안에 2개로 공개된 게, 6개 미완성보다 100배 낫다."

## 프로젝트 선언문 작성

한 문장으로 프로젝트 전체를 정의한다.

```
형식:
나는 [페르소나]가 [문제]를 해결하도록
[해결책]을 4주 안에 만들어 [공개 방법]으로 배포한다.
```

예시:

```
나는 1인 컨설턴트가 고객 문서에서 빠르게 답변을 찾도록
문서 Q&A 봇을 4주 안에 만들어 Vercel에 공개 배포한다.
```

선언문이 길면 범위가 좁혀지지 않은 것이다.

---

## AI에게 선언문 검토받기

```
내 캡스톤 선언문이야:
[선언문]

1. 이 범위가 4주 안에 혼자 완성 가능한가?
2. 범위를 더 좁혀야 한다면 어떻게 줄이겠어?
3. 만들다 멈추게 될 가장 큰 위험이 뭔지 말해줘.
```

---

## 핵심 원칙

> 선택의 기준은 "가장 좋은 아이디어"가 아니라 "4주 안에 공개할 수 있는 아이디어"다.
> 범위 압축은 포기가 아니라 집중이다.
