## "검색창에 '비용 절감'이라 치면 '경비 삭감'은 안 나오잖아요"

사내 위키에 문서가 500개 있어요. 검색창에 "비용 절감 방안"이라 칩니다. 딱 그 단어가 들어간 문서 3개만 나와요. 그런데 "경비 삭감 전략"이라는 제목의 핵심 보고서는 안 잡힙니다. 같은 뜻인데도요.

[긴 문서를 AI에게 먹이는 법](/lessons/feeding-long-documents)에서 "관련 청크만 골라 넣기(Selective Injection)"가 RAG로 가는 다리라고 했죠. 그 다리의 첫 발판이 오늘의 주제 — **임베딩(Embedding)**이에요. 텍스트를 "의미의 좌표"로 바꾸면, 단어가 달라도 뜻이 비슷한 문서가 가까이 모입니다.

오늘 만들 건 **내 도메인 문장 10개의 코사인 유사도 비교 노트**예요.

## 토큰 vs 임베딩 — 무엇이 다른가

{/* Source · DOC · OpenAI "Embeddings" — https://developers.openai.com/api/docs/guides/embeddings */}
{/* Source · DOC · Anthropic "Contextual Retrieval" — https://www.anthropic.com/news/contextual-retrieval */}

LLM을 쓰면서 "토큰"이라는 단어는 익숙해졌을 거예요. 토큰은 텍스트를 **잘게 쪼갠 조각**이에요 — "비용 절감"이 ["비용", " 절", "감"] 같은 식으로 나뉘죠. 문제는 토큰이 의미를 모른다는 거예요. "비용"과 "경비"가 비슷하다는 사실을 토큰만으로는 알 수 없습니다.

**임베딩**은 텍스트 전체를 **고차원 벡터(숫자 리스트)**로 바꿔요. "비용 절감 방안"이 `[0.12, -0.34, 0.78, ...]` 같은 1536개(또는 모델에 따라 다른 수) 숫자 리스트로 떠오르는 거예요. 핵심은 이겁니다.

> 의미가 비슷한 문장은 **벡터 공간에서 가까이** 놓인다.

"비용 절감 방안"과 "경비 삭감 전략"의 벡터는 서로 가깝고, "오늘 점심 뭐 먹지"의 벡터는 멀리 떨어져 있어요. 이게 키워드 검색이 못 잡는 걸 **의미 검색(semantic search)**이 잡는 원리입니다.

## 코사인 유사도 — 방향이 가까우면 의미가 가깝다

{/* Source · PAPER · Lewis et al. (2020) "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" — https://arxiv.org/abs/2005.11401 */}

두 벡터가 "가깝다"를 어떻게 재나요? 가장 흔한 방법이 **코사인 유사도(cosine similarity)**예요. 두 벡터의 **방향**이 얼마나 같은지를 -1에서 1 사이로 나타냅니다.

| 코사인 유사도 | 해석 |
| --- | --- |
| 0.9 이상 | 거의 같은 의미 |
| 0.7~0.9 | 관련 있는 주제 |
| 0.3~0.7 | 느슨한 관련 |
| 0.3 미만 | 거의 무관 |

(이 구간은 모델·도메인마다 다릅니다. 자기 데이터에서 직접 기준선을 측정하세요.)

수식으로 보면 두 벡터 A, B의 내적을 각각의 크기로 나눈 겁니다. 하지만 수식은 몰라도 돼요. 중요한 직관 한 줄:

> 벡터의 **길이(크기)**는 무시하고 **방향**만 본다.

짧은 문장이든 긴 문단이든, 의미의 방향이 같으면 유사도가 높게 나옵니다.

## 의미 검색이 키워드 검색을 이기는 곳 (그리고 안 이기는 곳)

| 상황 | 키워드 검색 | 의미 검색 (임베딩) |
| --- | --- | --- |
| 동의어 / 다른 표현 | 못 잡음 | 잡음 |
| 오타가 심한 쿼리 | 부분 매칭 의존 | 의미로 잡을 수 있음 |
| 정확한 코드명 / 고유명사 | 정확 | 오히려 비슷한 일반 단어에 끌림 |
| 숫자·날짜 기반 필터 | 필터 우수 | 의미적 유사도와 무관 |
| 불리언 조합 (AND/OR/NOT) | 잘 됨 | 지원 안 됨 (별도 필터 필요) |

핵심: 의미 검색이 **항상** 좋은 게 아니에요. "문서번호 A-2024-0317"을 찾을 때는 키워드 검색이 압도적입니다. 그래서 실무에서는 둘을 합친 **하이브리드 검색**을 씁니다 — 이건 [벡터 검색(Vector Search) 기본](/lessons/vector-search-basics)에서 다뤄요.

## 임베딩 모델 선택 — 무엇을 기준으로

{/* Source · DOC · OpenAI "Embeddings" — https://developers.openai.com/api/docs/guides/embeddings */}

임베딩을 만들려면 **임베딩 모델**이 필요합니다. LLM과는 다른 종류의 모델이에요 — 텍스트를 받아 벡터를 반환하는 전용 모델입니다.

| 기준 | 설명 |
| --- | --- |
| 차원 수 | 벡터의 길이. 1536, 3072 등. 클수록 표현력 ↑, 저장·검색 비용 ↑ |
| 다국어 지원 | 한국어 성능이 중요하면 다국어 모델 필수 |
| 비용 | API 호출 과금 (토큰당). 대량 임베딩 시 누적됨 |
| 최대 입력 길이 | 한 번에 임베딩할 수 있는 토큰 수 (보통 512~8192) |

OpenAI `text-embedding-3-small`(1536차원)이 가성비 시작점으로 많이 쓰여요. 한국어 특화가 필요하면 다국어 오픈소스 모델(예: multilingual-e5)도 선택지입니다.

(모델 사양은 분기마다 바뀝니다. 위 공식 문서를 다시 확인하세요.)

## 손으로 해 보기 — 10문장 코사인 비교

직관을 만드는 가장 빠른 방법은 직접 돌려보는 거예요. 아래는 OpenAI 임베딩 API로 문장 쌍의 유사도를 비교하는 개념 흐름입니다.

```
1. 내 도메인 문장 10개 준비
   - 같은 주제 5쌍 (예: "비용 절감 방안" ↔ "경비 삭감 전략")
   - 다른 주제 5쌍 (예: "비용 절감 방안" ↔ "팀 회식 장소 추천")

2. 각 문장을 임베딩 API에 보내 벡터 획득

3. 쌍별 코사인 유사도 계산

4. 결과를 표로 정리
   | 문장 A | 문장 B | 유사도 | 예상 | 실제 |
   |--------|--------|--------|------|------|
   | 비용 절감 방안 | 경비 삭감 전략 | 0.91 | 높음 | 높음 ✅ |
   | 비용 절감 방안 | 팀 회식 장소 | 0.23 | 낮음 | 낮음 ✅ |
```

예상과 실제가 어긋난 쌍에서 가장 많이 배워요. "이 두 문장은 비슷한데 왜 유사도가 낮지?" — 그 어긋남이 임베딩 모델의 한계를 한 번에 보여줍니다. 그 한계를 손으로 만져본 사람이 RAG 파이프라인을 더 잘 짜요.

## 임베딩의 함정 3가지

1. **임베딩은 "정확한 사실"을 모릅니다.** "서울의 인구는 1000만"과 "서울의 인구는 500만"은 벡터가 거의 같아요. 의미 구조가 같으니까. 사실 검증은 임베딩의 일이 아닙니다 — [환각(Hallucination)을 잡는 검증법](/lessons/hallucination-verification)의 환각 검증이 별도로 필요해요.

2. **도메인 전문 용어는 일반 모델이 약합니다.** 의학·법률·금융 약어가 일반 임베딩에서 엉뚱한 위치에 놓일 수 있어요. 이때는 도메인 특화 임베딩이나 파인튜닝을 검토합니다.

3. **한국어 토큰화 품질이 다릅니다.** 같은 모델이라도 한국어 문장의 임베딩 품질이 영어보다 낮을 수 있어요. 반드시 한국어 데이터로 직접 테스트하세요.

## 그래서 오늘 만들 한 장

오늘 만들 건 **임베딩 유사도 비교 노트** (`embedding-experiment.md`)예요.

| 항목 | 무엇 |
| --- | --- |
| 문장 | 내 도메인 문장 10개 (같은 주제 쌍 + 다른 주제 쌍) |
| 임베딩 | API 호출로 벡터 획득 |
| 비교 표 | 문장 A · 문장 B · 예상 유사도 · 실제 유사도 · 일치 여부 |
| 놀람 기록 | 예상과 실제가 달랐던 쌍 + 왜 다른지 추정 |
| 한 줄 결론 | 내 도메인에서 의미 검색이 키워드 검색을 이기는 지점 |

이 한 장을 손으로 한 번 만들면 다음 레슨([문서 청크 전략](/lessons/document-chunking-strategy))에서 "어떤 청크가 잘 검색되는가"를 가늠하는 눈이 자동으로 따라옵니다. 임베딩의 감을 한 번 잡은 사람과 안 잡은 사람의 RAG 품질이 갈려요.
