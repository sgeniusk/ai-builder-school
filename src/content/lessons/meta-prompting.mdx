## 도입 — 학습자에게 말 거는 한 장면

개발자가 아니어도 이런 장면은 익숙합니다. “이 문서 좀 보기 좋게 정리해줘.” “이번 회의록에서 할 일 뽑아줘.” “이 프로젝트를 다음 단계로 진행해줘.” 말은 맞지만 AI 입장에서는 빈칸이 많아요. 어디까지 봐야 하는지, 어떤 기준을 지켜야 하는지, 끝났다고 판단할 조건이 무엇인지 모릅니다.

코딩 예시로 보면 차이가 더 선명합니다. 사람이 막연히 쓴 프롬프트는 “Pydantic v1을 v2로 마이그레이션해줘. 테스트 통과시켜.” 정도예요. 메타 프롬프팅을 쓰면 먼저 AI에게 좋은 프롬프트를 만들게 합니다. 그러면 대상 파일, 바꿔야 할 규칙, 금지할 변경, 테스트 기준이 들어간 작업지시서가 나옵니다. 같은 일을 시키더라도 시작점이 달라지는 거예요.

오늘 배우는 메타 프롬프팅은 이름이 조금 낯설지만 원리는 쉽습니다. **프롬프트를 직접 쓰는 대신, 프롬프트를 만드는 프롬프트를 먼저 쓴다**는 뜻입니다.

## Lesson 2.4 다음에 왜 이 레슨인가

[Lesson 2.1](/lessons/structure-of-good-prompts) 에서 우리는 좋은 프롬프트를 네 축으로 나눴어요. Task, Context, Constraints, Output입니다. [Lesson 2.4](/lessons/build-personal-prompt-library) 에서는 그렇게 만든 프롬프트를 흩어지지 않게 라이브러리로 모았습니다.

이제 한 단계 더 올립니다. 4축을 사람이 매번 채우는 대신, AI에게 내 일을 보고 4축을 채우게 하는 거예요. 이것이 메타 프롬프팅입니다. 새 개념을 추가하는 것이 아니라, 이미 배운 4축을 자기 자신에게 한 번 더 적용하는 방식이에요.

이 레슨은 Phase 2의 마무리에 잘 맞습니다. 프롬프트를 잘 쓰는 법, 출력 형식을 고정하는 법, 긴 문서를 넣는 법, 라이브러리로 관리하는 법을 배웠습니다. 이제 프롬프트 자체를 자산으로 만드는 마지막 조각을 더합니다. 이렇게 만든 정밀 프롬프트는 [Lesson 3.1](/lessons/automate-report-drafts) 같은 Phase 3 자동화 파이프라인의 재료가 됩니다.

## 핵심 개념 — Goal, Context, Constraints, Done When

메타 프롬프트의 네 칸은 Lesson 2.1의 네 축과 1대1로 이어집니다.

| Lesson 2.1 | 메타 프롬프팅 | 질문 |
| --- | --- | --- |
| Task | Goal | 무엇을 달성해야 하나요? |
| Context | Context | 어떤 배경과 자료를 봐야 하나요? |
| Constraints | Constraints | 무엇을 지키고 피해야 하나요? |
| Output | Done When | 무엇이 나오고 어떻게 확인하면 끝인가요? |

여기서 핵심은 Done When입니다. Output이 “마크다운 표로 출력”이라면 Done When은 “표가 나오고, 빠진 필드가 없고, 실제 검증 기준을 통과하면 완료”까지 포함합니다. AI가 일을 끝냈다고 말하는 기준과 사람이 완료로 인정하는 기준이 같아져야 합니다.

메타 프롬프트는 이렇게 시작할 수 있어요.

```md
내가 아래 작업을 AI에게 맡기려고 합니다.
이 작업을 더 잘 시키기 위한 정밀 프롬프트를 작성해 주세요.

Goal, Context, Constraints, Done When 네 섹션으로 나눠 주세요.
내 역할과 현재 자료를 보고 빠진 정보가 있으면 먼저 질문해 주세요.

[작업 설명]
```

이 한 번의 요청이 사람이 오래 고민할 프롬프트 초안을 빠르게 만들어 줍니다. 물론 그대로 복사하면 안 됩니다. AI가 만든 프롬프트도 내 업무 맥락에 맞는지 확인하고, 과한 제약이나 빠진 기준을 고쳐야 해요. [Lesson 1.1](/lessons/common-skills-of-future-proof-people) 의 판단 체크리스트가 여기서 다시 필요합니다.

## 그래서 오늘 할 일

오늘 만들 것은 `meta-prompt-template.md`입니다. 먼저 내 일에서 반복되는 작업 후보 3개를 고릅니다. 보고서 초안, 회의록 정리, 고객 답변, 블로그 글, 코드 리뷰, 채용 공고 정리처럼 자주 돌아오는 일이 좋습니다.

각 작업에 대해 메타 프롬프트를 실행합니다. “이 작업을 더 잘 시키기 위한 정밀 프롬프트를 만들어 달라”고 요청하고, Goal, Context, Constraints, Done When 네 칸으로 받습니다. 결과가 나오면 바로 저장하지 말고 세 가지를 확인하세요. 목표가 너무 넓지 않은지, 맥락이 내 실제 자료와 맞는지, 완료 기준이 검증 가능한지입니다.

그다음 [Lesson 2.4](/lessons/build-personal-prompt-library) 에서 만든 라이브러리에 저장 위치를 정합니다. 예를 들어 `prompt-library/reports/weekly-status.md`, `prompt-library/reviews/code-review.md`, `prompt-library/_meta/write-better-prompt.md`처럼요. 메타 프롬프트 자체도 자산입니다. 한 번 잘 만들어두면 다음 프롬프트를 더 빨리 만들 수 있어요.

마지막으로 직접 쓴 프롬프트와 AI가 만든 프롬프트를 비교합니다. AI가 더 구체화한 부분은 무엇인지, 오히려 복잡하게 만든 부분은 무엇인지 한 줄씩 적습니다. 이 비교가 다음 버전의 라이브러리를 더 좋게 만듭니다.

## 마무리

메타 프롬프팅은 “AI가 다 알아서 해준다”는 뜻이 아닙니다. 내가 하던 프롬프트 설계의 초안을 AI에게 맡기고, 사람은 더 좋은 기준을 고르는 쪽으로 이동한다는 뜻이에요. Phase 2에서 배운 4축은 이제 내가 쓰는 도구이면서, AI에게 프롬프트를 만들게 하는 질문지가 됩니다.

나중에 [Lesson 4.7](/lessons/nightly-loop-pattern) 에서 세션 밖 자동 루프를 설계할 때도 이 방식이 다시 나옵니다. 자동 루프가 잘 도는지는 처음 목표 문장이 얼마나 분명한지에 크게 좌우돼요. 오늘 만든 메타 프롬프트 템플릿은 Phase 3의 업무 자동화와 Phase 4의 코딩 에이전트 루프를 이어주는 다리입니다.
