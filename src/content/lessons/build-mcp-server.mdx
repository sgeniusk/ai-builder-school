## 도입 — 도구는 만들었는데, 다른 곳에서는 안 통해요

[지난 레슨](/lessons/prompt-injection-defense)에서 당신은 도구를 호출하는 AI를 외부 입력으로부터 지키는 법을 익혔어요. 그런데 그 도구들 — Function Calling으로 묶어둔 함수들 — 을 다시 보면 한 가지가 걸립니다. 그 도구는 당신이 만든 그 앱 안에서만 살아 있어요.

같은 "사내 위키 검색" 기능을 Claude Desktop에서도, 동료가 쓰는 다른 AI 클라이언트에서도 쓰고 싶어요. 그런데 클라이언트가 바뀔 때마다 연결 코드를 새로 짭니다. 도구는 하나인데 꽂는 방식이 제각각이에요. 표준 플러그가 없는 거죠.

## 왜 이 레슨인가

Function Calling은 강력하지만 한 앱 안에 갇힌 도구예요. 도구를 여러 AI에서 재사용하려면, 그 도구를 앱 바깥으로 떼어내 **표준 인터페이스**로 노출해야 합니다. 그게 MCP(Model Context Protocol)예요.

오늘 만들 건 거창한 서버가 아니라, 당신의 도구 하나를 MCP 서버로 떼어내는 설계 한 장입니다. 한 번 표준에 맞춰두면, MCP를 지원하는 어떤 클라이언트든 같은 방식으로 꽂아 씁니다.

## 핵심 개념 — 빠르게 짚고 갑니다

### MCP는 도구의 USB-C 표준이에요
예전엔 기기마다 충전 단자가 달랐죠. USB-C가 그걸 하나로 묶었어요. MCP도 똑같습니다. 도구를 MCP 서버로 노출하면, AI 클라이언트는 "MCP를 안다"는 사실만으로 그 도구를 꽂아 쓸 수 있어요. 클라이언트마다 연결 코드를 새로 짜지 않아도 됩니다.

### MCP 서버는 Tools와 Resources를 노출해요
**Tools**는 AI가 실행하는 동작이에요 — 메일 보내기, 검색 실행, 파일 쓰기처럼 무언가를 "한다". **Resources**는 AI가 읽는 데이터예요 — 문서, 설정값, 로그처럼 무언가를 "가져온다". 같은 도구라도 "동작"인지 "데이터"인지 구분해서 노출하면, 클라이언트가 더 정확하게 활용합니다.

### 전송 방식은 stdio와 HTTP 두 갈래예요
서버와 클라이언트가 대화하는 통로를 전송(transport)이라 불러요. **stdio**는 같은 컴퓨터 안에서 프로세스로 주고받는 로컬 방식이고, **HTTP**는 네트워크 너머 원격 서버에 붙는 방식입니다. 당신 노트북의 도구를 노출하는 거라면 stdio로 시작하세요. 가장 단순하고, 설정할 게 거의 없어요.

```json
{ "command": "node", "args": ["my-mcp-server.js"] }
```

이렇게 클라이언트 설정에 서버 실행 명령만 적으면 stdio 연결이 끝납니다.

## 마무리

MCP 서버 하나를 만들어두면, 그 도구는 더 이상 한 앱에 갇히지 않아요. 이번 회고에서 스스로 물어보세요 — 내가 만든 도구 중 "여러 AI에서 다시 쓰고 싶은데 매번 새로 연결하던" 것은 무엇이었나요. 그 도구가 바로 첫 MCP 서버 후보예요.
