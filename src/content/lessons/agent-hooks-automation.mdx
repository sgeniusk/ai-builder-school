## 도입 — "다음엔 꼭 돌릴게요"

[지난 레슨](/lessons/claude-md-four-principles)에서 당신은 CLAUDE.md에 "커밋 전 테스트를 돌릴 것"이라고 적었어요. 에이전트도 사람도 읽으면 고개를 끄덕입니다. 그런데 바쁜 날, 작은 수정 하나, "이건 그냥 오타니까"라는 생각 — 그렇게 한 번 건너뛴 커밋이 빌드를 깹니다.

당신이 게으른 게 아니에요. 규약은 "기억해서 지키는 것"이고, 기억은 바쁠 때 가장 먼저 흔들립니다. 문제는 의지가 아니라 구조예요. 규약에 적은 검증은 지키면 좋고 안 지켜도 아무도 안 막아요. 그게 함정입니다.

## 왜 이 레슨인가

CLAUDE.md에 적는 건 출발선이에요. 진짜 안전장치는 "안 지키면 진행이 막히는" 구조입니다. 검증을 기억에 맡기면 언젠가 빠지고, 구조에 박으면 빠질 수가 없어요.

오늘 만들 건 hook입니다. 작업 전후·커밋 전 같은 정해진 시점에 lint·typecheck·test가 자동으로 돌고, 실패하면 그 자리에서 진행을 막는 차단형(blocking) 게이트예요.

## 핵심 개념 — 빠르게 짚고 갑니다

### 규약은 기억, hook은 구조
CLAUDE.md의 "커밋 전 테스트"는 읽는 사람이 떠올려야 작동해요. hook은 떠올릴 필요가 없습니다. 정해진 시점이 되면 harness가 알아서 실행하고, 당신이 깜빡해도 구조가 대신 기억해요.

### 차단형 hook이 핵심이다
hook에는 알려만 주는 것과 막는 것이 있어요. 경고만 띄우는 hook은 무시당하면 그만입니다. **차단형 hook**은 검증이 실패하면 종료 코드를 0이 아닌 값으로 돌려서 커밋·작업 자체를 멈춰요. 막혀야 비로소 게이트입니다.

```bash
# 커밋 전 차단형 게이트 (실패 시 exit 1 → 커밋 중단)
npm run lint && npm run typecheck && npm test || exit 1
```

### 실패 출력은 에이전트의 다음 입력
hook이 막으면 거기서 끝이 아니에요. 막힌 이유 — lint 에러, 타입 불일치, 깨진 테스트 — 그 출력을 그대로 에이전트에게 돌려주면, 에이전트는 그걸 읽고 스스로 고칩니다. 게이트는 벌이 아니라 피드백 루프의 일부예요.

## 그래서 오늘 할 일

오늘 만들 한 장은 `hooks-setup.md`입니다. 작업 전후와 커밋 전 어느 시점에 무엇을 검증할지 정하고, 그걸 차단형 hook으로 걸고, hook이 막았을 때 출력을 에이전트에 돌려 자가 수정하게 하는 루프까지 한 장에 적는 거예요.

[다음 레슨](/lessons/subagent-context-design)에서 다룰 서브에이전트 설계도 같은 원리 위에 섭니다 — 각 에이전트가 자기 게이트를 통과해야만 다음으로 넘어가도록, 검증을 구조에 박아두는 것이죠.

## 마무리

hook은 한 번 걸어두면 당신이 잊어도 평생 작동합니다. 이번 회고에서 스스로 물어보세요 — 지금 내 프로젝트에서 "기억에 기대고 있는" 검증은 무엇인가요. 그 하나를 hook으로 옮겼다면, 당신은 검증을 의지에서 구조로 넘긴 거예요.
