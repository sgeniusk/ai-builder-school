## 같은 부탁을 두 번 했는데, 결과가 완전히 달랐어요

월요일 아침에 "지난 주 회의록 요약해줘" 부탁했더니 깔끔한 3줄 요약이 나왔어요. 좋아서 이번 주에 같은 부탁을 했죠. 이번엔 6 문단짜리 분석문이 나옵니다. 회의록 내용은 별 차이가 없는데요.

문제는 모델이 아니라 **프롬프트가 모호했다는 것**이에요. "회의록 요약" 한 줄에는 *"누가 읽나"*, *"몇 줄로"*, *"어떤 톤으로"*, *"무슨 형식으로"* 가 다 빠져 있어서 모델이 매번 재량껏 메우게 됩니다. 재량의 폭이 클수록 결과 분산도 커지죠.

오늘 만들 건 **재현 가능한 4축 프롬프트 템플릿**입니다. 같은 입력에 같은 톤으로 답이 오게 하는 가장 작은 단위.

## Phase 1 다음에 왜 이 레슨인가

[Lesson 1.4](/lessons/checks-before-trusting-ai-output) 에서 답을 *받은 뒤* 의심하는 5축 체크를 만들었어요. 그건 *방어*입니다. 이번엔 한 발 앞 — 받기 전부터 답이 흔들리지 않게 만드는 *공격* 도구를 짜요.

방어 + 공격이 합쳐지면, 이런 사이클이 생깁니다.

> 좋은 프롬프트 → 안정된 답 → 5분 트러스트 체크 통과 → 그대로 발송

이 사이클이 자리 잡으면 매일 *AI 검수 스트레스* 가 절반으로 줄어요.

## 핵심 개념 — Task / Context / Constraints / Output 4축

좋은 프롬프트는 항상 네 가지를 분리해 명시합니다. 이걸 머릿속에서 흩뿌려 두면 매번 결과가 달라져요. 정해진 자리에 박아두는 게 핵심.

### Task — *무엇을 해 달라는가*

AI 가 만들 *최종 결과물*. 동사 + 목적어 한 문장으로.

> "지난 주 회의록을 요약하라" (X — 너무 일반적)
>
> "지난 주 마케팅 회의록에서 결정된 Action Item 만 추출해, 담당자·기한과 함께 정리하라" (O)

### Context — *AI 가 알아야 할 배경*

누가 읽는지, 어떤 도메인인지, 무엇을 이미 알고 있는지. 빠지면 모델이 "일반 직장인 평균" 으로 가정해버려요.

> "독자: 마케팅 디렉터. 시간 부족, 의사결정 직전. 회사는 B2B SaaS, 회의 빈도 주 2회."

### Constraints — *지켜야 할 제약*

길이, 톤, 금지 표현, 허용 표현. 모델은 친절해서 *덤* 으로 더 쓰는 경향이 있어요. 명시 안 하면 항상 길어집니다.

> "5줄 이내. 마크다운 불릿. 추측이나 의견 금지 — 회의에서 *명시* 된 것만."

### Output — *결과물의 형식·예시*

표인지, JSON 인지, 마크다운인지. 가능하면 **예시 한 줄을 직접 보여주는 것** 이 가장 확실합니다.

````
| 담당자 | Action | 기한 |
| --- | --- | --- |
| 김민지 | 신규 캠페인 카피 v2 | 6/20 |
````

이 예시 하나가 "톤·구조·세부 형식" 세 가지를 한 번에 못 박아요. AI 는 예시를 거의 안 어깁니다.

## 그래서 오늘 할 일

매일 호출되는 작업 3개를 골라 4축 템플릿으로 만들어요. 권장 후보,

- 매주 쓰는 보고서 / 회의록
- Slack 회신 / 클라이언트 메일
- 코드 리뷰 / 버그 리포트 정리

권장 작업 흐름,

1. **Before** 칸에 지금 쓰는 모호한 한 줄 프롬프트를 그대로 적습니다 (5분).
2. 4 축 표를 만들고, 각 칸에 위 예시처럼 구체로 채웁니다 (15분).
3. 같은 입력 데이터로 *Before* 와 *After* 를 두 번씩 실행해 결과 분산을 비교합니다 (10분).
4. After 결과가 일관되면 `prompt-templates/<task>.md` 로 저장하고, Before 는 휴지통으로 (5분).
5. 같은 패턴으로 3개 작업을 반복.

산출물 템플릿은 [`four-axis-prompt.md`](/lessons/structure-of-good-prompts) — 4축 칸 + Before/After + 결과 비교 슬롯이 한 페이지 안에 들어 있어요.

## 마무리 — 입력의 무게가 출력의 신뢰를 결정합니다

오늘 4축이 익으면, 다음 [Lesson 2.2](/lessons/enforcing-output-format) 에서는 *Output* 축을 더 강하게 — JSON 스키마로 강제하는 법을 배웁니다. 자동화 파이프라인의 시작이에요.

매일 같은 자리에서 같은 4축으로 부르는 게 핵심. 처음 한 주 동안은 표를 들고 일하는 느낌이지만, 한 달이 지나면 머리가 자동으로 4축으로 정리합니다. 그때부터 *프롬프트 직관* 이 생긴 거예요.
