## "어제는 잘 되더니 오늘은 왜?"

같은 질문을 어제는 멋지게 답해주던 AI가 오늘 아침엔 엉뚱한 답을 줍니다. 이상해서 동료한테 같은 질문을 시켜봤어요. 또 다른 답이 나옵니다. 이 즈음 사람들이 농담처럼 말하죠 — "AI가 점점 멍청해지는 거 아니야?"

사실은 우리 **프롬프트가 매번 달랐던 것**이 진짜 원인이에요. AI 입장에서는 부탁받을 때마다 다른 사람이 다른 톤으로, 다른 맥락에서, 다른 길이로 시키는 셈입니다. 그러니 답도 매번 다르게 나오죠.

오늘 만들 한 장은 **그 변동성을 줄이는 4축 구조**예요. 부탁이 아니라 명세처럼 쓰는 법.

## 프롬프트 = 부탁 → 명세

{/* Source · DOC · OpenAI Academy "Prompting fundamentals" — https://openai.com/academy/prompting/ */}
{/* Source · DOC · Anthropic "Prompt engineering overview" — https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/overview */}

[Lesson 04](/lessons/checks-before-trusting-ai-output)에서 만든 5분 트러스트 체크가 "받은 답을 검증하는" 도구였다면, 오늘부터는 **답이 처음부터 잘 오게 만드는** 입력 쪽으로 옮겨갑니다. 검증할 게 줄어들면 5분 체크도 덜 빡빡해지니까요.

핵심 한 줄:

> 프롬프트는 부탁이 아니라 **명세(Spec)**다.

부탁은 듣는 사람의 해석에 맡기는 일이에요. 명세는 입력·출력·제약을 다 적어둔 계약서입니다. 같은 사람한테 부탁 100번 하면 100가지 다른 결과가 나오지만, 같은 명세로 호출하면 답이 훨씬 좁은 폭 안에 모입니다. 이게 재현 가능성(reproducibility)에 가까워지는 길이에요. (LLM은 본질적으로 확률적이라 완전한 같음은 보장되지 않아요. `temperature`를 낮추면 더 결정적으로 가까워집니다.)

## 4축 — Task · Context · Constraints · Output

명세를 4칸으로 분리합니다. 이 네 칸만 의식하면 80%의 흔들림이 사라져요.

### 1. Task (작업) — 무엇을

추상명사 대신 동사로 적어요.

- ❌ "이 데이터 분석해 줘"
- ✅ "이 데이터에서 강점 3가지·약점 3가지를 표로 정리해 줘"

차이가 보이시죠? 첫 번째는 "분석"이라는 단어 안에 무한 가지 가능성이 들어있어요. 두 번째는 결과의 형태가 이미 정해져 있습니다.

### 2. Context (맥락) — 왜·누구에게·어디에

AI가 같은 작업을 누구의 입장에서, 어디에 쓸지 알아야 답이 맞춰집니다. 빼놓으면 모델이 가장 흔한 평균치 답으로 회귀해요.

- ❌ "보고서 써 줘"
- ✅ "B2B SaaS 영업 매니저 입장에서, 3분기 실적이 목표 대비 -8%인 상황을 임원진 4명에게 보고하는 보고서를 써 줘. 임원진은 숫자보다 시나리오와 액션 아이템을 본다."

길어 보이지만 이게 답을 두 단계 위로 끌어올립니다.

### 3. Constraints (제약) — 톤·길이·금지

가드레일을 명시합니다.

- 톤: "전문가 톤" / "친근한 톤" / "내부 메모 톤"
- 길이: "300자 이내" / "5문단 이내"
- 금지: "이모지 금지" / "이름 없는 인용 금지" / "추정 금지 — 모르면 모른다고"
- 허용: "한 줄짜리 농담 1개 허용"

명시 안 하면 모델이 자기 평균치로 채워요. 예를 들어 길이를 안 정하면 어떤 모델은 짧게, 어떤 모델은 길게 줍니다.

### 4. Output (출력) — 형식·예시 ← 가장 강력

{/* Source · PAPER · Brown et al. (2020) "Language Models are Few-Shot Learners" (GPT-3) — https://arxiv.org/abs/2005.14165 · 예시 기반 in-context learning */}
{/* Source · DOC · OpenAI "Structured model outputs" — https://developers.openai.com/api/docs/guides/structured-outputs */}
{/* Source · DOC · Anthropic "Structured outputs" — https://platform.claude.com/docs/en/build-with-claude/structured-outputs */}

이게 4축 중 **가장 영향력 있는 축**이에요. 한 줄로 답의 모양을 보여주면, 모델은 그 모양을 거의 그대로 따라옵니다.

```
Output:
- 형식: Markdown 표
- 예시:
  | 항목 | 강점 | 약점 |
  | --- | --- | --- |
  | A 제품 | 빠른 출시 속도 | 마진 낮음 |
  | B 제품 | 고객 retention | 가격 경쟁력 약함 |
```

{/* Note: "예시 한 줄이 길어요" 같은 정량 비교는 외부 검증 자료 없음. in-context learning 일반 효과는 Brown 2020 참조. */}

표를 글자로 그려 한 줄만 넣어도, 답이 흔들리지 않아요. JSON·체크리스트·시퀀스 다이어그램 — 어느 형식이든 **예시 한 줄**이 길게 풀어쓴 설명보다 결과를 단단하게 만듭니다.

## 그래서 오늘 할 일

이걸 다 외울 필요는 없어요. 오늘 만들 건 한 장의 라이브러리예요 — `prompt-library-v1.md`. 거기에 **내가 매주 가장 자주 쓰는 프롬프트 3개**를 4축 구조로 다시 씁니다.

| 단계 | 무엇 |
| --- | --- |
| Before | 평소 쓰던 모호한 프롬프트 원문 그대로 |
| After | Task / Context / Constraints / Output 4섹션으로 다시 |
| 비교 | 같은 질문을 2번 호출 — 답이 얼마나 일치하나 1~5점 |
| 효과 | 어느 축을 명시했을 때 가장 크게 달라졌나 한 줄 |

이 라이브러리가 한 장 생기면 다음부터는 매번 새로 짜지 않습니다. **호출하면 됩니다.**

## 이 한 장이 가져다 주는 것

[Lesson 02](/lessons/what-llms-are-good-and-bad-at)의 강·약 매핑 지도가 "AI에 무엇을 맡길지" 정해줬다면, 이 라이브러리는 "어떻게 맡길지" 정해줍니다. 둘이 합쳐지면 매주 받는 답의 품질이 올라가요. 검증할 게 줄어들고 (Lesson 04 체크가 빠르게 끝나고), 같은 작업 반복할 때 처음부터 시작 안 하게 됩니다.

다음 레슨([출력 형식 강제](/lessons/enforcing-output-format))에서는 4축의 마지막 축인 Output을 더 깊이 파요 — 자동화 파이프라인을 만들 수 있는 수준의 JSON 스키마 강제까지. 그 전에 4축의 기본 골격을 먼저 손에 쥐는 게 오늘입니다.
