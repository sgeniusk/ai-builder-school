## 도입 — 학습자에게 말 거는 한 장면

챗 앱이 제법 잘 돌아갑니다. API도 안전하게 붙였고, 응답은 스트리밍으로 흐르고, 추천 결과는 스키마 검증까지 통과합니다. 그런데 새로고침을 누르는 순간 모든 대화가 사라져요. 어제 사용자가 어떤 질문을 했는지, 어떤 답이 도움이 됐는지, 어떤 결과가 실패했는지 남아 있지 않습니다.

처음에는 “대화 기록은 나중에 붙이면 되겠지”라고 생각하기 쉽습니다. 하지만 저장 구조는 늦게 붙일수록 더 어려워져요. 메시지 원문, 요약, 모델 이름, 토큰 비용, 오류 로그, 개인정보 마스킹 여부가 뒤섞이면 나중에 정리하기가 힘듭니다. 대화가 길어질수록 전체 원문을 매번 모델에 보내는 비용도 커집니다.

오늘은 작은 AI 앱의 기억 구조를 설계합니다. 거창한 데이터베이스 이론이 아니라, 세션과 메시지와 요약을 어떻게 나눠 저장할지 정하는 일입니다.

## Lesson 5.3 다음에 왜 이 레슨인가

Phase 5의 앞 세 레슨은 하나의 호출 흐름을 만들었습니다. [Lesson 5.1](/lessons/connect-ai-api) 에서 안전한 서버 라우트를 만들었고, [Lesson 5.2](/lessons/streaming-response-ui) 에서 응답을 흐르게 했고, [Lesson 5.3](/lessons/structured-output-handling) 에서 결과 모양을 검증했어요. 이제 그 흐름을 다음 턴으로 넘겨야 합니다.

대화 저장은 단순히 “채팅 로그를 남긴다”가 아닙니다. 앱 안에 루프를 박는 일입니다. 사용자의 새 메시지는 이전 맥락을 참고하고, 모델의 답은 저장되고, 오래된 내용은 요약되고, 다음 요청은 그 요약을 다시 씁니다. 이 구조가 있어야 AI 앱이 매번 처음 만나는 도구가 아니라, 사용자의 맥락을 이어받는 제품이 됩니다.

다음 Phase 6에서는 [Lesson 6.1](/lessons/understanding-embeddings) 로 넘어가 문서와 대화를 검색 가능한 형태로 바꿉니다. 그 전에 무엇을 저장하고 무엇을 버릴지 정해야 해요. 저장을 잘못하면 RAG도 흐려집니다.

## 핵심 개념 — Session, Message, Summary

Session은 사용자가 한 번의 작업 맥락에서 이어 가는 대화 묶음입니다. 예를 들어 “랜딩페이지 카피 만들기”라는 세션이 있고, 그 안에 여러 메시지가 들어갑니다. Thread라는 말을 쓰는 도구도 많아요. 이름보다 중요한 건 “어떤 대화들이 같은 목표에 속하는가”입니다.

Message는 실제 발화입니다. 사용자가 보낸 말, AI가 답한 말, 시스템이 넣은 지시를 구분해 저장합니다. 각 메시지에는 역할, 내용, 생성 시간, 모델 이름, 토큰 사용량, 오류 여부 같은 메타데이터가 붙을 수 있어요. 모든 것을 저장하면 추적은 쉬워지지만 개인정보와 비용 리스크가 커집니다. 그래서 저장 전 마스킹 규칙을 정해야 합니다.

Summary Rolling은 오래된 메시지를 짧은 요약으로 압축하는 방식입니다. 대화 50턴 전체를 매번 모델에 보내면 비용도 크고 속도도 느립니다. 대신 최근 몇 턴은 원문으로 두고, 오래된 부분은 “사용자는 B2B SaaS 창업자이며, 가격 페이지 카피를 만들고 있다”처럼 요약해 넘깁니다. 요약은 편하지만 손실이 있습니다. 그래서 원문 보존 기간, 요약 주기, 요약 품질 점검 기준이 필요합니다.

Trace는 개발자를 위한 발자국입니다. 어떤 프롬프트와 어떤 모델이 어떤 결과를 만들었는지 남기는 기록이에요. 제품 분석과 디버깅에는 유용하지만, 사용자가 볼 필요는 없습니다. 사용자 경험용 대화 기록과 개발자용 trace를 섞지 않는 것이 좋습니다.

## 그래서 오늘 할 일

오늘 만들 것은 “대화 저장 스키마 + API” 설계입니다. 실제 DB가 SQLite든 Postgres든 Supabase든 상관없어요. 먼저 테이블의 역할을 정합니다. `sessions`는 대화 묶음, `messages`는 각 발화, `summaries`는 압축된 맥락, 필요하면 `traces`는 개발자 로그를 맡습니다.

그다음 저장 흐름을 씁니다. 사용자가 메시지를 보내면 먼저 `messages`에 저장합니다. 서버 라우트가 모델을 호출하고, 응답이 끝나면 assistant 메시지를 저장합니다. 일정 턴 수를 넘으면 오래된 메시지를 요약해 `summaries`에 저장하고, 다음 호출에는 최근 메시지와 최신 요약만 넣습니다. 실패한 호출도 완전히 버리지 말고, 오류 상태와 requestId 정도는 남겨야 디버깅할 수 있어요.

산출물 [`conversation-storage-design.md`](/lessons/conversation-storage-design) 에는 테이블 초안, 저장 흐름, 개인정보 마스킹 규칙, 요약 주기, 비용 점검 표를 적습니다. 처음부터 완벽한 데이터 모델을 만들 필요는 없지만, 나중에 제품으로 키울 생각이라면 삭제 정책과 내보내기 정책도 한 줄씩 남기세요.

## 마무리

대화 저장은 AI 앱의 기억 장치입니다. 기억이 없으면 매 요청이 새 출발이고, 기억이 너무 많으면 비용과 개인정보 위험이 커집니다. 좋은 설계는 그 사이에서 필요한 맥락만 남깁니다. 이 레슨으로 Phase 5의 작은 앱 사이클이 닫힙니다. API 연결, 스트리밍 UI, 구조화 출력, 대화 저장이 이어지면 “한 번 답하는 데모”가 “계속 쓸 수 있는 앱”으로 바뀝니다. 다음 Phase 6에서는 이 저장된 맥락과 외부 문서를 검색으로 연결해 RAG의 세계로 들어갑니다.
