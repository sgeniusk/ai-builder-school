## "어제 대화를 이어하려는데 다 날아갔어요"

미니 채팅 앱을 만들었습니다. [AI API 붙이기](/lessons/connect-ai-api)의 API 연결, [스트리밍 응답 UI 만들기](/lessons/streaming-response-ui)의 스트리밍 UI, [구조화 출력(Structured Output) 다루기](/lessons/structured-output-handling)의 구조화 출력까지 다 붙었어요. 잘 됩니다 — **브라우저를 닫기 전까지는요.** 다음 날 아침에 열면 대화가 전부 사라져 있습니다. "어제 그 레시피 다시 보여줘"가 안 돼요.

메모리에만 있는 대화는 휘발됩니다. 오늘은 대화를 **영속(Persist)**시키는 저장소를 설계해요.

## 대화의 구조 — Session · Message · Summary

{/* Source · DOC · OpenAI "Chat Completions API" — https://developers.openai.com/api/docs/guides/text-generation · messages 배열 구조 */}
{/* Source · DOC · Anthropic "Messages API" — https://docs.claude.com/en/api/messages · role/content 메시지 형식 */}

AI 대화의 기본 단위는 **메시지(Message)** 예요. 그리고 메시지의 묶음이 **세션(Session)** 입니다.

```
Session (대화방)
├── Message 1 (user: "레시피 추천해 줘")
├── Message 2 (assistant: "파스타는 어떠세요?")
├── Message 3 (user: "재료는?")
├── Message 4 (assistant: "올리브 오일, 마늘...")
└── ...
```

OpenAI와 Anthropic 모두 `messages` 배열을 받습니다. 각 메시지는 `role`(user/assistant/system)과 `content`로 이루어져요. 이 구조를 그대로 DB에 저장하면 됩니다.

하지만 대화가 길어지면 **두 가지 문제**가 생겨요.

## 긴 대화의 두 가지 적 — 비용과 품질

{/* Source · DOC · OpenAI "Rate limits" — https://developers.openai.com/api/docs/guides/rate-limits */}

### 1. 비용 폭주

매 호출마다 **이전 대화 전체**를 보내야 합니다. 100턴 대화면 100턴 분량의 토큰을 매번 보내는 거예요. 입력 토큰도 과금됩니다.

```
10턴:  약 2K 토큰 × $0.003 = $0.006/호출
50턴:  약 10K 토큰 × $0.003 = $0.03/호출
200턴: 약 40K 토큰 × $0.003 = $0.12/호출
```

(가격은 모델·벤더마다 다릅니다. 핵심은 **턴이 늘수록 비용이 선형 증가**한다는 구조예요.)

### 2. 품질 저하

[긴 문서를 AI에게 먹이는 법](/lessons/feeding-long-documents)의 "Lost in the Middle" 기억하시죠? 대화가 길어지면 **초반 맥락을 잊어버립니다**. 사용자가 10턴 전에 한 말을 모델이 못 잡는 순간, "얘가 내 말 듣고는 있는 거야?" 체감이 뚝 떨어져요.

해법이 **Summary Rolling** 이에요.

## Summary Rolling — 요약으로 압축하기

{/* Note: Summary Rolling은 LangChain의 ConversationSummaryBufferMemory 등에서 비슷한 개념으로 구현됩니다. 명칭은 이 학교 운영자의 정리입니다. */}

오래된 메시지를 **요약으로 압축**하고, 최근 N턴의 원문만 유지합니다.

```
[요약: 사용자가 파스타 레시피를 요청, 재료 목록 전달, 조리 시간 확인]
├── Message 47 (user: "소금은 얼마나?")
├── Message 48 (assistant: "파스타 물 1L당 소금 10g...")
├── Message 49 (user: "타이머 맞춰 줘")
└── Message 50 (assistant: "8분 타이머 시작...")
```

이러면 매 호출에 보내는 토큰은 **요약 + 최근 N턴**으로 일정하게 유지됩니다. 50턴이든 500턴이든 비용이 비슷해요.

| 설정 | 추천 시작점 (실험용 — 본인 도메인에서 직접 조정) |
| --- | --- |
| 최근 N턴 원문 유지 | 예: 10~20턴 |
| 요약 주기 | 예: 10턴마다 요약 갱신 |
| 요약 모델 | 메인 모델보다 작은 모델 (비용 절감) |

## 스토리지 선택 — SQLite · Postgres · Redis

{/* Source · DOC · SQLite "About" — https://www.sqlite.org/about.html */}
{/* Source · DOC · PostgreSQL "About" — https://www.postgresql.org/about/ */}

대화를 어디에 저장할까요? 세 선택지를 비교합니다.

| 스토리지 | 장점 | 단점 | 적합 |
| --- | --- | --- | --- |
| **SQLite** | 설정 0 · 파일 1개 · 빠름 | 동시 쓰기 약함 · 스케일 한계 | 프로토타입 · 개인 도구 |
| **Postgres** | 트랜잭션 · JSON 지원 · 풀텍스트 검색 | 서버 운영 필요 | 프로덕션 · 팀 서비스 |
| **Redis** | 초고속 읽기/쓰기 | 영속성이 기본은 아님 · 비용 | 세션 캐시 · TTL 기반 |

MVP에서는 **SQLite로 시작하고, 사용자가 늘면 Postgres로 마이그레이션**하는 경로가 가장 현실적이에요. Redis는 "최근 대화 캐시" 레이어로 Postgres 앞에 두는 패턴이 일반적입니다.

## 테이블 설계 — 세 테이블이면 충분

```sql
-- 1. sessions: 대화방
CREATE TABLE sessions (
  id          TEXT PRIMARY KEY,
  user_id     TEXT NOT NULL,
  title       TEXT DEFAULT '새 대화',
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. messages: 개별 메시지
CREATE TABLE messages (
  id          TEXT PRIMARY KEY,
  session_id  TEXT REFERENCES sessions(id),
  role        TEXT CHECK(role IN ('system', 'user', 'assistant')),
  content     TEXT NOT NULL,
  token_count INTEGER,
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 3. summaries: 요약 스냅샷
CREATE TABLE summaries (
  id          TEXT PRIMARY KEY,
  session_id  TEXT REFERENCES sessions(id),
  content     TEXT NOT NULL,
  up_to_msg   TEXT REFERENCES messages(id),
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

이 세 테이블의 관계:

| 테이블 | 역할 | 핵심 필드 |
| --- | --- | --- |
| `sessions` | 대화방 목록 | `user_id`, `title` |
| `messages` | 모든 메시지 원문 | `session_id`, `role`, `content` |
| `summaries` | 주기적 요약 | `session_id`, `up_to_msg` (어디까지 요약했는지) |

`up_to_msg`가 중요해요 — 이 필드가 있어야 "어디까지 요약했고 어디서부터 원문을 보내야 하는지" 경계가 명확합니다.

## CRUD 기본 동선

```typescript
// 1. 세션 생성
async function createSession(userId: string): Promise<string> { ... }

// 2. 메시지 저장
async function saveMessage(sessionId: string, role: string, content: string): Promise<void> { ... }

// 3. 컨텍스트 조립 (요약 + 최근 N턴)
async function buildContext(sessionId: string, recentN: number = 10) {
  const summary = await getLatestSummary(sessionId);
  const recentMessages = await getRecentMessages(sessionId, recentN);

  return [
    ...(summary ? [{ role: "system" as const, content: `이전 대화 요약: ${summary.content}` }] : []),
    ...recentMessages,
  ];
}

// 4. 요약 갱신 (N턴마다)
async function updateSummary(sessionId: string): Promise<void> { ... }
```

`buildContext`가 핵심 함수예요. 이 함수가 매 호출마다 "요약 + 최근 원문"을 조립해서 모델에 보냅니다.

## 프라이버시 — 대화에는 개인정보가 들어간다

대화 저장소에는 사용자가 입력한 모든 것이 남아요. 이름, 이메일, 건강 정보, 금융 정보 — 의식하지 않아도 들어갑니다.

| 대응 | 어떻게 |
| --- | --- |
| 데이터 보존 기간 | 예: 30/90일 등 서비스 정책·법규에 맞춘 TTL |
| 사용자 삭제 요청 | "내 대화 삭제" 기능 필수 |
| PII 마스킹 | 저장 전에 이메일·전화번호 패턴을 `[REDACTED]`로 |
| 접근 제어 | 세션은 생성한 사용자만 조회 가능 |

MVP에서도 **"내 대화 삭제"** 버튼 하나는 넣어야 해요. 규정 이전에 사용자 신뢰의 문제입니다.
