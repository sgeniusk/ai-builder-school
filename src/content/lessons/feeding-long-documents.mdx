## 100쪽 PDF 를 통째로 넣었더니 앞부분을 잊어버렸어요

분기 보고서 PDF 100쪽. 통째로 모델에 던지고 *"가장 중요한 의사결정 5개 뽑아줘"* 라고 부탁했죠. 답이 왔는데, 5개 모두 PDF 의 *마지막 30쪽* 에서만 나왔습니다. 앞부분의 핵심 결정 — *"신규 사업 진출 보류"* 같은 — 은 누락됐어요.

원인은 단순해요. **컨텍스트 윈도우가 길어졌다고 모델의 *주의 분포* 가 균등한 건 아닙니다.** 긴 문맥의 가운데는 흐려지고, 끝과 시작에 무게가 실리는 *Lost in the Middle* 현상이 잘 알려져 있죠.

오늘은 100쪽 짜리 문서를 *맥락 손실 없이* 모델에게 먹이는 세 가지 전략을 비교해요. 같은 문서, 같은 질문, 다른 결과 — 차이를 직접 측정합니다.

## 출력 잠근 다음에 왜 이 레슨인가

[Lesson 2.1](/lessons/structure-of-good-prompts) 에서 4축 중 *Context* 축을 배웠어요. [Lesson 2.2](/lessons/enforcing-output-format) 에서 *Output* 을 잠갔고요. 이번엔 그 *Context* 가 *너무 클 때* 어떻게 다루는지가 주제예요.

이 레슨은 또한 **Phase 6 (RAG) 의 씨앗** 이에요. 오늘 배우는 *Selective Injection* 이 RAG 의 핵심 메커니즘이거든요. 지금은 손으로, 나중엔 자동으로.

## 핵심 개념 — 세 가지 전략

### 1) Full Context — *통째로 넣기*

가장 단순. 모델 컨텍스트 윈도우가 충분히 크면 그냥 던져요. 단, 위험,

- *Lost in the Middle* — 중간이 흐려짐
- 비용이 토큰 수에 비례 — 자주 부르면 부담
- 한 번에 못 넣는 길이는 자동 절단

언제 쓰나, **30쪽 미만 + 한 번 호출** 이면 보통 OK.

### 2) Map-Reduce — *조각 요약 → 통합 요약*

긴 문서를 의미 단위로 잘라(*chunk*) 각 조각을 따로 요약하고, 그 요약들을 다시 합쳐 최종 답을 만들어요.

```
Doc → [chunk1, chunk2, ..., chunkN]
      ↓ 각 청크 요약
      [s1, s2, ..., sN]
      ↓ 요약들의 요약
      Final Summary
```

장점,

- 컨텍스트 한계 우회
- 중간 청크의 정보 손실 적음 — 모델이 매번 *작은 문서* 만 보니까

단점,

- 호출 수 ↑ 비용·지연 ↑
- 청크 경계에서 맥락 끊김 (한 결정이 두 청크에 걸친 경우)

언제 쓰나, **요약·키 포인트 추출** 같은 *집계형* 질문에 강함.

### 3) Selective Injection — *질문 관련 부분만 골라 넣기*

질문을 미리 분석해서 문서에서 *관련 청크 3~5개만* 뽑아 모델에 주입해요. RAG 의 본질.

```
Query → 관련 청크 검색 → 상위 5개 + Query → 답
```

장점,

- 가장 저렴 (적은 토큰)
- 가장 빠름
- 질문에 *집중된* 답

단점,

- 검색이 틀리면 답도 틀림 (관련 청크를 놓치면 답이 빈칸)
- "전체 분위기" 가 필요한 질문에 약함 ("이 문서의 톤이 어떤가?" 같은)

언제 쓰나, **구체 질문 + 큰 문서** 에 가장 강함.

## 그래서 오늘 할 일

40~60쪽 짜리 *내 도메인의 긴 문서* 하나를 골라 세 전략을 *같은 질문* 으로 비교해요.

권장 후보,

- 회사 분기 보고서 / IR 자료
- 학술 논문 / 백서
- 회의록 묶음 (1년치)
- 제품 사양서 / API 문서

작업 흐름,

1. 문서 1개 + 비교 질문 1개를 정합니다 (10분).
2. 전략 1 (Full) — 통째로 넣고 답 받기 (5분).
3. 전략 2 (Map-Reduce) — 청크 5~7개로 잘라 각 요약 → 통합 (15분).
4. 전략 3 (Selective) — 질문에 키워드로 가장 관련 깊은 3 청크만 골라 주입 (10분).
5. 세 답을 *완전성·정확성·비용·시간* 4 축으로 비교 표 작성 (10분).

산출물은 [`long-doc-strategy-comparison.md`](/lessons/feeding-long-documents) — 비교 표 + 발견 로그 + 도메인 추천 슬롯.

## 마무리 — 입력 디자인이 모델 성능을 절반은 결정합니다

세 전략 비교를 한 번 해보면, *내 도메인에 어떤 전략이 맞는지* 의 직관이 생겨요. 이 직관이 [Lesson 2.4](/lessons/build-personal-prompt-library) 에서 만들 *프롬프트 라이브러리* 의 메타 결정 — *언제 어떤 템플릿을 부를지* — 에 직접 쓰입니다.

그리고 *Selective Injection* 을 자동화한 게 RAG 예요. **Phase 6** ([Lesson 6.1](/lessons/understanding-embeddings) 부터) 에서 임베딩 + 벡터 검색으로 이걸 시스템화합니다.
