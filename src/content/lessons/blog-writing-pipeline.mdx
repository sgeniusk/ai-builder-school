## "AI가 쓴 글, 읽어보면 다 '그런 글'이에요"

블로그 글을 AI에게 시켰습니다. 5분 만에 2000자가 나옵니다. 문법도 맞고, 구조도 깔끔해요. 그런데 읽어보면 어딘가 밋밋합니다. 동료한테 보여줬더니 한마디 — "이거 AI가 쓴 거지?" 내 글에는 있던 날카로운 비유, 짧은 호흡, 갑자기 튀어나오는 질문 — 그게 다 사라져 있어요.

AI가 글을 "잘" 쓰는 건 맞아요. 하지만 **"나처럼" 쓰는 건** 다른 문제예요. 오늘은 AI의 속도를 살리면서 내 목소리를 유지하는 파이프라인을 만듭니다. Phase 3의 마지막 한 장이에요.

## 왜 AI 글은 다 비슷할까

{/* Source · DOC · Anthropic "Prompt engineering overview" — https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/overview */}

이유는 단순해요. AI는 학습 데이터의 **평균**으로 돌아가려는 경향이 있습니다. 수백만 개의 글을 봤으니, 가장 "무난한" 문체로 수렴해요.

| AI 기본 문체 | 개인 문체 |
| --- | --- |
| 균일한 문장 길이 | 길었다 짧았다 불규칙 |
| 부드러운 연결어 | 갑작스러운 전환 |
| 중립적 톤 | 의견이 분명한 톤 |
| "~합니다" 체 일관 | 체를 섞어 쓰기도 |

이 차이를 메우려면 **내 글 샘플을 컨텍스트로 넣어야** 해요. [Lesson 09](/lessons/automate-report-drafts)의 템플릿 그라운딩과 같은 원리 — AI에게 "이게 내 스타일이야"를 보여주는 거예요.

## 스타일 샘플 주입 — 내 목소리를 가르치기

{/* Source · DOC · Anthropic "Context windows" — https://platform.claude.com/docs/en/build-with-claude/context-windows */}

내 글 중 **가장 "나다운" 글 2~3개**를 골라 컨텍스트에 넣습니다.

```
Context:
- 아래는 내가 직접 쓴 블로그 글 3개입니다. 내 문체의 특징을 분석해 주세요.
- [글 1 전문 또는 핵심 문단]
- [글 2 전문 또는 핵심 문단]
- [글 3 전문 또는 핵심 문단]

Task:
위 글에서 나타나는 문체 특징을 5가지로 정리해 주세요.
(문장 길이, 비유 스타일, 톤, 전환 방식, 자주 쓰는 표현 등)
```

이 분석 결과를 **스타일 가이드**로 저장해 두세요. 다음부터 글을 쓸 때마다 이 가이드를 프롬프트에 넣으면, AI가 내 문체에 가까운 글을 생성합니다.

핵심은 **한 번 분석해 놓으면 계속 쓸 수 있다**는 거예요. [Lesson 08](/lessons/build-personal-prompt-library)의 프롬프트 라이브러리에 스타일 가이드를 넣어두면 매번 샘플을 다시 넣을 필요 없이 호출할 수 있어요.

## 4단 편집 루틴 — 한 번에 다 하지 말기

{/* Note: 4단 편집 루틴(아웃라인→초안→구조 검토→톤 검토)은 이 학교 운영자가 정리한 명칭이며, 외부 표준명은 아닙니다. */}

한 번의 호출로 완벽한 글을 뽑으려 하면 실패해요. 대신 **4단계로 나누면** 각 단계에서 한 가지에만 집중할 수 있습니다.

| 단계 | 집중 | AI / 사람 |
| --- | --- | --- |
| 1. 아웃라인 | 구조와 흐름 | AI 제안 → 사람 확정 |
| 2. 초안 | 내용 채우기 | AI (스타일 가이드 적용) |
| 3. 구조 검토 | 논리 흐름 + 사실 확인 | AI 1차 → 사람 확정 |
| 4. 톤 검토 | 내 목소리와 일치 | 사람 (가장 중요) |

**4단계가 사람인 이유**가 있어요. AI는 문법·논리는 잘 잡지만, "이 문장이 나답냐"는 결국 본인만 판단할 수 있습니다. 이 단계를 생략하면 "AI가 쓴 거지?"가 계속 따라다녀요.

## 1단계: 아웃라인 — 방향이 크게 결정됩니다

아웃라인을 잘 잡으면 나머지 단계가 수월해요. 아웃라인 프롬프트의 핵심은 **독자와 목적을 명시하는 것**입니다.

```
Task:
아래 주제로 블로그 글 아웃라인을 작성해 주세요.

Context:
- 주제: [주제]
- 독자: [누구]
- 목적: [읽은 후 독자가 무엇을 하게 되기를 원하는지]
- 길이: [2000자 / 3000자 등]

Output:
- H2 섹션 5~7개
- 각 섹션별 핵심 메시지 1문장
- 도입부의 훅(첫 문장)
- 마무리의 CTA(행동 유도)
```

[Lesson 05](/lessons/structure-of-good-prompts)의 4축이 여기서도 그대로 쓰이죠. 특히 Context에 독자와 목적을 넣으면 아웃라인의 방향이 잡혀요.

아웃라인이 나오면 **사람이 확정**합니다. "이 섹션은 빼고, 이건 순서를 바꾸자." 이 확정 없이 바로 초안으로 가면 나중에 구조를 고치느라 시간이 더 들어요.

## 2단계: 초안 — 스타일 가이드를 붙여서

확정된 아웃라인 + 스타일 가이드를 함께 넣고 초안을 생성합니다.

```
Context:
- 확정된 아웃라인: [아웃라인]
- 내 문체 가이드: [스타일 가이드 5가지 특징]

Task:
위 아웃라인에 따라 블로그 글 초안을 작성해 주세요.

Constraints:
- 문체 가이드를 따르세요
- 주장에는 근거를 붙이세요 (없으면 "[확인 필요]" 표시)
- 도입부 훅은 아웃라인의 것을 살려 주세요
```

이 초안은 **출발점**이에요. 구조는 갖췄지만, 여기서부터 사람의 손이 들어갑니다.

## 3단계: 구조 검토 — AI에게 한 번 더

{/* Source · DOC · Anthropic "Prompt engineering overview" — https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/overview */}

초안을 AI에게 다시 주면서 논리 흐름과 사실 확인을 시킵니다.

```
Task:
아래 블로그 글 초안을 검토해 주세요.

확인 항목:
1. 각 섹션의 흐름이 자연스러운가 (논리적 비약 없는가)
2. 주장에 근거가 붙어 있는가 ("[확인 필요]"가 남아 있는가)
3. 중복되는 내용이 있는가
4. 독자가 읽다가 이탈할 만한 지점이 있는가
```

[Lesson 09](/lessons/automate-report-drafts)의 자기 검토, [Lesson 10](/lessons/meeting-notes-pipeline)의 누락 체크와 같은 패턴이에요. **한 호출이 놓친 걸 다음 호출이 잡아줍니다.**

## 4단계: 톤 검토 — 이건 사람만

AI가 잡은 논리·사실 이슈를 반영한 뒤, 마지막은 **소리 내어 읽기**예요.

- 이 문장이 나답나?
- 이 비유를 내가 쓸까?
- 이 전환이 내 글 호흡과 맞나?

아니면 고치세요. 이 단계가 "AI가 쓴 거지?"와 "이건 네 글이네"의 분기점이에요.

## 그래서 오늘 만들 한 폴더

오늘 만들 건 `blog-pipeline/` 폴더예요.

| 파일 | 역할 |
| --- | --- |
| `style-analysis.md` | 내 글 3개 → 문체 특징 5가지 추출 프롬프트 |
| `outline.md` | 1단계 — 아웃라인 생성 프롬프트 (독자·목적 명시) |
| `draft.md` | 2단계 — 스타일 가이드 적용 초안 프롬프트 |
| `review.md` | 3단계 — 구조 검토 프롬프트 (논리·사실·중복) |
| `tone-checklist.md` | 4단계 — 사람용 톤 검토 체크리스트 |

이 다섯 파일이 있으면 반복 작업은 AI가 덜어주면서, 마지막 판단에서 내 목소리를 살릴 수 있어요.

## Phase 3를 닫으며

Phase 3 (일상 자동화)에서 만든 4개 파이프라인을 모아볼게요.

- [Lesson 09](/lessons/automate-report-drafts) — **보고서 초안 파이프라인** (템플릿 그라운딩 + 3단 루프)
- [Lesson 10](/lessons/meeting-notes-pipeline) — **회의록 → Action Item 파이프라인** (3섹션 구조화 + JSON)
- [Lesson 11](/lessons/research-workflow) — **리서치 파이프라인** (질문 분해 + 인용 강제 + 근거 표)
- **Lesson 12 (오늘)** — **블로그 글쓰기 파이프라인** (스타일 주입 + 4단 편집)

이 네 가지가 있으면 매일의 정보 업무 — 정리, 분석, 생성 — 에서 AI가 가속기가 됩니다. 매번 새로 짜지 않아요. **라이브러리에서 호출하면 됩니다.**

다음 Phase ([Phase 4 — 코딩 에이전트](/curriculum/phase-4-coding-agents))에서는 코드의 세계로 들어갑니다. 비개발자도 AI를 통해 코드를 읽고, 만들고, 자동화할 수 있어요.
