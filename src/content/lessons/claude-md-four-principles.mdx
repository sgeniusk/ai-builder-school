## "65줄짜리 텍스트 파일이 깃헙 10만 스타를 받았어요"

Andrej Karpathy가 자기 프로젝트에 깐 `CLAUDE.md` 한 장이 있어요. 65줄. 코드도 아니고 라이브러리도 아닌 **그냥 텍스트 파일**입니다. 그런데 이 파일을 추가하면 Claude Code의 코딩 품질이 눈에 띄게 달라져요.

[Lesson 13](/lessons/coding-agent-setup)에서 CLAUDE.md를 만들었지만, "어떻게 잘 쓰나"는 따로 다루지 않았어요. 오늘은 Karpathy의 65줄 안에 든 **4가지 원칙**을 풀어 봅니다.

## AI 코딩의 3가지 고질병

{/* Source · DOC · Andrej Karpathy "CLAUDE.md" — https://github.com/karpathy/nanochat/blob/master/CLAUDE.md */}
{/* Note: 본문의 "3가지 고질병 / 4원칙" 정리는 Karpathy의 글을 이 학교 운영자가 한국어로 압축한 것입니다. */}

Karpathy가 CLAUDE.md에 손을 댄 이유는 명확해요. AI 코딩 에이전트한테 일을 시키면 항상 같은 패턴으로 망가집니다.

| 고질병 | 증상 |
| --- | --- |
| **잘못된 가정을 한다** | 묻지도 않고 멋대로 해석한 뒤 그대로 달려갑니다 |
| **코드를 부풀린다** | 간단한 일을 1000줄짜리 추상화로 만들고, 죽은 코드를 치우지 않습니다 |
| **코드를 멋대로 고친다** | 부탁하지도 않은 옆 코드를 바꾸고 주석을 지웁니다 |

Karpathy의 결론은 한 줄로 압축돼요.

> "명령하지 말고, 성공 기준을 주고 지켜보세요."

명령은 모호하면 흐릿해지지만, 성공 기준은 검증 가능합니다. 이 한 줄이 4원칙의 토대예요.

## 4원칙 — 65줄 안의 핵심

### 1. Think Before Coding — 코딩하기 전에 먼저 생각하세요

```
Don't assume. Don't hide confusion. Surface tradeoffs.
```

가정하지 말고, 혼란을 숨기지 말고, **트레이드오프를 드러내세요**. AI에게 모호한 요청을 하면 묻지 않고 자기 해석으로 코드를 시작해버려요. 한 번 시작하면 200줄이 와르르 나옵니다.

같은 "로그인 기능 추가해줘" 요청에 두 가지 반응을 비교해 보세요.

| ❌ 원칙 없는 AI | ✓ 원칙 적용된 AI |
| --- | --- |
| "알겠습니다. JWT 기반 인증으로 구현했습니다" | "방식이 여러 가지 있는데 어느 쪽으로 갈까요? JWT는 stateless라 SPA에 적합, 세션은 보안 관리가 쉬움, OAuth는 외부 인증 활용. 프로젝트에 next-auth가 깔려 있어 OAuth+세션이 자연스러울 것 같은데 맞나요?" |
| → 코드 200줄 와르르 출력 | → 한 번 묻고 사람의 한 줄 컨펌으로 결정 |
| → 회사는 OAuth만 쓰는데 처음부터 다시 짜야 함 | → 처음부터 맞는 방향으로 |

### 2. Simplicity First — 문제를 푸는 최소한의 코드만 짜세요

```
No defensive programming. No premature abstraction. No options not asked for.
```

방어적 프로그래밍 금지. 추상화 미리 만들기 금지. 부탁하지도 않은 옵션 추가 금지. AI는 "혹시 모를 케이스"를 위해 try/catch 5겹과 옵션 12개를 만들고 싶어해요. 그게 그대로 코드 부풀림이 됩니다.

핵심 한 줄.

> 지금 풀려는 문제를 푸는 코드만 짜세요.

미래의 다른 케이스는 그때 가서 보태도 충분합니다.

### 3. Surgical Changes — 꼭 필요한 곳만 수술하듯 고치세요

```
Touch only what's needed. Preserve what works. Don't rewrite for style.
```

필요한 것만 만지고, 작동하는 코드는 보존하고, 스타일 때문에 다시 쓰지 마세요. AI에게 "버그 고쳐줘" 시키면 옆 코드도 정리하고 변수명도 바꾸고 주석도 지웁니다. PR이 100줄로 부풀어요.

수술하듯 고치는 사람의 PR은 5줄이에요. 변경 범위가 작으면 리뷰가 빠르고 사고 확률이 떨어집니다.

### 4. Goal-Driven Execution — 성공 기준을 정의하고, 검증될 때까지 반복하세요

```
Define done. Verify before claiming. Loop until the goal is met.
```

"끝"의 정의를 명시하고, 끝났다고 주장하기 전에 검증하고, 목표 달성까지 반복합니다. AI에게 "테스트 통과시켜줘"가 아니라 "`npm run test`가 exit 0을 반환해야 함"이라고 주는 거예요. 그러면 테스트만 끄지 않고, 실제로 통과하는지 검증하고 반복합니다.

이게 Karpathy의 한 줄과 직결돼요. **명령이 아니라 성공 기준**.

## 4원칙을 내 CLAUDE.md에 어떻게 넣나

[Lesson 13](/lessons/coding-agent-setup)에서 만든 CLAUDE.md에 다음 섹션 한 장을 추가하면 됩니다.

```markdown
## How to work in this repo

### 1. Think before coding
- Don't assume. Surface tradeoffs.
- Confused? Ask. Don't hide it in code.

### 2. Simplicity first
- Solve only what was asked.
- No defensive try/catch unless required.

### 3. Surgical changes
- Touch only what's needed. Preserve working code.
- No rewrites for style.

### 4. Goal-driven execution
- Define "done" as a runnable command.
- Verify before claiming complete.
```

이 한 장이 들어가면 Claude Code가 매 메시지마다 이 규약을 먼저 읽고 따라가요. [Lesson 13](/lessons/coding-agent-setup)의 "상시 프롬프트" 개념이 본격 작동하는 거예요.
