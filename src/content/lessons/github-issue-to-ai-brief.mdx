## "버그 수정해줘" — 이슈 하나로 에이전트가 움직일 수 있을까?

금요일 오후, GitHub에 이슈가 올라옵니다. 제목은 **"로그인 버그 수정해주세요"**. 본문에는 "로그인이 안 됩니다. 급합니다." 한 줄. 사람이 봐도 모호한데, 코딩 에이전트에게 이걸 던지면 어떻게 될까요?

에이전트는 "로그인"이 어떤 페이지인지, 어떤 상태에서 안 되는지, 성공 조건이 뭔지 모릅니다. 그래서 가장 눈에 띄는 `login.ts` 파일을 열고 추측으로 고치기 시작해요. 결과는 — 기존에 되던 소셜 로그인까지 깨지는 사고.

[좋은 프롬프트의 구조](/lessons/structure-of-good-prompts)에서 "프롬프트는 부탁이 아니라 명세"라고 했죠. GitHub 이슈도 똑같아요. 에이전트가 읽을 이슈는 **부탁이 아니라 브리프(Brief)** 여야 합니다.

## 이슈 vs 브리프 — 무엇이 다른가

| 항목 | 일반 이슈 | AI 브리프 |
| --- | --- | --- |
| 제목 | "로그인 버그" | "이메일 로그인 시 500 에러 — `POST /api/auth/login`" |
| 재현 스텝 | 없음 | 1. 회원가입 → 2. 이메일/비밀번호 입력 → 3. 500 반환 |
| 수용 조건 | "되게 해 주세요" | "200 반환 + JWT 발급 + 기존 소셜 로그인 미영향" |
| 영향 범위 | 없음 | `src/app/api/auth/login/route.ts`, `src/lib/auth.ts` |
| 검증 명령 | 없음 | `npm run test -- --grep "login"` |

왼쪽은 사람 사이에서도 오해가 생기는 이슈예요. 오른쪽은 에이전트가 **첫 시도부터** 올바른 파일을 열고, 올바른 방향으로 고칠 수 있는 브리프입니다.

## 브리프의 4축 — L05의 코딩 에이전트 버전

{/* Source · DOC · Anthropic "Claude Code overview" — https://docs.anthropic.com/en/docs/claude-code/overview */}

[좋은 프롬프트의 구조](/lessons/structure-of-good-prompts)의 4축(Task · Context · Constraints · Output)을 코딩 에이전트용으로 번역하면 이렇게 됩니다.

| L05 4축 | AI 브리프 버전 | 예시 |
| --- | --- | --- |
| Task (작업) | 무엇을 고칠/만들 것인가 | "이메일 로그인 시 500 에러를 해결한다" |
| Context (맥락) | 재현 스텝 + 영향 파일 | "POST /api/auth/login → 500. 관련 파일: route.ts, auth.ts" |
| Constraints (제약) | 건드리지 말 것 + 코드 스타일 | "소셜 로그인 로직 미수정, 기존 테스트 통과" |
| Output (출력) | 수용 조건 + 검증 명령 | "200 반환 + `npm run test` 통과" |

이 4칸이 채워진 이슈는 사람이 읽어도 명확하고, 에이전트가 읽어도 바로 시작할 수 있어요.

## `gh issue view` — 터미널에서 이슈 읽기

{/* Source · DOC · GitHub CLI "gh issue view" — https://cli.github.com/manual/gh_issue_view */}

코딩 에이전트는 브라우저가 아니라 터미널에서 동작합니다. GitHub 이슈를 에이전트에게 전달하는 가장 빠른 방법은 **GitHub CLI**예요.

```bash
# 이슈 본문을 터미널에 출력
gh issue view 42

# 마크다운 형식으로 출력 (에이전트가 파싱하기 좋음)
gh issue view 42 --json title,body,labels,assignees
```

이 출력을 에이전트에게 넘기면 됩니다. Claude Code라면 이런 식이에요:

```
이 이슈를 읽고 브리프로 변환해 줘:
$(gh issue view 42)
```

하지만 이슈 본문이 모호하면 에이전트도 모호한 브리프를 만들어요. **이슈 자체가 브리프 형태로 작성되어야** 하는 이유입니다.

## 이슈 템플릿 — 처음부터 브리프로 쓰기

{/* Source · DOC · GitHub "Creating issue templates" — https://docs.github.com/en/communities/using-templates-to-encourage-useful-issues-and-pull-requests/configuring-issue-templates-for-your-repository */}

GitHub는 이슈 템플릿을 지원합니다. `.github/ISSUE_TEMPLATE/ai-brief.md`를 만들어두면 이슈를 쓰는 사람이 **처음부터 4축 구조로** 작성하게 됩니다.

```markdown
---
name: AI Brief (코딩 에이전트용)
about: 코딩 에이전트가 바로 시작할 수 있는 이슈 형식
labels: ai-brief
---

## Task (무엇을)
<!-- 한 줄로: 어떤 버그를 고치거나, 어떤 기능을 만들 것인가 -->

## Context (맥락)
### 재현 스텝
1.
2.
3.

### 영향 파일
- `src/...`

### 현재 동작 vs 기대 동작
| 현재 | 기대 |
| --- | --- |
| | |

## Constraints (제약)
- 건드리지 말 것:
- 코드 스타일:
- 시간 제한:

## Output (수용 조건)
- [ ] (기능적 조건)
- [ ] (비기능적 조건)
- [ ] 검증: `npm run test` 통과
```

이 템플릿이 깔리면 팀 전체의 이슈 품질이 올라가요. 사람이 읽어도, 에이전트가 읽어도 명확한 브리프가 됩니다.

## 이슈 → 브리프 변환 스크립트

이미 수십 개의 모호한 이슈가 쌓여 있다면? 하나씩 다시 쓸 수는 없어요. 변환 스크립트를 만듭니다.

```bash
#!/bin/bash
# issue-to-brief.sh — 기존 이슈를 AI 브리프로 변환
ISSUE_NUM=$1
ISSUE_JSON=$(gh issue view "$ISSUE_NUM" --json title,body,labels)

# Claude Code에게 변환 요청
cat <<EOF
다음 GitHub 이슈를 AI 브리프 형식으로 변환해 줘.

원본 이슈:
$ISSUE_JSON

변환 규칙:
1. Task: 한 줄 동사형 요약
2. Context: 재현 스텝 (추론 가능하면 작성, 불가능하면 "정보 부족 — 이슈 작성자에게 확인 필요")
3. Constraints: 원본에서 언급된 제약 + "기존 테스트 통과" 기본 추가
4. Output: 수용 조건 체크리스트 3~5개
5. 정보가 부족한 칸은 비워두지 말고 "[확인 필요]"로 표시

형식: Markdown. 앞뒤 설명 없이 브리프만 출력.
EOF
```

이 스크립트의 핵심은 마지막 규칙이에요 — **"정보가 부족한 칸은 비워두지 말고 '[확인 필요]'로 표시"**. 에이전트가 빈 칸을 추측으로 채우는 것보다, 모르는 걸 모른다고 표시하는 게 훨씬 안전합니다. [출력 형식(Output Format)을 강제하는 법](/lessons/enforcing-output-format)의 3단 방어선에서 배운 "형식을 강제하면 실패가 보인다"는 원리와 같아요.

## 그래서 오늘 만들 한 폴더

오늘 만들 건 `issue-to-brief/` 폴더예요. 이슈 템플릿과 변환 스크립트를 한 자리에 모읍니다.

| 파일 | 역할 |
| --- | --- |
| `ai-brief.md` | GitHub 이슈 템플릿 (4축 구조) |
| `issue-to-brief.sh` | 기존 이슈 → 브리프 변환 스크립트 |
| `examples/before.md` | 변환 전 모호한 이슈 예시 |
| `examples/after.md` | 변환 후 AI 브리프 예시 |

이 폴더가 있으면 새 이슈는 템플릿으로, 기존 이슈는 스크립트로 변환합니다. 에이전트에게 "이슈 42번 해결해"가 아니라 "이 브리프대로 해결해"가 됩니다.

## 브리프의 가치 — 속도가 아니라 방향

브리프를 쓰면 시간이 더 걸린다고 느낄 수 있어요. 하지만 모호한 이슈를 던져서 에이전트가 잘못된 방향으로 달린 뒤 되돌리는 비용을 생각하면, 브리프를 쓰는 몇 분이 되돌리는 시간을 줄여주는 경우가 많아요.

다음 레슨([AI에게 코드베이스 읽히기](/lessons/letting-ai-read-codebase))에서는 브리프가 가리키는 코드베이스를 에이전트가 **제대로 이해하게 만드는 법**을 다룹니다. 브리프가 "무엇을 할지"를 정해주고, 코드베이스 맵이 "어디서 할지"를 정해줍니다.

시작해 볼까요?
