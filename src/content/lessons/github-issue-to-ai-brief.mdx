## 도입 — 학습자에게 말 거는 한 장면

GitHub Issue 에 “모바일에서 버튼이 이상해요”라고 적혀 있습니다. 사람 팀원이라면 대충 맥락을 물어보고 화면을 보며 추측할 수 있어요. 하지만 코딩 에이전트에게 이 문장은 너무 넓습니다. 어떤 버튼인지, 어떤 화면인지, 무엇이 정상인지, 고치면 어떻게 확인할지 아무것도 없어요.

결국 에이전트는 파일을 넓게 뒤지고, 비슷해 보이는 스타일을 바꾸고, 마지막에는 “수정했습니다”라고 말합니다. 그런데 실제 화면에서는 그대로예요. 이 실패의 원인은 모델 성능보다 작업지시서의 빈칸입니다. 오늘은 애매한 이슈를 에이전트가 바로 움직일 수 있는 AI 브리프로 바꿉니다.

## Lesson 4.1 다음에 왜 이 레슨인가

[Lesson 4.1](/lessons/coding-agent-setup) 에서 권한, 비밀, 로그의 울타리를 만들었어요. 이제 그 울타리 안으로 실제 작업을 넣어야 합니다. 작업의 가장 흔한 입구는 GitHub Issue 예요. 이슈가 애매하면 아무리 안전한 환경이어도 에이전트는 헤맵니다.

여기서 [Lesson 2.1](/lessons/structure-of-good-prompts) 의 4축을 다시 호출합니다. Task 는 무엇을 고칠지, Context 는 어디에서 발생하는지, Constraints 는 건드리면 안 되는 범위, Output 은 완료 기준이에요. 오늘 산출물 [`issue-to-brief.md`](/lessons/github-issue-to-ai-brief) 는 이 4축을 GitHub Issue 양식으로 고정합니다.

## 핵심 개념 — Acceptance Criteria 와 Reproduction Steps

### Acceptance Criteria — 끝났다고 말할 수 있는 조건

“버튼을 고친다”는 완료 조건이 아니에요. “모바일 390px 화면에서 결제 버튼이 하단 safe area 를 침범하지 않고, 탭하면 `/checkout` 으로 이동한다”가 완료 조건입니다. AI 는 완료의 정의가 흐리면 자기 기준으로 끝냅니다.

좋은 수용 조건은 세 가지를 포함해요.

| 요소 | 질문 | 예시 |
| --- | --- | --- |
| 관찰 가능성 | 눈으로 보거나 테스트로 확인할 수 있나? | 버튼이 화면 밖으로 나가지 않는다. |
| 범위 | 어디까지만 고치나? | 결제 페이지 모바일 레이아웃만 수정한다. |
| 검증 | 무엇을 실행해 통과를 확인하나? | `npm run typecheck` 와 모바일 화면 확인. |

### Reproduction Steps — 다시 볼 수 있어야 고칠 수 있다

버그는 재현되지 않으면 고쳐지지 않습니다. 에이전트도 마찬가지예요. “가끔 깨짐” 대신 “Chrome 모바일 폭 390px, 장바구니 2개, 결제 페이지 진입, 하단 버튼 확인”처럼 순서를 줍니다. 나중에 [Lesson 4.6](/lessons/bug-reproduction-loop) 에서 이 재현 스텝이 최소 예제와 회귀 테스트로 이어져요.

### Impact Scope — 어디를 건드리면 안 되는가

AI 브리프에는 고칠 곳만큼 건드리지 말아야 할 곳도 들어갑니다. 결제 버튼 문제라면 인증, 가격 계산, API 라우트는 범위 밖일 수 있어요. 이 한 줄이 불필요한 리팩터링을 막습니다. [Lesson 1.4](/lessons/checks-before-trusting-ai-output) 의 Scope 체크를 구현 전에 당겨오는 방식이에요.

## 그래서 오늘 할 일

오늘 만들 것은 “GitHub Issue 를 AI 작업지시서로 바꾸는 템플릿”입니다.

1. 최근에 애매했던 이슈 3개를 고릅니다.
2. 각 이슈에서 문제 장면, 기대 동작, 실제 동작을 분리합니다.
3. 재현 스텝을 3~7단계로 씁니다.
4. 수용 조건을 체크박스 3개 이상으로 만듭니다.
5. 영향 범위와 제외 범위를 적습니다.
6. 에이전트에게 줄 첫 요청 문장을 씁니다.
7. 산출물 [`issue-to-brief.md`](/lessons/github-issue-to-ai-brief) 로 저장하고 다음 [Lesson 4.3](/lessons/letting-ai-read-codebase) 의 코드베이스 읽기 가이드와 함께 사용합니다.

처음에는 시간이 더 걸리는 것처럼 보여요. 하지만 이슈를 한 번 브리프로 바꿔두면 에이전트의 첫 시도 품질이 달라집니다. 질문을 덜 하고, 엉뚱한 파일을 덜 열고, 검증까지 자연스럽게 이어져요.

## 마무리

AI 에게 좋은 일을 시키는 시작점은 “좋은 이슈”입니다. 오늘 만든 브리프는 사람에게도 좋고 에이전트에게도 좋아요. 다음 레슨에서는 이 브리프를 들고 실제 저장소 안으로 들어갑니다. [Lesson 4.3](/lessons/letting-ai-read-codebase) 에서 에이전트가 코드베이스를 읽는 순서와 지도를 만들어 볼 거예요.
