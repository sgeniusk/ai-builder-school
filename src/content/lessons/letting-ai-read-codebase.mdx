## 도입 — 학습자에게 말 거는 한 장면

작은 수정 하나를 부탁했는데 에이전트가 저장소 전체를 훑기 시작합니다. 비슷한 이름의 컴포넌트가 세 개라서 어디를 고쳐야 할지 헷갈리고, 테스트 파일 위치도 못 찾고, 예전 패턴을 따라 새 코드를 씁니다. 사람이라면 “아, 우리 프로젝트는 이 폴더가 진짜야”라고 바로 알 텐데요.

코드베이스는 문서보다 암묵지가 많아요. 폴더 이름, 테스트 관습, 오래된 코드와 새 코드의 경계, 절대 건드리면 안 되는 파일 같은 것들입니다. AI 는 처음 보는 저장소에서 이 암묵지를 모릅니다. 오늘은 에이전트가 코드를 읽기 전에 먼저 볼 지도와, 실제 변경 전에 탐색만 하는 Scout Run 루틴을 만듭니다.

## Lesson 4.2 다음에 왜 이 레슨인가

[Lesson 4.2](/lessons/github-issue-to-ai-brief) 에서 작업지시서를 만들었어요. 하지만 좋은 브리프만으로는 부족합니다. 브리프가 “결제 버튼 수정”이라고 말해도, 저장소 안에서 결제 버튼이 어디에 있는지 모르면 에이전트는 추측합니다.

여기서 [Lesson 2.3](/lessons/feeding-long-documents) 의 긴 문서 다루기 감각을 다시 씁니다. 코드베이스도 거대한 문서예요. 한 번에 전부 넣는 대신 구조를 요약하고, 필요한 부분만 읽게 하고, 변경 전에 위험을 보고하게 해야 합니다. 산출물 [`codebase-reading-guide.md`](/lessons/letting-ai-read-codebase) 는 그 안내문입니다.

## 핵심 개념 — Repo Map 과 Scout Run

### Repo Map — 저장소를 한 화면으로 줄인다

Repo Map 은 폴더 목록을 복사한 것이 아니에요. 에이전트가 “어디를 먼저 봐야 하지?”를 판단하게 해주는 지도입니다. 좋은 지도에는 네 가지가 들어갑니다.

| 항목 | 설명 |
| --- | --- |
| 주요 폴더 | 앱, 컴포넌트, 데이터, 테스트, 스크립트 위치. |
| 로컬 관습 | import alias, 파일 이름 규칙, 서버와 클라이언트 컴포넌트 기준. |
| 검증 명령 | 타입체크, 테스트, 빌드, 콘텐츠 검증 명령. |
| 위험 영역 | 건드리기 전 사람 확인이 필요한 파일과 이유. |

이 지도는 [Lesson 4.1](/lessons/coding-agent-setup) 의 권한 문서와 붙어 있어야 해요. 권한 문서가 울타리라면 Repo Map 은 길 안내입니다.

### Scout Run — 고치기 전에 읽기만 한다

Scout Run 은 구현이 아닙니다. 에이전트에게 “지금은 파일을 바꾸지 말고 읽기만 해요”라고 시키는 단계예요. 결과물은 변경안이 아니라 탐색 리포트입니다. 어떤 파일을 읽었는지, 어떤 파일이 후보인지, 어떤 위험이 있는지, 어떤 테스트가 필요할지 적게 합니다.

이 한 단계가 폭주를 크게 줄여요. 에이전트가 바로 수정하려고 할 때보다, 먼저 읽고 요약하게 했을 때 사람이 범위를 잡아주기 쉽습니다. 다음 [Lesson 4.4](/lessons/plan-with-ai) 의 구현 계획도 Scout Run 결과를 입력으로 삼으면 훨씬 구체적이 됩니다.

### 최신성 — 지도는 오래되면 오히려 위험하다

Repo Map 은 한 번 만들고 끝내는 문서가 아니에요. 폴더가 바뀌거나 테스트 명령이 바뀌면 업데이트해야 합니다. 오래된 지도는 없는 지도보다 위험할 수 있어요. 그래서 마지막에 “마지막 확인 날짜”와 “업데이트 책임자”를 넣습니다.

## 그래서 오늘 할 일

오늘 만들 것은 “코드베이스 읽기 가이드”입니다. 비전공자라도 자기 저장소를 열고 에이전트에게 이 순서로 읽으라고 줄 수 있어야 해요.

1. 저장소의 주요 폴더를 5~8개로 묶어 설명합니다.
2. 자주 쓰는 명령을 읽기, 검증, 실행으로 나눕니다.
3. 파일 이름, import, 테스트 위치 같은 로컬 관습을 적습니다.
4. 절대 바로 고치면 안 되는 위험 영역을 표시합니다.
5. Scout Run 프롬프트를 작성합니다.
6. 최근 이슈 하나를 골라 Scout Run 을 실제로 실행합니다.
7. 산출물 [`codebase-reading-guide.md`](/lessons/letting-ai-read-codebase) 를 저장하고 [Lesson 4.4](/lessons/plan-with-ai) 에서 계획 입력으로 씁니다.

핵심은 에이전트에게 모든 코드를 다 읽히는 것이 아니에요. “이 작업에는 이 근처를 먼저 봐야 한다”는 방향을 주는 겁니다. 사람에게 온보딩하듯 AI 에게도 온보딩이 필요해요.

## 마무리

코드베이스를 잘 읽히면 에이전트가 갑자기 똑똑해진 것처럼 보입니다. 사실은 추측할 공간이 줄어든 거예요. 다음 [Lesson 4.4](/lessons/plan-with-ai) 에서는 Scout Run 으로 얻은 후보 파일과 위험 리포트를 바탕으로, 구현 전에 합의하는 계획서를 만들겠습니다.
