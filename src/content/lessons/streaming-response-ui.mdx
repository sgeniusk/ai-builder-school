## 도입 — 학습자에게 말 거는 한 장면

챗 앱을 켰습니다. 질문을 입력하고 보내기를 눌렀어요. 화면에는 로딩 점 세 개만 깜박입니다. 5초, 10초, 20초가 지나도 아무 글자가 안 보입니다. 사용자는 앱이 멈춘 건지, 답을 쓰고 있는 건지 알 수 없어요. 결국 새로고침을 누르고 같은 질문을 다시 보냅니다. 서버는 같은 일을 두 번 하고, 사용자는 앱을 덜 믿게 됩니다.

AI 응답은 종종 길어요. 보고서 초안, 코드 리뷰, 긴 문서 요약은 한 번에 끝나지 않습니다. 그런데 사용자는 “완성된 답”보다 “앱이 살아 있다는 신호”를 먼저 원합니다. 한 글자씩, 한 문장씩 흘러나오면 기다림이 훨씬 견딜 만해져요.

오늘은 [Lesson 5.1](/lessons/connect-ai-api) 에서 만든 안전한 서버 호출을 스트리밍 UI로 바꿉니다. 모델이 답을 완전히 끝낼 때까지 기다리는 대신, 들어오는 조각을 즉시 화면에 붙여 보여주는 흐름을 만들 거예요.

## Lesson 5.1 다음에 왜 이 레슨인가

API 연결만으로는 앱의 느낌이 완성되지 않습니다. 서버 라우트가 안전하게 답을 가져오는 것은 1단계이고, 사용자가 그 답을 기다리는 경험을 설계하는 것이 2단계예요. 특히 AI 앱에서는 응답 시간이 들쭉날쭉합니다. 짧은 질문은 2초에 끝나고, 긴 요청은 30초도 걸립니다. 이 차이를 UI가 감당해야 해요.

Phase 4에서 배운 루프 관점으로 보면 스트리밍은 피드백 루프를 짧게 만드는 장치입니다. 사용자는 “내 요청이 처리되고 있다”는 증거를 계속 봅니다. 개발자도 중단, 재시도, 네트워크 끊김을 더 빨리 발견합니다. 다만 [Lesson 2.2](/lessons/enforcing-output-format) 에서 배운 구조화 출력과는 긴장이 있어요. 스트리밍 중간에는 JSON이 아직 완성되지 않았을 수 있기 때문입니다. 그래서 텍스트 스트림과 구조화 결과를 어떻게 나눌지 처음부터 생각해야 합니다.

## 핵심 개념 — Streaming 과 Cancellation

Streaming은 서버가 응답을 한 덩어리로 다 만든 뒤 보내는 대신, 작은 조각으로 나눠 보내는 방식입니다. 사용자는 “최종 답”을 기다리지 않고 “생성 중인 답”을 봅니다. 기술적으로는 Server-Sent Events, ReadableStream, chunked response 같은 이름을 만날 수 있어요. 지금은 이름을 외우기보다 흐름을 이해하면 됩니다. 서버가 조각을 보내고, 클라이언트가 그 조각을 누적해서 화면에 붙입니다.

누적 렌더링은 생각보다 조심해야 합니다. 새 조각이 올 때마다 전체 메시지를 다시 갈아끼우면 스크롤이 튀거나 커서가 어색해질 수 있어요. 긴 답에서는 자동 스크롤을 유지하되, 사용자가 위로 올려 읽고 있다면 억지로 아래로 끌어내리지 않는 배려도 필요합니다. 작은 챗 UI라도 이 차이가 앱의 완성도를 크게 바꿔요.

Cancellation은 사용자가 중단할 권리입니다. 질문을 잘못 보냈거나 답이 엉뚱한 방향으로 가고 있으면 멈출 수 있어야 해요. 브라우저에서는 `AbortController` 같은 도구로 요청을 취소합니다. 중요한 건 버튼만 바꾸는 게 아니라 실제 네트워크 요청이 끊기는지 확인하는 것입니다. 중단 버튼이 UI 장식에 그치면 비용은 계속 나갈 수 있어요.

## 그래서 오늘 할 일

오늘 만들 것은 “스트리밍 챗 UI 예제”입니다. 서버 쪽에는 스트리밍 버전의 라우트를 만들고, 클라이언트 쪽에는 들어온 조각을 하나의 메시지로 누적하는 작은 훅이나 컴포넌트를 만듭니다.

작업 순서는 단순합니다. 먼저 [Lesson 5.1](/lessons/connect-ai-api) 의 `/api/chat` 호출을 스트림을 반환하는 형태로 바꿉니다. 다음으로 UI 상태를 `idle`, `streaming`, `done`, `error`, `cancelled`처럼 나눕니다. 답변 조각이 도착할 때마다 현재 assistant 메시지에 이어 붙입니다. 중단 버튼은 요청을 실제로 취소하고, 재시도 버튼은 같은 입력을 다시 보내되 이전 실패 상태를 남깁니다.

산출물 [`streaming-ui-pattern.md`](/lessons/streaming-response-ui) 에는 상태표, 이벤트 흐름, 중단 버튼 점검, 스크롤 체크를 적습니다. JSON 형태의 결과가 필요한 기능이라면 중간 토큰을 그대로 파싱하려 하지 말고, 다음 [Lesson 5.3](/lessons/structured-output-handling) 에서 구조화 출력을 별도 단계로 다루세요.

## 마무리

스트리밍 UI는 화려한 효과가 아니라 신뢰의 신호입니다. 사용자는 기다리는 동안 앱이 일하고 있다는 증거를 보고, 필요하면 멈출 수 있어요. 오늘 만든 패턴은 앞으로 요약 앱, 코딩 도우미, 리서치 앱 어디에나 붙습니다. 다음 레슨에서는 이 흐르는 텍스트를 앱이 안전하게 읽을 수 있는 데이터로 바꿉니다. 빠르게 보이는 답과 정확하게 다루는 답은 서로 다른 문제이고, 둘 다 갖춰야 AI 앱이 실제 제품에 가까워집니다.
