## 아이디어를 제품으로 만드는 첫 단계

아이디어가 많다는 건 좋은 신호다. 하지만 아이디어가 많을수록 선택이 어려워진다.

```
"AI 일정 관리 앱 만들까?"
"아니면 이메일 자동 답장 툴?"
"아, SNS 콘텐츠 생성기가 더 팔릴 것 같은데..."
```

결국 셋 다 반쯤 만들고 멈춘다.

브리프(brief)는 이 순환을 끊는 도구다. 1페이지 브리프 하나가 "무엇을 왜 먼저 만드는가"를 명확하게 해준다.

---

## 브리프가 없으면 생기는 일

- 코딩 에이전트에게 "AI 일정 관리 앱 만들어줘"라고 하면 10가지 기능이 달린 앱이 나온다
- 그중 실제로 필요한 기능은 2개다
- 나머지 8개를 만드느라 4주가 사라진다

브리프는 "이것만 먼저"를 정하는 문서다.

---

## 1페이지 브리프의 4가지 칸

### 1. 문제 (페르소나 + 통증)

```
[페르소나]는 [상황]에서 [통증]을 겪는다.
```

예시:

```
프리랜서 디자이너는 클라이언트 미팅 후
회의 내용을 정리하는 데 매번 30분을 쓰는데,
나중에 다시 보면 뭘 결정했는지 기억이 안 난다.
```

"조금 불편하다" 수준이 아니라 "이게 없으면 지금도 고통스럽다" 수준이어야 제품이 된다.

### 2. Narrow MVP (기능 3개 이하)

전체 기능 목록을 쓰고, "이 기능이 없으면 MVP가 아닌가?"를 물어보며 줄인다.

```
✅ 회의 녹음 → 요약 자동 생성
✅ 결정 사항 자동 추출
❌ 캘린더 연동 (나중에)
❌ 팀 공유 기능 (나중에)
❌ 모바일 앱 (나중에)
```

### 3. 가장 위험한 가정

제품이 성공하려면 반드시 맞아야 하는 가정 중, 아직 확인 안 된 것.

```
가장 위험한 가정:
"프리랜서 디자이너가 회의 내용을 AI가 요약해주면 실제로 사용할 것이다."
```

이게 틀리면 MVP 전체가 무의미해진다.

### 4. 검증 방법 (2주 안에 가능한 것)

```
검증 방법:
→ 프리랜서 디자이너 5명에게 Zoom 인터뷰 + 간단한 프로토타입 시연
→ "이게 있으면 쓸 것 같아?" 직접 물어보기
```

## 브리프 작성 실습

**Step 1.** AI에게 아이디어 후보를 보여주고 물어본다:

```
나 지금 3개 아이디어 중에 뭘 먼저 만들지 고민 중이야.
[아이디어 1, 2, 3 설명]

각 아이디어에 대해 '이 제품이 없으면 누가 가장 힘든가?'를
한 줄로 써줘. 그리고 어떤 아이디어의 통증이 가장 선명한지 말해줘.
```

**Step 2.** 통증이 가장 선명한 아이디어를 고른다.

**Step 3.** 브리프 4칸을 채운다.

**Step 4.** 채운 브리프를 AI에게 검토받는다:

```
이 1페이지 브리프를 검토해줘.
1. 가장 위험한 가정이 더 있는지
2. 기능이 3개 이하로 정말 압축됐는지
3. 검증 방법이 2주 안에 가능한지
```

## 완성된 브리프 예시

```
문제:
프리랜서 디자이너는 클라이언트 미팅 후 회의 요약에
매번 30분을 쓰고, 나중에 결정 사항을 잊는다.

Narrow MVP (기능 3개):
1. 회의 녹음 파일 업로드
2. 요약 + 결정 사항 자동 추출
3. Notion에 자동 저장

가장 위험한 가정:
"프리랜서 디자이너가 회의 녹음을 실제로 올릴 것이다."

검증 방법:
→ 디자이너 5명 인터뷰 + Loom으로 시연 후 반응 확인
```

이 브리프가 있으면 코딩 에이전트에게 이렇게 말할 수 있다:

```
이 브리프에 맞는 최소 MVP를 만들어줘.
기능은 정확히 3개만. 그 이상은 넣지 마.
```

---

## 핵심 원칙

> 아이디어를 좁히는 게 브리프의 목적이 아니다.
> 검증받지 않은 가정을 드러내는 게 브리프의 진짜 목적이다.
