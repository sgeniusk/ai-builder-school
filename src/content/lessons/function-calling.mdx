## 도입 — 학습자에게 말 거는 한 장면

작은 문서 Q&A 앱을 만들었습니다. 사용자가 “이번 주 회의록에서 내가 해야 할 일만 찾아줘”라고 물으면 AI가 문서를 읽고 답해요. 여기까지는 Phase 6에서 만든 RAG 흐름과 비슷합니다. 그런데 사용자가 한 걸음 더 나갑니다. “그 할 일을 내 투두 목록에 추가해줘”라고 말해요.

일반 프롬프트만으로는 여기서 멈춥니다. 모델은 “추가했습니다”라고 말할 수는 있지만, 실제 투두 앱에 항목을 만들지는 못해요. 데이터베이스를 조회하거나, 캘린더에 일정을 넣거나, 이메일 초안을 저장하려면 모델 바깥의 함수가 필요합니다.

Function Calling은 이 경계를 여는 첫 번째 문입니다. AI에게 아무 도구나 쥐어주는 것이 아니라, 이름과 인자와 실행 조건이 정해진 작은 함수를 보여 주고, 모델이 필요할 때 그 함수를 호출하게 만드는 방식이에요. 직원에게 회사 카드를 줄 때 한도와 용처를 정하듯, 모델에게도 “이 도구는 이럴 때, 이런 값으로만 쓴다”는 규칙을 붙입니다.

## Phase 6 다음에 왜 이 레슨인가

[Lesson 6.4](/lessons/grounded-rag-answers) 에서는 RAG 답변에 출처를 붙이고, 근거가 없을 때 답하지 않는 법을 만들었어요. 그때의 앱은 주로 “찾고 답하는” 앱이었습니다. Phase 7에서는 AI가 찾은 결과를 바탕으로 다음 행동까지 제안하거나 실행합니다. 검색이 도구가 되고, 요약이 도구가 되고, 저장도 도구가 됩니다.

그 전에 [Lesson 5.3](/lessons/structured-output-handling) 의 습관을 다시 가져와야 해요. 함수 호출은 사실 구조화 출력의 다음 단계입니다. 모델이 자유 문장으로 “서울 날씨를 찾아볼게요”라고 말하는 대신, `getWeather({ city: "Seoul" })`처럼 앱이 읽을 수 있는 인자를 내보냅니다. [Lesson 2.2](/lessons/enforcing-output-format) 에서 배운 형식 강제도 여기서 다시 살아납니다.

중요한 변화는 실행 책임입니다. 구조화 출력은 대체로 데이터를 읽는 문제였지만, 함수 호출은 실제 일을 일으킬 수 있어요. 그래서 스키마와 검증과 로그가 함께 있어야 합니다.

## 핵심 개념 — Tool Schema 와 Argument Validation

Tool Schema는 모델에게 보여 줄 함수 설명서입니다. 함수 이름, 설명, 필요한 인자, 각 인자의 타입과 제한을 적습니다. 예를 들어 날씨 조회 함수라면 `city`는 문자열이고, `unit`은 `celsius` 또는 `fahrenheit` 중 하나라고 정해요. 설명이 흐리면 모델도 흐리게 호출합니다. “정보를 가져온다”보다 “도시 이름을 받아 현재 날씨 요약을 반환한다”가 낫습니다.

Argument Validation은 모델이 만든 인자를 실행 전에 확인하는 단계입니다. 모델이 `city`에 빈 문자열을 넣거나, `unit`에 엉뚱한 값을 넣을 수 있어요. 이때 바로 외부 API를 부르면 실패가 비용으로 이어집니다. 앱은 먼저 스키마 검증을 하고, 잘못된 값이면 실행하지 않고 다시 묻거나 오류를 기록해야 합니다.

실행 경로도 분리해야 합니다. 모델은 “어떤 함수를 어떤 인자로 부르고 싶다”까지만 말합니다. 실제 실행은 우리 코드가 맡아요. 이 분리가 있어야 권한을 조절할 수 있습니다. 조회 함수는 자동 실행해도 되지만, 이메일 발송이나 결제 변경 같은 함수는 사람 확인을 거쳐야 합니다.

마지막은 로그입니다. 어떤 사용자 요청에서 어떤 함수가 어떤 인자로 호출됐는지 남겨야 나중에 실패를 되짚을 수 있어요. 함수 호출은 모델의 말보다 더 강한 행동이기 때문에, 말풍선보다 실행 기록이 더 중요해집니다.

## 그래서 오늘 할 일

오늘 만들 것은 “Function Calling 예제”입니다. 가장 안전한 예제로 시작합니다. 외부 세계를 바꾸지 않는 조회 함수가 좋아요. 날씨 조회, 문서 검색, 상품 정보 조회처럼 읽기 전용 함수부터 다룹니다.

먼저 함수 하나를 고릅니다. 그다음 Tool Schema를 씁니다. 함수 이름은 동사로 시작하고, 설명은 한 문장으로 좁힙니다. 인자는 최소로 둡니다. 모델이 선택할 수 있는 값은 enum으로 제한하고, 필수 값과 선택 값을 나눕니다.

다음으로 검증 레이어를 둡니다. 모델이 만든 인자를 바로 실행하지 않고, 타입과 빈값과 허용 목록을 확인합니다. 실패하면 “인자가 부족하다”는 관찰을 모델에게 돌려주거나, 사용자에게 다시 물어봅니다. 마지막으로 실행 로그를 남깁니다. 요청 ID, 함수 이름, 인자, 성공 여부, 오류 메시지를 적으면 충분해요.

산출물 [`function-calling-recipe.md`](/lessons/function-calling) 에는 함수 후보, 스키마, 검증 규칙, 실행 권한, 테스트 질문을 한 장으로 정리합니다. 이 레시피가 다음 [Lesson 7.2](/lessons/mini-agent-loop) 의 재료가 됩니다. 단일 함수 호출이 안정적이어야, 여러 번 도구를 쓰는 작은 에이전트 루프도 안전하게 돌 수 있어요.

## 마무리

Function Calling은 AI에게 손을 달아주는 작업입니다. 하지만 손을 달아준다는 말은 책임도 커진다는 뜻이에요. 오늘은 하나의 읽기 전용 함수로 시작해, 스키마와 검증과 로그를 붙입니다. Phase 7의 나머지 레슨에서는 이 작은 도구가 반복 루프 안으로 들어가고, 권한 울타리와 인젝션 방어를 만나면서 제품에 넣을 수 있는 에이전트 구조로 자랍니다. Phase 11이나 Phase 12에서 실제 출시형 제품을 만들 때도 첫 질문은 같습니다. 이 AI가 어떤 도구를, 어떤 조건에서, 어떤 기록을 남기며 쓰는가입니다.
