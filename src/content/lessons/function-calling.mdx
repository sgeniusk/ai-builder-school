## "검색해 줘"라고 했는데 직접 검색한 게 아니었어요

챗봇에게 "오늘 서울 날씨 알려줘"라고 물었어요. 답이 그럴듯하게 나옵니다 — "맑고 기온 24도." 그런데 실제로 밖에 나가 보니 비가 옵니다. 모델은 날씨 API를 호출한 게 아니라, **학습 데이터에서 그럴듯한 답을 지어낸** 거예요. LLM은 텍스트를 잘 만들지만, 외부 세계에 손을 뻗어 데이터를 가져오거나 동작을 실행하는 능력은 기본적으로 없습니다.

{/* Source · DOC · Anthropic "Tool use overview" — https://docs.claude.com/en/docs/agents-and-tools/tool-use/overview */}
{/* Source · DOC · OpenAI "Function calling" — https://developers.openai.com/api/docs/guides/function-calling */}

그래서 등장한 패턴이 **Function Calling (Tool Use)**이에요. 모델에게 "네가 부를 수 있는 함수 목록"을 알려주면, 모델은 텍스트 대신 **함수 호출 요청**을 반환합니다. 우리 코드가 그걸 받아 실제 함수를 실행하고, 결과를 다시 모델에 돌려주는 구조예요.

오늘은 이 단발 도구 호출의 전체 흐름을 손으로 한 번 만들어 봅니다.

## 호출 흐름 한눈에 — 4단계

Function Calling의 전체 흐름은 네 단계입니다.

| 단계 | 누가 | 무엇 |
| --- | --- | --- |
| 1. 도구 정의 | 개발자 | 함수 이름 · 설명 · 파라미터 JSON Schema를 모델에 등록 |
| 2. 모델 판단 | LLM | 사용자 질문을 보고 "이 도구를 이 인자로 호출하겠다" 결정 |
| 3. 실행 | 개발자 코드 | 모델이 요청한 인자를 검증하고, 실제 함수를 실행 |
| 4. 결과 반환 | 개발자 → LLM | 실행 결과를 모델에 돌려주면, 모델이 최종 답을 생성 |

핵심은 **모델이 직접 함수를 실행하지 않는다**는 점이에요. 모델은 "이 함수를 이 인자로 불러달라"고 요청할 뿐, 실제 실행은 우리 코드가 합니다. 이 분리가 안전의 기본입니다.

## 도구 스키마 — 모델에게 메뉴판 주기

{/* Source · DOC · Anthropic "Tool use overview" — https://docs.claude.com/en/docs/agents-and-tools/tool-use/overview */}

모델에게 도구를 알려주는 방식은 **JSON Schema**예요. 함수 이름, 설명, 파라미터의 타입과 제약을 적어줍니다.

```json
{
  "name": "get_weather",
  "description": "지정한 도시의 현재 날씨를 조회합니다.",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "도시 이름 (예: 'Seoul', 'Busan')"
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "온도 단위"
      }
    },
    "required": ["city"]
  }
}
```

이 스키마가 **모델의 메뉴판**이에요. 메뉴판에 없는 함수는 모델이 부를 수 없습니다. 그래서 스키마를 잘 쓰는 게 곧 도구 호출의 품질을 결정해요.

스키마 작성 팁 세 가지:

- **description을 충분히 쓰세요.** 모델은 이 설명을 보고 "이 도구를 써야 하나?"를 판단합니다.
- **enum으로 선택지를 제한하세요.** 자유 텍스트보다 enum이 모델의 인자 정확도를 올립니다.
- **required를 명시하세요.** 빠뜨리면 모델이 필수 인자를 생략할 수 있어요.

## Anthropic vs OpenAI — 용어와 포맷 차이

{/* Source · DOC · Anthropic "Tool use overview" — https://docs.claude.com/en/docs/agents-and-tools/tool-use/overview */}
{/* Source · DOC · OpenAI "Function calling" — https://developers.openai.com/api/docs/guides/function-calling */}

두 벤더의 핵심 개념은 같지만, 용어와 API 포맷이 다릅니다.

| 항목 | Anthropic (Claude) | OpenAI (GPT) |
| --- | --- | --- |
| 용어 | Tool Use | Function Calling |
| 스키마 위치 | `tools[].input_schema` | `tools[].function.parameters` |
| 호출 응답 | `content[].type: "tool_use"` | `tool_calls[].function` |
| 결과 반환 | `role: "user"` + `type: "tool_result"` | `role: "tool"` + `tool_call_id` |
| 다중 호출 | 한 턴에 여러 tool_use 블록 가능 | `parallel_tool_calls` 옵션 |

포맷은 다르지만 **패턴은 동일**해요: 스키마 등록 → 모델 판단 → 개발자 실행 → 결과 반환. 한쪽을 이해하면 다른 쪽도 30분 안에 적용할 수 있습니다.

## 인자 검증 — 모델을 믿지 마세요

모델이 만들어낸 인자를 그대로 함수에 넣으면 안 돼요. 모델은 스키마를 "참고"하지만 100% 지키지는 않습니다.

[Lesson 06](/lessons/enforcing-output-format)에서 배운 3단 방어선의 2선(스키마 강제)이 여기서도 적용됩니다.

```typescript
// 인자 검증 예시 (Zod)
import { z } from "zod";

const GetWeatherArgs = z.object({
  city: z.string().min(1),
  unit: z.enum(["celsius", "fahrenheit"]).default("celsius"),
});

function executeGetWeather(rawArgs: unknown) {
  const args = GetWeatherArgs.parse(rawArgs); // 여기서 실패하면 모델에 에러 반환
  return fetchWeatherAPI(args.city, args.unit);
}
```

검증을 통과한 인자만 실행 경로로 보내세요. 실패하면 에러 메시지를 모델에 돌려줍니다 — 모델은 자기 실수를 고치는 데 생각보다 능숙해요.

## MCP — 도구를 표준으로 공유하기

{/* Source · DOC · Model Context Protocol — https://modelcontextprotocol.io/ */}

도구가 3개, 5개 늘어나면 "이 도구 정의를 매번 복사하는 게 맞나?" 의문이 들어요. **Model Context Protocol (MCP)**은 도구·리소스·프롬프트를 표준 인터페이스로 공유하는 프로토콜이에요. 서버 하나에 도구를 등록하면, MCP를 지원하는 어떤 클라이언트든 그 도구를 발견하고 호출할 수 있습니다.

오늘은 MCP 자체를 깊이 다루지는 않아요. 핵심만 기억하세요 — **Function Calling이 "모델이 도구를 부르는 방법"이라면, MCP는 "도구를 여러 모델/앱에서 재사용하는 방법"**입니다. L28에서 에이전트 루프를 만들 때 이 구분이 다시 나옵니다.

## 실패 패턴 3가지

단발 도구 호출에서 가장 흔한 실패 세 가지예요.

| 실패 | 원인 | 대응 |
| --- | --- | --- |
| 도구를 안 부름 | description이 모호하거나 질문과 안 맞음 | description에 "언제 이 도구를 써야 하는지" 명시 |
| 잘못된 인자 | 스키마에 enum/format이 없어 자유 해석 | required + enum + description 보강 |
| 환각 도구 호출 | 등록 안 한 함수를 부르려 함 | `tool_choice: "auto"` 대신 목록 제한, 검증 레이어 |

세 번째가 의외로 자주 생겨요. 모델이 학습 데이터에서 본 함수 이름을 지어내는 경우입니다. 스키마에 없는 도구 호출은 무조건 거부하는 게 안전해요.

## 그래서 오늘 만들 한 장/폴더

오늘 만들 건 `examples/function-calling/` 폴더예요. 미니 도구 1개(날씨 조회 또는 환율 조회)를 정의하고, LLM이 호출하는 전체 흐름을 동작하는 코드로 만듭니다.

| 파일 | 역할 |
| --- | --- |
| `tool-schema.json` | 도구 스키마 정의 (메뉴판) |
| `validate.ts` | Zod로 인자 검증 |
| `call.ts` | API 호출 → 도구 실행 → 결과 반환 전체 루프 |
| `test-cases.md` | 성공 케이스 2개 + 실패 케이스 2개 로그 |

이 네 파일이 한 자리에 있으면, 다음 레슨([미니 에이전트 루프](/lessons/mini-agent-loop))에서 도구를 여러 개로 늘리고 반복 호출하는 루프를 올릴 때 그 기반이 됩니다.
