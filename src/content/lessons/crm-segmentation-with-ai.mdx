## 세그먼트를 만들기 전에 물어야 할 것

고객 세그먼트를 만들면 뭘 할 건가? 이 질문에 답이 없으면 세그먼트를 아무리 정교하게 만들어도 쓸 수가 없다.

세그먼테이션의 출발점은 데이터가 아니라 **다음 행동**이다.

- 잠자는 고객에게 재활성화 캠페인을 보낼 것 → '잠자는 고객' 기준이 필요
- 로열티 프로그램 대상을 선정할 것 → '충성 고객' 기준이 필요
- 이탈 위험을 선제 방어할 것 → '이탈 신호' 기준이 필요

목표가 정해지면 기준이 역으로 나온다.

---

## AI와 세그먼트 규칙 공동 설계

기준을 혼자 만들지 않는다. AI와 대화로 설계한다.

**프롬프트 예시:**
```
우리는 B2C SaaS 서비스를 운영하고 있어요.
다음 달 재활성화 이메일 캠페인을 위해 '잠자는 고객'을 정의해야 해요.
보유 데이터: last_login_date, last_purchase_date, email_open_rate(30일), total_purchases

어떤 기준으로 '잠자는 고객'을 정의하면 좋을까요?
조건을 불리언으로 작성해주세요.
```

AI가 제안한 기준을 그대로 쓰지 않는다. **비즈니스 맥락에 맞게 조정한다.**

예를 들어 AI가 `last_login_date > 60일`을 제안했지만 우리 서비스가 월 1회 이용이 정상이라면 기준을 `> 90일`로 바꿔야 한다.

---

## 규칙을 불리언으로 문서화하기

모호한 규칙은 사람도, AI도 일관되게 적용하지 못한다.

**나쁜 예:**
> 잠자는 고객 = 요즘 안 들어오는 고객

**좋은 예:**
```
잠자는 고객 (DORMANT):
  - last_login_date < 현재일 - 60일
  - AND (last_purchase_date IS NULL OR last_purchase_date < 현재일 - 90일)
  - AND email_open_rate_30d < 0.05

우선순위: last_login이 최신이면 DORMANT 제외
          (로그인하지만 구매 안 한 경우는 별도 세그먼트)
```

불리언 조건으로 적으면 세 가지가 가능해진다.
- 사람이 수동으로 적용할 수 있다
- AI 배치 레이블링에 그대로 넣을 수 있다
- SQL WHERE 조건으로 변환할 수 있다 (L44 연결)

---

## 엣지 케이스 선제 정의

규칙 문서가 있어도 AI는 애매한 케이스에서 흔들린다. 미리 예시를 만들어둔다.

| 케이스 | 데이터 상태 | 올바른 레이블 | 이유 |
|--------|------------|--------------|------|
| 최근 로그인했지만 오래 전 구매 | login=5일 전, purchase=120일 전 | ACTIVE | 로그인 기준 우선 |
| 구매 없이 이메일만 열어봄 | login=45일 전, purchase=NULL, open_rate=0.3 | ENGAGED_NO_PURCHASE | 별도 세그먼트 |
| 모든 데이터 누락 | NULL | UNKNOWN | 레이블 불가 |
| 가입 3일 이내 | created_at=3일 전 | NEW | 세그먼트 판단 보류 |

이 예시를 배치 프롬프트 안에 포함시킨다.

---

## 배치 레이블링 프롬프트 구조

```
[세그먼트 규칙]
ACTIVE: last_login_date < 14일 AND (last_purchase_date < 30일 OR total_purchases > 3)
DORMANT: last_login_date >= 60일 AND last_purchase_date >= 90일
ENGAGED_NO_PURCHASE: login < 30일 AND total_purchases = 0
CHURNED: last_login_date >= 180일
UNKNOWN: 위 조건 해당 없음 또는 필수 데이터 누락

[엣지 케이스 예시]
... (위 표의 내용)

[레이블링 지시]
아래 고객 데이터를 위 규칙으로 레이블링하세요.
- 각 행마다 세그먼트 이름 하나만 출력하세요
- 규칙에 맞는 게 없으면 반드시 UNKNOWN으로 표시하세요
- 이유를 추측하지 마세요

[데이터]
user_id, last_login_date, last_purchase_date, email_open_rate_30d, total_purchases
1001, 2025-04-20, 2025-03-15, 0.12, 5
1002, 2025-02-01, 2024-11-20, 0.01, 2
...
```

**핵심은 UNKNOWN 폴백이다.** 규칙에 해당 없으면 UNKNOWN — 이 지시가 없으면 AI가 가장 비슷한 세그먼트를 추측해 붙인다. UNKNOWN이 많이 나오면 규칙이 불완전하다는 신호다.

---

## 검증 루프

10~20행 샘플로 먼저 돌린다.

1. 가상 데이터 10행 생성 (각 세그먼트마다 2개씩 + 엣지 케이스 2개)
2. 손으로 레이블 계산
3. AI 레이블링 결과와 비교
4. 불일치 발견 → 규칙 수정 or 엣지 케이스 추가
5. 재검증 후 전체 데이터 적용

한 번에 전체 데이터를 돌리지 않는다. 작게 검증하고 확장한다.

## 산출물 구성

**batch-labeling-prompt.md** — 규칙 + 엣지 케이스 + UNKNOWN 폴백이 포함된 재사용 배치 프롬프트. 매달 새 데이터를 이 프롬프트에 넣어 돌린다.

## 핵심 원칙

> 세그먼트는 데이터에서 만들지 않는다. 다음 행동에서 역으로 설계한다.
> 규칙이 모호하면 AI도 일관되게 레이블링하지 못한다.
> UNKNOWN이 많으면 규칙을 고칠 때다.
