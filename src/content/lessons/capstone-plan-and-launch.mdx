## "많이 배웠는데, 보여줄 게 없어요"

32개 레슨을 거의 다 끝냈습니다. 프롬프트를 구조화하고, 출력을 강제하고, 긴 문서를 먹이고, 자동화를 짜고, API를 연결하고, RAG를 붙이고, 에이전트를 돌리고, 비용까지 관리했어요. 그런데 동료가 "그래서 뭐 만들었어?" 하고 물으면 — 말문이 막힙니다. 레슨마다 나온 산출물은 있지만, **처음부터 끝까지 내가 기획하고, 만들고, 사람한테 써 보게 한 것**은 아직 없어요.

{/* Source · BOOK · Eric Ries, "The Lean Startup" (2011) — https://theleanstartup.com · MVP와 Build-Measure-Learn 루프 */}

오늘은 그 "보여줄 것"을 만드는 레슨이에요. 캡스톤 — 졸업 작품입니다. 4주 안에 하나의 AI 앱을 기획하고, 만들고, 런칭하고, 회고합니다.

## 지금까지 우리가 만든 것

시작하기 전에 한 번 돌아볼게요. Phase 1부터 11까지, 레슨마다 손에 남은 산출물이 있었습니다.

| Phase | 무엇을 배웠나 | 손에 남은 것 |
| --- | --- | --- |
| 1 — 기초 | AI의 강·약, 신뢰 체크 | 5분 트러스트 체크리스트, 강·약 매핑 지도 |
| 2 — 프롬프트 | 4축 구조, 출력 강제, 장문 처리, 라이브러리 | prompt-library v2 폴더, 3단 방어선 예제 |
| 3 — 자동화 | 보고서 초안, 분류기, 슬랙 봇 | 일상 자동화 파이프라인 |
| 4 — 코딩 에이전트 | 코딩 에이전트 셋업, 이슈→브리프, 코드베이스 읽기 | AI-assisted 개발 워크플로우 |
| 5 — AI 개발 | API 연결, 스트리밍 UI, 구조화 출력, 대화 저장 | 작동하는 AI 챗 앱 |
| 6 — RAG | 임베딩, 청킹, 벡터 검색, 근거 기반 답변 | RAG 파이프라인 |
| 7 — 에이전트 | 함수 호출, 미니 에이전트 루프, 도구 권한 | 도구를 쓰는 에이전트 |
| 8 — 보안 | 프롬프트 인젝션 방어 | 보안 체크리스트 |
| 9~10 — 팀/평가 | 팀 도입, 품질 평가 | 도입 플레이북, 평가 프레임워크 |
| 11 — 비용 | 토큰 비용 추적, 모델 라우팅 | cost-tracker 폴더 |

이 모든 조각이 오늘의 캡스톤에 재료로 들어갑니다. 하나도 버릴 게 없어요.

## Narrow MVP — 가장 좁게, 가장 빠르게

{/* Source · BOOK · Eric Ries, "The Lean Startup" (2011) — https://theleanstartup.com · MVP / Build-Measure-Learn 관점 */}

캡스톤이라고 거대한 걸 만들 필요가 없습니다. Eric Ries의 Lean Startup이 말하는 MVP(Minimum Viable Product)의 핵심은 **가장 위험한 가정을 테스트할 수 있는, 가능한 한 단순한 것**이에요.

AI 캡스톤에서 "가장 위험한 가정"은 대개 이 셋 중 하나입니다:

1. **이 문제를 진짜 겪는 사람이 있는가?** (문제 가정)
2. **AI가 이 문제를 사람보다 빠르게/싸게 풀 수 있는가?** (기술 가정)
3. **이 사람이 이 해결책에 시간이나 돈을 쓸 의향이 있는가?** (가치 가정)

셋 중 가장 불확실한 하나를 골라, 그것만 테스트하는 앱을 만듭니다. 나머지는 다음 이터레이션에서 확인해요.

## 4주 계획 — Week by Week

{/* Note: 4주 프레임워크는 이 학교 운영자가 정리한 캡스톤 로드맵이며, 외부 표준 프레임워크는 아닙니다. */}

### Week 1 — Spec & Design (명세와 설계)

| 항목 | 무엇을 하나 |
| --- | --- |
| 문제 정의 | 누구의 어떤 문제를 AI로 푸는가? (한 문장) |
| 기술 가정 | AI가 이 문제를 풀 수 있다는 근거 (프로토타입 1개로 증명) |
| 범위 확정 | "이것만 되면 v1" — 기능 3개 이하 |
| 와이어프레임 | 화면 3개 이하의 흐름도 |
| 기술 스택 | [Lesson 19](/lessons/connect-ai-api) API 연결 + [Lesson 20](/lessons/streaming-response-ui) 스트리밍 기반 |

Week 1의 산출물은 **1페이지 브리프**예요. 이 한 장이 나머지 3주의 나침반이 됩니다.

### Week 2 — Build (핵심 기능 구현)

{/* Source · DOC · Vercel "Deploying" — https://vercel.com/docs */}

| 항목 | 무엇을 하나 |
| --- | --- |
| 핵심 루프 | 사용자 입력 → AI 처리 → 결과 표시의 한 바퀴 |
| 데이터 흐름 | [Lesson 22](/lessons/conversation-storage-design) 대화 저장 + [Lesson 26](/lessons/grounded-rag-answers) RAG (필요 시) |
| 에이전트 (선택) | [Lesson 28](/lessons/mini-agent-loop) 미니 에이전트 루프 (필요 시) |
| 배포 | Vercel / Netlify / Railway 등에 첫 배포 — "URL이 존재하는 상태" |

Week 2 끝에는 **작동하는 URL**이 있어야 합니다. 예쁠 필요 없어요. 돌아가면 됩니다.

### Week 3 — Test & Iterate (테스트와 반복)

| 항목 | 무엇을 하나 |
| --- | --- |
| 첫 사용자 5명 | 지인·동료·커뮤니티에서 5명을 모아 써 보게 함 |
| 피드백 수집 | "가장 유용했던 순간 / 가장 답답했던 순간 / 계속 쓸 의향" 3문항 |
| 버그 수정 | 크리티컬 버그 우선 — [Lesson 18](/lessons/bug-reproduction-loop) 버그 재현 루프 활용 |
| 비용 확인 | [Lesson 31](/lessons/ai-app-cost-and-usage) cost-tracker로 실제 비용 측정 |
| 보안 점검 | [Lesson 30](/lessons/prompt-injection-defense) 프롬프트 인젝션 방어 체크 |

5명이 써 보면 **혼자서는 안 보이던 문제가 보입니다**. "이게 뭐 하는 건지 모르겠어요"가 가장 흔한 피드백이에요 — 그건 문제 정의가 명확하지 않다는 신호입니다.

### Week 4 — Launch & Retrospective (런칭과 회고)

{/* Source · DOC · Product Hunt "Launching on Product Hunt" — https://www.producthunt.com/launch */}

| 항목 | 무엇을 하나 |
| --- | --- |
| README | 무엇을 · 왜 · 어떻게 쓰는지 · 스크린샷 2장 |
| 공개 | GitHub 저장소 공개 + 데모 URL |
| 공유 | 커뮤니티 1곳(인디해커스 / 트위터 / 디스코드 / 오픈카톡)에 소개 글 |
| 회고 | 지표·피드백·한계·다음 단계를 `retrospective.md`에 기록 |

런칭은 "완벽한 상태"가 아니라 **"피드백을 받을 수 있는 상태"** 에서 합니다. 완벽을 기다리면 런칭은 영원히 안 옵니다.

## 캡스톤 아이디어 — 어디서 시작할지 모르겠다면

아이디어가 안 떠오르면, 이 커리큘럼에서 만든 산출물을 확장하는 것도 좋은 출발이에요.

| 아이디어 | 기반 레슨 | 핵심 기술 |
| --- | --- | --- |
| 사내 문서 Q&A 봇 | L23~L26 (RAG) | 임베딩 + 벡터 검색 + 근거 인용 |
| 회의록 → Action Item 자동화 | L06 (출력 강제) + L09 (보고서 자동화) | 구조화 출력 + 파이프라인 |
| 개인 프롬프트 라이브러리 앱 | L05, L08 (프롬프트 라이브러리) | CRUD + 변수 치환 |
| 코드 리뷰 어시스턴트 | L13~L15 (코딩 에이전트) | 코드베이스 읽기 + 이슈 브리프 |
| AI 비용 대시보드 | L31 (비용 관리) | 토큰 로깅 + 시각화 |

어떤 아이디어든 괜찮아요. 중요한 건 **4주 안에 5명이 써 보는 상태까지 가는 것**입니다.

## 회고 — 만든 것보다 중요한 것

{/* Source · BOOK · Eric Ries, "The Lean Startup" (2011) — https://theleanstartup.com · Build-Measure-Learn 루프의 Learn 단계 */}

4주가 끝나면 `retrospective.md`를 씁니다. Lean Startup의 Build-Measure-Learn 루프에서 **Learn이 가장 중요한 단계**예요.

회고에 들어갈 5가지:

1. **처음 가정 vs 실제** — "이 문제를 겪는 사람이 있을 것이다"가 맞았는가?
2. **사용자 피드백 요약** — 5명의 3문항 답변에서 패턴
3. **기술적 한계** — AI가 못 한 것, 예상보다 비용이 든 것
4. **내가 과소평가한 것** — 시간이 가장 많이 들어간 작업
5. **다음 이터레이션** — v2에서 가장 먼저 바꿀 것 1가지

이 회고가 남아 있으면 6개월 뒤 다시 볼 때 "그때 내가 뭘 배웠지"가 선명해요. 없으면 증발합니다.

## 커리큘럼을 닫으며

이 레슨이 32번째이자 마지막입니다.

Phase 1에서 "AI가 잘하는 것과 못하는 것"을 구분하는 것부터 시작했어요. 그때는 프롬프트 한 줄 쓰는 게 전부였죠. 지금은 프롬프트를 구조화하고, 출력을 강제하고, 긴 문서를 청킹하고, API를 연결하고, 스트리밍 UI를 붙이고, RAG로 근거를 대고, 에이전트로 도구를 쓰고, 보안을 점검하고, 비용을 추적하고, 그 위에 하나의 제품을 세울 수 있습니다.

"AI 빌더"란 AI를 잘 아는 사람이 아니에요. **AI로 무언가를 만들어 사람에게 쓰게 하는 사람**입니다. 이 캡스톤이 그 첫 번째 증거가 됩니다.

4주 뒤, 작동하는 URL 하나와 회고 한 장을 들고 돌아와 주세요. 그게 졸업입니다.

시작해 볼까요?
