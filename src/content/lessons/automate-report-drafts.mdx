## "매주 금요일, 보고서 틀 채우느라 분석할 시간이 없어요"

금요일 오후 4시. 주간 보고서를 써야 합니다. 구글 독스를 열고 지난주 보고서를 복사해 날짜를 고치고, 이번 주 수치를 채우고, 요약을 다시 씁니다. 1시간 반이 지나면 내용 자체보다 **형식을 맞추는 데** 시간을 더 쓴 자신을 발견해요. 정작 "이 수치가 왜 이렇게 나왔는지"를 생각할 시간은 없었습니다.

이게 보고서의 함정이에요. 형식은 매번 같은데 매번 새로 채워야 하고, 채우느라 분석이 밀립니다. [Lesson 05](/lessons/structure-of-good-prompts)에서 만든 4축 프롬프트와 [Lesson 08](/lessons/build-personal-prompt-library)의 라이브러리가 있으니, 오늘은 그걸 **실제 보고서 파이프라인**에 붙여 봅시다.

## 보고서는 왜 자동화하기 좋은가

{/* Source · DOC · Anthropic "Prompt engineering overview" — https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/overview */}
{/* Source · DOC · Anthropic "Context windows" — https://platform.claude.com/docs/en/build-with-claude/context-windows */}

보고서에는 자동화가 잘 먹히는 세 가지 조건이 다 들어 있어요.

1. **형식이 반복된다** — 섹션 구성, 톤, 길이가 매번 비슷. 회사마다 템플릿이 정해져 있죠.
2. **입력이 정형화되어 있다** — 수치 데이터, 이전 보고서, 회의 메모가 소스.
3. **사람이 검토한다** — 최종 검수가 반드시 붙으니 AI가 중간 실수를 해도 잡을 수 있어요.

문제는 "그냥 보고서 써 줘"라고 하면 회사 템플릿과 전혀 다른 답이 나온다는 거예요. AI가 우리 템플릿을 모르니까요.

## 템플릿 그라운딩 — AI에게 "우리 양식"을 가르치기

{/* Note: 템플릿 그라운딩(Template Grounding)은 이 학교 운영자가 정리한 명칭이며, 외부 표준명은 아닙니다. */}

핵심은 **과거 보고서 2~3개를 컨텍스트로 함께 주는 것**이에요. AI가 패턴을 읽고 "우리 회사 보고서는 이런 모양이구나"를 한 호출 안에서 학습합니다.

```
Context:
- 아래는 우리 팀의 지난 3주 주간보고서입니다.
- [보고서 1 전문]
- [보고서 2 전문]
- [보고서 3 전문]

Task:
위 보고서의 공통 구조(섹션 순서, 톤, 길이, 수치 표현 방식)를 분석하고,
아래 이번 주 데이터로 같은 형식의 주간보고서 초안을 작성해 주세요.

이번 주 데이터:
- [수치/이벤트/메모]
```

[Lesson 07](/lessons/feeding-long-documents)에서 배운 컨텍스트 윈도우를 떠올려 보세요. 보고서 3개면 보통 3000~6000 토큰. 현재 모델들의 윈도우에 충분히 들어갑니다. 보고서가 길다면 핵심 섹션만 골라 넣는 Selective 전략을 쓰세요.

## 3단 파이프라인 — 초안 → 자기 검토 → 사람 검수

한 번에 완벽한 보고서를 뽑으려고 하면 실패합니다. 대신 **세 단계로 나누면** 각 단계의 부담이 줄어요.

| 단계 | 무엇 | 누가 |
| --- | --- | --- |
| 1. 초안 생성 | 템플릿 + 데이터 → 초안 | AI |
| 2. 자기 검토 | 초안을 다시 읽고 빠진 섹션·수치 오류 체크 | AI (다른 호출) |
| 3. 사람 검수 | 톤·맥락·민감 정보 최종 확인 | 사람 |

2단계가 핵심이에요. 같은 AI에게 "이 초안에서 빠진 섹션, 수치 오류, 톤 불일치를 찾아줘"라고 한 번 더 시키면 1단계에서 빠뜨린 것의 상당 부분이 잡힙니다. [Lesson 06](/lessons/enforcing-output-format)의 3단 방어선과 같은 원리 — 한 단계가 놓치면 다음 단계가 잡아 줍니다.

## 자기 검토 프롬프트 — "상사의 눈으로 읽어줘"

{/* Source · DOC · Anthropic "Prompt engineering overview" — https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/overview */}

자기 검토를 시킬 때 가장 잘 먹히는 패턴은 **검토자 역할을 명시하는 것**이에요.

```
Context:
- 당신은 우리 팀의 팀장입니다. 아래 주간보고서 초안을 검토해 주세요.
- 우리 팀의 보고서 기준: [보고서 템플릿 또는 과거 보고서 1개]

Task:
아래 초안을 읽고 다음을 확인하세요.
1. 템플릿의 모든 섹션이 채워져 있는가
2. 수치에 단위·기간이 명시되어 있는가
3. 고유명사(프로젝트명, 사람 이름)가 정확한가
4. 톤이 우리 팀 기존 보고서와 일치하는가

Output:
- 문제 목록 (섹션별 · 심각도 상/중/하)
- 수정 제안 (원문 → 수정안)
```

이 검토 프롬프트를 [Lesson 08](/lessons/build-personal-prompt-library)의 라이브러리에 넣어두면 매주 복사 없이 호출할 수 있어요.

## 수치 검증 — AI가 가장 약한 지점

보고서 자동화에서 가장 주의할 지점이 **수치**예요. AI는 텍스트 생성에는 강하지만, 수치 계산과 인용에서는 실수가 잦습니다. 반드시 확인해야 할 체크리스트:

| 확인 항목 | 왜 위험한가 | 대응 |
| --- | --- | --- |
| 수치의 출처 | AI가 "그럴듯한 숫자"를 만들어냄 | 원본 데이터와 대조 |
| 증감률 계산 | 분모/분자를 뒤바꾸거나 기간을 혼동 | 수식을 직접 확인 |
| 고유명사 | 프로젝트명·담당자를 비슷한 이름으로 바꿈 | 원본 명단과 대조 |
| 날짜·기간 | "이번 주"와 "지난 주"를 혼동 | 날짜를 명시적으로 입력 |

이 네 가지만 잡아도 보고서 자동화의 신뢰도가 눈에 띄게 올라가요.

## 그래서 오늘 만들 한 폴더

오늘 만들 건 `report-pipeline/` 폴더예요. 이 한 폴더가 매주 보고서의 시작점이 됩니다.

| 파일 | 역할 |
| --- | --- |
| `template.md` | 우리 팀 보고서 템플릿 (섹션 구성 + 톤 가이드) |
| `generate-draft.md` | 1단계 — 초안 생성 프롬프트 (4축 구조) |
| `self-review.md` | 2단계 — 자기 검토 프롬프트 (검토자 역할) |
| `checklist.md` | 3단계 — 사람 검수용 체크리스트 (수치·고유명사·톤) |

이 네 파일이 한 자리에 있으면, 금요일 오후의 보고서 루틴이 바뀝니다. 1시간 반이 걸리던 일이 30분 안에 끝나고, 남는 시간에 "이 수치가 왜 이렇게 나왔는지"를 생각할 수 있어요.
