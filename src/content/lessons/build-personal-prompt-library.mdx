## 그 잘 만든 프롬프트, 어디 두셨어요?

지난 주에 *진짜 좋게 나오는* 회의록 요약 프롬프트를 만들었어요. 한 시간 정도 다듬어서 결과가 깔끔했죠. 그런데 오늘, 같은 작업을 하려고 보니 — 어디 있죠?

ChatGPT 사이드바? Claude 대화 기록? Notion 어딘가? Slack DM 자기 자신? Apple Notes? 메신저 PC 백업?

찾는 데 15분, 못 찾으면 다시 만들기. 이게 매주 반복되면 *프롬프트 자산이 매번 0 부터 다시 쌓이는* 상태예요. Phase 2 의 마지막 한 가지가 이 누수를 막는 *시스템* 입니다.

## Phase 2 의 마무리 — 흩어진 자산을 한 곳으로

[Lesson 2.1](/lessons/structure-of-good-prompts) 의 4축, [Lesson 2.2](/lessons/enforcing-output-format) 의 스키마, [Lesson 2.3](/lessons/feeding-long-documents) 의 입력 전략 — 이 세 가지가 모이면 *재현 가능한 좋은 프롬프트* 가 만들어져요.

문제는 *그 좋은 프롬프트가 어디 있냐* 죠. 자산이 흩어진 채로는 4축도, 스키마도, 전략도 *매번 다시 짜야* 합니다. 결국 안 쓰게 되고, 안 쓰니 잊혀지고.

오늘은 그 자산을 *한 곳* 에 모으고 *버전* 을 추적해, **Phase 3 자동화로 곧장 넘길 수 있는 형태** 로 정리합니다.

## 핵심 개념 — Library / Variable / Version

### Prompt Library — *카테고리 + 파일 단위 저장소*

흩어진 프롬프트를 *역할별 폴더 + 파일* 로 정리합니다.

```
prompt-library/
├── README.md
├── research/
│   ├── literature-review.md
│   └── topic-explainer.md
├── reports/
│   ├── weekly-status.md
│   └── action-items.md
└── reviews/
    ├── code-review.md
    └── design-feedback.md
```

각 파일은 한 가지 작업만 책임. 폴더 구조 자체가 *작업 카탈로그* 가 됩니다.

### Template Variable — *치환 가능한 자리표시자*

프롬프트 안에 `{audience}`, `{topic}`, `{tone}` 같은 *변수* 를 두고, 사용 시 채워 넣어요.

```markdown
당신은 {audience} 를 위해 글을 씁니다.
주제는 {topic} 이고, 톤은 {tone} 입니다.
...
```

변수가 있으면 *한 템플릿 → 여러 상황 재사용*. 변수가 없으면 매번 복붙·수정.

### Version Front Matter — *추적 가능한 버전*

각 템플릿 상단에 메타 정보를 YAML front matter 로.

```yaml
---
purpose: "주간 보고 초안 자동 생성"
version: "1.3"
last_tested: 2026-05-01
inputs: [topic, key_metrics, blockers]
output_format: "Markdown — 4 섹션"
---
```

수정될 때마다 version 을 올리고, 몇 번 써보면 *어떤 버전이 가장 안정적이었는지* 가 자연스럽게 기록돼요. *팀에 공유* 할 땐 이 메타가 핵심 — *왜 이 버전인지* 가 한눈에 보이니까.

## 그래서 오늘 할 일

매주 5번 이상 호출되는 작업 5~7개 를 골라 *프롬프트 라이브러리* 한 폴더로 이관합니다.

권장 작업 흐름,

1. 지금 흩어져 있는 프롬프트를 30분만 *전수 조사* — Slack DM, Notion, ChatGPT 사이드바, 메모장 모두 (10분 권장하지만 보통 30분 걸림).
2. 카테고리 3~4 개 결정 — 자기 일에서 가장 자주 부르는 묶음으로. 예, *research / reports / reviews* (5분).
3. `prompt-library/` 폴더 만들고 카테고리 하위 폴더 생성 (2분).
4. 템플릿 Front Matter 표준 정의 — 위 예시 그대로 시작 (5분).
5. 자주 쓰는 5개 를 *4축 구조 + Front Matter* 로 이관 (40분).
6. README 작성 — *이 라이브러리는 어떻게 쓰나*, *언제 어떤 템플릿을 부르나* (10분).
7. Git 저장소 또는 Notion DB 에 올림 (5분).

산출물은 [`prompt-library-skeleton.md`](/lessons/build-personal-prompt-library) — 폴더 구조 + Front Matter 표준 + 첫 템플릿 예시 한 페이지.

## 마무리 — Phase 2 를 닫으며

여기까지 오면 매주 일과가 이렇게 흐릅니다.

> *작업 시작* → *라이브러리에서 템플릿 검색* → *변수 채워서 호출* → *결과 받음* → *5분 트러스트 체크* → *발송*

같은 작업을 *3번째* 부터는 5분 안에 끝나요. 첫 번째 만드는 데 한 시간이었던 게, 다섯 번째엔 5분. 그 차이가 *프롬프트 자산화* 의 효과입니다.

**Phase 3 (AI Work OS)** 부터는 이 라이브러리를 *자동화 파이프라인* 에 연결해요. [Lesson 3.1](/lessons/automate-report-drafts) 에서 *주간 보고서 자동 초안*, [Lesson 3.2](/lessons/meeting-notes-pipeline) 에서 *회의록 → Action Item* 의 자동 라우팅. 오늘 만든 라이브러리가 그 모든 자동화의 *재료 창고* 가 됩니다.

흩어진 자산을 모으는 데 90분. 이후 평생 회수하는 시간은 그 100 배 이상이에요.
