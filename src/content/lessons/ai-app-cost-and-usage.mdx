## "어느 날 갑자기 API 청구서가 월급보다 커졌어요"

사이드 프로젝트로 AI 챗봇을 만들었습니다. 친구 5명이 써 봤을 때는 하루 $0.30이었어요. 재미있다며 회사 슬랙에 공유했더니 사용자가 50명으로 늘었고, 일주일 뒤 청구서를 열어보니 하루 $15. 한 달이면 $450. 아직 수익은 $0인데요.

{/* Source · DOC · OpenAI "Pricing" — https://openai.com/api/pricing/ */}
{/* Source · DOC · Anthropic "Pricing" — https://www.anthropic.com/pricing */}

비용 문제는 프로토타입에서는 안 보이다가 **사용량이 늘어날 때 갑자기 보입니다**. [AI API 붙이기](/lessons/connect-ai-api)에서 첫 호출을 성공시켰다면, 오늘은 그 호출이 **지갑을 어떻게 열고 닫는지** 관리하는 법을 배웁니다.

오늘 만들 건 `examples/cost-tracker/` — 호출마다 토큰과 비용을 로깅하고, 일별 리포트를 뽑는 미니 트래커예요.

## 토큰 단가 — 인풋과 아웃풋은 가격이 다르다

{/* Source · DOC · OpenAI "Pricing" — https://openai.com/api/pricing/ */}
{/* Source · DOC · Anthropic "Pricing" — https://www.anthropic.com/pricing */}

LLM API는 **토큰 단위로 과금**됩니다. 그런데 인풋 토큰과 아웃풋 토큰의 가격이 다릅니다 — 보통 아웃풋이 인풋보다 비싸요.

| 항목 | 설명 |
| --- | --- |
| 인풋 토큰 | 시스템 프롬프트 + 사용자 메시지 + 대화 히스토리 |
| 아웃풋 토큰 | 모델이 생성하는 답변 |
| 가격 차이 | 일부 모델에서 아웃풋이 인풋보다 비쌈 (벤더 페이지 직접 확인) |

(모델별 정확한 단가는 위 공식 pricing 페이지에서 확인하세요. 분기마다 바뀝니다.)

이 구조를 알면 비용 최적화의 방향이 보입니다 — **아웃풋을 줄이는 게 인풋을 줄이는 것보다 단가 기준으로는 효과가 큽니다**. [출력 형식(Output Format)을 강제하는 법](/lessons/enforcing-output-format)에서 배운 "스키마 강제"가 여기서도 빛을 발해요. JSON 스키마로 출력을 고정하면 모델이 불필요한 서두를 안 붙여서 아웃풋 토큰이 줄어듭니다.

## 비용이 폭발하는 3가지 패턴

사용자가 느는 것 외에도, 비용을 키우는 숨은 패턴이 있습니다.

### 1. 대화 히스토리 누적

매 턴마다 이전 대화 전체를 인풋으로 보내면, 대화가 길어질수록 인풋 토큰이 기하급수로 늡니다. [긴 문서를 AI에게 먹이는 법](/lessons/feeding-long-documents)의 "컨텍스트 윈도우" 개념을 떠올려 보세요 — 윈도우에 다 들어간다고 다 넣으면 비용도 따라갑니다.

**대응**: 대화 요약(summarization) 또는 슬라이딩 윈도우. 최근 N턴만 보내고 이전은 요약문으로 대체합니다.

### 2. 반복 질문에 매번 새 호출

같은 질문이 반복되는데 매번 모델을 호출합니다. FAQ 봇이라면 상위 20개 질문이 전체 트래픽의 대부분을 차지하는 경우가 많아요.

**대응**: 응답 캐싱. 동일하거나 유사한 질문에 대해 이전 답변을 재사용합니다.

### 3. 모든 작업에 최대 모델 사용

요약 한 줄에도 가장 비싼 모델을 쓰면 비용 대비 효과가 떨어집니다.

**대응**: 모델 라우팅. 작업 난이도에 따라 모델을 다르게 배정합니다.

## 비용을 낮추는 4가지 전략

{/* Note: 4가지 비용 최적화 전략은 일반 관행을 이 학교 운영자가 정리한 것이며, 외부 표준명은 아닙니다. */}

### 전략 1 — 프롬프트 캐싱

{/* Source · DOC · Anthropic "Prompt caching" — https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching */}
{/* Source · DOC · OpenAI "Prompt caching" — https://platform.openai.com/docs/guides/prompt-caching */}

긴 시스템 프롬프트나 문서를 반복 전송할 때, 벤더의 캐싱 기능을 쓰면 캐시 히트 시 인풋 비용이 크게 줄어듭니다. 특히 시스템 프롬프트가 수천 토큰인 앱에서 효과적이에요.

### 전략 2 — 응답 캐싱 (앱 레벨)

같은 질문 → 같은 답이 예상되는 시나리오에서는 앱 레벨에서 응답을 캐싱합니다. Redis 같은 캐시 스토어에 `(질문 해시, 모델, 답변)` 을 저장해 두고, 히트하면 API 호출을 건너뜁니다.

- ✅ FAQ 봇, 분류기, 정형화된 요약
- ❌ 대화형 챗봇 (매 턴 컨텍스트가 다름)

### 전략 3 — 모델 라우팅 (다운그레이드)

{/* Source · DOC · OpenAI "Pricing" — https://openai.com/api/pricing/ */}
{/* Source · DOC · Anthropic "Pricing" — https://www.anthropic.com/pricing */}

모든 작업에 최상위 모델을 쓸 필요가 없습니다. 분류·추출·간단 요약은 소형 모델로 충분하고, 복잡한 추론·코드 생성만 대형 모델을 씁니다.

| 작업 유형 | 추천 모델 티어 | 이유 |
| --- | --- | --- |
| 감정 분류, 키워드 추출 | 소형/경제형 | 정답이 짧고 명확 |
| 1~2문단 요약, 번역 | 중형 | 품질과 비용의 균형 |
| 복잡한 추론, 긴 코드 생성 | 대형/최신 | 정확도가 비용보다 중요 |

(모델 티어 명칭은 벤더마다 다릅니다. 위 공식 pricing에서 본인 벤더의 모델 라인업을 확인하세요.)

### 전략 4 — 사용자별 제한 (Rate Limit / Budget Cap)

사용자 한 명이 하루에 쓸 수 있는 토큰 또는 금액에 상한을 둡니다. 상한에 도달하면 "오늘 사용량을 다 써서, 내일 다시 이용해 주세요" 같은 안내를 보여줍니다.

이건 비용 통제이기도 하지만 **남용 방지** 이기도 해요. 한 명의 파워유저가 전체 예산을 잡아먹는 걸 막습니다.

## 비용을 보려면 — 사용량 추적 인프라

{/* Source · DOC · OpenAI "Usage API" — https://developers.openai.com/docs/api-reference/usage */}
{/* Source · DOC · Helicone (LLM observability) — https://www.helicone.ai/ */}
{/* Source · DOC · Langfuse (LLM observability) — https://langfuse.com/ */}

비용을 관리하려면 먼저 **볼 수 있어야** 합니다. API 응답의 `usage` 필드에는 인풋·아웃풋 토큰 수가 들어 있어요. 이걸 매 호출마다 기록하면 비용 추적의 기초가 됩니다.

```
// API 응답 예시 (개념)
{
  "usage": {
    "prompt_tokens": 1250,
    "completion_tokens": 340,
    "total_tokens": 1590
  }
}
```

좀 더 본격적인 관찰(observability)이 필요하면 Helicone이나 Langfuse 같은 도구가 있어요. 프록시 한 줄 설정으로 모든 호출의 토큰·비용·레이턴시를 대시보드로 볼 수 있습니다. 하지만 오늘은 **직접 손으로 한 번 만들어 보는 게** 목표예요.

## 트레이드오프 — 한 표로

{/* Note: 비용 절감 효과는 도메인·사용 패턴에 따라 다릅니다. 아래는 감을 잡기 위한 정성 비교이며, 구체 수치는 본인 도메인에서 직접 측정하세요. */}

| 전략 | 구현 난이도 | 비용 절감 효과 | 품질 영향 |
| --- | --- | --- | --- |
| 프롬프트 캐싱 | 낮음 (벤더 기능 활성화) | 중간 | 없음 |
| 응답 캐싱 | 중간 (캐시 스토어 필요) | 높음 (히트율에 비례) | 없음 (동일 답 재사용) |
| 모델 라우팅 | 중간 (분류 로직 필요) | 높음 | 주의 필요 (다운그레이드 시 품질 확인) |
| 사용자별 제한 | 낮음 (카운터 + 안내) | 직접 (사용량 상한) | UX 마찰 발생 |

"항상 모델 라우팅이 답"이 아닙니다. **자기 앱의 트래픽 패턴**에 맞는 전략을 고르세요. 반복 질문이 많으면 응답 캐싱, 작업 종류가 다양하면 모델 라우팅이 먼저입니다.

## 그래서 오늘 만들 폴더

오늘 만들 건 `examples/cost-tracker/` — 호출마다 토큰과 비용을 기록하고 일별 리포트를 뽑는 미니 트래커예요.

| 파일 | 역할 |
| --- | --- |
| `logger.ts` | 매 API 호출 후 `usage` 필드에서 토큰 수를 추출해 JSON Lines 파일에 기록 |
| `pricing.ts` | 모델별 인풋·아웃풋 토큰 단가 테이블 (공식 pricing에서 가져온 값) |
| `report.ts` | JSON Lines 로그를 읽어 일별·모델별 비용 합산 리포트 생성 |
| `budget-guard.ts` | 사용자별 일일 예산 상한 체크 — 초과 시 호출 차단 |
| `README.md` | 설정법 + "내 앱에 붙이는 법" 가이드 |

[좋은 프롬프트의 구조](/lessons/structure-of-good-prompts)에서 만든 4축 프롬프트, [출력 형식(Output Format)을 강제하는 법](/lessons/enforcing-output-format)에서 만든 3단 방어선, [개인 프롬프트 라이브러리 만들기](/lessons/build-personal-prompt-library)의 프롬프트 라이브러리 — 이 모든 산출물이 결국 **비용을 수반하는 API 호출** 위에서 돌아갑니다. 오늘 만드는 트래커는 그 비용을 투명하게 만드는 기반이에요.
