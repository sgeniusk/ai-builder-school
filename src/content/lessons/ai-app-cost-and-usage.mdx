## 도입 — 학습자에게 말 거는 한 장면

작은 AI 앱이 드디어 돌아갑니다. 서버 라우트에서 API를 부르고, 화면에는 답이 스트리밍되고, 대화도 저장돼요. 친구에게 링크를 보냈더니 꽤 재밌다며 세 명에게 더 공유했습니다. 여기까지는 기분 좋은 순간입니다.

그런데 바로 다음 질문이 옵니다. “친구 10명이 하루에 다섯 번씩 쓰면 한 달에 얼마 나와요?” 데모를 만들 때는 한 번의 응답이 신기했지만, 운영으로 가면 한 번의 응답이 비용이 됩니다. AI API 호출은 전기 스위치보다 수도꼭지에 가까워요. 토큰이라는 물이 흐르고, 쓴 만큼 요금이 붙습니다.

오늘은 겁먹자는 시간이 아닙니다. 비용을 모르면 출시가 막연하고, 비용을 알면 선택지가 생깁니다. 어떤 모델은 비싼 대신 정확하고, 어떤 요청은 캐시하면 거의 다시 돈을 쓰지 않아도 되고, 어떤 사용자는 무료로 열어 둬도 괜찮지만 어떤 사용자는 유료 전환점이 필요해요.

## Phase 10 다음에 왜 이 레슨인가

Phase 5와 6에서 만든 앱은 “작동하는 데모”에 가까웠습니다. [Lesson 5.1](/lessons/connect-ai-api) 에서 붙인 그 API 호출은 실제로 돈이 드는 호출이고, [Lesson 5.4](/lessons/conversation-storage-design) 에서 저장한 대화는 다음 호출에 다시 들어가며 비용을 키울 수 있어요. 대화 전체를 매번 보내면 사용자는 편하지만, 앱은 매번 더 무거운 물을 흘려보내는 셈입니다.

RAG도 마찬가지입니다. [Lesson 6.3](/lessons/vector-search-basics) 에서 만든 벡터 검색은 임베딩을 만들 때도 비용이 들고, 검색된 청크를 답변에 넣을 때도 토큰이 듭니다. [Lesson 7.2](/lessons/mini-agent-loop) 의 에이전트 루프는 한 바퀴 돌 때마다 계획, 도구 호출, 관찰, 재계획을 반복하니 작은 실험보다 비용 곡선이 빨리 커질 수 있어요.

Phase 10이 배포 전 안전과 검증을 다뤘다면, Phase 11은 “이걸 사람들이 써도 운영할 수 있는가”를 봅니다. 비용은 회계팀만의 일이 아닙니다. 빌더가 제품을 책임지는 방식이에요.

## 핵심 개념 — 토큰, 추적, 가드레일

Token Accounting은 입력 토큰과 출력 토큰을 나눠 보는 습관입니다. 사용자가 보낸 말, 시스템 프롬프트, 이전 대화 요약, 검색된 문서 청크가 입력 토큰에 들어갑니다. 모델이 만든 답변은 출력 토큰이에요. 보통 출력 토큰이 더 비싸거나, 모델마다 입력과 출력 단가가 다르게 잡힙니다. 그래서 “요청 1회”보다 “요청 1회가 몇 토큰인가”가 더 중요합니다.

Usage Tracking은 사용량을 제품 안에서 기록하는 일입니다. 최소한 요청 수, 입력 토큰, 출력 토큰, 모델 이름, 사용자 ID, 세션 ID, 캐시 적중 여부를 남겨야 해요. 이 기록이 있어야 한 사용자가 비용을 많이 쓰는지, 특정 기능이 비용을 키우는지, 새 모델로 바꿨을 때 월 비용이 어떻게 변하는지 볼 수 있습니다.

Cost Guardrail은 비용이 새는 지점을 막는 장치입니다. 예산 알림, 사용자별 일일 제한, 월 예산 한도, 고비용 기능의 확인 버튼, 긴 대화 요약, 저렴한 모델로 먼저 초안 만들기 같은 것들이 여기에 들어갑니다. 캐싱은 빗물 받아쓰기와 비슷해요. 같은 질문과 같은 문서 조합이 반복된다면, 매번 수도꼭지를 틀지 말고 이미 받아 둔 물을 먼저 씁니다.

Free Tier와 Paid Trigger도 정해야 합니다. 무료 사용자는 몇 번까지 편하게 써도 되는지, 어떤 순간부터 유료 플랜을 안내할지, 유료 사용자는 월 사용량을 어디까지 보장할지 정합니다. 가격은 감으로 붙이면 흔들려요. 비용표를 보고 정하면 “무료로 열어도 되는 기능”과 “유료에서만 열어야 하는 기능”이 보입니다.

## 그래서 오늘 할 일

오늘 만들 한 장은 `cost-and-usage-dashboard.md`입니다. 실제 대시보드 코드를 바로 만들기보다, 먼저 어떤 숫자를 볼지 정하는 운영 노트예요.

먼저 내 앱의 대표 사용 시나리오를 세 개로 나눕니다. 가벼운 사용자, 평균 사용자, 많이 쓰는 사용자를 상상해요. 각 시나리오마다 하루 요청 수, 요청당 평균 입력 토큰, 요청당 평균 출력 토큰, 한 달 사용 일수를 적습니다. 그다음 모델 단가를 넣어 월 비용을 계산합니다. 정확한 숫자보다 중요한 건 범위입니다. “사용자 10명은 괜찮지만 100명부터는 알림이 필요하다”처럼 판단할 수 있어야 해요.

다음으로 로그 필드를 정합니다. `requestId`, `userId`, `model`, `inputTokens`, `outputTokens`, `estimatedCost`, `cacheHit`, `featureName` 정도면 첫 버전은 충분합니다. 이 필드는 나중에 비용 대시보드의 축이 됩니다.

마지막으로 가드레일을 고릅니다. 하루 요청 제한, 월 예산 알림, 긴 대화 요약, 반복 질문 캐시, 무료 티어 제한 중 내 앱에 바로 필요한 세 가지만 선택하세요. 너무 많은 정책을 한 번에 붙이면 제품 경험이 답답해질 수 있습니다. 비용을 줄이는 일과 사용자가 계속 쓰고 싶은 경험 사이에 균형이 필요합니다.

마무리 전에 네 가지 트랙 카드로 감각을 정리해 봅니다.

| 트랙 카드 | 비용 감각이 하는 일 |
| --- | --- |
| 개인 사용 | 그냥 써도 되는 범위와 아껴야 하는 습관을 알게 해요. |
| 부업 | 한 건의 작업 가격이 API 비용보다 충분히 큰지 보게 해요. |
| 사업 | 무료 체험, 유료 전환, 마진을 숫자로 계산하게 해요. |
| 포트폴리오 | “만들 줄 안다”를 넘어 운영 비용까지 의식한다는 신호를 줘요. |

## 마무리

비용 계산은 창의성을 줄이는 일이 아닙니다. 오히려 더 오래 버티는 제품을 만들게 해요. 토큰은 수도 요금처럼 쓴 만큼 나오고, 캐싱은 빗물처럼 반복 비용을 줄여 줍니다. 사용량을 추적하고 가드레일을 세우면 데모가 운영으로 넘어갈 준비를 합니다.

비용까지 챙겼으면, 이제 진짜 내 것 하나를 끝까지 만들 차례예요. 다음 Phase 12에서는 지금까지 만든 프롬프트, 앱, RAG, 에이전트, 비용 감각을 모아 공유 가능한 캡스톤 프로젝트로 묶습니다.
