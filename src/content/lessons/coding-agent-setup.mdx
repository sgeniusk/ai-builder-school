## 도입 — 학습자에게 말 거는 한 장면

새 프로젝트를 열고 Codex 나 Claude Code 를 켰어요. “이 파일 좀 정리해줘”라고 말하려는 순간 손이 멈춥니다. 이 도구가 어디까지 파일을 고칠 수 있는지, 터미널 명령을 실행해도 되는지, `.env` 파일을 읽거나 바꾸지는 않는지 확신이 없어요. 잘 쓰면 하루가 줄어들 것 같은데, 잘못 쓰면 저장소가 망가질 것 같죠.

이 불안은 자연스러워요. 코딩 에이전트는 일반 챗봇보다 훨씬 가까운 곳에서 일합니다. 파일을 읽고, 수정하고, 테스트를 돌리고, 때로는 외부 도구를 호출해요. 그래서 첫 번째 레슨은 “무엇을 만들까”가 아니라 “어떤 울타리 안에서 일하게 할까”입니다. 오늘 만들 한 장은 에이전트에게 보여줄 온보딩 문서예요.

## Phase 3 다음에 왜 이 레슨인가

[Lesson 3.4](/lessons/blog-writing-pipeline) 까지는 AI 가 문서와 글을 잘 다루도록 작업 흐름을 만들었어요. 이제 Phase 4 부터는 같은 원리를 코드 작업에 옮깁니다. 차이는 위험도예요. 글 초안이 마음에 안 들면 버리면 되지만, 코드 변경은 테스트 실패, 비밀 유출, 엉뚱한 리팩터링으로 이어질 수 있어요.

그래서 [Lesson 1.4](/lessons/checks-before-trusting-ai-output) 의 트러스트 체크를 코드 환경 안으로 끌고 옵니다. 답을 믿기 전에 검토했던 것처럼, 실행을 맡기기 전에 권한과 멈춤 지점을 정해요. 이 레슨의 산출물 [`agent-onboarding.md`](/lessons/coding-agent-setup) 은 앞으로 모든 코딩 에이전트 작업의 첫 장이 됩니다.

## 핵심 개념 — 빠르게 짚고 갑니다

### Tool Permissions — 할 수 있는 일과 못 하는 일을 적는다

에이전트에게 “알아서 해줘”라고 말하면, 모델은 도움이 된다고 판단한 작업을 넓게 시도할 수 있어요. 로컬 파일을 고치고, 테스트를 돌리고, 설치 명령을 제안합니다. 문제는 이 범위가 사람 머릿속에만 있으면 매번 흔들린다는 점이에요.

권한 문서는 세 칸으로 나누면 충분해요.

| 구분 | 예시 | 원칙 |
| --- | --- | --- |
| 항상 허용 | 파일 읽기, 테스트 실행, 타입체크 | 결과를 되돌릴 수 있고 위험이 낮은 일. |
| 확인 후 허용 | 패키지 설치, 대규모 리팩터링, 설정 변경 | 비용이나 영향 범위가 있는 일. |
| 항상 금지 | 비밀 파일 수정, 강제 reset, 무단 push | 한 번 실행하면 회복 비용이 큰 일. |

이 표가 있어야 에이전트도 “여기서는 멈춰야 하는구나”를 배웁니다. [Lesson 2.1](/lessons/structure-of-good-prompts) 의 Constraints 축을 코드 작업 환경에 적용하는 셈이에요.

### Secret Handling — 비밀은 절대 대화 재료가 아니다

API 키, 토큰, 운영 DB 주소는 에이전트가 추측하거나 복사해도 되는 정보가 아니에요. “`.env`는 읽지 않는다”, “샘플 값은 `.env.example`에만 둔다”, “로그에 토큰을 출력하지 않는다”처럼 문장으로 박아둡니다. 비전공자에게도 이 규칙은 중요해요. 코드를 잘 몰라도 비밀을 어디에 두면 안 되는지는 정할 수 있어요.

### Logs — 나중에 되짚을 수 있어야 한다

에이전트가 한 일을 기억에 의존하면 안 됩니다. 어떤 파일을 읽었고, 어떤 파일을 바꿨고, 어떤 검증을 돌렸는지 마지막 메시지에 남기게 하세요. 작은 로그 습관이 다음 [Lesson 4.2](/lessons/github-issue-to-ai-brief) 에서 이슈와 작업지시서를 연결하는 발판이 됩니다.

## 그래서 오늘 할 일

오늘 만들 것은 “코딩 에이전트 온보딩 문서”입니다. 새 저장소를 열 때마다 에이전트에게 먼저 읽히는 한 장이에요.

1. 실습용 저장소 이름을 정합니다. 예시는 `my-ai-builder-lab` 이에요.
2. 에이전트가 항상 해도 되는 일을 5개 적습니다.
3. 반드시 사람 확인을 받아야 하는 일을 5개 적습니다.
4. 절대 하면 안 되는 일을 5개 적습니다.
5. 비밀 관리 규칙을 `.env`, `.env.example`, 로그 세 가지 기준으로 씁니다.
6. 마지막 응답에 포함해야 할 작업 로그 형식을 정합니다.
7. 산출물 [`agent-onboarding.md`](/lessons/coding-agent-setup) 로 저장하고 다음 레슨에서 이슈 브리프의 공통 컨텍스트로 재사용합니다.

처음부터 완벽한 보안 정책을 만들 필요는 없어요. 중요한 건 코딩 에이전트가 “내 컴퓨터에서 일하는 동료”라는 사실을 문서에 반영하는 것입니다. 무엇을 맡길지보다 무엇을 맡기지 않을지가 먼저예요.

## 마무리

Phase 4 의 첫 단추는 멋진 코드가 아니라 안전한 작업장입니다. 오늘 만든 온보딩 문서가 있으면 다음 [Lesson 4.2](/lessons/github-issue-to-ai-brief) 에서 GitHub Issue 를 에이전트가 바로 이해할 수 있는 작업지시서로 바꿀 수 있어요. 권한, 비밀, 로그의 울타리가 잡혔으니 이제는 그 안에서 일을 작게 잘라 맡기는 법으로 넘어갑니다.
