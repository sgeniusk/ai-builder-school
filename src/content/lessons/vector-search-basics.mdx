## 도입 — 학습자에게 말 거는 한 장면

임베딩 파일이 생겼습니다. 문서 조각 300개를 벡터로 바꿔 JSON 파일에 저장했어요. 처음에는 충분해 보입니다. 질문 하나가 들어오면 모든 벡터와 유사도를 계산해서 가장 가까운 5개를 고르면 되니까요. 그런데 문서가 3,000개, 30,000개로 늘어나면 이야기가 달라집니다. 검색이 느려지고, 필터가 필요해지고, 업데이트도 번거로워져요.

사용자 입장에서는 “내 질문에 맞는 문서를 바로 찾아줘”가 전부입니다. 하지만 앱 입장에서는 질문 벡터를 만들고, 저장소에서 가까운 청크를 찾고, 권한이나 날짜 같은 조건을 거르고, 상위 결과를 모델에게 넘기는 일이 필요합니다.

오늘은 그 검색 저장소를 만듭니다. 벡터 DB나 벡터 인덱스라는 말이 어렵게 들릴 수 있지만, 핵심은 도서관 사서가 빠르게 비슷한 책장을 찾아오게 만드는 색인입니다.

## Phase 5 다음에 왜 이 레슨인가

[Lesson 6.1](/lessons/understanding-embeddings) 에서는 문장을 의미 좌표로 바꾸는 법을 봤고, [Lesson 6.2](/lessons/document-chunking-strategy) 에서는 문서를 검색 가능한 청크로 자르는 전략을 세웠습니다. 이제 두 재료를 저장하고 검색해야 해요.

작은 실험에서는 파일 하나로 충분합니다. 하지만 문서 Q&A 봇을 실제로 쓰려면 새 문서가 들어오고, 오래된 문서가 수정되고, 특정 팀만 볼 수 있는 문서를 제외해야 합니다. 이때 벡터 검색 파이프라인이 필요합니다. RAG에서 Retrieval은 “대충 관련 있어 보이는 것 찾기”가 아니라, 질문에 답할 근거 후보를 재현 가능하게 가져오는 과정입니다.

Phase 5의 서버 라우트 관점도 다시 이어집니다. 사용자가 질문을 보내면 서버는 곧바로 생성 모델을 부르지 않습니다. 먼저 질문을 임베딩하고, 벡터 검색으로 관련 청크를 찾고, 그 결과를 프롬프트에 넣어 답을 요청합니다.

## 핵심 개념 — Top-K, 필터, 하이브리드 검색

Top-K 검색은 질문과 가장 가까운 청크 K개를 가져오는 방식입니다. K가 5라면 상위 5개 청크를 가져옵니다. K가 너무 작으면 필요한 근거를 놓칠 수 있고, 너무 크면 모델에게 불필요한 문맥을 많이 넣게 됩니다. 처음에는 3, 5, 8처럼 작은 후보를 비교해 보는 것이 좋아요.

벡터 인덱스는 가까운 벡터를 빠르게 찾기 위한 구조입니다. 모든 벡터를 하나씩 비교하는 방식은 데이터가 작을 때는 단순하지만, 커질수록 느립니다. pgvector, Pinecone, Weaviate, Qdrant 같은 도구는 이 검색을 빠르게 하도록 돕습니다. 어떤 도구를 쓰든 핵심 필드는 비슷합니다. 청크 ID, 원문, 임베딩, 문서 제목, 섹션, 권한, 업데이트 날짜가 필요합니다.

필터는 검색 전에 범위를 줄이는 장치입니다. 예를 들어 사용자가 한국어 문서만 검색하고 싶거나, 특정 프로젝트 문서만 보고 싶거나, 권한이 없는 문서는 제외해야 할 수 있어요. 벡터 유사도만 믿으면 “의미는 비슷하지만 사용자가 보면 안 되는 문서”가 나올 수 있습니다.

하이브리드 검색은 벡터 검색과 키워드 검색을 함께 쓰는 방식입니다. 벡터 검색은 표현이 달라도 의미가 비슷한 내용을 잘 찾습니다. 반면 제품 코드, 에러 메시지, 사람 이름, 법 조항 번호처럼 정확한 문자열이 중요한 질문에는 키워드 검색이 강합니다. 둘을 합치면 “의미도 맞고 정확한 단어도 맞는” 결과를 고를 수 있어요.

## 그래서 오늘 할 일

오늘 만들 것은 “벡터 검색 기본 파이프라인”입니다. [Lesson 6.2](/lessons/document-chunking-strategy) 에서 만든 청크를 가져와 각 청크에 임베딩을 붙이고, 간단한 검색 함수로 질문에 대한 Top-K 결과를 반환합니다.

먼저 저장할 필드를 정합니다. `chunkId`, `documentId`, `title`, `section`, `content`, `embedding`, `source`, `updatedAt` 정도면 시작할 수 있어요. 다음으로 인덱스 생성 스크립트를 만들고, 샘플 문서 20개 이상을 넣습니다. 그다음 질문 5개를 정해 Top-K 결과를 확인합니다. 가능하면 키워드 검색 결과와 벡터 검색 결과를 나란히 비교합니다.

산출물 [`vector-search-recipe.md`](/lessons/vector-search-basics) 에는 저장소 선택, 스키마, 인덱스 생성, 검색 함수, 품질 점검 질문을 적습니다. 중요한 건 특정 도구 이름보다 흐름입니다. 질문을 임베딩하고, 후보 청크를 가져오고, 사람이 봐도 납득되는 근거인지 확인하는 흐름을 재현할 수 있어야 합니다.

## 마무리

벡터 검색은 RAG의 기억 저장소에서 필요한 조각을 꺼내 오는 단계입니다. 임베딩만 있으면 좌표가 있고, 청킹만 있으면 조각이 있지만, 검색 파이프라인이 있어야 앱이 질문마다 알맞은 근거를 가져옵니다. 다음 [Lesson 6.4](/lessons/grounded-rag-answers) 에서는 이 검색 결과를 모델에게 넘기고, 답에 출처를 붙입니다. 그때부터 문서 Q&A 봇은 “아마 그럴 거예요”가 아니라 “이 문서의 이 부분을 근거로 답해요”라고 말할 수 있습니다.
