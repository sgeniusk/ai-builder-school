## "JSON 파일에 벡터 넣어놓으니 10개까지는 됐는데 500개 넘으니 10초 걸려요"

임베딩 실험이 잘 됐어요. 문장 10개의 코사인 유사도를 비교하니 감이 옵니다. 그래서 회사 문서 500개를 전부 임베딩해 JSON 파일에 저장했습니다. 검색할 때마다 500개 벡터를 다 읽어서 하나하나 비교합니다. 10개일 때는 눈 깜짝할 사이인데, 수가 늘어나면 **눈에 띄게 느려지기 시작합니다**. 1만 개가 되면? 쓸 수 없는 속도가 됩니다.

[Lesson 23](/lessons/understanding-embeddings)에서 임베딩을, [Lesson 24](/lessons/document-chunking-strategy)에서 청킹을 배웠어요. 오늘은 그 청크 벡터들을 **빠르게 검색할 수 있는 장소** — 벡터 DB — 에 넣습니다.

오늘 만들 건 **10~30개 문서로 동작하는 미니 벡터 검색** (`examples/vector-search/`)이에요.

## 벡터 DB가 하는 일

{/* Source · DOC · OpenAI "Embeddings" — https://developers.openai.com/api/docs/guides/embeddings */}

벡터 DB는 한 마디로 **"가장 가까운 벡터를 빠르게 찾아주는 저장소"**예요. 일반 DB가 "이름이 '김태우'인 행을 찾아줘"라면, 벡터 DB는 "이 벡터와 가장 비슷한 벡터 Top-5를 찾아줘"입니다.

내부적으로는 **ANN(Approximate Nearest Neighbor)** 알고리즘을 씁니다. 모든 벡터를 하나하나 비교하는 대신, 인덱스 구조를 만들어 후보를 빠르게 좁히는 방식이에요. 100만 개 벡터에서도 밀리초 단위로 검색됩니다.

(ANN은 "근사" 검색이라 정확도와 속도 사이 트레이드오프가 있어요. 본인 데이터에서 Recall을 직접 측정해 허용 범위를 정하세요.)

## 벡터 DB 선택지 — 한 표로

{/* Source · DOC · Pinecone documentation — https://docs.pinecone.io/ */}
{/* Source · DOC · pgvector (PostgreSQL extension) — https://github.com/pgvector/pgvector */}
{/* Source · DOC · Chroma (open-source embedding database) — https://docs.trychroma.com/ */}

| 항목 | pgvector | Pinecone | Chroma |
| --- | --- | --- | --- |
| 종류 | PostgreSQL 확장 | 관리형 클라우드 | 오픈소스 (로컬/클라우드) |
| 설치 | 기존 PostgreSQL에 추가 | 가입 후 바로 사용 | `pip install chromadb` |
| 장점 | SQL DB 옆에 두기 — 기존 데이터와 JOIN 가능 | 스케일링 자동, 관리 부담 0 | 로컬에서 10줄로 시작 |
| 단점 | 대규모 시 성능 튜닝 필요 | 비용 (규모에 비례) | 대규모 프로덕션엔 한계 |
| 적합한 때 | 이미 PostgreSQL 사용 중 | 빠르게 프로덕션, 인프라 부담 최소 | 프로토타입·로컬 실험 |
| 무료 시작 | 오픈소스 | 무료 티어 있음 | 완전 무료 |

### SQL DB 옆에 두기 vs 별도 서비스

이 선택이 실무에서 가장 많이 고민되는 지점이에요.

**SQL DB 옆에 두기 (pgvector)**
- 기존 사용자 테이블과 벡터를 **같은 DB에서 JOIN** 가능
- "이 사용자의 문서 중 가장 관련 있는 청크"를 SQL 한 방에 쿼리
- 인프라가 하나 — 운영 부담 낮음
- 단점: 벡터 전용 최적화(인덱스 종류, 디스크 레이아웃)가 제한적

**별도 서비스 (Pinecone, Chroma 등)**
- 벡터 검색에 특화된 인덱스·캐싱
- 대규모(100만+)에서 성능 우위
- 단점: 인프라가 두 개 — 동기화·장애 포인트 증가

10만 개 이하 + 이미 PostgreSQL 쓰고 있으면 **pgvector가 가성비 1등**이에요. 빠르게 실험하려면 **Chroma 로컬**이 마찰 최소입니다. 규모가 크고 인프라 관리를 맡기고 싶으면 **Pinecone**.

## Top-K 검색 — "가장 가까운 K개를 줘"

벡터 검색의 기본 단위는 **Top-K retrieval**이에요. 쿼리 벡터와 가장 유사한 K개 청크를 가져옵니다.

```
쿼리: "인증 방식이 뭐야?"
  ↓ 임베딩
쿼리 벡터: [0.12, -0.34, ...]
  ↓ 벡터 DB에서 Top-5 검색
결과:
  1. "OAuth 2.0 인증 플로우..." (유사도 0.89)
  2. "JWT 토큰 갱신 절차..." (유사도 0.85)
  3. "API 키 발급 방법..." (유사도 0.81)
  4. "사용자 권한 설정..." (유사도 0.72)
  5. "로그인 화면 디자인..." (유사도 0.65)
```

K를 몇으로 할지가 중요합니다.

| K | 장점 | 단점 |
| --- | --- | --- |
| 3 | LLM 컨텍스트 절약, 노이즈 적음 | 관련 청크를 놓칠 수 있음 |
| 5~10 | 대부분의 관련 정보 포함 | 비관련 청크가 섞일 수 있음 |
| 20 이상 | 놓치는 게 거의 없음 | LLM 컨텍스트 소비 ↑, "Lost in the Middle" 위험 |

시작은 **K=5**. 답이 불완전하면 올리고, 노이즈가 많으면 줄이세요.

## 하이브리드 검색 — 키워드 + 벡터의 합작

{/* Source · DOC · Anthropic "Contextual Retrieval" (2024) — https://www.anthropic.com/news/contextual-retrieval */}

[Lesson 23](/lessons/understanding-embeddings)에서 "문서번호 A-2024-0317을 찾을 때는 키워드 검색이 압도적"이라고 했죠. 실무에서는 **벡터 검색과 키워드 검색을 같이** 돌립니다. 이게 하이브리드 검색이에요.

```
쿼리: "A-2024-0317 인증 방식"

키워드 검색 결과: [문서번호 정확 매칭]
벡터 검색 결과: [의미적으로 "인증 방식" 관련 청크]

→ 두 결과를 합쳐서 순위 매김 (RRF 등)
→ 최종 Top-K
```

Anthropic의 Contextual Retrieval 연구에서도 벡터 검색 단독보다 **BM25(키워드) + 임베딩(의미)을 합쳤을 때** 검색 실패율이 크게 줄어든다고 보고합니다.

| 검색 방식 | 잘하는 것 | 못하는 것 |
| --- | --- | --- |
| 키워드 (BM25) | 정확한 용어, 고유명사, 코드명 | 동의어, 다른 표현 |
| 벡터 (임베딩) | 의미적 유사, 동의어 | 정확한 매칭, 숫자/날짜 |
| 하이브리드 | 둘 다 | 구현 복잡도 ↑ |

## 미니 벡터 검색 만들기 — 흐름

오늘 만들 미니 검색의 전체 흐름이에요.

```
1. 문서 10~30개 준비 (Markdown 또는 텍스트)
2. Lesson 24의 전략으로 청킹
3. 각 청크를 임베딩 API로 벡터화
4. 벡터 DB에 저장 (Chroma 로컬 추천 — 설치 한 줄)
5. 쿼리를 받아 Top-K 검색하는 함수 작성
6. 테스트 질문 5개로 검색 품질 확인
```

이 6단계가 RAG 파이프라인의 **앞쪽 절반**이에요. 뒤쪽 절반 — 검색 결과를 LLM에 넣어 답을 만드는 것 — 은 다음 레슨([근거 박힌 RAG 답변](/lessons/grounded-rag-answers))에서 완성합니다.

## 검색 품질을 측정하는 눈

검색을 만들었으면 "잘 되는지" 확인해야 해요. 간단한 방법 두 가지:

**1. 수동 관련도 체크**
테스트 질문 5개에 대해 Top-5 결과를 눈으로 보고 관련도 1~5점을 매깁니다. 평균 3점 이하면 청킹이나 임베딩을 재검토합니다.

**2. 기대 청크 비교**
"이 질문에는 이 청크가 나와야 한다"를 미리 정해놓고, 실제로 Top-K 안에 들었는지 확인합니다. Recall@K라고 불러요.

| 지표 | 뜻 | 목표 |
| --- | --- | --- |
| Recall@5 | 기대 청크가 Top-5 안에 있는 비율 | 도메인별 직접 설정 (초기 목표 80% 예시) |
| MRR (Mean Reciprocal Rank) | 기대 청크가 몇 번째에 나오는지의 역수 평균 | 0.5 이상 |

(이 목표값은 시작점이에요. 도메인 난이도에 따라 조정하세요.)

## 그래서 오늘 만들 폴더

오늘 만들 건 `examples/vector-search/` 폴더예요.

| 파일 | 역할 |
| --- | --- |
| `documents/` | 테스트용 문서 10~30개 (Markdown 또는 텍스트) |
| `chunk.py` (또는 `.ts`) | 문서를 청킹하는 스크립트 |
| `embed.py` | 청크를 임베딩해 벡터 DB에 저장 |
| `search.py` | 쿼리를 받아 Top-K 검색 |
| `eval.md` | 테스트 질문 5개 + 검색 결과 + 관련도 점수 |

이 폴더가 있으면 다음 레슨에서 LLM 연결만 붙이면 됩니다. 검색 파이프라인이 이미 돌아가고 있으니까요.
