| 항목 | 내용 |

## 1. 비밀 키 보관

| 항목 | 결정 |
| --- | --- |
| 환경변수 이름 | `OPENAI_API_KEY` 또는 `ANTHROPIC_API_KEY` |
| 키를 읽는 위치 | 서버 라우트 안에서만 |
| 클라이언트 노출 여부 | 아니오 |
| 키가 없을 때 동작 | 개발자 로그 기록 + 사용자에게 설정 오류 안내 |

확인 질문.

- 브라우저 코드에 API 키 문자열이 직접 들어가 있지 않나요?
- `NEXT_PUBLIC_` 접두사가 붙은 값에 비밀 키를 넣지 않았나요?
- 배포 환경에도 같은 환경변수가 등록되어 있나요?

## 2. 서버 라우트 설계

```txt
Route: POST /api/chat

Input:
- message: string
- threadId: optional string

Server steps:
1. 환경변수 확인.
2. 입력 길이와 필수 필드 확인.
3. 모델 호출에 timeout 적용.
4. 재시도는 최대 1회만 허용.
5. 사용자 메시지와 개발자 로그를 분리.

Output:
- ok: boolean
- message: string
- requestId: string
```

---

## 3. 실패 처리 표

| 실패 종류 | 사용자에게 보일 말 | 로그에 남길 것 | 재시도 |
| --- | --- | --- | --- |
| 입력이 비어 있음 | 질문을 입력해 주세요. | validation_empty_message | 아니오 |
| API 키 없음 | 앱 설정을 확인해야 해요. | missing_api_key | 아니오 |
| 모델 타임아웃 | 응답이 지연됐어요. 다시 시도해 주세요. | timeout + requestId | 1회 |
| 제공자 오류 | 잠시 후 다시 시도해 주세요. | provider_status + requestId | 상황별 |

## 4. 첫 수동 테스트

| 테스트 | 기대 결과 | 실제 결과 |
| --- | --- | --- |
| 정상 질문 1개 전송 | 답변이 돌아온다 | |
| 빈 질문 전송 | 검증 메시지가 보인다 | |
| API 키 임시 제거 | 사용자용 오류가 보인다 | |
| 네트워크 지연 상황 | timeout 뒤 복구된다 | |
