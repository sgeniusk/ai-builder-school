## "고쳤는데 또 나왔어요"

금요일 오후, 결제 페이지에서 버그가 터져요. 특정 카드사에서 결제가 안 된다는 신고. 코딩 에이전트한테 "이거 고쳐줘"라고 던지니까 5분 만에 수정 PR이 올라옵니다. 머지하고 퇴근. 월요일 아침, 같은 버그가 다시 올라옵니다. "분명 고쳤는데요?" 코드를 보니 에이전트가 **증상은 막았지만 원인은 안 건드렸어요.** 결제 금액이 소수점일 때 반올림 로직이 빠져있었는데, 에이전트는 그냥 에러를 catch해서 삼켜버린 거예요.

"고쳤는데 또 나오는" 버그는 같은 패턴이 자주 반복돼요. **재현 없이 수정했기 때문**입니다. 증상만 보고 고치면 원인은 살아남아요.

오늘 만들 건 **재현 → 최소 예제 → AI 가설 → 수정 → 회귀 테스트**의 5단 루프예요. 이 루프를 한 번 돌면 같은 버그가 두 번 오지 않습니다.

## 재현이 수정보다 먼저인 이유

{/* Source · DOC · Anthropic "Claude Code overview" — https://docs.claude.com/en/docs/claude-code/ */}

에이전트에게 버그 리포트를 그대로 던지면 어떻게 될까요? 에이전트는 성실하게 코드를 읽고 "이 부분이 문제인 것 같습니다"라고 말하며 고칩니다. 그런데 그게 진짜 원인인지 **확인할 방법이 없어요.** 재현 스텝이 없으니까.

재현 = **"이 입력을 넣으면 이 에러가 나온다"를 누구나 반복할 수 있는 절차.**

재현이 있으면 세 가지가 달라져요.

1. **수정 전에 확인**: 정말 이 버그가 존재하는지 내 눈으로 봄
2. **수정 후에 확인**: 고친 뒤 같은 절차를 돌려 사라졌는지 확인
3. **회귀 테스트 변환**: 재현 절차를 테스트 코드로 바꿔 자동으로 감시

재현 없이 고치면 이 세 가지가 다 불가능해요. "고쳤다고 생각한다"와 "고쳤다는 걸 보여줄 수 있다"의 차이입니다.

## Minimal Reproduction — 가장 작은 재현

버그 리포트가 이렇게 옵니다: "결제 페이지에서 카드 등록하고 상품 담고 쿠폰 넣고 결제하면 에러가 나요." 이 전체 흐름을 재현하려면 10분이 걸려요. 그런데 진짜 원인은 **소수점 금액 반올림** 하나예요.

**Minimal Reproduction(최소 재현)**은 버그를 일으키는 가장 작은 입력/코드를 찾는 것이에요. 10단계의 재현을 1단계로 줄이면:

```typescript
// 최소 재현
calculatePayment(99.99, "KR") // → NaN (기대값: 99990)
```

이 한 줄이면 버그가 보여요. 에이전트에게 이걸 주면 원인을 정확하게 찾아줍니다. 10단계 재현을 주면 에이전트도 헤매요.

최소 재현을 찾는 과정은 이래요.

| 단계 | 하는 일 |
| --- | --- |
| 1. 전체 재현 | 리포트 그대로 따라해서 버그 확인 |
| 2. 절반 제거 | 흐름의 절반을 빼고 재현되는지 확인 |
| 3. 반복 | 재현이 유지되는 한 계속 줄임 |
| 4. 고립 | 더 줄일 수 없으면 그게 최소 재현 |

이 과정 자체를 에이전트에게 시킬 수도 있어요. "이 버그의 최소 재현을 찾아줘. 한 번에 하나씩 줄여가며 확인해"라고 하면 에이전트가 체계적으로 줄여줍니다.

## git bisect — 언제부터 깨졌나 찾기

{/* Source · DOC · Git "git-bisect" — https://git-scm.com/docs/git-bisect */}

"이 버그가 언제부터 있었지?"가 궁금할 때 `git bisect`가 친구예요. 이진 탐색으로 "마지막으로 정상이던 커밋"과 "처음 깨진 커밋" 사이를 좁혀갑니다.

```bash
git bisect start
git bisect bad          # 현재 커밋은 깨져있음
git bisect good abc123  # 이 커밋은 정상이었음
# → git이 중간 커밋을 체크아웃해줌
# → 테스트하고 good/bad 판정
# → 반복하면 원인 커밋을 찾음
```

에이전트에게 이 과정을 시킬 수 있어요. "git bisect로 이 테스트가 처음 깨진 커밋을 찾아줘"라고 하면 에이전트가 자동으로 good/bad를 판정하면서 찾아갑니다.

`git bisect`로 원인 커밋을 찾으면 **diff가 좁아져요.** "지난 2주간의 변경 전체"를 보는 대신 "이 커밋의 변경 30줄"만 보면 됩니다. 에이전트의 가설도 훨씬 정확해집니다.

## AI에게 가설 세우게 하기

{/* Source · DOC · Anthropic "Extended thinking" — https://docs.claude.com/en/docs/build-with-claude/extended-thinking */}

최소 재현을 찾았으면 에이전트에게 던집니다. 이때 "고쳐줘"가 아니라 **"가설을 세워줘"**라고 시키세요.

```
아래 코드에서 버그가 발생해요.

최소 재현:
calculatePayment(99.99, "KR") → NaN (기대값: 99990)

관련 코드: [파일 경로와 함수]

이 버그의 가능한 원인 가설을 3가지 세워줘.
각 가설에 대해 확인 방법도 알려줘.
```

에이전트가 "고쳐줘" 대신 "가설을 세워줘"를 받으면 더 신중하게 접근해요. "이것 같습니다"라고 바로 고치는 대신, **여러 가능성을 나열하고 각각을 확인하는 방법**을 알려줍니다.

가설 예시:

| 가설 | 확인 방법 |
| --- | --- |
| 소수점 문자열 파싱 실패 | `parseFloat("99.99")` 결과 확인 |
| 통화 변환 곱셈 순서 오류 | `99.99 * 1000` vs `Math.round(99.99 * 1000)` 비교 |
| KR 통화 코드 미등록 | 통화 코드 맵에 "KR" 키 존재 여부 확인 |

세 가설 중 하나를 확인해서 원인을 특정한 뒤에 수정을 시키면, 증상이 아니라 **원인을 고치는** 수정이 나옵니다.

## 회귀 테스트 — 같은 버그의 재발 방지

{/* Source · BOOK · Kent Beck "Test-Driven Development: By Example" (2002) · 회귀 테스트의 원리 */}

수정이 끝났으면 **재현 스텝을 테스트 코드로 바꿉니다.** [Lesson 17](/lessons/write-tests-with-coding-agent)에서 배운 패턴 그대로예요.

```typescript
// 의도: 소수점 금액이 올바르게 원 단위로 변환되어야 한다 (버그 #42 회귀 방지)
test("소수점 금액 → 원 단위 변환", () => {
  expect(calculatePayment(99.99, "KR")).toBe(99990);
});

// 의도: 0원 결제는 거부되어야 한다 (같은 반올림 로직 영역)
test("0원 결제 → 에러", () => {
  expect(() => calculatePayment(0, "KR")).toThrow();
});
```

이 테스트가 CI에 들어가면 같은 원인의 버그가 다시 나올 때 **코드가 머지되기 전에** 빨간불이 켜집니다. "고쳤는데 또 나왔어요"가 구조적으로 불가능해지는 거예요.

회귀 테스트의 주석에 **버그 번호를 남기는 것**이 중요해요. 3개월 뒤에 이 테스트가 깨졌을 때 "이건 버그 #42 때문에 만든 테스트니까 함부로 지우면 안 된다"는 맥락이 살아있어야 합니다.

## 5단 루프 한 장 정리

{/* Note: 5단 루프(재현 → 최소화 → 가설 → 수정 → 회귀)는 일반적인 디버깅 관행을 이 학교 운영자가 한국어로 정리한 것입니다. */}

| 단계 | 시간 | 하는 일 | 에이전트 역할 |
| --- | --- | --- | --- |
| 1. 재현 | 5분 | 버그 리포트를 따라 재현 확인 | 재현 스크립트 생성 |
| 2. 최소화 | 5~10분 | 재현을 가장 작은 형태로 줄임 | 이진 탐색 / 절반 제거 |
| 3. 가설 | 5분 | 원인 가설 3개 + 확인 방법 | 가설 생성 + 확인 |
| 4. 수정 | 10~20분 | 원인을 고치는 코드 변경 | 수정 코드 작성 |
| 5. 회귀 | 5분 | 재현 → 테스트 변환 + CI 등록 | 테스트 코드 작성 |

전체 30~45분. "에이전트한테 고쳐달라고 던지고 5분 뒤에 머지"보다 길어요. 하지만 월요일 아침에 같은 버그를 또 만나는 시간을 빼면 **되돌아오는 버그를 다시 보는 시간을 줄일 수 있어요.**

## 그래서 오늘 만들 한 장

오늘 만들 건 **한 실제 버그의 5단 루프 기록**이에요.

| 항목 | 무엇 |
| --- | --- |
| 버그 선정 | 최근 겪은 실제 버그 1개 (또는 연습용 버그) |
| 재현 기록 | 전체 재현 스텝 → 최소 재현 (줄여가는 과정 포함) |
| 가설 기록 | 에이전트가 세운 가설 3개 + 확인 결과 |
| 수정 기록 | 원인과 수정 내용 (diff 또는 설명) |
| 회귀 테스트 | 재현을 변환한 테스트 코드 + `// 의도:` 주석 |

이 한 장이 생기면 다음 버그부터 "일단 고쳐줘"가 아니라 **"먼저 재현해줘"**로 시작하게 됩니다. 그 한 마디가 디버깅의 품질을 바꿔요.
