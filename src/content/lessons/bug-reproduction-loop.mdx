## 도입 — 학습자에게 말 거는 한 장면

지난주에 고친 버그가 다시 나타났어요. 이번에는 조금 다른 데이터에서만 발생합니다. 에이전트에게 다시 맡기려니 설명하기도 어렵습니다. “가끔 저장이 안 돼요”라고 말하면 또 비슷한 곳을 고칠 것 같고, “이전처럼 고쳐줘”라고 해도 근본 원인이 같은지 확신이 없어요.

버그 수정에서 가장 먼저 필요한 것은 해결책이 아니라 재현입니다. 같은 문제를 다시 볼 수 있어야 원인을 찾고, 최소 예제로 줄이고, 수정 후 다시 깨지지 않게 테스트로 잠글 수 있어요. 오늘은 재현 → 최소 예제 → 수정 → 회귀 테스트의 루프를 한 장으로 만듭니다.

## 핵심 개념 — Minimal Reproduction 과 Regression Test

### Minimal Reproduction — 가장 작은 실패 장면

최소 재현은 버그를 보여주는 가장 작은 입력과 코드입니다. 전체 앱을 켜야만 보이는 버그라도, 원인을 좁히다 보면 특정 함수, 특정 props, 특정 데이터 조합으로 줄일 수 있어요. 이 과정이 없으면 에이전트는 증상 근처만 계속 만집니다.

최소 재현에는 세 가지가 들어갑니다.

| 항목 | 질문 |
| --- | --- |
| 입력 | 어떤 데이터나 사용자 행동이 문제를 만들었나요? |
| 기대값 | 정상이라면 무엇이 일어나야 하나요? |
| 실제값 | 지금은 무엇이 다르게 일어나나요? |

이 표를 채우면 “왜 깨지는지”보다 “어떻게 깨지는지”가 먼저 보입니다. 원인은 그다음이에요.

### Regression Test — 같은 버그가 다시 못 들어오게 잠근다

회귀 테스트는 과거 버그를 자동으로 기억하는 장치입니다. 오늘 수정한 버그가 한 달 뒤 리팩터링으로 다시 생길 수 있어요. 회귀 테스트가 있으면 그 순간 빨간불이 켜집니다. [Lesson 4.5](/lessons/write-tests-with-coding-agent) 의 Red → Green 순서를 그대로 씁니다. 먼저 버그를 재현하는 실패 테스트를 만들고, 수정 후 통과시킵니다.

### Root Cause Note — 증상과 원인을 분리한다

“버튼이 안 눌림”은 증상입니다. “disabled 상태 계산이 빈 문자열을 false 로 처리하지 못함”은 원인입니다. 에이전트가 증상만 고치면 다른 화면에서 같은 문제가 반복될 수 있어요. 마지막에 근본 원인을 한 문장으로 남기면, 다음 버그에서 검색 가능한 지식이 됩니다.

## 그래서 오늘 할 일

오늘 만들 것은 “버그 재현과 수정 루프 템플릿”입니다.

1. 최근 반복되었거나 설명이 애매했던 버그 하나를 고릅니다.
2. 실제 사용자 장면을 3~7단계 재현 스텝으로 씁니다.
3. 기대값과 실제값을 나눠 적습니다.
4. 에이전트에게 최소 재현 후보를 만들게 합니다.
5. 실패 테스트를 먼저 만들고 Red 상태를 기록합니다.
6. 수정 후 Green 상태와 검증 명령 결과를 기록합니다.
7. 산출물 [`bug-repro-loop.md`](/lessons/bug-reproduction-loop) 에 근본 원인과 재발 방지 메모를 남깁니다.

처음에는 번거롭지만, 같은 버그를 두 번 설명하는 것보다 훨씬 싸요. 특히 비전공자라면 “원인”을 바로 말하려 하지 말고, 재현 장면을 잘 적는 데 집중하면 됩니다. 좋은 재현은 에이전트가 원인을 찾을 수 있는 가장 강한 입력입니다.
