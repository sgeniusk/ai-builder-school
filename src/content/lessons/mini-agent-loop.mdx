## 도입 — 학습자에게 말 거는 한 장면

사용자가 이렇게 묻습니다. “우리 제품 환불 정책을 찾아서, 핵심만 요약하고, 고객 답변 초안으로 저장해줘.” 이 요청은 한 번의 답변으로 끝나지 않습니다. 먼저 관련 문서를 찾아야 하고, 찾은 내용을 읽어야 하고, 답변 초안을 만들어야 하고, 마지막으로 파일이나 저장소에 남겨야 해요.

[Lesson 7.1](/lessons/function-calling) 에서 만든 단일 함수 호출은 한 번의 손동작에 가깝습니다. 날씨를 조회하거나, 문서를 검색하거나, 파일을 저장하는 한 가지 행동이에요. 그런데 실제 작업은 손동작 여러 개가 순서대로 이어집니다. 그래서 필요한 것이 미니 에이전트 루프입니다.

미니 에이전트 루프는 거대한 자율 에이전트가 아닙니다. “생각하고, 도구를 부르고, 결과를 보고, 다음 단계를 정한다”를 몇 번만 반복하는 작은 작업 흐름이에요. 마치 초보 직원에게 “자료 찾아 보고, 요약하고, 초안으로 남겨 주세요. 다만 5단계 안에서 멈추세요”라고 맡기는 것과 비슷합니다.

## Lesson 7.1 다음에 왜 이 레슨인가

단일 함수 호출은 AI가 도구를 정확히 집는 연습입니다. 이제는 그 도구를 순서 있게 쓰는 연습이 필요해요. 검색 한 번으로 충분하지 않으면 다시 검색하고, 검색 결과가 부족하면 질문을 좁히고, 요약이 끝나면 저장 도구를 부르는 식입니다.

이 흐름은 [Lesson 1.0](/lessons/what-is-a-loop) 의 7-step 루프를 세션 안에 작게 접어 넣은 모습입니다. 문제를 보고, 개념을 적용하고, 미션을 실행하고, 검증하고, 회고하는 큰 구조가 몇 번의 Think, Act, Observe로 줄어든 거예요. 그래서 이 레슨은 “루프”라는 말을 앱 코드 안에서 처음 직접 구현하는 자리입니다.

또 하나의 연결점은 [Lesson 4.7](/lessons/nightly-loop-pattern) 입니다. 그 레슨은 잠든 사이 도는 외부 루프였어요. 오늘의 미니 에이전트 루프는 내 손으로 지켜보는 인-세션 루프입니다. 둘은 형제지만 시간감이 다릅니다. 오늘은 요청 하나를 처리하는 동안 작게 돌고, Lesson 4.7은 사람이 자리를 비운 동안 정해진 예산 안에서 돕니다.

## 핵심 개념 — ReAct 와 Stopping Criteria

ReAct는 Reasoning과 Acting을 엮은 패턴입니다. 우리식으로는 생각, 도구 호출, 관찰의 반복이라고 보면 됩니다. 모델이 먼저 “환불 정책 문서를 찾아야 한다”고 판단합니다. 그다음 `searchDocs` 도구를 호출합니다. 도구 결과를 관찰한 뒤 “이제 고객 답변 초안을 쓸 수 있다”거나 “결과가 부족하니 다시 검색해야 한다”고 정해요.

Observation은 도구가 돌려준 결과입니다. 검색 결과 목록, 파일 저장 성공 메시지, API 오류, 권한 거부 같은 것들이 모두 관찰이에요. 관찰을 잘 기록해야 다음 판단이 좋아집니다. 모델에게 도구 결과를 그냥 긴 문자열로 던지기보다, 성공 여부와 핵심 데이터와 오류 이유를 나눠 주는 편이 안정적입니다.

Stopping Criteria는 멈춤 조건입니다. 에이전트 루프에서 가장 위험한 실패는 끝없이 도는 것입니다. 그래서 최대 스텝 수, 최대 시간, 성공 조건, 실패 조건을 코드로 둡니다. 예를 들어 “최대 5스텝”, “저장 성공 시 종료”, “같은 도구가 같은 인자로 두 번 실패하면 사람에게 넘김”처럼 적습니다.

작은 루프일수록 로그가 중요합니다. 사용자가 보는 최종 답과 별도로, 개발자는 어떤 생각으로 어떤 도구를 불렀는지 볼 수 있어야 해요. 디버그 UI가 있으면 좋지만, 처음에는 Markdown 로그나 콘솔 로그만으로도 충분합니다.

## 그래서 오늘 할 일

오늘 만들 것은 “검색, 요약, 저장 3도구 미니 에이전트”입니다. 큰 일을 시키지 않습니다. 사용자가 질문을 던지면, 에이전트는 관련 문서를 검색하고, 결과를 요약하고, 초안을 저장합니다.

먼저 도구 세 개를 정의합니다. `searchDocs`는 질문을 받아 후보 문서를 돌려줍니다. `summarizeContext`는 검색 결과를 짧은 답변 재료로 바꿉니다. `saveDraft`는 최종 초안을 파일이나 DB에 저장합니다. 각 도구는 [Lesson 7.1](/lessons/function-calling) 처럼 스키마와 검증 규칙을 가져야 해요.

그다음 루프를 만듭니다. 시작 상태에는 사용자 목표와 빈 로그가 있습니다. 매 스텝마다 모델은 다음 행동을 고릅니다. 도구를 부르면 관찰을 로그에 붙이고 다음 스텝으로 갑니다. 완료 답변을 내면 루프를 끝냅니다. 최대 스텝을 넘기면 “여기까지 진행했고, 다음은 사람이 확인해야 해요”라고 멈춥니다.

산출물 [`mini-agent-loop-design.md`](/lessons/mini-agent-loop) 에는 도구 목록, 루프 상태, 종료 조건, 실패 처리, 관찰 로그 형식을 적습니다. 특히 같은 도구가 반복 호출되는 상황을 꼭 테스트하세요. 미니 에이전트는 똑똑해 보이는 것보다 멈출 줄 아는 것이 먼저입니다.

## 마무리

미니 에이전트 루프는 Phase 7의 중심입니다. AI가 도구를 한 번 쓰는 데서 끝나지 않고, 관찰을 다음 판단의 재료로 삼기 시작하기 때문이에요. 하지만 오늘 만든 것은 작은 형제입니다. 잠든 사이 오래 도는 [Lesson 4.7](/lessons/nightly-loop-pattern) 의 외부 루프가 아니라, 요청 하나 안에서 사람이 추적할 수 있는 인-세션 루프예요. 다음 레슨에서는 이 루프가 쓸 수 있는 도구와 쓰면 안 되는 도구를 나눕니다. 코딩 에이전트에게 쳤던 울타리를, 이제 우리가 만든 에이전트에게도 쳐야 합니다.
