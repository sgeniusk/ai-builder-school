## "누가 뭐 하기로 했지? — 일주일 전 회의의 저주"

월요일 아침 스탠드업. "지난주에 누가 그거 하기로 했잖아요." 모두 고개를 갸웃거립니다. 회의록은 있어요 — 40줄짜리 줄글이 공유 드라이브에 올라가 있긴 합니다. 하지만 **누가, 언제까지, 무엇을** 해야 하는지는 그 줄글 속 어딘가에 파묻혀 있어요.

[Lesson 06](/lessons/enforcing-output-format)에서 만든 3단 방어선이 "AI 출력을 기계가 읽을 수 있게" 만들었다면, 오늘은 그 기법을 **회의록이라는 비정형 입력**에 적용합니다. 줄글을 구조화된 Action Item으로 바꾸는 파이프라인이에요.

## 회의록의 진짜 문제

회의록 자체가 문제가 아니에요. 문제는 **회의록에서 Action Item을 뽑아내는 과정**이에요.

| 현재 | 이상 |
| --- | --- |
| 줄글로 된 회의 메모 | 구조화된 3섹션 (결정/Action Item/오픈 이슈) |
| "그거 하기로 했잖아" 구두 추적 | 담당자·마감·산출물이 명시된 표 |
| 정리할 시간이 없어 나중에 | 회의 직후 5분 안에 공유 |

이 간극을 메우는 게 오늘의 파이프라인이에요.

## 3섹션 구조화 — 결정 · Action Item · 오픈 이슈

{/* Source · DOC · Anthropic "Structured outputs" — https://platform.claude.com/docs/en/build-with-claude/structured-outputs */}
{/* Source · DOC · Anthropic "Prompt engineering overview" — https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/overview */}

회의에서 나오는 정보는 크게 세 종류예요.

1. **결정 사항 (Decisions)** — "A 방향으로 가기로 했다." 누가 결정했는지, 근거가 뭐였는지.
2. **Action Item** — "김태우가 금요일까지 보고서 초안을 완성한다." 담당자·마감·산출물 셋이 다 있어야 함.
3. **오픈 이슈 (Open Issues)** — 결론이 안 난 것. 다음 회의에서 다시 다룰 것.

이 세 가지가 분리되면 회의록의 가치가 달라집니다. "누가 뭐 하기로 했지?"에 5초 안에 답할 수 있어요.

## 구조화 프롬프트 — 비정형을 정형으로

{/* Note: 3섹션 분류(Decisions / Action Items / Open Issues)는 일반 회의 관행을 이 학교 운영자가 정리한 것이며, 외부 표준명은 아닙니다. */}

비정형 회의 메모를 3섹션으로 바꾸는 프롬프트의 핵심은 **분류 기준을 명시하는 것**이에요.

```
Task:
아래 회의 녹취/메모를 읽고 세 섹션으로 분류해 주세요.

Constraints:
- 결정 사항: 합의된 결론 + 근거 한 줄
- Action Item: 담당자 / 마감일 / 산출물 세 필드 필수
  - 담당자가 불명확하면 "[미정]"으로 표시
  - 마감이 언급 안 됐으면 "[미정]"으로 표시
- 오픈 이슈: 결론이 안 난 것 + 다음 단계 한 줄
- 원문에 없는 내용을 추가하지 마세요

Output:
## 결정 사항
| # | 결정 | 근거 |
|---|------|------|
| 1 | ... | ... |

## Action Items
| # | 담당자 | 마감 | 산출물 |
|---|--------|------|--------|
| 1 | ... | ... | ... |

## 오픈 이슈
| # | 이슈 | 다음 단계 |
|---|------|-----------|
| 1 | ... | ... |
```

[Lesson 05](/lessons/structure-of-good-prompts)의 4축이 여기서도 쓰이죠. 특히 Output 섹션에 표 예시를 넣으면 답이 표 형태를 유지하기 쉬워져요.

## 발화자 귀속 — "누가 말한 건지"

녹취록에서 가장 자주 빠지는 정보가 **발화자**예요. "그건 좀 리스크가 있지 않을까요?"가 누구 입에서 나왔는지에 따라 Action Item의 무게가 달라집니다.

발화자 귀속을 잘 하려면 입력 단계에서 도와줘야 해요.

| 방법 | 정확도 | 추가 작업 |
| --- | --- | --- |
| 화자 분리된 녹취 (Otter.ai 등) | 높음 | 녹취 도구 설정 |
| 메모에 이니셜 표기 (TW: ...) | 중간 | 회의 중 수기 표기 |
| AI에게 추측 맡기기 | 낮음 | 없음 (비추천) |

가장 현실적인 건 두 번째예요. 회의 중 메모를 적을 때 `TW:`, `MJ:` 같은 이니셜을 앞에 붙이는 습관만 들여도 AI가 발화자를 구분하는 데 큰 도움이 됩니다.

## JSON 출력 — 자동화로 가는 다리

{/* Source · DOC · Anthropic "Structured outputs" — https://platform.claude.com/docs/en/build-with-claude/structured-outputs */}

사람이 읽을 회의록은 마크다운 표면 충분해요. 하지만 Notion·Slack·Jira에 자동으로 보내려면 **JSON**이 필요합니다.

[Lesson 06](/lessons/enforcing-output-format)의 3단 방어선을 여기 그대로 적용하세요.

```json
{
  "decisions": [
    { "id": 1, "decision": "...", "rationale": "..." }
  ],
  "action_items": [
    { "id": 1, "owner": "김태우", "due_date": "2025-10-10", "deliverable": "..." }
  ],
  "open_issues": [
    { "id": 1, "issue": "...", "next_step": "..." }
  ]
}
```

1선 프롬프트에 "순수 JSON, 앞뒤 설명 금지"를 명시하고, 2선 API 스키마 강제를 깔면, 이 JSON이 Notion API에 바로 들어갑니다.

## Action Item 누락 체크 — AI도 놓친다

AI가 회의 메모에서 Action Item을 뽑을 때 가장 자주 놓치는 유형이 있어요.

- **암묵적 Action Item** — "그러면 다음 주에 한 번 봐야겠네요"는 Action Item인데, AI가 감상으로 분류
- **조건부 Action Item** — "만약 A가 확정되면 B를 시작하자"는 조건이 붙어서 분류가 흐릿
- **복수 담당자** — "둘이 같이 하자"에서 AI가 한 명만 잡음

이런 케이스를 잡으려면 **2차 검토 호출**을 쓰세요. 초안 결과를 다시 AI에게 주면서 "위 원문에서 Action Item으로 분류되지 않았지만 실행이 필요한 항목이 있나요?"라고 한 번 더 물어보는 거예요. [Lesson 09](/lessons/automate-report-drafts)의 자기 검토 패턴과 같은 원리입니다.

## 그래서 오늘 만들 한 폴더

오늘 만들 건 `meeting-pipeline/` 폴더예요.

| 파일 | 역할 |
| --- | --- |
| `structure-prompt.md` | 3섹션 구조화 프롬프트 (마크다운 표 출력) |
| `json-prompt.md` | JSON 출력용 프롬프트 (자동화 연결) |
| `review-prompt.md` | Action Item 누락 체크 프롬프트 |
| `checklist.md` | 사람 검수용 체크리스트 (담당자·마감·발화자) |

이 네 파일이 있으면 회의 직후에 바로 구조화된 회의록을 만들고, "누가 뭐 하기로 했지?" 같은 질문에 빠르게 답할 수 있는 팀이 돼요.
