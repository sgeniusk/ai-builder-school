## 회의 끝나자마자 사라지는 결정들

한 시간짜리 회의가 끝나요. 머릿속엔 *"오늘 결정된 게 중요했지"* 의 잔상만 남고, 다음 회의 시작 5 분 전이라 정리할 시간이 없어요. 일주일 뒤, *"지난주에 누가 캠페인 카피 V2 까지 작성하기로 했죠?"* 슬랙이 떨어집니다. 아무도 기억 못 해요.

진짜 문제는 *기억력* 이 아니라 **회의 → 행동 사이의 다리가 없는 것**. 결정은 회의에서 나오는데 정리는 따로 안 되니, 회의가 의사결정이 아니라 *대화* 로 끝나버려요. 오늘 만들 건 그 다리예요. 회의 끝남 → 5 분 뒤 자동으로 *Action Item · 결정 · 오픈 이슈* 가 정리된 노트가 손에 들어옵니다.

## 보고서 자동화 다음에 왜 이 레슨인가

[Lesson 3.1](/lessons/automate-report-drafts) 에서 *정기 출력* 을 자동화했어요. 이번엔 *불규칙 입력* 을 자동화합니다. 회의는 매번 다르지만 — *추출하는 구조* 는 같아요 (결정 / Action / Open). 입력은 자유, 출력은 강제.

[Lesson 2.2](/lessons/enforcing-output-format) 에서 만든 *JSON 스키마 + 재시도 루프* 가 여기서 핵심으로 등장합니다. 회의록 자동화의 절반은 *구조화 출력* 의 정확도예요.

## 핵심 개념 — Action Item / Speaker Attribution / Decision Lock

### Action Item — *세 칸이 다 차야 진짜다*

| 칸 | 없으면 |
| --- | --- |
| **담당자** | 책임 분산 → 아무도 안 함 |
| **기한** | 시간 무한정 → 한 달 뒤에도 그대로 |
| **산출물** | "검토하기" 같은 모호한 동사 → 완료 시점 불명확 |

세 칸이 다 안 차 있으면 그건 Action Item 이 아니라 *대화의 일부* 예요. 모델에게 *"세 칸이 비면 'incomplete' 로 표시하라"* 를 명시.

### Speaker Attribution — *누가 무슨 말을 했는지가 책임을 정한다*

같은 *"제가 할게요"* 가 누구 입에서 나왔는지에 따라 책임자가 달라져요. 녹취가 있다면 화자 정보를 같이, 메모만 있다면 *"발화자 추정 + Confidence"* 로 표기.

### Decision Lock — *결정과 토론은 다르다*

회의에서 *"고민해 보자"* 와 *"결정했다"* 가 같은 톤으로 나옵니다. AI 한테 *결정은 명시적 신호* (예 — *"OK"*, *"가자"*, *"확정"*, *"do it"*) 를 기준으로 분류하라고 명시. 그 외엔 *"오픈 이슈"* 로 회수.

## 그래서 오늘 할 일

가장 최근에 끝난 *Action Item 이 많이 나왔던* 회의 1 개를 골라 파이프라인으로 돌려봐요.

작업 흐름,

1. **입력 준비** — 녹취 또는 메모 텍스트. 길이는 상관 없음 (5분)
2. **스키마 박기** — 아래 JSON 을 시스템 프롬프트에 (5분)

````
{
  "decisions": [{ "topic": "...", "outcome": "...", "trigger": "..." }],
  "action_items": [
    {
      "assignee": "string|null",
      "task": "string",
      "due_date": "YYYY-MM-DD|null",
      "deliverable": "string",
      "status": "complete|incomplete"
    }
  ],
  "open_issues": [{ "topic": "...", "next_step": "..." }]
}
````

3. **재시도 루프 부착** — [Lesson 2.2](/lessons/enforcing-output-format) 의 코드 그대로. 파싱 실패 시 *"수정된 JSON 만 반환"* (10분)
4. **첫 실행 + 검증** — 5축 트러스트 체크 ([Lesson 1.4](/lessons/checks-before-trusting-ai-output)) 중 **Fact + Origin** 두 축에 집중. 누락된 Action 이 있는지 손으로 확인 (10분)
5. **배포 자리 결정** — Slack DM 자기 자신 / Notion DB / GitHub Issue 자동 생성 중 하나로. 매번 같은 자리에 떨어지게 (5분)
6. **저장** — `prompt-library/work-os/meeting-notes.md` 로 (Lesson 2.4) (5분)

산출물 [`meeting-pipeline.md`](/lessons/meeting-notes-pipeline) — 스키마 + 재시도 코드 + 배포 자리 + 첫 5 회 측정 표.

## 마무리 — 회의를 *기록 가능한 단위* 로 만듭니다

이 파이프라인이 자리 잡으면 회의의 가치가 두 배가 돼요. 회의 *중* 에는 발언에 집중하고, 회의 *후* 에는 5 분 안에 정리된 노트가 손에 들어옵니다.

[Lesson 3.3](/lessons/research-workflow) 에선 *입력은 더 흩어진* 형태 (웹 검색·문서·메모) 를 같은 방식으로 묶는 *리서치 파이프라인* 으로 확장. 패턴은 같지만 입력의 폭이 커집니다.

회의록은 가장 자주 깨지는 자동화 후보예요. 매주 *5 분 측정* 으로 1 회 통과율을 추적하면 한 달 안에 95% 안정이 들어옵니다.
