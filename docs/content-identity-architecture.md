# 콘텐츠·정체성 구조 — 레슨·프로젝트·템플릿 + 캐릭터

> AI Builder School 의 콘텐츠 3층(레슨·프로젝트·템플릿)을 어떻게 연결하고, 그 진척을 캐릭터 성장으로 잇는지 정의한다. 새 콘텐츠·기능을 더할 때 이 문서를 기준으로 삼는다.
> 작성 2026-05-18 — 설계 대화 정리본.

## 1. 왜 이 문서

레슨 84 · 프로젝트 13 · 템플릿 24 가 쌓였는데 세 콘텐츠 타입의 관계가 흐릿하다. 특히 템플릿이 어느 프로젝트에도 연결되어 있지 않아 독립적으로 떠 있다. 이 문서는 세 층의 역할을 고정하고, 연결·대시보드·업적·캐릭터까지의 구현 순서를 정한다.

## 2. 3층 모델

| 층 | 역할 | 한 줄 | 방침 |
|---|---|---|---|
| 레슨 | 이론·개념 바탕 | "왜·무엇" | 핵심. 컴팩트 유지 — 무분별히 안 늘림 |
| 프로젝트 | 실전 목표 | "이걸 만든다" | 진열창. 적극적으로 키움 |
| 템플릿 | 재사용 실행 자산 | "이걸로 한다" | 프로젝트에 종속 — 독립 콘텐츠 아님 |

**여정(journey)은 4번째 층이 아니다.** 여정은 페르소나별 *경로·렌즈* 다 — 레슨·프로젝트를 "당신에게 맞는 순서"로 묶어 보여주는 오버레이일 뿐, 그 자체로 콘텐츠를 갖지 않는다.

**권고** — 여정의 핵심 일은 "페르소나 → 프로젝트" 연결이다. 사람을 끄는 건 프로젝트이므로, 여정 페이지는 매칭되는 프로젝트(`project.targetJourneys`)를 앞세워야 한다.

### 관계

```
레슨  ──준비──▶  프로젝트  ──실행 지원──▶  템플릿
                    ▲
            여정이 경로로 묶어 보여줌
```

## 3. 연결 스펙 — 끊긴 곳을 잇는다

현재 연결 상태.

| 연결 | 상태 |
|---|---|
| 프로젝트 → 레슨 (`keyLessons`, 마일스톤 `fallbackLesson`) | 있음 |
| 레슨 → 프로젝트 (역방향) | 없음 |
| 프로젝트 ↔ 템플릿 | 없음 — 핵심 공백 |

### 할 일

- `Project` 에 `templateSlugs: string[]` 추가 — 이 프로젝트가 쓰는 템플릿.
- 프로젝트 상세 페이지에 "이 프로젝트에 쓰는 템플릿" 섹션.
- 템플릿 쪽에 역방향 "이 템플릿을 쓰는 프로젝트".
- 레슨 페이지에 역방향 "이 레슨이 준비시키는 프로젝트" (`keyLessons` 역참조).
- `validateContent` 에 `templateSlugs` 슬러그 검증 추가.

**권고 — 연결은 동시에 감사다.** 13×24 를 잇다 보면 (1) 어느 프로젝트에도 안 쓰이는 잉여 템플릿, (2) 받쳐줄 템플릿이 없는 프로젝트가 드러난다. 잉여 템플릿은 *프로젝트에 배정하거나 잘라낸다* — 연결만 하고 죽은 템플릿을 남기면 "애매함"이 그대로다. (콘텐츠 삭제는 사용자 승인 후.)

**권고 — 연결은 프로젝트 단위로 충분하다.** 마일스톤별 템플릿 참조까지 가지 않는다. 프로젝트의 `starterPrompt`(마일스톤 내장 1회용)와 템플릿(여러 곳에서 재사용)은 구분을 유지한다 — 둘을 합치지 않는다.

## 4. 대시보드 — "지금 뭘 만들 수 있나"

`/me` 빌더 대시보드에 프로젝트 층을 추가한다.

**권고 — 새 추적 장치를 만들지 않는다.** 프로젝트 준비도는 이미 있는 레슨 진척에서 파생한다.

- 프로젝트 준비도 = 완료한 `keyLessons` 수 / 전체 `keyLessons` 수.
- 상태 3단계 — 잠김(0%) · 준비 중(일부) · 시작 가능(`keyLessons` 100%).
- `/me` 에 "내 프로젝트" 섹션 — 내 여정에 맞는 프로젝트를 준비도와 함께. "시작 가능" 은 강조.
- 여정 페이지에도 매칭 프로젝트를 준비도와 함께 노출.

localStorage 에 새로 저장할 것 없음. "시작했음" 같은 명시 상태는 필요해지면 그때 추가한다.

## 5. 업적(achievement) 시스템

**권고 — 업적도 저장하지 않고 파생한다.** 빌더 랭크(`builderRank`)가 이미 쓰는 패턴 — "현재 진척 상태에 대한 조건"으로 정의하면 이벤트 시스템도 persistence 도 필요 없다.

- 업적 = `{ id, label, domain, test(progress) => boolean }`.
- `domain` 은 여정 계열(creator·engineer·founder·practitioner·adopter·explorer) + 활동 계열(streak·first-deliverable 등).
- 예 — "창작 새싹 = creator 여정 레슨 5개 완료", "첫 빌드 = 프로젝트 1개 '시작 가능' 도달".
- `src/lib/achievements.ts` 에 정의 목록 + 판정 함수.
- 노출 — 캐릭터 프로필 모달에 업적 격자 (캐릭터와 한 화면).

## 6. 캐릭터 — 액세서리 레이어

목표 — "창작 업적 쌓은 토끼 → 노래부르는 토끼" 같은 정체성 성장.

**권고 — 풀 리드로우 금지, 액세서리 레이어로.** 동물 6 × 여정 변형을 통째로 그리면 수십 장이다. 기본 동물은 그대로 두고, 업적이 작은 SVG 소품을 위에 합성한다.

- 액세서리 6~8개 — 마이크(creator) · 렌치(engineer) · 왕관(founder) 등, 32×32 픽셀 소품.
- 한 번 그리면 동물 6종 전부에 합성 — 6 + 8 조각으로 48 조합을 공짜로 얻는다.
- `CharacterAvatar` 에 `accessory` 레이어 추가. 획득 액세서리는 업적에서 파생.
- 여러 개 획득 시 표시할 것을 사용자가 고른다 (동물 선택기와 같은 패턴).
- 최상위 마일스톤 1~2개만 "풀 변형" 스프라이트로 와우 포인트를 남긴다 — 선택 사항, 후순위.

## 7. 구현 순서

| Phase | 내용 | 비고 |
|---|---|---|
| 1 | 프로젝트↔템플릿 연결 + 템플릿 감사 | 구조 토대. 먼저 |
| 2 | 대시보드 프로젝트 준비도 + 여정 페이지 프로젝트 노출 | 1 위에 |
| 3 | 업적 정의·판정·노출 | 진척에서 파생 |
| 4 | 캐릭터 액세서리 레이어 | 업적 → 액세서리 |
| 5 | 프로젝트 키우기 — 도구별 실전 프로젝트 추가 | 1 이후 상시. 새 프로젝트는 템플릿을 달고 태어남 |

**권고 — 프로젝트는 Phase 1 이전에 키우지 않는다.** 구조 없이 키우면 끊긴 콘텐츠만 늘어난다.

## 8. 진행·열린 결정

- Phase 1 완료 — 프로젝트↔템플릿 연결. 감사 결과 24개 중 22개 연결, 고아 2개.
- Phase 2 완료 — 프로젝트 준비도(keyLessons 파생)를 `/me`·프로젝트 카드·여정 페이지에 노출.
- 잉여 템플릿 처리 — 고아 2개(`github-issue-brief-mission`·`pr-review-mission`, 팀 협업 코딩 워크플로). **결정 — Phase 5에서 팀 협업 코딩 프로젝트를 추가해 흡수(C안).** 그때까지 보존.
- 이 묶음(Phase 1~4)의 로드맵 위치 — `future-roadmap.md` 의 v0.8(모션·일관성)·v0.9(연습장)와 어떻게 배치할지. 별도 마일스톤으로 빼는 게 자연스럽다.
- 캐릭터 "풀 변형" 스프라이트를 실제로 만들지 — Phase 4 마무리에 재검토.
