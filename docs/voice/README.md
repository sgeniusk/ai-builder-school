# 보이스 정의서 — 지음의 사람 목소리

> 레슨 산문이 "AI가 쓴 글"로 수렴하는 것을 막는 단일 기준점.
> lesson-writer 스킬이 집필·수정 전에 읽고, `npm run eval:rubric`이 채점 프롬프트에 통째로 주입한다.

## 왜 있나

"해요체·콜론 금지" 같은 규칙은 오류를 막을 뿐 목소리를 만들지 못한다. AI는 규칙을 완벽하게 지키면서도 AI처럼 쓸 수 있고, 89개 레슨이 같은 규칙으로 수렴할수록 균질함은 오히려 강해진다. 그래서 기준을 둘로 나눈다.
- **골든 샘플 (positive)** — 운영자가 직접 쓴 글. 모사할 목소리의 원본.
- **AI-스멜 금지 목록 (negative)** — 모델 평균 문체의 지표. 집필 때 피하고 채점 때 검출한다.

## 골든 샘플

`docs/voice/sample-*.md`로 둔다. **아직 비어 있다 — 운영자가 채워야 작동하는 절반이다.**

| 항목 | 기준 |
|---|---|
| 수량 | 3~5편 |
| 출처 | 운영자가 처음부터 끝까지 직접 쓴 글만 — AI 초안을 다듬은 글은 제외 |
| 종류 | 블로그 · 강의 스크립트 · 길게 쓴 메모 등 무언가를 설명하는 글이면 충분 |
| 길이 | 한 편당 500~2000자면 충분 (전문이 아니어도 됨) |

각 샘플 첫머리에 출처 한 줄을 적는다 (예 `> 출처 — 2025-11 블로그 "..."`).

**공개 저장소 주의** — 이 저장소는 GitHub Pages로 공개돼 있다. 공개해도 되는 글만 넣는다.

샘플이 없는 동안에도 아래 금지 목록은 그대로 작동한다.

## AI-스멜 금지 목록

실제 레슨에서 발견된 패턴 기준. 하나하나는 무해해 보여도, 반복되면 "사람이 아니라 시스템이 썼다"는 신호가 된다.

1. **수사 의문문 남발** — "차이가 보이시죠?", "시작해 볼까요?". 한 레슨에 최대 1회.
2. **섹션마다 펀치라인 마무리** — 모든 섹션이 강조 한 줄로 끝나는 균질 리듬. 어떤 섹션은 그냥 끝나도 된다.
3. **정밀 계량 과장** — "답을 두 단계 위로 끌어올립니다", "80%의 흔들림이 사라져요" 같은 근거 없는 수치·단계 비유.
4. **em-dash(—) 남용** — 문단마다 등장하면 스멜. 쉼표나 두 문장으로 풀 수 있는지 먼저 본다.
5. **균질한 문단 리듬** — 모든 문단이 2~3문장으로 비슷한 길이. 사람 글은 길이가 들쭉날쭉하다.
6. **깔끔한 삼분 병렬의 반복** — "A하고, B하고, C합니다" 구조가 글 전체에서 거듭 등장.
7. **전형적 가상 시나리오** — "매주 금요일, 김 대리는..." 식 지어낸 인물·장면. 실제 겪은 일화가 최선이고, 없으면 시나리오는 짧게 끝낸다.
8. **빈 강조 부사** — "정말", "꽤", "굉장히"로 채운 문장. 구체적 사실로 대체하거나 뺀다.

## 실제 일화 — 가장 강한 인간화

이질감의 절반은 문체가 아니라 "실제 일어난 일"의 부재에서 온다. AI는 그럴듯한 가상 시나리오를 무한히 쓰지만 실제 사건은 쓰지 못한다. 레슨에 운영자가 실제로 겪은 일화(만들다 실패한 것, 실제로 받은 질문, 실제로 날린 비용)가 들어갈 자리가 보이면, 집필 에이전트는 지어내지 말고 `{/* TODO(운영자): 실제 일화 */}` 주석으로 자리만 비워 둔다.

## 기계 연결

- **lesson-writer 스킬** — 산문을 쓰거나 고치기 전에 이 문서와 샘플을 읽는 것이 의무.
- **`npm run eval:rubric`** — 이 문서 + `sample-*.md`를 채점 프롬프트에 주입. tone 축이 골든 샘플과의 거리로 채점되고, AI-스멜은 flag로 나온다.
