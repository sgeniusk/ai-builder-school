# 모델 1위는 한 달짜리 정보다 — 라우팅과 오케스트레이션이 빌더의 새 척추다

> 2026 상반기 프런티어 AI 특강 · narrative draft · 검토 시한 2026-11-30

## Thesis

빌더가 이 강의를 닫으며 가져갈 단 한 줄.

> **"어느 모델이 1위냐"는 한 달짜리 정보다. 빌더의 진짜 경쟁력은 여러 모델을 한 워크플로 안에서 어떻게 짜는가 — 라우팅과 오케스트레이션 — 에서 나온다.**

이 글의 모든 사실은 위 한 줄을 증명하기 위해 동원된다. 사실은 결론이 아니라 증거다.

---

## 1. 한 달 반 동안 SWE 1위가 세 번 바뀌었다

2026년 4월 16일, Anthropic이 Claude Opus 4.7을 GA로 내놨다. SWE 부문에서 다른 모든 모델을 따돌렸다. 일주일 뒤 4월 23일, OpenAI가 GPT-5.5를 공개하며 14개 벤치마크 SOTA를 주장했다. 다시 한 달도 안 된 5월 19일 Google I/O에서 Gemini 3.5 Flash가 Terminal-Bench 2.1에서 76.2%를 찍으며 자사 3.1 Pro를 능가했다 — 출력 속도는 4배 빠르고 가격은 한 토막이다.

빌더가 4월 16일에 "이제 Opus 4.7로 통일하자"라고 결정했다면, 그 결정은 **35일 뒤 무효가 됐다**.

이 빠르기가 흔들기다. 분기·연 단위였던 모델 경쟁이 주·월 단위로 바뀌었다. 어제의 최강이 오늘의 차선이고 내일의 평균이다.

## 2. 빌더가 흔히 빠지는 함정 — "최강 하나 정하자"

이 흔들림 앞에서 빌더가 가장 자주 하는 생각은 이렇다.

- "Claude로 다 풀자, 가장 똑똑하니까."
- "GPT로 통일하자, 가장 많이 쓰여서 자료가 많으니까."
- "Gemini로 옮기자, 가장 싸니까."

전부 같은 패턴이다 — **단일 모델로 모든 작업을 처리하겠다는 함정**. 이 함정의 문제는 두 가지다.

첫째, 발표마다 1위가 바뀌니까 그 선택이 한 달 안에 outdated가 된다. 두 번째 갈아탈 때 추상화가 없으면 코드 전체를 다시 짠다.

둘째, 어떤 모델도 모든 작업에서 1위가 아니다. Opus 4.7은 어려운 SWE에서 최강이지만 출력 토큰 $25/M로 라이트 작업까지 다 돌리면 비용이 폭탄이다. GPT-5.5는 컴퓨터 사용·도구 호출에 강하지만 일부 한국어 카피라이팅에서는 다른 모델보다 어색하다. Flash는 빠르고 싸지만 가장 어려운 추론에서는 Opus·GPT-5.5에 밀린다.

**"내 작업"이라는 단일한 게 없다.** 빌더가 만드는 워크플로는 보통 5~10가지 다른 모양의 작업으로 쪼개진다. 그 모든 작업에 최강인 모델은 존재하지 않는다.

## 3. 세 빅테크가 같은 신호를 보내고 있다 — "단일 모델 시대는 끝났다"

흥미로운 건, 모델을 만드는 회사들 본인이 빌더에게 같은 메시지를 던지고 있다는 점이다.

### Anthropic의 발신

4월 16일 Opus 4.7을 GA로 낸 뒤, 5월 6일 Code with Claude 컨퍼런스가 있었다. 그 컨퍼런스의 헤드라인이 무엇이었는가 — **신규 플래그십 모델이 없었다.** 대신 Claude Code의 도구·플러그인 생태계 확장이었다.

이 의도가 명확하다. 이미 Claude Code는 작업 난이도에 따라 Sonnet과 Opus를 자동 전환한다 — 즉 Anthropic 본인이 빌더에게 "한 모델을 더 쓰지 말고 둘을 라우팅해서 써라"라고 가르치고 있다. 새 플래그십을 내놓는 것보다 라우팅 환경을 두텁게 만드는 게 더 가치 있다고 본 것이다.

게다가 Opus 4.7 가격($5/M 입력 · $25/M 출력)이 그대로 유지됐다. 이건 우연이 아니라 신호다 — **다회차 에이전트 워크플로에서 Opus만 쓰면 비용이 누적될 것이고, 그러니 라이트 작업은 다른 모델로 보내라는 것**.

### OpenAI의 발신

4월 23일 GPT-5.5를 공개했지만 거기서 멈추지 않았다. 그 뒤 한 달 안에 **방향이 다른 세 가지를 동시에 풀어놓았다**.

첫째, 5월 7일 **GPT-5.5-Cyber** — 보안 특화 모델을 검증된 보안팀에 별도 출시. 범용 GPT-5.5와 같은 모델 군 안에서 도메인별로 가지치기.

둘째, 5월 중순 **"Work with Codex from anywhere"** — ChatGPT 모바일 앱에 Codex를 탑재하고, Remote SSH·Hooks를 정식 기능으로 제공. 빌더는 휴대폰에서 데스크탑 Codex 세션의 활성 스레드·승인·플러그인·프로젝트 맥락을 이어 다룰 수 있다. 스크린샷·터미널 출력·diff·테스트 결과가 실시간으로 전달되며, 파일/자격증명은 보안 릴레이 계층으로 원격 머신에 그대로 유지된다.

셋째, Business/Enterprise를 위한 **Programmatic access tokens** — Codex를 CI·자동화 파이프라인에 박을 수 있는 채널.

세 발표의 공통점은 **"모델 자체"가 아니라 "어디서 어떤 작업에"** 라는 차원의 확산이다. OpenAI도 단일 GPT-5.5로 모든 걸 다 처리하는 게 아니라, "런타임 × 도메인 × 모델"의 3차원으로 잘게 쪼개는 방향으로 움직였다. **빌더에게 보내는 신호 — 작업별로 다른 환경을 골라 써라.**

### Google의 발신

I/O 2026의 카드 구성을 보라.

- **Gemini 3.5 Flash** — 빠르고 싸고 좋게. 라이트 작업 대량 처리용.
- **Gemini 3.5 Pro** — 다음달. 어려운 작업용.
- **Gemini Spark** — Gmail·Drive·Calendar 통합 24/7 에이전트. 24시간 운영되는 라이트 에이전트는 Flash 위에서 돈다.
- **Gemini Omni** — 영상 입출력 멀티모달. 매체별 특화.
- **AI Ultra $100** — 컴퓨트 기반 과금. 사용량 모델 전환.

한 모델로 통일이 아니라 **모델 라인업 자체를 가격·속도·매체별로 분화**한 것이다. Spark가 Flash 위에서 도는 구조는 빌더에게 그대로 모범이다 — 어떤 작업이 어디서 어떤 모델로 돌아갈지 라우팅 설계를 빅테크가 먼저 보여줬다.

### 세 회사의 공통 신호

다 다른 회사인데 같은 방향을 가리키고 있다.

**단일 모델 시대는 끝났다. 모델 × 런타임 × 도메인의 매트릭스에서 라우팅하는 시대로 들어갔다.**

이걸 모델 회사들이 빌더에게 먼저 신호로 보내고 있다. 빌더가 이 신호를 못 읽고 "1위 모델 하나"에 집착하면, 빅테크가 이미 떠난 정거장에 혼자 서 있게 된다.

## 4. 빌더의 새 척추 — 다섯 축

그래서 빌더의 일은 어디로 옮겨가야 하는가. 모델 발표가 와도 그대로 얹히는 다섯 가지 구조 위로.

### 4-1. 추상화 — vendor 잠금에서 벗어나기

모델 호출을 한 줄 교체로 끝낼 수 있게 만든다. Vercel AI Gateway 같은 통합 API를 쓰거나, 자체 어댑터 한 겹을 둔다. 프롬프트·tool 정의·검증 로직을 모델 SDK와 분리한다.

이게 없으면 다음 모델 발표에 갈아타는 비용이 무한이다. 빌더가 가장 먼저 만들어야 하는 단 하나의 추상이다.

### 4-2. 평가셋 — "내 일"에 맞는 검증

공개 벤치마크는 평균치다. SWE-Bench Verified 1위 모델이 내 PR 리뷰에서도 1위는 아니다.

빌더가 가장 자주 하는 5개 작업을 골라 각각 10건씩 모은다. 입력·기대 출력·중요 평가 기준을 묶으면 50건짜리 내 평가셋이 된다. 모델이 새로 나올 때마다 이 50건만 돌리면 1~2시간 안에 진짜 1위가 가려진다.

이 평가셋이 발표 수치보다 신뢰할 수 있는 유일한 의사결정 자산이다.

### 4-3. 라우팅 — 작업별로 모델을 가르기

다섯 작업이 있다고 가정해보자.

1. 회의록 요약 — 빠르고 싸야 한다 → Flash 또는 Haiku 급
2. PR 리뷰 코멘트 — 정확도가 중요하지만 비용도 본다 → Sonnet 또는 GPT-5.5
3. 어려운 리팩토링 제안 — 최고 추론력 필요 → Opus 4.7
4. 이미지·다이어그램 분석 — 비전 강한 모델 → Opus 4.7 (고해상도) 또는 Gemini Omni
5. 모바일에서 코드 실행 — 컴퓨터 사용 도구 필요 → GPT-5.5 + Codex iOS

이 매핑을 코드로 박는다. 작업 타입을 인자로 받아 적절한 모델을 호출하는 라우팅 함수 하나. Claude Code가 Sonnet/Opus를 자동 전환하는 것과 같은 패턴을 빌더가 자기 워크플로에 적용한다.

### 4-4. 하네스 — 6축으로 에이전트 환경 설계

모델은 한 부품이다. 결과를 만드는 건 그 주변 여섯 가지다 — **도구·피드백·복구·제약·토폴로지·평가**.

같은 GPT-5.5도 어떤 도구를 쓰게 하고, 실패를 어떻게 잡고, 무엇을 못하게 막고, 누구와 협업시키고, 어떻게 점수 매기느냐에 따라 결과가 5배 차이 난다.

모델은 6개월마다 갈아엎힌다. 잘 설계한 하네스는 모델 위에 그대로 얹힌다. 모델보다 하네스에 투자하는 게 자산이 쌓이는 방향이다.

### 4-5. 오케스트레이션 — 다중 모델을 한 워크플로에서 조율

라우팅이 한 단계의 결정이라면, 오케스트레이션은 여러 단계가 엮인 흐름이다.

예 — 이메일 응답 자동화 워크플로.

1. 메일 분류 — Flash로 빠르게
2. 컨텍스트 검색 — Embeddings로
3. 응답 초안 — Sonnet으로 (정확하지만 비용 적당)
4. 톤·문법 검증 — Haiku로 가볍게
5. 어려운 케이스만 — Opus로 에스컬레이션

이 5단계가 한 워크플로 안에서 작업 단위로 다른 모델에 위임된다. 빌더의 새 일은 이런 흐름을 설계하는 것이다 — 모델 하나를 고르는 것이 아니라.

## 5. 행동 — 이 강의를 닫고 한 시간 안에 할 일

다섯 축이 추상적으로 들리면 가장 시작하기 쉬운 한 가지부터 해본다 — **평가셋**.

1. 자기가 자주 하는 작업 5개를 적는다
2. 각 작업에 대해 가장 중요한 평가 기준 2~3개를 정한다
3. 각 작업의 실제 사례 3~5건을 모은다
4. 슬라이드 끝의 평가셋 빌더에 입력하고 프롬프트를 복사한다
5. GPT-5.5 · Opus 4.7 · Gemini Flash 셋에 같은 프롬프트를 던지고 표로 비교한다

한 시간이면 끝난다. 그러나 그 한 시간이 다음 모델 발표에 흔들리지 않는 의사결정 자산을 만든다. 다음 특강이 갱신되어도 이 50건짜리 평가셋은 그대로 살아남는다.

## 6. 이 강의는 낡는다, 그러나 메시지는 그대로다

이 특강에 적힌 모델 이름·버전 번호·벤치마크 수치는 곧 옛것이 된다. Mythos가 풀리고, GPT-6이 나오고, Gemini 4.0이 Flash를 또 한 단계 끌어올릴 것이다. 검토 시한(2026-11-30)이 지나면 이 글은 새 사실로 다시 쓴다.

그러나 메시지는 그대로다.

다섯 축 — 추상화·평가셋·라우팅·하네스·오케스트레이션 — 은 어떤 모델이 1위가 되든 그 위에 그대로 얹힌다. 빌더가 모델 발표에 휘둘리지 않는 이유는 이 다섯 축이 모델 위에 있기 때문이다. 모델이 부품으로 갈아끼워질 때, 척추가 그 부품을 받아낸다.

**모델이 부품이라면, 빌더는 그 부품을 조립하는 설계자다.** 이 강의가 끝났을 때 가져갈 것은 어떤 모델의 이름이 아니라, 그 부품들을 어떻게 조립하는가에 대한 다섯 가지 척추다.
