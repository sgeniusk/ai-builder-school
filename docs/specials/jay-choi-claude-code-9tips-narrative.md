## 발표자의 thesis (영상 핵심 한 줄)

> **"신선한 컨텍스트가 비대한 컨텍스트를 이긴다."**

발표자가 직접 "이 문장 하나만 기억하셔도 됩니다"라고 말한 영상의 중심 메시지. 동시에 영상 전체 구조의 골격이기도 하다 — 좋은 걸 다 깔고 다 시키는 게 답이 아니라, 안 할 것을 안 하고 진짜 필요한 것만 명시적으로 시키는 게 답.

## 9가지 정수

### Part 1 — 하지 말 것 3가지

**1. 처음부터 풀 세팅하지 말 것**
- 핵심 — 기본을 파악하기도 전에 스킬·라이브러리를 다 깔아놓는 함정.
- 증거 — Claude Code 창시자 보리스가 X에서 공개 — "내 세팅은 놀랍게도 바닐라다." 본인이 만든 툴인데 거의 커스텀 안 하고 쓴다.
- 처방 — 기본 Claude Code로 시작 → 같은 실수·같은 워크플로 반복이 보일 때 그 패턴 하나에 대해서만 스킬을 만든다. 실제 니즈에 따라 자라나는 세팅이 가장 좋다.

**2. FOMO에 휩쓸리지 말 것**
- 핵심 — 매일 쏟아지는 새 기능을 다 따라가야 한다는 불안을 다스리기.
- 증거 — 발표자도 매주 새 기능 (루틴·데스크탑 리디자인·Claude Design 등)이 쏟아지면서 뽐뿌를 느낀다고 인정. 하지만 솔직히 그 기능들은 당장 내 워크플로와 안 맞거나 필요 없을 확률이 높다.
- 처방 — 새 기능을 무시하라는 게 아니라 "안 쓸 거면서 안 쓰면 뒤처질까 봐 일단 깔고 보는" 불안에서 자유로워질 것.

**3. "알아서 해줘" 식 프롬프트 금지**
- 핵심 — AI가 혼자 다 해주길 바라지 말고 인간이 검증한 방식을 패키지로 만들어 명확히 지시.
- 증거 — Anthropic도 공식적으로 이 방향을 발신 중. "이거 만들어 줘", "이 버그 알아서 고쳐 줘" 식 프롬프트로는 단번에 완성도 있는 결과가 거의 안 나온다.
- 비유 — 자율주행 차를 타도 어디로 갈지 내비게이션은 직접 찍어야 한다. 목적지는 AI가 아니라 빌더가 정한다.

### Part 2 — 해야 할 것 3가지

**4. 컨텍스트 관리 (가장 중요한 스킬)**
- 핵심 — **"신선한 컨텍스트가 비대한 컨텍스트를 이긴다."** 정보 많이 주면 더 잘할 거라는 직관은 LLM에서 틀리다.
- 증거 — 대화 메시지·파일·명령어가 쌓일수록 성능이 떨어진다. 관련 없는 정보가 많을수록 Claude가 잘못된 방향으로 빠진다.
- 처방 — CLAUDE.md는 어차피 80% 정도만 따른다고 알려져 있고 길수록 중요한 규칙이 무시될 확률이 높다. **150~200줄 안에서 끝내라.**

**5. 진짜 계획 + 명시적 지시**
- 핵심 — "플랜 모드 켜라"가 아니다. 진짜 계획 — 어떤 식으로 구현할지, 완료 기준이 뭔지를 먼저 정리.
- 증거 — 특정 스킬을 만들어 놨어도 그냥 "이거 구현해 줘"라고만 하면 AI가 그 스킬을 안 쓰고 자기 멋대로 한다.
- 처방 — 플랜 모드를 그냥 수락하지 말고 **적극적으로 편집**할 것. 그리고 AI가 가진 도구·스킬·규칙을 진짜 쓰게 하려면 알아서 잘해주길 바라지 말고 **대놓고 명시적으로 지시**. "너무 세세하게 하나하나 다 지시해야 하는 거 아닌가?" — 그게 AI를 잘 쓰는 거다.

**6. 검증 장치 마련 + 그 장치 감시 (보리스가 단 하나 꼽으면 이것)**
- 핵심 — Claude에게 스스로 일을 검증할 방법을 주기만 해도 결과물 품질이 2~3배 올라간다.
- 증거 — 검증 장치가 없으면 빌더가 유일한 피드백 루프가 된다. 매번 사람이 확인·피드백·고치기 사이클을 돌린다.
- 함정 — AI가 검증 장치를 **악용**한다. 테스트가 실패하면 코드를 고치는 게 맞는데, AI가 테스트 자체를 코드에 맞춰서 통과만 되도록 의미 없이 수정해버린다.
- 처방 — 검증 루프 만들어줬다고 끝이 아니라 **테스트 자체가 올바른 걸 검증하고 있는지** 그 테스트를 사람이 본다.

### Part 3 — 매일 쓰는 실전 꿀팁 3가지

**7. 세션 관리 — 리와인드 적극 활용**
- 핵심 — Claude Code 팀이 "컨텍스트 관리 잘하는 사람의 단 하나의 습관"으로 꼽은 게 리와인드.
- 시나리오 — A를 시켰는데 실패. 그냥 "A 말고 B로 해줘"라고 보내면 A의 실패 기록이 컨텍스트에 그대로 남아 Claude가 그 실패에 계속 영향을 받는다. 리와인드를 쓰면 A 시도 기록이 없는 깨끗한 상태에서 B를 시도한다 → 결과물이 훨씬 깨끗.
- 처방 — 대화에서 **ESC×2** = 리와인드. 작업 끝났을 때 **/clear** = 세션 마침표. 요약할 때 **/compact "이번엔 A에 집중하고 B는 버려"** 식으로 방향 직접 지시.

**8. 스킬 & 플러그인 생태계**
- 핵심 — 항상 유지되는 규칙(CLAUDE.md) 대신, 필요할 때만 로드되는 스킬을 자산으로 쌓아라.
- 증거 — 현재 Claude 생태계가 이 방향으로 움직이는 중. CLAUDE.md는 세션 시작 시 무조건 로드되지만 스킬은 실제 필요할 때만 로드.
- 처방 — 먼저 GitHub의 유명 리포지토리(예 디스텍 슈퍼파)에서 남이 만든 좋은 스킬을 가져다 쓴다. 본인이 반복하는 작업이 있다면 그것만 스킬로 저장해 가며 쌓는다. 스킬은 유지 비용이 없고 자산이라고 할 만큼 유용.

**9. 열린 사고 (태도)**
- 핵심 — 영상 초반 "FOMO 피하라"와 모순처럼 들리지만 다르다. FOMO는 불안, 열린 사고는 사고.
- 차이 — FOMO — "이거 안 쓰면 뒤처지는 거 아닌가" 불안에 쫓겨 깔고 본다. 열린 사고 — "이 기능은 왜 생긴 거지? 내 프로젝트엔 어떻게 적용할 수 있을까?" 질문을 던지는 태도. 당장 안 써도 괜찮다, 이해하고 있으니까.
- 마지막 메시지 — 진짜 좋은 건 남의 워크플로를 그대로 복사하는 게 아니라 그걸 **이해해서 내 상황에 맞게 번역해서 적용하는 능력**. 자동화할 수 없고, 시간이 지날수록 자산이 되는 부분. 이게 AI 시대를 가장 건강하게 살아가는 방식.
