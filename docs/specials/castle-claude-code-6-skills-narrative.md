## 원전 thesis (마지막 슬라이드에서 직접 명시)

> **"모델은 실행하고, 사람은 결정한다. 6가지 스킬은 그 결정을 선명하게 만드는 도구다."**

Jay Choi 영상의 "신선한 컨텍스트 > 비대한 컨텍스트"가 *원리*였다면, 캐슬은 그 원리를 **6개의 구체적 명령어 스킬**로 풀어낸다. 두 특강이 같은 결의 메시지를 다른 추상화 레벨에서 다룬다.

## 6 스킬 정수

### 1. grill-me — 의도를 캐묻는다

**핵심.** 사용자 요청을 받자마자 코드를 쓰지 않고, 핵심 질문 3~10개를 던진다. 답변이 모이면 그제서야 의도가 명료해진다.

**왜.** "한 줄 요청에서 200줄 사고가 난다." 사용자는 "이런 거 만들어줘" 한 줄로 시작 → 모델은 그 한 줄에서 전체 명세 추측 → 결과 잘못된 방향으로 200줄 만든 뒤 다시 고친다. 진짜 문제는 모델이 아니라 요청자도 자신이 뭘 원하는지 흐릿하다는 것.

**무엇.** 핵심 질문 4축
- 누가 이걸 쓸 것인가
- 성공 기준은 무엇인가 — 어떤 상태가 되어야 끝인가
- 빼도 되는 건 무엇인가 — 범위 밖
- 이미 비슷한 게 있나 — 재활용

**언제.** 새 기능 시작 시(가장 큰 효과). 모호한 요청 ("그냥 좀 깔끔하게 정리해줘" 류). 해석이 둘 이상일 때. 변형 `grill-with-docs`는 관련 코드·문서를 먼저 읽은 뒤 캐묻는 변종. 버그 수정처럼 의도 명확한 작업엔 굳이 쓰지 않는다.

**결정적 예시 — 결제 페이지.**
- BAD — "결제 페이지 만들어줘" → 바로 코딩 → 카드·계좌·페이팔 다 구현 → 사용자 "PG는 토스만 쓸 건데..." → 절반 재작성.
- GOOD — grill-me 발동 → "PG 정해졌나? 결제 수단 몇 개? 모바일/PC 둘 다? 정기결제?" → 30초 답변 → 토스 단일, 카드만, PC만 → 정확한 범위로 코딩.

**효과.** 앞에서 30초, 뒤에서 30분 절약. 잘못된 방향으로 가는 비용이 거의 0. 사용자도 자기 요구사항을 명확히 인식 (부수 효과). 다음 단계 to-prd 품질이 통째로 올라간다. "이게 아닌데..."가 사라진다 → 신뢰의 시작점.

### 2. to-prd — 합의를 기획서로 굳힌다

**핵심.** grill-me에서 모은 답변을 한 페이지 마크다운 PRD로 정리. PRD는 모델과 사람 모두의 외부 메모리.

**왜.** "말로 한 합의는 휘발된다." 채팅으로 합의한 건 1시간 뒤면 흐려진다. 코딩 시작하면 "어 그게 뭐였지" 다시 물어보게 된다. 문서가 없으면 모델은 매 세션마다 처음부터 추측한다.

**무엇.** PRD 한 페이지 — 목표 / 사용자 시나리오 / 기능 명세 / 범위 밖 (Non-goals) / 성공 기준. Product Requirements Document. 한 페이지 마크다운으로 충분.

**언제.** grill-me 직후 (합의가 휘발되기 전). 기능 크기가 1일 이상 걸릴 때. 여러 세션에 걸쳐 작업할 때 (휘발 방지). 팀원과 공유해야 할 때. 30분짜리 단발 작업엔 과하다, 단 다시 들춰볼 일 있으면 그래도 쓴다.

**효과.** 새 세션이 제로콜드로 시작 — PRD 파일 하나만 읽히면 끝. "그거 어떻게 하기로 했죠?" 같은 질문 사라짐. 모델이 임의로 범위 확장하는 일 줄어든다 ("범위 밖" 명시 효과). 사람이 휴가 가도 다른 사람이 PRD만 읽고 이어받을 수 있다.

### 3. to-issues — 작업 단위로 쪼갠다 (수직 슬라이스)

**핵심.** PRD를 수직 슬라이스 단위로 분해. 큰 PRD를 통째로 시키면 아무도 못 읽는다.

**왜.** 200줄짜리 PR보다 20줄짜리 PR 10개가 안전하다. 모델은 작은 단위에 집중도가 훨씬 높다. 롤백·디버깅이 작은 단위에서만 의미 있다. 여러 사람·세션이 병렬로 작업할 수 있다.

**무엇 — 핵심 개념 수평 X · 수직 ✓.**
- **수평 분해 (BAD).** Issue 1 API만 / Issue 2 UI만 / Issue 3 DB만 → 다 합쳐야만 동작 → 중간 진행도 0 또는 100.
- **수직 슬라이스 (GOOD).** Issue 1 비로그인 결제 차단 / Issue 2 카드 결제 1건 / Issue 3 결제 실패 메시지 → 각자 끝나면 그것만으로 동작 → 매 슬라이스가 머지 가능.

**언제.** PRD가 만들어진 직후 (to-prd → to-issues가 표준 순서). 한 PR이 200줄 넘을 것 같을 때. 여러 사람 병렬 작업. 실패해도 부분만 롤백하고 싶을 때.

**효과.** PR 리뷰 시간이 1/5 수준으로 줄어든다. 한 슬라이스 단위로 머지 → 매일 production에 진척이 보인다. 모델 응답도 짧고 정확해진다 (작은 작업 = 적은 추측). 실패해도 한 슬라이스만 되돌리면 된다.

### 4. caveman — 답변을 단순화 (75% 토큰 절감)

**핵심.** 모델 응답에서 fluff를 제거해 응답 토큰을 75% 절감 (Julius Brussee 케이스). "동굴인 말투"로 응답.

**왜.** "Sure! I'd be happy to..." 같은 fluff가 한 세션에서 수십 KB로 누적된다 — 컨텍스트도, 비용도, 속도도. 응답이 짧을수록 사용자가 핵심을 빨리 찾는다.

**무엇.**
- **제거** — a/an/the, just/really/basically, pleasantries (sure/of course)
- **유지** — 기술 용어, 코드, 에러 메시지 (정확성 100%)
- Fragment OK · 짧은 동의어 사용 (big not extensive)
- **코드/커밋/PR은 정상 문체** — 캐주얼 모드 꺼짐

**언제.** 긴 대화 세션 — 컨텍스트 부담이 큰 작업. 반복 작업 — lint 고치기, 패턴 적용 등 빠른 사이클. 의식적으로 컨텍스트 윈도우를 아끼고 싶을 때. 토글 — `/caveman lite | full | ultra`.

**결정적 예시 — 같은 답, 4분의 1 길이.**
- BEFORE (일반 모드, ~280토큰) — "Sure! I'd be happy to help you with that. The issue you're experiencing is likely caused by an off-by-one error in the auth middleware..."
- AFTER (caveman full, ~60토큰) — "Bug in auth middleware. Token expiry check use `<` not `<=`. Fix:"

**효과.** 응답 토큰 약 75% 절감. 응답 속도 체감 상승 (출력이 짧으니 바로 시작). 핵심만 남으니 리뷰도 빨라진다. 큰 작업을 한 세션에서 끝낼 확률 상승.

### 5. handoff — 세션 인수인계 (메타 스킬)

**핵심.** 컨텍스트가 가득 차기 전, 현재 세션의 상태를 HANDOFF.md 한 파일에 정리. 새 세션은 이 파일을 읽고 시작한다.

**왜.** "긴 세션은 결국 잃어버린다." 세션이 길어지면 컨텍스트 자동 압축 → 정보 손실. 모델이 앞 단계 의도를 잊고 같은 결정을 다시 추측. "방금 결정한 거 다시 알려줘"가 반복된다. 새 세션 시작이 인지 비용 0이 아니다 — 정리 없으면.

**무엇 — HANDOFF.md 4섹션.**
- 지금까지 한 것 — 완료된 항목
- 어디까지 됐는가 — 진행 중인 작업의 정확한 상태
- 다음 차례 — 즉시 할 다음 액션
- 주의사항·미해결 질문 — 함정과 결정 대기 항목

**언제.** 컨텍스트가 60% 이상 찼을 때 (자동 압축 직전). 작업이 자연스럽게 멈추는 지점 (큰 단위 완료 후). 다음 날, 다른 사람에게 넘길 때. 디버깅처럼 상태가 많은 작업의 중간 지점.

**HANDOFF.md 샘플.**
```
# Handoff — 결제 페이지 / Issue #2
## 완료
- 토스 SDK 설치, ENV 키 등록 (.env.local)
- /pay 라우트 추가, 카드 결제 호출 OK
## 진행 중
- 결제 성공 webhook 핸들러 작성 중
- src/api/pay/webhook.ts 에서 서명 검증 부분 막힘
## 다음 차례
1. webhook 서명 검증 (TOSS_SECRET 사용)
2. 성공 시 user_courses 테이블에 row 추가
## 주의
- 결제 실패 시 retry는 v2에서 — 지금은 ✗
- 환불은 PRD 범위 밖
```

**효과.** 세션 길이 한계가 실질적으로 무력화. 협업 시 인수인계 비용이 1/10 수준. 모델 응답 품질 유지 — 자동 압축 손실을 사전 회피. 사람 입장에서도 "어디까지 했더라" 사라진다.

### 6. improve-codebase — 주기적 정리 (Deep module)

**핵심.** John Ousterhout의 *A Philosophy of Software Design* — **Deep module** 원칙으로 코드베이스를 일관되게 유지. 좁은 인터페이스, 깊은 내부.

**왜.** "정리는 저절로 안 된다." 기능이 늘면 같은 일 하는 코드가 3곳에 흩어진다. 디렉터리 구조가 점점 직관과 어긋난다. "다음에 정리하자" → 영원히 안 한다. 모델에게 정리를 시키는 주기적 루틴이 필요.

**무엇 — Deep module 핵심.**
- **Shallow module (BAD)** — 많은 함수 노출 / 각 함수는 단순 래퍼 / 호출자가 내부를 다 알아야 함 → 추상화가 적자
- **Deep module (GOOD)** — 한두 개 함수만 노출 / 내부에서 복잡도를 흡수 / 호출자는 인터페이스만 알면 됨 → 추상화가 흑자

**언제.** 한 큰 PR이 끝났을 때 — 정기 점검 한 번씩. 비슷한 코드가 3곳에서 보일 때. 디렉터리 구조가 직관과 어긋나기 시작할 때.

**예시 명령.**
```
improve-codebase: shallow한 모듈, 중복 패턴,
                  죽은 코드 찾아서 리포트만 작성.
                  수정은 다음 단계에서.
```

**효과.** 기술 부채가 임계점을 못 넘는다 (작게 자주 정리). 모델이 잘 작동하는 코드베이스 유지 — 간결하고 일관됨. 새 기능 추가 비용이 점점 줄어든다. 인수인계(handoff)도 쉬워진다 — 정돈된 코드는 설명도 짧다.

## 큐레이션 위치

이 영상은 **원전 화자(캐슬)의 thesis**를 따른다. 빌더 스쿨의 새 thesis를 위에 얹지 않는다. 우리가 추가하는 가치는 — (1) 41페이지를 9장으로 압축한 시각적 정수, (2) 한국어 학습자가 빠르게 흡수할 수 있는 인터랙티브 구조, (3) **마지막 슬라이드 한 장에서 빌더 스쿨 학습자가 이 6스킬을 어떤 Stage·레슨과 짝지어야 하는지**.

빌더 스쿨 ↔ 6스킬 매핑.
- `grill-me`·`to-prd` → Stage 2 "Ask & Refine"의 핵심 도구
- `to-issues` → Stage 3 "Collaborate"의 작업 분해
- `caveman` → Stage 4 "Build & Verify"의 응답 최적화 (그리고 우리의 다섯 척추 중 "컨텍스트 관리"와 같은 결)
- `handoff` → 세션 간 인수인계 — 빌더 스쿨의 학습 사이클이 길어졌을 때 그대로 적용
- `improve-codebase` → Stage 5 "Ship"의 정리 단계, 또는 진행한 프로젝트의 주기적 정비
