# Fast Campus "전현준의 하네스 엔지니어링" 강의 분석

> **출처** — https://fastcampus.co.kr/data_online_harness
> **분석일** — 2026-05-12 (듀이)
> **분석 범위** — 가격·혜택·강사 프로필 제외, 커리큘럼·메시지·교육 방식·인포그래픽 패턴 중심

## Context (왜 이 분석을 했는가)

AI Builder School은 "초보자가 빌더로 성장하는 12-Phase 커리큘럼"을 만드는 중이고, 같은 시기에 Fast Campus가 **AI 코딩 에이전트 + 하네스 엔지니어링**을 정면으로 다루는 강의를 출시했다. 주제와 톤이 직접 경쟁·참고 관계에 있다. 이 강의의 페르소나 카피, 학습 로드맵 구조, 키워드 인포그래픽 배치, "환경 설계"를 USP로 잡는 방식은 AI Builder School의 phases/journeys 설계와 랜딩 카피에 그대로 응용 가능하다.

## 1. 랜딩 페이지 메시지 흐름 (Hero → Persona → USP)

페이지는 다음 5개 명시 섹션으로 구성된다 (가격/강사 제외하면 사용자가 봐야 할 흐름).

| # | 섹션 | 역할 |
|---|------|------|
| Hero | 강의명 + 한 줄 소개 + 태그 | 누구를 위한, 무엇을 약속하는지 |
| Persona | 3단 "왜 차이가 벌어지나?" 훅 | 통증 환기 |
| 정의 | "*하네스 엔지니어링이란?" 펼침형 | 개념 정의 + 메타포 |
| 01 | 강의 핵심 포인트 (4 카드) | USP 4축 |
| 03 | 커리큘럼 핵심 키워드 (이미지 슬라이드 8개) | 학습할 키워드를 한눈에 |
| 04 | 커리큘럼 자세히 (학습 로드맵 5 STEP + 8개 파트) | 실제 구조 |

### 1.1 Hero 카피

- 메인 — **"전현준의 하네스 엔지니어링: Claude code · Codex 완벽 가이드"**
- 서브 — *"현업 전문가들이 먼저 알아본 전현준의 하네스 엔지니어링 노하우. Claude Code · Codex를 10배 더 잘 활용하는 방법부터, 스스로 일하는 환경을 설계하고 프로젝트에 적용하는 방법까지 40+개의 실습을 통해 실무에 바로 적용 가능한 수준으로 완성합니다."*
- 태그 — `#AI #멀티에이전트 #코덱스 #클로드코드 #하네스엔지니어링` (해시태그 = 검색 SEO + 학습 키워드 노출)
- 강의 정보 칩 — **온라인 · 9파트 · 약 20시간 · 사전지식 필요 · 평생소장 · 학습자료 제공 · 커뮤니티 운영**

### 1.2 Persona 훅 (3-Step Reveal)

페이지는 거대한 인포그래픽 이미지 + 짧은 카피를 3번 반복한다.

1. **도발** — "클로드 코드 쓰고 생산성 올라갔다? **정말 만족하고 계신가요?**"
2. **격차 제시** — "똑같은 코딩 툴을 써도 누군가는 **하루 걸릴 일을 1시간에 끝내고** 누군가는 **수정 요청만 반복**합니다."
3. **호기심 유발** — "똑같이 AI를 쓰는데 **왜 이렇게 차이가 벌어질까요?**"

→ 정답 카피 — **"내가 잠 자는 동안에도 24시간 돌아가는 환경 설계의 핵심"**

### 1.3 하네스 엔지니어링 정의 (펼침형)

> "AI가 '말'이라면 하네스는 그 말을 원하는 방향으로 움직이게 만드는 **장비(마구)** 입니다."

4가지 구성요소를 명시한다.

- **컨텍스트** — AGENTS.md, CLAUDE.md 등 프로젝트 문서
- **구조** — 아키텍처 제약, 모듈 경계 규칙
- **피드백 루프** — 린터, 타입 체커, 테스트, pre-commit hook
- **자동화** — CI/CD 파이프라인, PR 검증 프로세스

결론 — *"AI가 단순히 코드를 생성하는 것이 아니라 스스로 수정하고 개선할 수 있는 구조를 만드는 것"*

### 1.4 핵심 카피 (반복 출현)

> **"이제 코드를 작성하는 사람이 아니라 에이전트가 안정적으로 일하도록 환경을 설계하는 사람이 되어야 합니다."**

이 한 줄이 강의 전체의 thesis다. AI Builder School도 "도구 사용자 → 빌더"라는 자기 thesis를 한 문장으로 결정해두면 좋다.

## 2. 01. 강의 핵심 포인트 — 4축 USP 카드

각 카드는 `핵심 0N` 라벨 + 굵은 헤드라인 + 1줄 설명으로 동일 포맷.

| # | 헤드라인 | 설명 |
|---|--------|------|
| 01 | **Claude Code × Codex로 완성하는 10x 개발 워크플로우** | 기초 설치/환경 세팅부터 두 에이전트의 강점을 조합해 실제 개발 작업에 활용 |
| 02 | **하네스 엔지니어링까지 담은 국내 유일의 커리큘럼** | 에이전트를 **사용**하는 수준을 넘어 개발 시스템을 **설계**하는 Agent Harness |
| 03 | **실제 프로덕트에 활용하기 위한 40개+ 프로젝트 실습** | 아이디어 빠른 구현(MVP) → 기존 코드 확장(실무 프로젝트) |
| 04 | **매 달 찾아가는 Live 월간 하네스 세미나** | "AI 도구는 빠르게 변화 → 지속 업데이트로 최신 반영" |

**구조의 시사점**

- 4축이 **기능(01) → 차별점(02) → 실습(03) → 지속성(04)** 의 논리적 사다리
- USP 02의 "국내 유일" 같은 단정형 카피는 *교육 시장에서 효과적*. AI Builder School은 "초보자를 빌더로 데려가는 한국어 풀스택 커리큘럼" 같은 단정형 카피를 만들 수 있다.
- USP 04의 "월간 라이브 세미나" = **재방문/리텐션 장치**. 평생소장의 약점(콘텐츠 노후화)을 보완. AI Builder School에 적용한다면 "월간 빌더 라운지", "분기별 커리큘럼 업데이트 로그" 같은 후속 콘텐츠 약속.

## 3. 03. 커리큘럼 핵심 키워드 — 8개 인포그래픽 슬라이드

페이지에서 가장 강한 시각 자산. 키워드별 카드 1장씩 (2단 4열, 슬라이드 캐러셀).

| # | 키워드 슬라이드 제목 | 한 줄 설명 카피 |
|---|---------------------|----------------|
| 1 | **Claude Code · Codex 기능** | "설치 방법 & 다양한 기능을 배우고, 함께 사용하는 하이브리드 전략을 학습합니다." |
| 2 | **메모리 · 컨텍스트** | "AI가 프로젝트를 제대로 이해하도록 컨텍스트를 설계하고 관리하는 방법을 학습합니다." |
| 3 | **Git · 권한 · 샌드박스** | "AI가 코드베이스를 안전하게 수정할 수 있도록 Git 기반 워크플로와 권한·실행 환경을 설정합니다." |
| 4 | **Planning · SDD · Test Loop** | "작업 목표와 완료 기준을 명확히 정의하고 Spec 기반 계획과 테스트 검증 루프를 설계합니다." |
| 5 | **Agent Skills** | "반복 작업을 줄이고 에이전트의 행동을 안정화하는 Skill 기반 자동화 구조를 이해합니다." |
| 6 | **Hooks** | "특정 작업 상황에서 자동으로 동작하는 Hook 이벤트를 활용한 개발 자동화 전략을 학습합니다." |
| 7 | **Multi Agent (Agent Teams)** | "여러 에이전트를 역할별로 분리해 협업 구조로 운영하는 멀티 에이전트 패턴을 이해합니다." |
| 8 | **Agent Harness** | "문서, 규칙, 자동화 시스템을 포함해 AI 에이전트가 실제 개발 환경에서 일하도록 만드는 작업 시스템을 설계합니다." |

**구조 디자인 시사점**

- 8개 키워드를 **2행 × 4열 슬라이드**로 분할 — 한 행에 학습 난이도 흐름이 들어감 (행1은 기초 4종 / 행2는 자동화·하네스 4종).
- 키워드 = 명사구, 카피 = 동사로 끝나는 일관된 문체 ("~를 학습합니다 / ~을 설계합니다 / ~을 이해합니다") → **읽는 사람이 "어떤 행동을 배우는지" 즉시 파악**.
- AI Builder School은 12 Phase가 너무 많다고 느껴질 수 있으니 **상위 키워드 8개로 압축한 카드 슬라이더**를 만드는 게 효과적일 수 있다. 예시는 `프롬프트 기초 / 데이터 다루기 / 컴포넌트 만들기 / API 연결 / 자동화 / 배포 / 운영 / 빌더 마인드셋`.

## 4. 04. 학습 로드맵 — 5 STEP (메타 구조)

커리큘럼 자세히 보기 위에 먼저 **5단계 메타 로드맵**을 보여준다. 각 STEP은 라벨 + 헤드라인 + 1줄 설명 + "대표 학습 내용" 4~5개 불릿으로 구성.

| STEP | 제목 | 1줄 설명 | 대표 학습 내용 |
|------|------|---------|--------------|
| 01 | **Claude Code · Codex 시작하기** | 두 에이전트의 사용법 + 하이브리드 개발 방식 | 설치/사용 가이드, 작동 방식 이해, 비전공자용 터미널·Git, 코드베이스 분석·수정 |
| 02 | **AI에게 프로젝트 이해시키기** | CLAUDE.md · AGENTS.md로 컨텍스트 설계 | 공식 문서 활용 전략, WHAT·WHY·HOW 프롬프트, CLAUDE.md/AGENTS.md 설계, 컨텍스트·세션 관리 |
| 03 | **실무 개발 워크플로우 자동화하기** | Git, MCP, 훅으로 실무 연결 | 권한·승인·샌드박스, Git/리뷰 흐름, MCP 외부 도구, Agent Skills, Hooks·정책 |
| 04 | **멀티 에이전트 구조와 하네스 설계하기** | 역할 분리·협업 구조 설계 | 서브에이전트, 멀티 에이전트 패턴, Agent Harness 구조, 협업 구조 |
| 05 | **실제 프로젝트에 적용하기** | Agent Harness로 서비스 개발/개선 | PRD 기반 MVP(0→1), 기존 코드 확장(1→100), 하이브리드 활용, oh-my-claudecode 뜯어보기 |

**시사점**

- **2단 계층 구조** (상위 5 STEP → 하위 8 파트 73 클립) 가 학습자에게 인지 부담을 줄여준다. AI Builder School의 12 Phase도 상위 **3~5 Stage**로 묶어 보여주면 첫 인상이 훨씬 가벼워진다.
- STEP 05의 **"0→1 / 1→100" 프레이밍**이 매우 강력하다 — "어디까지 가르치는지"를 한 번에 전달. AI Builder School의 마지막 Phase도 "내 첫 빌더 프로젝트 출시 (0→1)" 같은 식으로 라벨링 가능.
- 각 STEP 박스의 **"대표 학습 내용"** 형식 = 학습자가 "내가 모르는 게 여기 있다"를 빠르게 발견하게 만드는 장치. AI Builder School의 Phase 카드에도 비슷한 3-5개 키 토픽 미리보기를 도입할 만하다.

### Part 1. AI 코딩 에이전트와 친해지기 (Claude code, Codex) (12 클립 · 3:01:31)

**Ch1. AI 코딩 에이전트가 바꾸는 개발 방식** (41:16)
- 1. 자동완성과 AI 코딩 에이전트는 무엇이 다른가

**Ch2. 비전공자도 따라오는 터미널과 Git 최소 기초** (46:20)
- 1. 터미널이 무섭지 않게 — 폴더 이동과 파일 다루기
- 2. Git이 꼭 필요한 이유 — 저장, 되돌리기, 브랜치 감각 키우기
- 3. 오늘 바로 쓰는 Git 기본 루틴 — clone → branch → commit → push

**Ch3. Claude Code 시작하기** (48:33)
- 1. Claude Code 설치와 로그인 한 번에 끝내기

**Ch4. Codex 시작하기** (45:22)
- 1. Codex 설치와 실행 모드 이해하기
- 2. 첫 대화 — 코드 구조 설명과 간단 수정 맡기기
- 3. AGENTS.md를 읽게 만들고 승인 모드를 체험해보기

### Part 2. 문서와 컨텍스트를 다루기 쉽도록 만들기 (10 클립 · 공개 예정)

**Ch1. Claude code, Codex 공식문서 활용법**
- 1. Claude Code / Codex 공식 문서를 통해 기능과 활용법을 이해하기
- 2. 공식문서에서 제공하지 않는 기능을 Claude Code, Codex 를 이용해서 찾아보기
- 3. 문서를 읽고 바로 실습으로 바꾸는 좋은 질문법

**Ch2. CLAUDE.md와 AGENTS.md로 AI를 팀원처럼 일하게 만들기**
- 1. 좋은 지시형 프롬프트는 무엇이 다른가 — WHAT, WHY, HOW
- 2. CLAUDE.md 잘 쓰기 — 전역, 프로젝트, 하위 폴더 규칙 잡기
- 3. AGENTS.md 잘 쓰기 — 여러 도구에 통하는 공통 지시문 만들기
- 4. 나쁜 지시문을 고치는 실전 리팩터링

**Ch3. 긴 대화를 무너지지 않게 관리하기 — 세션·컨텍스트·압축**
- 1. 긴 대화가 망가지는 이유와 새 세션을 열 타이밍
- 2. /compact, resume, memory를 언제 어떻게 쓸까
- 3. 1개의 대화 세션 컨텍스트를 지키는 컨텍스트-문서 분리 전략

### Part 3. 실무에서 안전하게 개발 작업을 위임하는 방법 (12 클립 · 공개 예정)

**Ch1. 안전하게 작업 맡기기 — 권한, 승인, 샌드박스**
- 1. AI에게 어디까지 맡길 것인가 — 승인 정책의 기본
- 2. Claude Code 권한과 샌드박스 실전 설정
- 3. Codex 승인 모드와 작업 공간 접근 범위 이해하기

**Ch2. Git 브랜치와 리뷰 흐름에 AI 붙이기**
- 1. Explore → Plan → Implement → Commit 흐름 만들기
- 2. 브랜치, diff, 리뷰 — AI가 만든 변경을 읽는 법
- 3. PR 생성, 코멘트 반영, 충돌 해결까지 한번에 이어가기

**Ch3. MCP로 외부 도구 연결하기 — GitHub·DB·브라우저까지**
- 1. MCP가 왜 중요한가 — AI를 코드 밖으로 꺼내기

## 6. 페이지 인포그래픽 / 비주얼 패턴

### 6.1 큰 가로 이미지 + 짧은 카피 페어링

- 페르소나 3단 훅마다 거대한 가로 이미지를 한 장씩 깔고, 그 아래 한 줄 카피 → "긴 카피를 안 읽어도 분위기만으로 메시지가 전달"

### 6.2 핵심 키워드 4 × 2 카드 슬라이드

- 1번 슬라이드(키워드 4개) → 2번 슬라이드(키워드 4개)로 분할
- 같은 비주얼 grid·동일 폰트·동일 라벨 위치 — **반복으로 인한 기억 강화**

### 6.3 학습 로드맵 — 가로 5단 STEP

### 6.4 프로젝트 카드 (0→1, 1→10)

- 같은 포맷의 카드 2장 — `[프로젝트 01]` / `[프로젝트 02]` 라벨 → 헤드라인 → "| 학습 포인트 |" 섹션 → 4~5개 불릿

## 8. 교육 방식 / 학습 설계 패턴

### 8.1 클립 시간 표준화

- Part 1의 모든 클립이 **14~17분** 사이로 통일 (가장 짧은 게 10:57, 가장 긴 게 17:01)
- 한 챕터 = 3~4 클립 = 약 45~50분
- 한 자리에서 한 챕터 소화 가능한 길이 → **"하루 1챕터" 학습 루틴**을 자연스럽게 유도
- AI Builder School의 lesson 단위도 비슷한 길이 가이드를 두면 학습자가 진도 빼기 쉽다

### 8.2 7-step 루프와 비교

AI Builder School은 `problem → concepts → mission → build → verify → ship → reflect` 7단계를 lesson마다 보존한다. Fast Campus는 **챕터 단위로** 다음과 같이 흐른다.

1. **인식 (Ch 첫 클립)** — "왜 차이가 나는가 / 자동완성과 무엇이 다른가"
2. **기초 도구 (Ch 중간 클립)** — 설치/사용법
3. **첫 산출물 (Ch 마지막 클립)** — 첫 CLAUDE.md 만들기 / 첫 PR 만들기

→ AI Builder School의 7-step 루프는 **더 세분화돼 있어서 학습자 관점에선 더 부담스럽게 느껴질 수 있다**. 다만 "verify"와 "reflect"는 Fast Campus 강의에는 없는 강점. 7-step을 유지하되 **"한 lesson을 25-40분 안에 끝낼 수 있다"는 시간 가드레일**을 추가하면 좋다.

### 8.3 "40+ 실습" — 정확한 실습 카탈로그 공개

페이지는 25개 프로젝트 더보기를 클릭하면 다음 3개 그룹으로 분류된 실습을 보여준다 (확장 시점에 미리 펼쳐진 6개 그룹 중 일부).

**그룹 1 — Claude Code · Codex 기능 이해**
1. Claude Code / Codex 공식 문서를 통해 기능과 활용법을 직접 확인
2. 공식 문서에 없는 기능을 AI를 활용해 탐색하는 방법 실습
3. 문서를 읽고 바로 실습으로 전환하는 질문 전략

**그룹 2 — 프로젝트 컨텍스트 설계**
4. CLAUDE.md / AGENTS.md를 작성해 AI가 프로젝트를 이해하도록 설계
5. 긴 대화가 망가지는 이유와 새 세션을 열 타이밍을 실습
6. 컨텍스트와 문서를 분리하는 관리 전략을 설계
7. /compact, resume, memory 기능 활용 방법

**그룹 3 — Git 개발 워크플로우 구축**
8. Explore → Plan → Implement → Commit 흐름
9. 브랜치, diff, 리뷰로 AI 변경 사항 읽기
10. PR → 코멘트 반영 → 충돌 해결 워크플로우
11. Claude Code 권한 및 샌드박스 설정
12. Codex 승인 모드와 작업 공간 접근 범위 이해

(나머지 25 프로젝트는 더 펼쳐야 하지만, 카테고리 6개 — Agent Skills, Hooks 기반 자동화, SDD 기반 개발 루프, Agent Harness, 멀티 에이전트 협업, 프로젝트 적용 — 으로 구성됨이 확인됨)

→ **시사점** — 단순히 "40개 실습"이라고 숫자만 던지지 않고 *카테고리별 실습 목록 미리보기*를 보여줘서 신뢰를 만든다. AI Builder School도 phases.ts의 lessons에서 *대표 산출물 미리보기 카드*를 만들면 효과적.

### 8.4 사례 연구 학습법 (Part 7)

실제 오픈소스 프로젝트(`oh-my-claudecode`, `oh-my-openagent`)를 **하나하나 뜯어보는** 챕터를 마지막에 배치. "본인 프로젝트로 이식"하는 단계를 항상 마지막에 둠 → **"가져가서 쓰세요" 메시지**.

→ AI Builder School은 templates.ts가 비슷한 역할. 다만 *"내 프로젝트에 이식하는 단계"*를 lesson 마지막에 명시적으로 두면 학습자가 행동으로 옮긴다. 현재의 "reflect" 스텝에 "**transplant**(이식)" 액션을 추가하는 옵션도 고려할 만하다.

## 10. 차별화 권장 (Fast Campus와 겹치지 말 것)

- **타겟 페르소나 더 넓게** — Fast Campus는 "사전지식 필요"라 명시. AI Builder School은 정반대로 *"코딩 0, 프롬프트 0이라도 OK"* 메시지를 더 강화.
- **도구 종속성을 낮춰라** — Fast Campus는 Claude Code · Codex 두 도구 중심. AI Builder School은 **"도구가 바뀌어도 살아남는 빌더 마인드셋"**으로 차별화.
- **한국어 + 친근 어조** — lesson-writer 스킬의 "친근한 말 거는 어투"는 Fast Campus의 약간 격식 있는 톤("~합니다") 대비 강한 차별 요소. 유지/강화할 것.
- **AI 코딩 외 영역도 포함** — Fast Campus는 코딩 한정. AI Builder School은 콘텐츠/이미지/영상/자동화까지 포괄 → "빌더" 정체성을 더 넓게 정의.
- **수료 후 출구** — Fast Campus는 "월간 하네스"로 유지. AI Builder School은 **"졸업 프로젝트 → 빌더 포트폴리오 → 빌더 커뮤니티"**로 명시적 출구 동선을 제안할 수 있다.
