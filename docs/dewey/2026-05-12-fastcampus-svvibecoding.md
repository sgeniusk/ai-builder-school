## Context

AI Builder School은 *"초보자가 빌더로 성장하는 한국어 12-Phase 커리큘럼"* 을 만든다. 이 강의는 정반대 극단을 노린다 — *"상위 1% 개발자"* 와 *"엔터프라이즈급 실무"* 라는 엘리트 프레이밍으로 *중급 이상 개발자*를 타겟. 같은 도구(Claude Code, Codex)를 가르치지만 청자가 정반대다. **차별화 자료**로서 가치가 크다 — "엘리트 프레이밍을 어떻게 빌더 친화 프레이밍으로 변환할지" 의 레퍼런스.

## 1. 랜딩 페이지 메시지 흐름 (Hero → Persona → USP)

### 1.1 Hero 카피
- **메인** — *"실리콘밸리 바이브코딩 — 상위 1% 개발자의 10가지 바이브코딩 스킬 with. Claude Code & Codex"*
- **서브** — *"토이 프로젝트가 아닌 엔터프라이즈급 실무까지 적용 가능한 실리콘밸리 엔지니어들의 바이브코딩 테크닉"*
- **태그** — `#Ai agent #바이브코딩 #클로드코드 #하네스엔지니어링`

### 1.2 페르소나 훅 (격차·도태)
- **격차** — *"상위 1% 개발자들의 바이브코딩은 다르다??"*

- **긴급** — *"실리콘밸리의 특급 노하우, 오픈 특가 할인중! 망설이면 가격이 올라가요"*

### 1.3 메시지 골격
페이지의 thesis = **"같은 도구를 써도 상위 1% 와 일반은 다르다 → 그 격차를 채워라 → 안 채우면 도태된다"**. 도태 위험 + 엘리트 참조점 조합. AI Builder School 의 thesis(*"누구나 빌더가 될 수 있다"*) 와 정반대 톤이라는 점이 중요하다.

## 2. 강의 핵심 포인트 (USP)

페이지에 "01. 강의 핵심 포인트" 카드 섹션은 명시적으로 노출되지 않는다. USP 는 분산된 카피로 운반된다.

| 카피 슬롯 | 내용 |
|----------|------|
| 권위 — 지역 | "실리콘밸리 엔지니어들의" |
| 권위 — 등급 | "상위 1% 개발자" |
| 스케일 차별 | "토이 프로젝트가 아닌 엔터프라이즈급 실무" |
| 도구 특정 | "with. Claude Code & Codex" |
| 결과 약속 | "대체 불가능한 AI Native 개발자" |

**구조의 시사점** — USP 카드 없이 *Hero 한 곳에 모든 권위·스케일·도구를 집중*. 페이지 스크롤이 짧아 보이는 효과. AI Builder School은 정반대로 USP 4축을 분리해서 *결정 부담을 분산*시키는 편이 적합 (페르소나가 결정 자신감이 낮음).

## 3. 커리큘럼 핵심 키워드 / 인포그래픽

페이지에 별도 키워드 슬라이더는 없다. 5개 파트 제목 자체가 키워드 역할을 한다.

| 파트 | 키워드 |
|------|------|
| 1 | 실리콘밸리 엔지니어의 하루 |
| 2 | Claude Code 마스터하기 |
| 3 | 4개 프로젝트로 증명하기 |
| 4 | 팀과 조직으로 확장하기 |
| 5 | 자동화 에이전트의 시대 |

**구조의 흥미로운 점** — 파트 제목이 *명사 + 동사 활용형*(마스터하기, 증명하기, 확장하기)으로 통일됨. 학습자가 "내가 할 행동"을 즉시 인지. AI Builder School 의 12 Phase 제목도 비슷한 일관성을 도입할 만하다.

## 4. 학습 로드맵

명시적 STEP N 로드맵 없음. 5개 파트가 *암묵적 선형 경로*를 형성.

**개인 기초 → 도구 마스터 → 프로젝트 실습 → 팀 확장 → 자동화 미래**

→ Fast Campus 하네스 엔지니어링 강의(앞서 분석한 73 클립 강의)는 STEP 1-5 메타 로드맵을 별도 시각화했지만 *이 강의는 그게 없다*. 페르소나 차이 — 이 강의 타겟(중급+)은 메타 로드맵 없이 파트 제목만 봐도 자기가 어디 있는지 안다는 가정.

## 8. 교육 방식 / 학습 설계 패턴

### 8.1 프로젝트 기반 부피
- 5 파트 중 *프로젝트* 파트가 클립 수 1위 (46개)
- 클립 평균 시간 추정 — 21시간 ÷ 115 = **약 11분**. 짧음
- "4개 프로젝트로 증명하기" — *증명* 이라는 어휘가 핵심. 학습 == 산출물 4개 만들기

### 8.2 팀 확장 파트 분리
- Part 4 "팀과 조직으로 확장하기"(12 클립) 가 별도 파트로 분리 — *개인 → 팀* 전환을 학습 경로에 명시적으로 포함
- AI Builder School 은 현재 *개인 빌더* 중심. 향후 "팀 빌더" 로 확장한다면 이 분리 패턴 참고

### 8.3 마지막 파트의 미래 전망
- Part 5 "자동화 에이전트의 시대"(5 클립) — *마지막 파트는 미래/전망*
- 학습자가 "지금 배운 것의 다음 단계"를 가져가도록. AI Builder School 도 마지막 Phase 에 *"다음 빌더로의 길"* 메타 콘텐츠를 둘 만하다

## 10. 차별화 권장 (이 강의와 겹치지 말 것)

- **"상위 1%" 프레이밍을 흉내내지 말 것** — 우리는 *수평적 입문* 톤이 정체성. 흉내내면 페르소나가 분열함
- **"엔터프라이즈급" 어휘 사용 X** — 빌더 입문자에게 무게감만 줌. 우리는 *"내 작은 도구"* / *"개인 자동화"* 같은 소규모 어휘가 적합
- **사전지식 가정 최소화** — Fast Campus는 "사전지식 필요" 명시. 우리는 정반대로 *"코딩 0, 프롬프트 0이라도 OK"*
- **도구 (Claude Code/Codex) 종속성 낮추기** — Fast Campus는 도구 마스터 파트가 43 클립. 우리는 *도구가 바뀌어도 살아남는 빌더 마인드*에 더 비중
- **"증명 / 도태" 같은 자기 입증 압박 어휘 회피** — 우리 톤은 *"같이 만들어요 / 손에 돌아와요"* 처럼 안심·환영
