# 페르소나 평가 보고서 — 이수민

## 1. 페르소나 자기소개 (1인칭, 1문단)

저는 35살 SaaS PM이고, 개발자는 아니지만 개발팀과 7년째 일하고 있어요. Notion, JIRA, Figma는 매일 쓰고 ChatGPT도 이미 업무에 꽤 붙여 쓰는 편입니다. 제가 궁금한 건 “나도 앱을 직접 만들 수 있나?”보다 “PM인 내가 팀의 AI 도입을 어떻게 리드할 수 있나?”예요. 코드를 전혀 몰라서 겁이 나는 지점도 분명해요. 괜히 기술 용어만 얕게 외우고 실제 팀 운영에는 못 붙이는 사람이 될까 봐요.

## 2. 커리큘럼 첫 인상 (스크롤하며 느낀 것)

처음 인상은 좋았어요. 제품 브리프의 “기획자/관리자는 조직에 AI를 어떻게 도입할지 모른다”는 문제 정의는 제 얘기였고, `Adopter` 여정의 “조직에 AI를 안전하게 들여오고, 위험과 품질을 함께 운영합니다”도 정확했어요. 그런데 12 Phase를 훑는 순간 구조가 살짝 흔들렸습니다. Phase 1에서 3까지는 제 언어인데, Phase 4부터 갑자기 터미널, GitHub CLI, `.env`, Zod, TypeScript가 앞으로 나와요. “코딩 모르는 초보자”를 위한 학교라기보다 “AI로 개발까지 해볼 사람”에게 무게중심이 기울어 보였습니다.

가장 큰 구조적 불안은 `Adopter` 경로가 약속은 큰데 실제 레슨 연결이 빈약하다는 점이에요. `journeys.ts`에서 Adopter 추천 레슨은 `checks-before-trusting-ai-output`, `research-workflow`, `ai-app-cost-and-usage`, `prompt-injection-defense` 네 개뿐이에요. 그런데 Phase 9와 Phase 10은 Adopter에게 핵심인데 `phases.ts`를 보면 lessonSlugs가 비어 있어요. PM 입장에서는 “도입 전략 + 검증 체크리스트”를 기대했는데, 실제로는 초반 생산성 레슨 몇 개 다음에 엔지니어링 보안 레슨으로 점프하는 느낌이었습니다.

## 3. 깊게 읽은 lesson 목록 (왜 이걸 골랐나)

제가 고른 기준은 세 가지였어요. 첫째, PM이 당장 업무에 붙일 수 있는가. 둘째, 팀 도입 가이드로 번역할 수 있는가. 셋째, 코드를 모르는 사람이 닫고 싶어지는 지점이 어디인가.

깊게 읽은 레슨은 `common-skills-of-future-proof-people`, `what-llms-are-good-and-bad-at`, `hallucination-verification`, `checks-before-trusting-ai-output`, `structure-of-good-prompts`, `enforcing-output-format`, `feeding-long-documents`, `automate-report-drafts`, `meeting-notes-pipeline`, `research-workflow`, `coding-agent-setup`, `github-issue-to-ai-brief`, `letting-ai-read-codebase`, `plan-with-ai`, `tool-permission-safeguards`, `prompt-injection-defense`, `ai-app-cost-and-usage`, `harness-engineering-roadmap`입니다.

앞의 10개는 제가 실제로 쓰고 싶어서 골랐고, 뒤의 8개는 제가 어려워할 것 같아서 일부러 읽었습니다. 특히 `github-issue-to-ai-brief`와 `plan-with-ai`는 개발자가 아니라 PM에게도 중요한데, 현재 경로상 Adopter에게 잘 보이지 않는 게 아쉬웠어요.

## 4. 너무 어려운 부분 / 막힌 지점 (구체 인용)

가장 먼저 닫고 싶어진 지점은 `enforcing-output-format`입니다. “사람이 읽을 답은 어느 정도 흔들려도 괜찮지만, 기계가 읽을 답은 99.9% 같은 모양이 나와야 해요”까지는 완전히 이해됐어요. 그런데 바로 뒤에 `response_format`, `json_schema`, `strict: true`, Zod 코드가 나옵니다. 저는 여기서 “아, 이건 개발자가 봐야 하는 레슨인가?”라고 느꼈습니다. PM으로서 알아야 할 판단 기준과 개발자가 구현해야 할 코드가 한 레슨 안에서 구분 없이 섞여 있어요.

`ai-app-cost-and-usage`도 비슷했습니다. “비용 문제는 프로토타입에서는 안 보이다가 사용량이 늘어날 때 갑자기 보입니다”는 훌륭한 문제 제기예요. PM이라면 무조건 알아야 합니다. 그런데 산출물이 `logger.ts`, `pricing.ts`, `report.ts`, `budget-guard.ts`로 제시되는 순간 제 일감이 아니라 개발자 티켓처럼 보여요. 제가 원한 건 “사용자 수, 호출 수, 평균 토큰, 모델 단가를 넣으면 월 비용이 나오는 스프레드시트”에 더 가까웠습니다.

`prompt-injection-defense`는 내용 자체가 중요하지만 “const INJECTION_PATTERNS = [...]”가 나오는 순간 학습 동기가 떨어졌어요. 저는 정규식을 직접 짜고 싶은 게 아니라, 고객지원 봇을 도입할 때 제품 요구사항에 어떤 보안 테스트를 넣어야 하는지 알고 싶었습니다. “이 네 파일이 있으면 새 에이전트를 만들 때마다 인젝션에 얼마나 버티는가를 테스트하는 기준선이 생깁니다”라는 문장은 맞지만, 저에게는 “그럼 이걸 개발팀에 어떤 acceptance criteria로 넘기지?”가 빠졌습니다.

`coding-agent-setup`의 “직접 파일을 읽고, 명령을 실행하고, 코드를 수정합니다”는 설명은 무섭지만 유익했어요. 다만 `package.json`, `.env`, `git push`가 한꺼번에 나오면 비개발 PM은 “내가 건드리면 사고 치겠다” 모드로 들어갑니다. 이 레슨은 좋은데, PM에게는 실행 레슨이 아니라 개발팀과 합의할 운영 가드레일 레슨으로 포지셔닝해야 해요.

## 5. 뜬금없는 요소 / "왜 여기 있나?" 부분

가장 뜬금없던 건 `Adopter`에게 추천된 `prompt-injection-defense`가 실제 메타에서는 `targetJourneys: ["engineer", "founder"]`라는 점이에요. 여정 문서에서는 도입자에게 이 레슨을 추천하는데, 레슨 자체는 도입자를 대상으로 삼지 않습니다. 그래서 제 입장에서는 “이거 나를 위한 거야, 개발자를 위한 거야?”가 생겼어요.

또 하나는 Phase 9와 Phase 10입니다. `Data Analysis & Decision AI`와 `Evals, Security & Responsible AI`는 PM의 핵심인데, 실제 `lessonSlugs`가 비어 있습니다. 커리큘럼 지도에는 기획자·관리자 필수 Phase로 보이는데, 막상 들어가면 몸통이 없는 느낌이에요. PM으로서는 오히려 여기서 “AI 도입 우선순위 점수표”, “팀 파일럿 운영 지표”, “AI 결과 품질 평가 회의 템플릿” 같은 게 나와야 합니다.

`blog-writing-pipeline`이 Adopter에도 들어간 건 조금 애매했어요. PM도 글을 쓰긴 하지만 조직 AI 도입을 리드하려는 사람에게 Phase 3의 핵심은 블로그보다 정책 문서, 세일즈/CS 대응 가이드, 내부 교육자료 자동화일 가능성이 높습니다.

## 6. 빠진 부분 / 내가 진짜 알고 싶었던 것

제가 진짜 알고 싶었던 건 “PM이 직접 빌더가 되지 않으면서 팀에 AI 도입을 리드하는 방법”입니다. 현재 커리큘럼은 개인이 AI를 잘 쓰는 방법과 개발자가 AI로 만드는 방법은 꽤 잘 다뤄요. 그런데 중간 역할, 즉 비개발 PM이 팀 단위 도입을 설계하는 경로가 약합니다.

빠진 건 이런 것들이에요. AI 도입 후보 업무를 고르는 우선순위 매트릭스. 보안·개인정보·법무·브랜드 리스크를 한 장으로 합의하는 정책 템플릿. 파일럿 팀을 고르고 2주 실험을 설계하는 로드맵. 성공 지표를 정하는 법. “AI가 절약한 시간”과 “품질 사고 위험”을 같이 보는 대시보드. 개발팀에 넘길 AI 기능 요구사항 문서. Notion, JIRA, Figma 안에서 AI 워크플로우를 운영하는 예시도 더 필요합니다.

지금은 “직접 만들자”의 압력이 강해서, PM이 “도입 리더”로 남아도 되는 경로가 잘 안 보입니다.

## 7. 좋은 부분 / 인상 깊은 lesson

좋았던 건 분명히 많아요. `common-skills-of-future-proof-people`의 “판단의 주체가 누구냐”는 문장은 PM에게 아주 잘 맞았습니다. AI를 결론 내는 기계가 아니라 초안 담당 팀원으로 본다는 관점은 팀 교육자료 첫 장에 넣고 싶었어요.

`what-llms-are-good-and-bad-at`도 좋았습니다. “AI에게 맡길까 말까”가 아니라 “어디까지 맡기고 어디서부터 내가 받아 검토할까”로 바꾸는 문장이 특히 실무적이에요. 개발 지식 없이도 업무 분류 워크숍으로 바로 돌릴 수 있습니다.

`meeting-notes-pipeline`은 PM에게 거의 즉시 적용 가능한 레슨입니다. “결정 사항 / Action Item / 오픈 이슈”로 나누는 구조는 JIRA와 Notion에 바로 연결돼요. `research-workflow`도 좋았어요. “검색 결과를 바로 요약하면 안 돼요. 중간에 근거 표를 두세요”는 리서치를 많이 하는 PM에게 정확한 가드레일입니다.

의외로 `github-issue-to-ai-brief`도 좋았습니다. 제목은 개발자용 같지만 본질은 PM의 요구사항 작성법이에요. “되게 해 주세요”와 “수용 조건 + 검증 명령”의 차이를 보여주는 표는 JIRA 티켓 교육에 바로 쓸 수 있습니다.

## 9. 종합 의견 + 구체 변경 제안 3개

### 9.1 종합 의견 (verdict 이유 포함)

제 결론은 “일부만 듣겠다”입니다. Phase 1에서 3까지는 듣겠습니다. PM이 AI를 더 안전하고 재현 가능하게 쓰는 데 도움이 돼요. 특히 판단 체크리스트, 강·약점 지도, 트러스트 체크, 회의록 파이프라인, 리서치 워크플로우는 바로 업무에 붙일 수 있습니다.

하지만 전체 12 Phase를 그대로 따라가지는 않을 것 같아요. 이유는 명확합니다. 저는 AI 도입을 리드하고 싶은 PM이지, 당장 코딩 에이전트로 앱을 직접 만들고 싶은 사람이 아니에요. 커리큘럼이 “AI 사용자에서 AI 메이커로”를 밀고 가는 건 이해하지만, PM에게는 “AI 도입 리더”라는 중간 목적지가 필요합니다. 지금은 그 목적지가 이름만 있고 레슨으로 충분히 구현되지 않았습니다.

### 9.2 변경 제안

1. **파일 — `src/content/journeys.ts`, `src/content/lessons.ts`** — 현재 Adopter 추천 레슨이 4개뿐이고, `github-issue-to-ai-brief`와 `plan-with-ai`처럼 PM에게 중요한 레슨이 Adopter 경로에 없습니다. Adopter 추천 레슨에 `github-issue-to-ai-brief`, `plan-with-ai`, `harness-engineering-roadmap`을 넣고, 각 레슨의 `targetJourneys`에도 `adopter`를 추가해 주세요. 단, 본문에는 “PM은 직접 구현자가 아니라 브리프와 가드레일의 소유자”라는 별도 안내를 넣어야 합니다.
2. **파일 — `src/content/phases.ts`, `src/content/lessons.ts`, `src/content/lessons/*.mdx`** — Phase 9와 Phase 10은 Adopter 필수인데 현재 `lessonSlugs`가 비어 있습니다. `ai-adoption-prioritization-scorecard`, `team-ai-pilot-roadmap`, `ai-output-eval-for-pms`, `responsible-ai-policy-template` 같은 PM용 레슨을 추가해 주세요. 산출물은 코드가 아니라 Notion/JIRA/Google Sheets 템플릿이어야 합니다.
3. **파일 — `src/content/lessons/ai-app-cost-and-usage.mdx`, `src/content/lessons/prompt-injection-defense.mdx`, `src/content/lessons/enforcing-output-format.mdx`** — 현재는 PM 판단 기준과 개발 구현이 한 흐름에 섞여 있습니다. 각 레슨 앞부분에 “PM lane”을 추가해 주세요. 예를 들어 비용 레슨은 먼저 “월 비용 추정 스프레드시트 + 예산 상한 정책 + 개발팀 전달용 요구사항”을 만들고, 그 다음을 “Engineer lane”의 `logger.ts` 구현으로 분리하면 좋겠습니다. 인젝션 방어도 정규식 코드 전에 “공격 시나리오 5개를 acceptance criteria로 쓰는 법”부터 나와야 합니다.
