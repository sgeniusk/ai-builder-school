# 페르소나 평가 보고서 — 이정호

## 1. 페르소나 자기소개 (1인칭, 1문단)

저는 54살이고 동네 카페를 7년째 하고 있습니다. 컴퓨터는 카톡, 네이버 검색, 주문서 확인 정도만 하고, 인스타는 작년에 겨우 시작했어요. 제가 AI에 관심을 가진 이유는 거창한 앱을 만들고 싶어서가 아니라, 메뉴 소개 글, 인스타 캡션, 계절 이벤트 문구, 단골 손님한테 보낼 안내문을 좀 덜 막막하게 만들 수 있나 싶어서입니다. 솔직히 `RAG`, `API`, `GitHub` 이런 단어가 보이면 바로 한 발 물러납니다. 이게 뭐예요? 제일 궁금한 건 하나예요. 이게 내 가게 운영에 진짜 도움이 되나, 아니면 IT 회사 사람들끼리 듣는 강의인가.

## 2. 커리큘럼 첫 인상 (스크롤하며 느낀 것)

첫 인상은 반반이었습니다. 제품 브리프의 “AI를 쓰는 사람에서 AI로 만드는 사람으로”라는 말은 멋있지만, 저는 처음부터 “만드는 사람”이라는 말이 조금 부담스러웠어요. 카페 사장 입장에서는 “AI로 오늘 오후 디저트 홍보글 하나 잘 쓰는 사람”도 충분히 큰 변화입니다.

좋았던 건 모든 레슨이 산출물로 끝난다는 점입니다. 그냥 듣고 끝나는 강의가 아니라 `judgment-checklist.md`, `prompt-library-v1.md`, `blog-pipeline/`처럼 손에 남는 것이 있다는 건 믿음이 갔어요. 그런데 `src/content/phases.ts`를 보니 Phase 0부터 “Codex CLI / Claude Code / GitHub / API 키”가 나옵니다. 여기서 저는 바로 긴장했습니다. 아직 챗GPT에 메뉴 사진 설명시키는 것도 어색한데, GitHub 저장소라니요. “비개발자 친화”라고 써 있지만 실제 길은 꽤 빨리 개발자 쪽으로 기울어집니다.

`src/content/journeys.ts`도 봤는데, 저 같은 사람은 정확히 들어갈 칸이 애매했습니다. `creator`는 블로그, 뉴스레터, 유튜브, 숏폼 운영자 이야기고, `practitioner`는 보고서, 이메일, 회의록을 매일 다루는 직장인 이야기입니다. 저는 매출표, 재고, 알바 스케줄, 인스타, 메뉴판, 단골 이벤트를 다룹니다. 비슷한 듯한데 이름표가 없습니다.

## 3. 깊게 읽은 lesson 목록 (왜 이걸 골랐나)

제가 고른 lesson은 “내 가게에 바로 써먹을 수 있나”와 “제목만 봐도 도망가고 싶은가” 두 기준으로 골랐습니다. 읽은 본문은 `common-skills-of-future-proof-people`, `ai-concept-learning-framework`, `what-llms-are-good-and-bad-at`, `hallucination-verification`, `checks-before-trusting-ai-output`, `structure-of-good-prompts`, `enforcing-output-format`, `feeding-long-documents`, `build-personal-prompt-library`, `automate-report-drafts`, `research-workflow`, `blog-writing-pipeline`, `coding-agent-setup`, `ai-app-cost-and-usage`, `capstone-plan-and-launch`입니다.

처음 네 개는 AI를 믿어도 되는지 알고 싶어서 읽었습니다. 프롬프트와 글쓰기 쪽은 인스타와 메뉴 소개에 연결될 것 같아서 골랐습니다. `coding-agent-setup`, `ai-app-cost-and-usage`, `capstone-plan-and-launch`는 제가 어려워할 부분을 일부러 읽었습니다. 내가 못 따라갈 지점이 어디인지 확인하고 싶었어요.

## 4. 너무 어려운 부분 / 막힌 지점 (구체 인용)

가장 먼저 막힌 곳은 `ai-concept-learning-framework`의 제목부터였습니다. “RAG, MCP, Agent... 매주 새 용어가 쏟아져요.” 이 문장 자체는 제 마음을 압니다. 그런데 바로 뒤에 RAG, MCP, Agent 사례가 이어지니 “그럼 결국 이걸 다 알아야 하나?” 싶었습니다. 특히 “Anthropic이 공통 프로토콜을 정의해서, 한 번 만든 MCP 서버는 모든 호환 클라이언트에서 쓸 수 있게 했어요”는 저한테 너무 멉니다. 서버요? 클라이언트요? 저는 네이버 플레이스도 아직 헷갈립니다.

`structure-of-good-prompts`의 “프롬프트는 부탁이 아니라 명세(Spec)다.”는 말은 이해가 됐지만, `Spec`이라는 괄호가 붙는 순간 다시 개발자 방에 들어온 느낌이었습니다. 그래도 이 레슨은 참고 넘을 수 있었어요. Task, Context, Constraints, Output은 카페에도 쓸 수 있습니다. “신메뉴 딸기라떼를 30대 동네 엄마 손님에게 소개하는 인스타 캡션으로, 300자 이내, 가격 직접 언급 금지”처럼요.

진짜 닫고 싶어진 건 `enforcing-output-format`입니다. “AI가 회의록을 받아 Action Item JSON을 뱉으면 그걸 Notion API에 그대로 던지는 흐름이에요.” 여기서 저는 멈췄습니다. JSON이 뭔지도 모르는데 뱉는다는 표현도 낯설고, Notion API는 또 뭔가요. 이어서 “`JSON.parse` 가 깨지면서 파이프라인 전체가 멈춰요”, “Zod / Pydantic — 런타임 검증을 같이 깔자”가 나오면 이건 제 강의가 아니라는 생각이 듭니다.

`coding-agent-setup`도 무섭습니다. “터미널에서 AI가 코드를 짜기 시작했는데, 멈출 수가 없어요”는 호기심보다 공포가 먼저 옵니다. “`.claude/settings.json`”, “`git push`”, “`rm -rf src/`” 같은 문구는 초보자에게 안전장치 설명이 아니라 사고 사례처럼 느껴졌습니다.

마지막으로 `capstone-plan-and-launch`의 “API를 연결하고, RAG를 붙이고, 에이전트를 돌리고, 비용까지 관리했어요”는 저한테 거의 산 정상 이야기였습니다. 저는 아직 AI에게 “오늘 비 오는 날 카페 이벤트 문구 5개”를 묻는 단계인데, 갑자기 4주 안에 AI 앱을 런칭하라니 숨이 찹니다.

## 5. 뜬금없는 요소 / "왜 여기 있나?" 부분

뜬금없다고 느낀 건 커리큘럼의 약속과 예시의 생활권이 자꾸 어긋나는 부분입니다. `blog-writing-pipeline`은 정말 제게 필요한 레슨인데 제목이 블로그라서 처음엔 지나칠 뻔했습니다. 저는 블로그보다 인스타 캡션, 메뉴판 문구, 네이버 플레이스 소식이 필요합니다. 본문에는 “내 글 샘플을 컨텍스트로 넣어야 해요”처럼 좋은 말이 많은데, 예시가 블로그 중심이라 자영업자는 스스로 번역해야 합니다.

`automate-report-drafts`도 마찬가지입니다. “매주 금요일, 보고서 틀 채우느라 분석할 시간이 없어요”는 직장인에게는 딱 맞겠지만 저는 보고서보다 “이번 주 매출이 왜 줄었지”, “비 오는 날엔 어떤 세트 메뉴를 밀지”가 더 급합니다. 구조는 쓸 수 있는데 예시가 회사 안쪽에 묶여 있습니다.

그리고 Phase 8이 `creator`에게 중요해 보이는데 `lessonSlugs`가 비어 있습니다. 콘텐츠 자동화가 제일 궁금한 사람에게 정작 중간 다리가 비어 있는 느낌입니다. 이게 왜 여기 비어 있지? 카페 사장인 저는 이미지, 자막, 숏폼보다 먼저 “인스타 한 달 운영표”가 필요합니다.

## 6. 빠진 부분 / 내가 진짜 알고 싶었던 것

제가 진짜 알고 싶었던 건 “내 가게 운영에 AI를 붙이는 아주 작은 첫날”입니다. 예를 들면 이런 겁니다. 메뉴 사진을 주고 설명 문구 5개 받기. 비 오는 날 손님을 부르는 쿠폰 문구 만들기. 악성 리뷰에 감정 상하지 않고 답글 초안 만들기. 이번 달 매출 메모를 보고 다음 이벤트 후보 뽑기. 알바 공고 문구를 더 따뜻하게 만들기.

지금 커리큘럼에는 이 재료들이 흩어져 있습니다. `what-llms-are-good-and-bad-at`은 숫자는 검산해야 한다고 알려줍니다. `checks-before-trusting-ai-output`은 5분 검토 습관을 줍니다. `blog-writing-pipeline`은 내 말투를 살리는 법을 줍니다. 그런데 이걸 “동네 카페 운영 키트”로 묶어주는 입구가 없습니다. 그래서 저는 계속 “이 좋은 걸 내가 알아서 내 가게 말로 바꿔야 하나?”라고 느꼈습니다.

## 7. 좋은 부분 / 인상 깊은 lesson

`what-llms-are-good-and-bad-at`은 정말 좋았습니다. “LLM의 능력은 ‘얼마나 똑똑한가’가 아니라 ‘어떤 모양의 일에 강한가’”라는 말은 제 불안을 줄였습니다. AI가 계산을 틀리면 제가 못 써서 그런 줄 알았는데, 애초에 계산기처럼 쓰면 안 된다는 설명이 시원했습니다.

`checks-before-trusting-ai-output`의 FOSCB도 좋았습니다. “5축, 각 1분”은 제가 따라할 수 있는 크기입니다. 카페 홍보글도 Fact, Origin, Scope, Confidence, Bias로 볼 수 있습니다. 특히 “Trust Floor와 Trust Ceiling”은 말은 어렵지만, 어떤 일은 그냥 써도 되고 어떤 일은 절대 AI만 믿으면 안 된다는 선을 세워줘서 좋았습니다.

`blog-writing-pipeline`은 제목만 바꾸면 제일 유용한 레슨입니다. “AI가 글을 ‘잘’ 쓰는 건 맞아요. 하지만 ‘나처럼’ 쓰는 건 다른 문제예요.” 이 문장은 인스타 하는 자영업자에게도 정확합니다. 제 가게 말투가 있는데, AI가 너무 백화점 광고처럼 쓰면 손님들이 바로 압니다.

## 9. 종합 의견 + 구체 변경 제안 3개

### 9.1 종합 의견 (verdict 이유 포함)

제 결론은 “일부만 듣겠다”입니다. Phase 1, Phase 2 일부, Phase 3의 글쓰기와 리서치 쪽은 듣겠습니다. 여기에는 제 가게 운영에 진짜 도움이 되는 뼈대가 있습니다. AI에게 무엇을 맡기고, 어디서 검증하고, 내 말투를 어떻게 유지할지 배우는 건 필요합니다.

하지만 Phase 4 이후로는 지금 제게 너무 이릅니다. 코딩 에이전트, API, RAG, 에이전트 루프, 비용 트래커는 “AI로 가게 운영을 조금 낫게 하고 싶은 50대 사장”에게는 멀어요. 언젠가 필요할 수는 있지만, 지금은 동기 부여보다 겁을 더 줍니다. 이 커리큘럼은 좋은 학교인데, 입구에 작은 가게 주인을 위한 안내 데스크가 아직 없습니다.

### 9.2 변경 제안
