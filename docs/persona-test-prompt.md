# AI Builder School — 20인 페르소나 베타 테스트 (Cowork 의뢰 프롬프트)

> 이 문서 전체를 Cowork에게 의뢰 프롬프트로 전달한다. Cowork는 아래 규격대로
> 20명의 가상 페르소나를 만들고, 10일 사용 시뮬레이션을 돌리고, 페르소나별 보고서
> 20장 + 종합 보고서 1장을 작성한다.

---

## 1. 배경

**AI Builder School**은 "AI를 쓰는 사람"에서 "AI로 만드는 사람"으로 학습자를 안내하는 한국어 AI 빌더 교육 사이트다.

- 구조 — 8개 Stage 사다리. 만나다 → 물어보다 → 확인하다 → 함께일하다 → 일감주다 → 시스템만들다 → 운영하다 → 공유하다
- 64개 lesson. 각 lesson은 7-step 러닝 루프(문제 이해 → 개념 학습 → 미션 → 빌드 → 검증 → 배포 → 회고)로 구성
- 6개 출발 여정(Journey) — Practitioner(실무자) · Adopter(도입자) · Creator(크리에이터) · Founder(파운더) · Engineer(엔지니어) · Explorer(탐험가)
- Stage 4·6은 sub-group을 가짐 — 4a 업무자동화 / 4b 콘텐츠·미디어 / 4c 데이터·분석, 6a API / 6b RAG / 6c Agent

**테스트 대상 사이트** — `{{SITE_URL}}` (의뢰자가 production 또는 preview URL을 채워 전달)
**소스 참고** — `github.com/sgeniusk/ai-builder-school`

현재 상태 — v0.4 진행 중. Stage 1은 책처럼 보완 완료(7 lesson), Stage 2~8은 보완 전이다. 테스터는 이 상태를 "있는 그대로" 평가한다.

---

## 2. 의뢰 작업

20명의 가상 페르소나를 만들어 각자 AI Builder School을 **10일간 사용**하는 과정을 시뮬레이션하고, 아래를 산출한다.

1. 페르소나별 보고서 20장 (§5 양식)
2. 종합 보고서 1장 (§6 양식)

각 페르소나는 실제 사람처럼 행동한다 — 막히면 멈추고, 지루하면 이탈하고, 좋으면 계속한다. "이상적인 학습자"가 아니라 "현실적인 사람"으로 시뮬레이션할 것.

## 3. 페르소나 생성 규격

20명을 아래 분포로 만든다.

### 3.1 여정 분포 (6 Journey × 20명)

| Journey | 인원 |
|---|---|
| Practitioner (실무자) | 4 |
| Adopter (도입자) | 3 |
| Creator (크리에이터) | 3 |
| Founder (파운더) | 3 |
| Engineer (엔지니어) | 4 |
| Explorer (탐험가) | 3 |

### 3.2 능력 분포 (4단계, 전체에 고르게 섞기)

- **L0 완전 입문** — 터미널·코드 경험 0, AI는 ChatGPT 가끔 사용 (5명)
- **L1 사용 경험** — AI 도구를 업무에 쓰지만 코딩은 모름 (7명)
- **L2 중급** — 약간의 코딩 경험, AI API는 미경험 (5명)
- **L3 숙련** — 개발자 또는 ML 경험자 (3명)

### 3.3 페르소나마다 정해야 할 속성

이름 · 나이 · 직업 · 소속 규모 · Journey · 능력 단계 · AI 친숙도 · 코딩 배경 · 학습 동기(왜 등록했나) · 하루 가용 시간 · 주 사용 기기(모바일/데스크톱/혼합) · 끈기 성향(쉽게 포기/보통/끈질김) · 의심 성향(검증 꼼꼼/보통/무비판).

다양성 요구 — 20대~50대를 고르게, 수도권 외 거주자 포함, 비IT 직군(자영업·교사·간호사·마케터·디자이너 등) 다수 포함, 모바일 주 사용자 최소 5명.

### 3.4 페르소나 카드 양식

```
### P{번호}. {이름} ({나이}, {직업})
- Journey: {journey} / 능력: {L0~L3}
- AI 친숙도: {한 줄}
- 코딩 배경: {한 줄}
- 학습 동기: {1~2문장, 1인칭}
- 하루 가용 시간: {분}
- 주 사용 기기: {모바일/데스크톱/혼합}
- 성향: 끈기 {상/중/하} · 의심 {상/중/하}
```

## 4. 테스트 프로토콜

### 4.1 10일 사용 시뮬레이션

각 페르소나는 자기 Journey의 추천 Stage를 따라 lesson을 진행한다. 하루 가용 시간 안에서 가능한 만큼만 진행하며, 10일 차에 멈춘다.

- 매일 한 일·진도·소요 시간을 일지로 기록
- lesson마다 7-step 루프를 따라가되, 막히면 그 지점을 정확히 기록하고 다음 행동(재시도/건너뛰기/이탈)을 정함
- 미션은 "실제로 해봤다"고 가정하고 — 미션 지시가 명확한지, 45분 안에 끝낼 만한지, 비개발자가 따라갈 수 있는지를 평가
- 능력·끈기·기기 성향이 행동에 반영되어야 함 (L0가 터미널 lesson에서 겪는 일과 L3가 겪는 일은 달라야 한다)

### 4.2 평가 두 축

**A. UI/UX 평가** — 사이트를 쓰면서 느낀 것
- 첫 화면 5초 인상 — 누구를 위한 곳인지 전달되는가
- 내비게이션 — 지금 내가 어느 Stage·lesson에 있는지 알 수 있는가
- 가독성 — 글 길이·폰트·여백·코드 블록
- 모바일 경험 — 모바일 사용자는 반드시 평가
- 진척 표시 — 얼마나 왔고 얼마나 남았는지 보이는가
- 마찰 지점 — 클릭이 헷갈리거나 길을 잃은 순간

**B. 레슨 콘텐츠 평가** — lesson을 진행하며 느낀 것
- 난이도 곡선 — lesson 간 도약이 너무 큰 지점
- 명확성 — 개념 설명이 이해되는가
- 미션 실행 가능성 — 지시대로 따라 할 수 있는가
- 가치 — 끝내고 손에 남는 게 실제로 쓸모 있는가
- 흐름 — lesson들이 한 권의 책처럼 이어지는가, 끊기는가
- 이탈 지점 — "여기서 닫고 싶다"가 든 정확한 lesson과 문장

### 4.3 점수 척도

모든 점수는 1~5점. 1 매우 나쁨 · 2 나쁨 · 3 보통 · 4 좋음 · 5 매우 좋음. 점수에는 반드시 한 줄 근거를 붙인다.

추천 의향은 NPS 0~10. "이 사이트를 비슷한 처지의 동료에게 추천하겠는가."

## 7. 작성 규칙

- 한국어로 작성. 친근하지만 보고서다운 톤
- 페르소나는 "현실적인 사람"으로 — 칭찬만 하지 말 것. 부정 피드백이 없으면 시뮬레이션이 부실한 것
- 모든 점수에 근거 한 줄 필수, 빈칸 금지
- lesson을 지목할 때는 slug를 정확히 쓸 것
- 한국어 문장을 콜론(:)으로 끝내지 말 것 — 종결은 마침표·물음표·느낌표
- 산출물은 페르소나별 보고서 20장 + 종합 보고서 1장. 페르소나 보고서는 `docs/persona-reviews/` 의 기존 형식과 호환되게 작성
